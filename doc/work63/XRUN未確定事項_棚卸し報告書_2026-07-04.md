## 1. 最大の新発見：Gen4 vs Gen5で特性激変

### 定量データ

| 指標 | Gen4（IR読込前） | Gen5（IR読込後） | 変化 |
|------|:---------------:|:----------------:|:----:|
| WS | 1,255 MB | 2,334 MB | **+1,079 MB** |
| Private | 1,354 MB | 2,486 MB | **+1,132 MB** |
| deltaArrival P50 | 5.33ms | 5.43ms | +0.1ms |
| deltaArrival P95 | **7.13ms** | **16.43ms** | **+9.3ms** |
| deltaArrival P99 | **7.66ms** | **25.73ms** | **+18.1ms** |
| deltaArrival >8ms | **0.3%** | **6.0%** | **20倍** |
| deltaArrival >15ms | **0.0%** | **5.4%** | **無限大** |
| XRUN密度 | 2.02/秒 | 1.56/秒 | やや低下 |

### この事実が意味すること

Gen4（IR未読込, WS=1.3GB）ではコールバック到着はほぼ理想的（P99=7.66ms）。
Gen5（IR読込後, WS=2.5GB）でP99が25.73msに悪化。

**これは「OSスケジューラだけが原因」という仮説では説明できない。** OSが「メモリ使用量2.5GBだからスケジューリングを変えた」という動作は、Windowsスケジューラの公称仕様として知られていない。

考えられる説明：
1. **メモリサブシステムの競合**: 2.5GBのWSによりTLBミス率・L2/L3キャッシュミス率が上昇。JUCE/Diver/OSのcallback配送にも影響。
2. **Page Faultの影響**: 起動直後のPageFaultサージ(+338,782)が落ち着いた後も、soft PageFaultがcallback配送に影響。
3. **MMCSSの制限**: メモリ負荷上昇によりMMCSSの効果が相対的に低下。
4. **DRAM帯域の競合**: IR処理のメモリアクセスパターンがシステム全体のDRAM帯域を消費。

### 重要な示唆

**WS=2.5GBの巨大プロセスは、OSのメモリ管理サブシステム全体に負荷をかけている可能性がある。** これはConvoPeqの「内部要因」とは呼べないが、ConvoPeqの設計（IRをメモリに常駐させる方式）に起因する間接的な副作用である。

## 2. 各未確定事項の詳細

### ① XRUNの正確な原因箇所（優先度：最高）

**現状の知識：**
- XRUN検出点は `processBlockDouble` 先頭（t0_start）
- deltaArrival（recordCallbackArrivalのts）でも同程度の値
- つまり遅延は **JUCE callback到達以降、processBlockDouble先頭より前** に発生

**残る候補（現時点では識別不能）：**

| 候補 | 内容 | 確率評価 |
|------|------|:--------:|
| OS Scheduler dispatch delay | Audioスレッドが実行可能(Runnable)だがディスパッチされない時間 | 中〜高 |
| DPC/ISR | 他デバイスの割り込み処理がCPUを占有 | 中 |
| Timer Coalescing | Windowsが複数タイマーをまとめて発火 | 低 |
| Voicemeeter Driver Stall | ASIOドライバ内部の処理遅延 | 中 |
| ConvoPeq入口オーバーヘッド | JUCE→getNextAudioBlock間に未測定のオーバーヘッド | 低 |
| RCU reader synchronize | Epoch-based RCUの読み取り側で待たされる | 低 |
| メモリサブシステム競合 | TLBミス・キャッシュミス・DRAM競合 | **中〜高**（Gen4/Gen5差の主因候補） |

**確定方法：**
1. **ETW (WPR)**: `Thread::ReadyThread` + `DPC::Routine` + `ISR::Routine` の同時取得 → 誰がCPUを占有していたか特定
2. **LatencyMon**: DPC/ISR実行時間の簡易計測
3. **2048Buffer試験**: 応急処置としての効果確認

### ② MMCSSが有効か

**コード上の実態：**

- バイナリ確認: `[MMCSS] registered: taskIndex=` 文字列が埋め込まれている ✅

**ログに出ない理由（未確定）：**

### ③ ConvoPeq内部の間接要因

**DSP時間に表れない間接要因の可能性：**

| 要因 | 影響の仕方 | 検出可能性 |
|------|-----------|:----------:|
| **TLBミス** | ページテーブルウォークによる数十〜数百nsの遅延。大量発生時はms級に。 | ❌ ログ不可、HW PMC要 |
| **L2/L3キャッシュミス** | 2.5GB WSによりキャッシュ圧力上昇。メモリレイテンシ増加。 | ❌ ログ不可、HW PMC要 |
| **DRAM帯域競合** | MKLのメモリ带宽集約型演算がシステム帯域を消費 | ❌ ログ不可 |
| **False Sharing** | 同一cache lineの異なるatomicを別スレッドが書込→無効化 | △ コード分析のみ |
| **RCU Synchronization** | Epoch進捗待ちでcallback開始が遅延 | △ ログから部分的に推定可能 |
| **NUMA Remote Access** | メモリがリモートNUMAノードに配置された場合のレイテンシ増加 | ❌ ETW要 |

**False Sharing評価（コード分析）：**

| 場所 | リスク | 説明 |
|------|:------:|------|
| `RTLocalState`全体 | **中** | 構造体に `alignas(64)` なし。`audioCallbackEpochCounter`, `lastCallbackEndTicks`, `lastCallbackEntryUs` 等が近接。ただし単一スレッド(Audio Thread)からの書き込みが主。 |
| `RTAuxMutable`内の `xRunDropCount` | **低** | 他の `debugLastReported*` 変数と近接。これらはTimer Threadが読取。ただしXRUN時のみ書込。 |
| `DiagPerTickCounter` 3つ直列 | **低** | 各 `alignas(64)` で独立cache line。 |
| `pendingLearningMode`, `adaptiveCaptureActiveRt`等 | **低** | `alignas(64)` 分離済み。 |

**結論：** False Sharingのリスクは認識されているが、DSP時間が安定していることから、XRUNへの寄与は軽微と判断される。

### ④ 21msピークの原因

**観測：** deltaArrivalに21-22msの明確なピーク（392件）。

**考えられる説明とその確度：**

| 説明 | 確度 | 理由 |
|------|:----:|------|
| A: **Timer Coalescing** | ★★★☆☆ | Windows 8以降のTimer Coalescingはデフォルトで有効。ただしAudioスレッドはMMCSSによりCoalescing対象外になる可能性。 |
| B: **DPC/ISRの長時間実行** | ★★★☆☆ | NIC/GPU/ストレージドライバのDPCがAudioスレッドをプリエンプト。21msはDPCとしては長めだが不可能ではない。 |
| C: **OS Scheduler量子超過** | ★★★☆☆ | 標準量子(≈30ms)を下回るが、優先度ブースト等の影響。 |
| D: **Audio Driverバッファ満杯待ち** | ★★☆☆☆ | Voicemeeter内部のWASAPI→ASIO変換でバッファリング。 |
| E: **単純な4コールバック消失** | ★☆☆☆☆ | 「OSが4個まとめて配送」はWindowsの仕様として知られていない。 |

**21msが4×5.33ms=21.3msであることの符合は偶然の可能性が高い。** 実際に観測される値は19.5ms〜23.0msに分布しており、これは21.3msを中心とする正規分布とは言えない。

### ⑤ 2048Bufferで全XRUN解決か

**理論値：**
- 現状の最大XRUN Interval: 10.22ms
- 2048 bufferの予算: 10.67ms × 1.5 = 16.0ms
- 10.22ms < 16.0ms → 現存する267件のXRUN **測定基準上は全件消失**

**しかし以下の問題が残る：**
1. deltaArrival >16ms が **1,355件（5.1%）** 存在
2. 2048 bufferでもコールバック到着遅延は解決されない（閾値を超えなくなるだけ）
3. 実際のASIOドライバ動作は理論通りとは限らない（ドライバ内部FIFOの動作）
4. レイテンシが5.33ms→10.67msに倍増するトレードオフ

**結論：** 2048 bufferでXRUN log上の判定は消失する可能性が高いが、音切れが完全に無くなるかは実機試験でのみ確認可能。「全XRUN解決」は条件付き。

### ⑥ RCU同期の影響

`AudioEngine` は `convo::RCUReader audioThreadRcuReader` を使用。`getRuntimeSampleRateHzFromWorld()` や `runtimeReadHandleRef` の取得にはRCU readerロックが必要。

RCUの仕組み上、Publishing側（Non-RT）がEpochを進める際に、Readerが古いEpochにいる場合は待たされることがある。ただしこれは通常 Reader側がフリーではない（非RTスレッドの処理完了待ち）場合に発生する。

Audio callback内でRCU readerが待たされることは通常ない（RCU reader側は無待機）。ただし **RCU writer側** が特定の状況で待たされることはあり、その間Publishが遅延する可能性はある。

**評価：** RCUがXRUNの直接原因である可能性は低いが、Publish遅延の間接要因としては可能性が残る。

### 結論

観測データは「DSP負荷超過ではない」を確定させる。しかし「ConvoPeqが原因ではない（内部要因100%無し）」は現時点では言えない。

**Gen4→Gen5でdeltaArrivalが激変した事実**は、ConvoPeqのメモリ使用量（IR読込によるWS 1.3→2.5GB）がコールバック到着に間接的に関与している可能性を示している。この間接要因を特定するにはETWが唯一の手段である。

**現時点で確定的に言えること：**
1. DSP実行時間は原因ではない
2. コールバック到着が遅延している
3. 遅延の程度はConvoPeqのメモリ使用量と相関する
4. バッファ2048でXRUN閾値超過は消失する可能性が高い（実機要確認）
5. **ConvoPeq内部・外部どちらが根本原因かは、ETWなしでは結論できない**
