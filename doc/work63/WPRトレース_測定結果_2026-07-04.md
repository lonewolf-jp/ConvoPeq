## 1. ETL トレースの問題点

### 1.1 イベントロスト

```

```

**原因：** `-start CPU -start Audio` のデフォルトバッファサイズでは不足していた。

### 1.2 DPC/CSwitchデータなし

`CPU` と `Audio` プロファイルでは **DPC・CSwitch・ReadyThread のキーワードが含まれない**。そのため、今回の ETL からは「誰が Audio スレッドをプリエンプトしたか」は解析不可。

**取得できたデータ：**

## 4. 次回の WPR トレース手順（改善版）

次回は **xperf** を使用してバッファサイズを明示的に指定し、DPC/CSwitch/ReadyThread を取得します。

```powershell
# 管理者 PowerShell で実行

# Step 1: トレース開始（バッファサイズ増量 + DPC/CSwitch/ReadyThread キーワード）
& "C:\Program Files (x86)\Windows Kits\10\Windows Performance Toolkit\xperf.exe" `
    -on PROC_THREAD+LOADER+DPC+INTERRUPT+CSWITCH+PROFILE `
    -stackwalk DPC+CSWITCH+ReadyThread `
    -BufferSize 2048 -MinBuffers 64 -MaxBuffers 256 -MaxFile 1024 -FileMode

# Step 2: ConvoPeq 起動・操作（IR読込 → PEQ → ANS → 音楽再生 30秒）

# Step 3: トレース停止
& "C:\Program Files (x86)\Windows Kits\10\Windows Performance Toolkit\xperf.exe" `
    -stop doc\work63\ConvoPeqTrace_v2.etl

# Step 4: WPA で解析
& "C:\Program Files (x86)\Windows Kits\10\Windows Performance Toolkit\wpa.exe" `
    doc\work63\ConvoPeqTrace_v2.etl
```

**主な改善点：**
| 項目 | 前回 | 今回 |
|------|:----:|:----:|
| バッファサイズ | デフォルト（256KB） | **2048KB** |
| 最小バッファ数 | デフォルト | **64** |
| 最大バッファ数 | デフォルト | **256** |
| 最大ファイルサイズ | デフォルト | **1024MB** |
| DPC キーワード | ❌ なし | ✅ DPC+INTERRUPT |
| CSwitch キーワード | ❌ なし | ✅ CSWITCH |
| ReadyThread | ❌ なし | ✅ PROC_THREAD |

## 5. 次回トレースで確認すべき項目

1. **ReadyThread P99 > 1ms** → AudioスレッドがRunnable状態で待たされた時間
2. **CSwitch (Preempting Thread)** → 誰がAudioスレッドを蹴り出したか
3. **DPC Duration > 500μs** → 長時間DPCの特定
4. **Audio Glitch vs ReadyThread の時系列一致** → XRUNとReady待ちの因果関係

これらのデータが揃えば、「ConvoPeq内部が原因か / OSスケジューラが原因か / DPC/ISRが原因か」を確定できます。
