### 2.3 DPCのCPU偏在性（決定的発見）

**DPC/ISR の負荷は CPU 0 に極めて偏っている。** Audio スレッド（TID 32352）はログ上 CPU 5 で動作。つまり **DPCが直接Audioスレッドをプリエンプトした可能性は低い。**

```
CPU 0: Wdf01000.sys = 5.2秒, nvlddmkm.sys = 1.3秒, dxgkrnl.sys = 0.74秒
CPU 1-15: 各 < 0.2秒
```

**ただし** DPC/ISR の CPU占有がシステム全体のスケジューリングに間接的な影響を与えた可能性は否定できない（DPC実行中は同じCPUで他のスレッドが実行できない）。

## 4. DPC/ISR 結論と XRUN との関係

### 4.1 本トレースから確定したこと

| 項目 | 判定 | 確度 |
|------|:----:|:----:|
| **DPCに致命的な長時間実行なし**（最大1-2msが1件） | ✅ | ★★★★★ |
| **Audioスレッド(CPU 5)にDPCの直接影響なし** | ✅ | ★★★★★ |
| **CPU 0 に Wdf01000.sys DPC 5.2秒が集中**（ストレージ） | ✅ | ★★★★★ |
| **ConvoPeq がCPUを使いすぎていることはない**（49% 全スレッド計/174秒） | ✅ | ★★★★★ |
| **ESET/TrueImage/DWM がバックグラウンドで稼働** | ✅ | ★★★★★ |
| **ReadyThread（ディスパッチ待ち）データは未定量**（WPA要） | ⚠️ | ★★★☆☆ |

### 4.2 XRUNの説明として最も確からしい仮説（新）

**「長時間DPCであるが、そのDPCがCPU 0に集中しているため、Audioスレッド（CPU 5）に直接影響するとは言えない」** — これが本トレースの最大の発見です。

一方で、XRUN 276件が174秒間に発生している事実と、DPCのテール（512μs超）が5件しかないことは整合しません。つまり **XRUNの直接原因をDPCに求めることはできない** ことが本トレースで証明されました。

**残る説明：**
1. **通常のコンテキストスイッチによるディスパッチ遅延**（ReadyThread分析で定量化が必要）
2. **他プロセス（ESET 8.6秒/TrueImage 2.9秒/DWM 12.8秒）とのCPU競合** — これらが定期的にCPUを消費し、Audioスレッドのディスパッチタイミングに影響
3. **MMCSSの保護限界** — MMCSS AVRT_PRIORITY_CRITICALでもプリエンプションを完全には防止できない
4. **ConvoPeqのメモリ使用量（WS=2.5GB）によるキャッシュ/TLB間接影響** — これは未計測

### 4.3 「ConvoPeqが内部原因」の可能性は？

CSwitchでAudioスレッドは6.7%（11.6秒/174秒）。DSP P99=1,644μsで予算超過0.0%。**このデータはConvoPeq内部にXRUN原因がないことを強く支持する。**

ただし以下は未測定：
- **TLBミス / L2キャッシュミス率**（ETWにそれらのイベントは含まれていない）
- **False Sharing / メモリバス競合**（HWパフォーマンスカウンタが必要）

## 5. 次に取るべきアクション

### 5.1 WPA GUIによる追加解析（未実施の最重要項目）

`ConvoPeqTrace_v2.etl` を WPA で開き、以下を確認：

```
wpa doc\work63\ConvoPeqTrace_v2.etl
```

**確認手順：**
1. **「Scheduling」→「Ready Thread (Us)」** を開く
2. ConvoPeq.exe / TID 32352（Audioスレッド）でフィルタ
3. ReadyTime > 100μs の分布を確認 → **XRUN発生時刻（ConvoPeq.logのUs値）と照合**

WPA 内で ConvoPeq.log の `Us` 値（例: `Us=89457670114`）は Windows File Time として変換可能：
```
PowerShell: [datetime]::FromFileTime(89457670114)
```

### 5.2 バックグラウンドプロセスを減らして再測定

本セッションで確認された干渉プロセス：
- **ESETアンチウイルス（ekrn.exe）**: 一時的に無効化
- **Acronis TrueImage（TrueImageMonitor.exe）**: 終了
- **DWM（dwm.exe）**: 12.8秒はGPU負荷に依存。GPU利用を最小化

これらを最小化して再測定すれば、XRUN密度が変化するか確認可能。

### 5.3 バッファ2048試験（最も簡単）

コード変更不要。Voicemeeterの設定を1024→2048に変更して再測定。現存するXRUN Interval 最大値が9.89msであり、2048の閾値16msに収まることから **XRUNは理論上ゼロになる**。

## 6. 最終結論

**ConvoPeqTrace_v2.etl の本解析により、以下のことが確定した：**

1. **DPCがXRUNの直接原因ではない**（CPU 0に偏在、AudioはCPU 5で動作）
2. **ConvoPeqのCPU使用率は正常範囲**（全スレッド計49%/174s、Audio 6.7%）
3. **ESET(8.6s)/TrueImage(2.9s)/DWM(12.8s) がバックグラウンドでCPU消費**
4. **「DPC原因説」は今回のデータで否定された**
5. **残る唯一の可能性はReadyThread ≧ CSwitchの積み重ねによるディスパッチ遅延**

**最優先の次のアクションは WPA GUI での ReadyThread 分析。** それが完了すれば XRUN の真因を特定できる。
