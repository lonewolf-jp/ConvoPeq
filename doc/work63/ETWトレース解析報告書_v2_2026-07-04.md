- 87区間(174秒)中 75区間で DPC 1-5%占有

- その他のCPU(1-15)はすべて <1%

**つまり DPC/ISR の負荷は CPU 0 に集中しており、Audio スレッドが CPU 0 に移動した場合のみ影響を受ける。**

## 3. 本トレースから確定したこと

### 3.1 確定的発見

| 項目 | 判定 |
|------|:----:|
| **DSP時間は原因でない**（6回連続確認） | ★★★★★ |
| **XRUN合計276件**（ICXビルドはMSVC比で変化なし） | ★★★★★ |
| **Gen4→Gen5特性激変**（6回連続確認） | ★★★★★ |
| **DPC/ISRに致命的な1ms超えは1件のみ**（異常値ではない） | ★★★★☆ |

## 4. 本トレースから見えたXRUN原因の方向性

### 新たに得られたエビデンス

1. **コンテキストスイッチ量は正常範囲**（他プロセス比較でもConvoPeqが突出していない）
2. **DPCに1-2msのテールあり**（1件のみだが重大）
3. **GPUドライバDPC(nvlddmkm.sys)がCPU 0で5.2秒間稼働** — これは長時間。ただしAudioスレッドはCPU 5で動作していたため直接の影響は限定的
4. **Wdf01000.sysのDPC(5.2ms)がCPU 0で顕著** — ストレージスタックの可能性。ミラーリング/バックアップソフト関連
5. **Acronis TrueImageMonitor(2.9s)とESETアンチウイルス(8.6s)がバックグラウンドで動作** — これらが定期的にCPUを消費

### XRUNの主因として最も可能性が高いもの（更新）

| 原因 | 確率 | 根拠 |
|------|:----:|------|
| **OSスケジューラのディスパッチ遅延** | ★★★★☆ | MMCSS適用下でも完全防止不可。CPU 0以外でのDPC競合あり。 |
| **他プロセスとのCPU競合** | ★★★★☆ | ESET(8.6s)、DWM(12.8s)、TrueImage(2.9s)がバックグラウンドで稼働 |
| **Wdf01000.sys DPC** | ★★★☆☆ | CPU 0集中だがAudioはCPU 5。間接的影響の可能性。 |
| **ConvoPeq内部要因** | ★★☆☆☆ | DSP時間正常。ただしTLB/Cacheミスは未測定。 |
| **単一1-2ms DPC** | ★★☆☆☆ | 173秒中1件のみ。XRUN 276件と相関せず。 |

## 5. 次回トレースへの推奨

### WPA GUI による追加解析（推奨）

既存の `ConvoPeqTrace_v2.etl` を WPA で開き、以下の手順を実行すれば ReadyThread の詳細が確認可能：

```powershell
wpa doc\work63\ConvoPeqTrace_v2.etl
```

WPA内で：
1. **「Scheduling」→「Ready Thread (Us)」** を開く
2. ConvoPeq.exe / Audioスレッド TID 32352 でフィルタ
3. ReadyTime > 1ms の分布を確認

### 次回トレースの改善点

| 項目 | 前回の問題 | 次回の改善 |
|------|-----------|-----------|
| バッファサイズ | 2048 | **4096**（イベントロスト防止） |
| 取得イベント | DPC+CSWITCH+PROC_THREAD | これに加え **DISK_IO** を追加（ストレージDPC特定） |
| スタックウォーク | DpcExecuteのみ | さらに **ReadyThread** の実際の取得を確認 |
| xperf 引数 | 試行錯誤あり | 最終的に動作するコマンドを保持 |

### ConvoPeqのバックグラウンドプロセス干渉を減らすには

本セッションで確認された他プロセス消費：
- **ESETアンチウイルス (ekrn.exe)**: 8.6秒 — 一時的に無効化して再試験
- **TrueImageMonitor.exe**: 2.9秒 — 終了して再試験
- **NVIDIA Container**: 1.8秒 — GPU負荷の影響確認

これらを最小化した状態で再測定すれば、原因の切り分けが可能。
