## 0. エグゼクティブサマリ

**本セッション（21:25:04）は267件のXRUN（1.56/秒）を記録。これは過去3回のセッション（0.86〜0.97/秒）から約60%増加している。**

**最大の発見はCPU Migrationパターンの完全な変化である。過去3回で確認された「96%がCPU 5に収束」という規則的パターンが消失し、AudioスレッドがCPU 0/2/4/1/3/6/7/14/13…と広範囲に分散している。このCPU分散がキャッシュミス・TLBミスを増加させ、XRUN密度の悪化に直結した可能性が極めて高い。**

**DSP処理時間はP99=1,428μs（予算5,333μsの26.8%）、予算超過率0.0%で依然として健全。ConvoPeq内部がXRUN原因である可能性は今回も完全に否定される。**

**加えて、MMCSSのログ出力がゼロであることから、`CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` が未定義であると判断。しかしMMCSSコードパス自体は `getNextAudioBlock()` から無条件で呼ばれており、**MMCSSは実際には有効である可能性が高い**（診断ログ出力のみが欠落）。この点は過去3報告書の「MMCSS未適用」判断を修正する必要がある。**

## 3. CPU Migration — **パターンが完全に変化（最重要発見）**

### 過去3回（17:09/18:52/19:07）
```
CPU 5収束率: 96% ← AudioスレッドがCPU 5に安定
CPU Migrationパターン: 規則的（CPU 0→5, 1→5, 2→5…）
```

### 本セッション（21:25）
```
CPU分布（n=80）:
  CPU 0: 14 (18%)   CPU 2: 14 (18%)
  CPU 4: 11 (14%)   CPU 1:  9 (11%)
  CPU 3:  9 (11%)   CPU 6:  5 (6%)
  CPU 7:  5 (6%)    CPU 14: 3 (4%)
  CPU 13: 3 (4%)    CPU 5:  2 (2%)  ← わずか2%！
  他: CPU 8/9/10/15 各1件
```

**過去3回の収束パターンが完全に消失し、Audioスレッドが広範囲のCPUに分散している。** CPU 5は過去96%→今回2%と激減。

**CPU Migration遷移パターンも無秩序化：**
```
過去: 96%が「n→5」の一点収束
今回: 4→0: 7回, 2→4: 7回, 4→2: 6回, 0→4: 3回, 0→1: 3回…
```

### 考察
- ThreadAffinityManagerの設定（Worker=CPU0, Learner=CPU1, Eval=CPU2, Heavy=CPU3）は過去と同一
- AudioスレッドはThreadAffinityManager非管理のためOSスケジューラに委ねられる
- システム全体のCPU負荷バランスが過去セッションと異なっている
- CPU Migrationの増加により **L1/L2キャッシュミス率が上昇**している可能性
- これがXRUN密度の約60%増加の主因と推定

## 5. MMCSSの真実 — 従来報告書の解釈を修正

**コード解析の結果、重要な発見があった。**

### コードパス
```
getNextAudioBlock() → [無条件] s_mmcssDoneフラグ → applyMmcssPriority()
```

`applyMmcssPriority()` は `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` に**依存せず**実行される。

### 診断ログ出力のみが条件付き
```cpp
void AudioEngine::applyMmcssPriority() noexcept {
    if (convo::consumeAtomic(useMmcssPriority, ...)) {
        AvSetMmThreadCharacteristicsA("Pro Audio", &taskIndex);  // ← 常時実行
        AvSetMmThreadPriority(hTask, AVRT_PRIORITY_CRITICAL);    // ← 常時実行
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
        diagLog("[MMCSS] registered: ...");  // ← この行のみ診断時
#endif
    }
}
```

### 結論
- **MMCSSは全4セッションで有効だった可能性が高い**
- `[MMCSS]` ログがゼロなのは `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` 未定義による診断ログ出力欠落
- ただし `useMmcssPriority` のデフォルト値は `true` だが、UIで"NativeRT"に変更されている可能性は否定できない（その場合 `[NATIVE_RT]` ログも同様に診断ガードあり）
- いずれにせよ **MMCSS適用済みでもXRUNは発生する** ことが今回のデータで示唆される
- MMCSS「Pro Audio」＋AVRT_PRIORITY_CRITICAL でも完全なXRUN防止はできない（OSスケジューラのプリエンプションを完全には排除できない）

## 10. 第4回セッションの特異点 — XRUN密度増加の原因考察

本セッションのXRUN密度（1.56/秒）が過去3回（0.86-0.97/秒）より約60%高い原因：

### 10.1 CPU Migrationパターンの変化（主因）
- **過去**: AudioスレッドがCPU 5に96%収束 → キャッシュウォーム → 安定動作
- **今回**: AudioスレッドがCPU 0/2/4/1/3/6/7/14/13…に分散 → CPU毎のキャッシュコールド → **スケジューリング遅延増加**
- CPU Migrationの多頻度化により、Callback到着間隔のジッターが増大

### 10.2 システム負荷の差異（推定）
- 本セッションは21:25開始（夜間）。前3回は17-19時台（夕方〜夜）。
- バックグラウンドプロセスの状態が異なる可能性
- Voicemeeterや他ASIOドライバの内部状態も影響

### 10.3 MMCSSの限界
- MMCSS「Pro Audio」＋AVRT_PRIORITY_CRITICAL でもプリエンプションは完全には防止できない
- WindowsのMMCSSはDPC（割り込み）を優先度昇格の対象としない
- 他ドライバ（ネットワーク、USB、GPU）のDPCがAudioスレッドより優先されるケース
