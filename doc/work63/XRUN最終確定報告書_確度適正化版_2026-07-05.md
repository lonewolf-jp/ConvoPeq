## 0. はじめに — 分析方法の限界

本調査は以下の手段で実施された：

| 分析手段 | 見えるもの | 見えないもの |
|---------|-----------|------------|
| ConvoPeqログ | DSP時間, XRUN検出, CB_ARRIVAL到着間隔 | OS内部状態, DPC/ISR実行 |
| ETW CSwitch | CPU可用率, CSwitch件数, スレッド別CPU時間 | ReadyThread **Wait Duration**（WPA GUI要） |
| ETW DPC/ISR | DPCモジュール別時間, CPU偏在, 時間分布 | **各DPCのスタック詳細**（WPA GUI要） |
| ETW dumper | イベント種類の一覧 | バイナリMOFイベントの中身（tracerpt非対応） |
| ETW tracerpt CSV | 限定的なイベントのみ（カーネルイベント非デコード） | CSwitch・ReadyThread・DPCの個別データ |

**CSwitch WaitTime（スレッドがRunnable状態→Runningまでの待ち時間）は、WPA GUIでのみ可視可能。** そのため「ReadyThread待ち時間はゼロ」とは断定できない。

## 1. 確定した観測事実（確度評価付き）

### 1.1 DSP負荷（★★★★★ — 極めて確実）

| 指標 | 値 | 根拠 |
|------|:---:|------|
| CALLBACK_STAGE P99 | **1,428-1,900μs** | 5セッション連続、79-187,536サンプル |
| 予算超過率 | **0.0%** | 全セッション、全サンプルで超過なし |
| CONV_TIME P50 | **224μs** | IR読込後も安定 |

**評価：DSP実行時間がXRUNの直接原因である可能性は極めて低い。** これは6セッション・延べ20万以上のコールバックサンプルで確認されている。

### 1.2 CPU使用率（★★★★☆ — 強い証拠）

| CPU | 可用率 | 備考 |
|:---:|:------:|------|
| CPU 0 | 92-95% | DPC/ISR 5-8%占有（Wdf01000.sys, nvlddmkm.sys, dxgkrnl.sys）|
| CPU 1-4 | 98-99% | 軽度のビジー |
| **CPU 5** | **99.9%+** | **Audioスレッド(TID 32352)動作CPU** |
| CPU 6-15 | 99.9%+ | ほぼ完全アイドル |

**評価：CPU 5 が常時飽和していることはない。** ただし：
- 平均可用率99.9%は **5msのAudioコールバック予算に対して最悪1ms未満の局部ビジーを示唆**
- この1ms未満のビジータイムが正確にいつ発生したかは **1秒刻みの集計では特定不能**
- **数msの局所的なレイテンシ（≒XRUNの原因）は平均99.9%可用でも発生しうる**

### 1.3 DPC分布（★★★★☆ — 強い証拠）

| DPCモジュール | CPU 0 (累積) | その他CPU |
|-------------|:----------:|:---------:|
| **Wdf01000.sys** | **5.21秒** | 0 |
| **nvlddmkm.sys** (NVIDIA GPU) | **1.30秒** | 0 |
| **dxgkrnl.sys** (DirectX) | **0.74秒** | 0 |
| **ntoskrnl.exe** | 0.12秒 | 全CPU分散(各0.05-0.16秒) |
| その他 | <0.02秒/CPU | 各CPU分散 |

**DPCの567,053件中、512μs超は5件のみ**。1-2ms超は1件のみ。

**評価：DPC/ISRによるCPU 5の直接的なプリエンプションは可能性が低い。** ただし：
- CPU 0 のDPCが **システム全体の割り込み分散・キャッシュ状態・メモリバス** に間接影響を与える可能性は否定できない
- **Wdf01000.sys**（Kernel-Mode Driver Framework）の 5.2秒の累積は、ストレージ/バックアップ関連のドライバスタックを示唆 — これらがISR→DPC→スレッド起床の経路でAudioに影響した可能性が残る

### 1.4 Gen4→Gen5 特性変化（★★★★★ — 極めて確実）

| 指標 | Gen4(IR前,WS=1.3GB) | Gen5(IR後,WS=2.5GB) |
|------|:------------------:|:-------------------:|
| deltaArrival P99 | **7.66ms** | **25.75ms** |
| deltaArrival >8ms | 0.3-0.4% | **5.6-6.2%** |
| deltaArrival >15ms | 0.0% | **4.9-5.4%** |

**5セッション全てで同一パターンが再現。** Gen4→Gen5の遷移は唯一無二の再現可能な実験条件。

**評価：ConvoPeqのメモリ使用量増加（WS 1.3→2.5GB）とdeltaArrival悪化に強い相関がある。** ただし因果関係のメカニズムは未確定（後述）。

## 2. 評価可能だが未確定の事項

### 2.1 ReadyThread WaitDuration（未定量 — WPA GUI要）

**現状：** tracerpt CSV はカーネルCSwitchイベントをデコードできないため、Audioスレッドの **Ready→Running 間のWaitTime** は未定量。

```
Required WPA steps:
1. wpa doc\work63\ConvoPeqTrace_v2.etl
2. "Scheduling" → "CPU Usage (Precise)"
3. Filter: ConvoPeq.exe / TID 32352
4. Right-click → "Summary Table" → Add "Wait Time (us)" column
5. Read P50/P90/P99/Max
```

**推定：** CPU 5 の可用率99.9%+から、WaitTimeが常時大きい可能性は低い。しかしXRUN周辺の局所的なWaitTimeスパイクは確認できていない。

### 2.2 AudioGlitchイベント（未取得）

`-start Audio` プロファイルでのトレース（1回目）では取得できたが、DPC/CSwitch取得を優先した2回目の xperf トレースでは AudioGlitch キーワードが含まれていない。Windows Audio Stack の内部状態は直接確認できていない。

### 2.3 キャッシュ/メモリサブシステム（未測定）

**HWパフォーマンスカウンタが必要な指標：**
- L2/L3 キャッシュミス率
- TLBミス率
- メモリバス帯域使用率
- リングバス/Infinity Fabric 競合
- LLC（Last Level Cache）競合（特にWS 2.5GB稼働時）

これらは Intel VTune / AMD uProf / Windows PerfMon のHW PMCで測定可能だが、今回のETWには含まれていない。

## 5. 最終結論

### 観測事実（確実）

1. **DSP実行時間はXRUNの直接原因ではない**（6セッション連続、予算超過率0.0%）
2. **CPU 5が常時飽和していることはない**（174秒間可用率99.9%+）
3. **DPC/ISRはCPU 0に偏在し、CPU 5への直接プリエンプションは稀**（DPC 512μs超は5件）
4. **Gen4→Gen5でdeltaArrival特性が激変する**（5セッション連続再現、WS 1.3→2.5GB）

### 推論（確度付き）

| 推論 | 確度 |
|------|:----:|
| CPU飽和・単純なスケジューラ競合が主因である可能性は**大幅に低下した** | ★★★★☆ |
| Voicemeeter ASIOドライバは有力な候補である | ★★★☆☆ |
| ConvoPeqのメモリ使用量増加は何らかの間接的影響を及ぼしている可能性がある | ★★★☆☆ |
| DPC/ISRの間接影響（Driver Queue経由）は否定できない | ★★☆☆☆ |
| 「ConvoPeq内部に原因がない」とまでは言えない（キャッシュ/TLB未測定） | ★★☆☆☆ |

### 次に取るべきアクション（優先度順）

```
1. WPA GUI で CSwitch WaitTime 確認（30分）← 現存ETLで可能
   wpa doc\work63\ConvoPeqTrace_v2.etl
   → "Scheduling"→"CPU Usage (Precise)"→ConvoPeq TID 32352→WaitTime

2. 2048 Buffer 試験（10分）← コード変更不要
   → Voicemeeter設定 1024→2048

3. ESET/TrueImage 停止 + 2048 Buffer で再測定（20分）
   → バックグラウンド最小化の効果確認

4. 2048 Buffer + xperf 同時取得（30分）
   → WaitTime定量 + XRUN発生有無の確定
```
