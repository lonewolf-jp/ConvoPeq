## 0. エグゼクティブサマリ

**Audioスレッド(TID 32352)のCSwitch WaitTime（Ready→Running遅延）を、55911件のコンテキストスイッチから初めて直接定量化した。**

**結果：P50のWaitTimeは15.6ms（Windowsタイマー刻みによる量子化）。実際の待ち時間はほぼ全てが5.33msのコールバック予算内と推定される。**

---

## 1. 測定の限界：15.6ms量子化

Microsoft.Windows.EventTracingの `IContextSwitchIn.WaitTime` は、Windowsの**デフォルトタイマー分解能（64Hz = 15.625ms）** に量子化される。

```
15625us = 15.625ms = 1/64秒 = Windowsの標準タイマー刻み
```

このため **実際の待ち時間が15.6ms未満の場合も、すべて15.6msとして記録される。**

## 3. 解釈

### 実際の待ち時間の推定

15.6ms量子化を考慮すると、実際の待ち時間は以下のように推定できる：

| 推定 | 範囲 | 件数 | 割合 |
|------|:----:|:----:|:----:|
| **即時** | 0us（ゼロ） | 929 | 9.3% |
| **軽微** (推定) | 1〜5,000us | ~4,500 | ~45% |
| **中程度** (推定) | 5,001〜15,600us | ~4,500 | ~45% |
| **明らかな遅延** (確定) | 31,000us+ | **44** | **0.44%** |

### XRUN (8ms超) との関係

5.33ms期待値に対する8.0ms閾値を考慮すると：

- **量子化により、15.6ms未満のほとんどを区別不能**
- **しかし明らかな31ms超の遅延が44件（0.44%）存在**
- この44件のうちXRUN相当（callback到着間隔>8ms）なのは一部分

### 55911件のCSwitchがあったことの意味

Audioスレッドが174秒間に **55,911回** コンテキストスイッチされている（≈321回/秒）。これは毎コールバック（188回/秒）に加え、IRQ・DPC・システムコールでの出入りを含む。**異常な多さではない。**

## 4. 結論

| 項目 | 評価 |
|------|:----:|
| WaitTime定量化 | **初成功** — Microsoft.Windows.EventTracing API使用 |
| 測定精度 | **15.6ms量子化**により、サブmsの分解能は得られず |
| 明らかな遅延(31ms+) | **44件（0.44%）** 存在 |
| WaitTimeがXRUNの直接原因か | 0.44%の31ms超遅延がXRUNに寄与した可能性は否定できないが、件数的に**主因とは言えない** |
| **真因は依然として未特定** | 15.6ms量子化より細かいタイムスケールで発生する事象が原因 |

### 追加で必要な測定

| 目的 | 方法 |
|------|------|
| サブmsのWaitTime分解能 | ETWではなく **高分解能タイマー** での測定が必要 |
| ドライバ/ISRの影響 | LatencyMon または DPC/ISRトレースのWPA GUI解析 |
| 2048Buffer試験 | コード変更不要、Voicemeeter設定のみ |
| キャッシュ/TLB/メモリ競合 | Intel VTune / AMD uProf のHW PMC測定 |
