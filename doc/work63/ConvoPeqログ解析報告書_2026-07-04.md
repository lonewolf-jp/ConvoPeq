## 1. エグゼクティブサマリ

本セッション（work63）のログを、work62報告書のフレームワークと改修計画書の測定目的に基づき詳細解析した結果、以下の結論を得た：

### 総合結論

**ConvoPeq 内部コンポーネント（DSPCore、Publish/Crossfade、Logger、コンボルバ）は全て正常動作しており、XRUNの原因がConvoPeq内部に存在する可能性は完全に否定された。CB_ARRIVAL到着間隔分布の初の定量化により、コールバック到着は74.3%が理想範囲（±20%）に収まる一方、6.0%がXRUN閾値（>8.0ms）を超えることが判明。CPU Migrationの96%がCPU 5に収束する規則的パターンと、MMCSSが本ビルドで未適用である事実から、XRUNの主因はOSスケジューラによるAudioスレッドのプリエンプションであると強く示唆される。**

### 核心エビデンス（定量データ）

```
DSP処理時間（CALLBACK_STAGE totalUs）: P50=706μs  P99=1,900μs  ← 予算5,333μsの36%
DSP予算超過率:                          0.0%          ← 全コールバックが予算内
コンボルバ（CONV_TIME）:               P50=231μs    ← DSP全体の33%
DiagDrain オーバーヘッド:              avg=63μs     ← Timer周期100msの0.063%
CB_ARRIVAL 理想到着率(±20%):           74.3%        ← 初定量化
CB_ARRIVAL XRUN率(>8.0ms):             6.0%         ← 初定量化
CPU Migration のCPU 5収束率:           96%          ← 新発見
MMCSS適用:                             未           ← 本ビルドでは未反映
```

## 6. DiagDrain / DiagStatistics 診断基盤評価

### 6.1 DIAG_DRAIN（Step 8a）

500tick（約50秒）ごとに出力。DiagEventリングバッファの消費状態を監視：

| tick | drained | drainUs | approxOcc/512 | hitLimit | 備考 |
|------|---------|---------|---------------|----------|------|
| 500 | 16/16 | 78μs | 496 (97%) | true | 初期状態 |
| 1000 | 16/16 | 73μs | 496 (97%) | true | 安定 |
| 1500 | 16/16 | 61μs | 496 (97%) | true | 安定 |
| 2000 | 16/16 | 40μs | 496 (97%) | true | セッション終盤 |

**評価**:
- ✅ 平均 drainUs = 63μs（Timer周期100msの0.063%）→ オーバーヘッド無視可能
- ⚠ **approxOccが常に496/512 (97%)**：バッファが常に満杯に近い状態
  - 原因: 100msあたり約19件のCB_ARRIVALがpushされるが、MaxDrainPerTick=16のため **3件/tick蓄積**
  - 512 / 3 ≈ 170tick（17秒）で満杯になるが、それ以上は溢れずに約496で定常化
- ✅ dropped=4（全セッション）→ データ損失は実質ゼロ

### 6.2 DIAG_STAT（Step 8.1）

100tick（約10秒）ごとに出力。イベントフローの健全性を監視：

```
セッション累計:
  pushed=0（per-tick pushは常に0 = DiagDrainループ外のカウンタ）
  popped=16（per-tick常に16 = 毎回MaxDrainPerTick消費）
  dropped=4（セッション末尾のみ）
  totalP=515, totalPop=32,000
  cbW=16（CB_ARRIVAL written/tick）
  cbD=3（CB_ARRIVAL dropped/tick、定常値）
```

**評価**:
- ✅ 診断基盤は安定動作。droppedは許容範囲内。
- ⚠ cbD=3/tick は **MaxDrainPerTick不足によりCB_ARRIVALの一部がdropされている可能性**。ただしdropされたCB_ARRIVALは最も古いエントリであるため、統計的偏りは軽微。

### 6.3 推奨チューニング

| パラメータ | 現状 | 推奨 | 理由 |
|-----------|------|------|------|
| MaxDrainPerTick | 16 | **32** | tickあたり約19件のpushに対して消費16件では不足。32に倍増でapproxOcc改善。 |
| DIAG_EVENT_POOL_SIZE | 512 | **1024** | バッファサイズ倍増でdropリスク低減。ただしメモリ増加は8KBのみ。 |

## 7. ThreadAffinity / CPU Migration 解析

### 7.1 ThreadAffinityManager 設定

```
worker=0x01 (CPU 0)    learner=0x02 (CPU 1)
eval=0x04   (CPU 2)    heavy=0x08  (CPU 3)
light=0x0F  (CPU 0-3)  ui=0x0F     (CPU 0-3)
```

- ✅ Worker（Rebuild含む）スレッドはCPU 0-3に固定
- ⚠ **Audioスレッドは未固定**（ThreadAffinityManagerの管理対象外）

### 7.2 CPU Migration の全貌（256:1サンプリング、97件）

**最も重要な発見：96%がCPU 5に収束**

```
遷移先CPU分布:
  CPU 5: 93回 (96%)
  CPU 4:  4回 (4%)

CPU遷移パターン top 10:
  CPU 0→5: 20回   CPU 1→5: 13回   CPU 2→5: 12回
  CPU 4→5: 10回   CPU 3→5: 8回    CPU 6→5: 6回
  CPU 5→5: 6回（同一CPUへの再割当＝サンプル誤差）
  CPU 10→5: 5回  CPU 14→5: 4回   CPU 12→5: 3回
```

**解釈**:
- ThreadAffinityで **RebuildスレッドをCPU 0-3に固定** した結果、Audioスレッドが **「空いたコア」= CPU 5に誘導** された
- CPU Migrationは無秩序ではなく、**OSスケジューラがAudioスレッドをCPU 5に集約する規則的な動作**
- CPU 5に固定された状態でもXRUNは発生（0.86/秒）→ **CPU Migration自体が原因ではない**（work62報告書を追認）

### 7.3 CPU Migration × CALLBACK_STAGE 相関

CPU Migration発生時のCALLBACK_STAGE totalUsを確認：

```
CPU_MIG cbIdx=256   → totalUs=253μs   drift=-697μs（余裕）
CPU_MIG cbIdx=512   → totalUs=294μs   drift=+199μs
CPU_MIG cbIdx=768   → totalUs=535μs   drift=+417μs
CPU_MIG cbIdx=1024  → totalUs=244μs   drift=-1022μs
CPU_MIG cbIdx=1280  → totalUs=689μs   drift=+640μs ← 最大だが予算内
```

**CPU Migration発生時もDSP処理時間は最大689μs（予算の13%）。キャッシュミスの影響は無視可能。**

## 10. 5つの否定された仮説の検証（本セッションのエビデンス）

| # | 仮説 | 判定 | 本セッションでのエビデンス |
|---|------|------|--------------------------|
| ① | Logger I/O → XRUN | ❌ **否定** | asyncSink正常動作(LOG_DROP=0)。XRUN 173件/ha発生。不変。 |
| ② | CPU Migration → XRUN | ❌ **否定** | 96%がCPU 5に収束する規則的パターン。MIG有無でDSP時間に有意差なし。 |
| ③ | DSP負荷超過 → XRUN | ❌ **否定** | CALLBACK_STAGE P99=1,900μs << 5,333μs。予算超過0.0%。 |
| ④ | メモリプレッシャー → XRUN | ❌ **否定** | 定常PageFault ~4,000/seq。サージは初期IR読込時2回のみ。 |

### 否定された仮説を踏まえた残る説明

**5件否定後の残る説明**: 「OSスケジューラによるAudioスレッドのプリエンプション」

**新たに得られた整合的エビデンス**:
1. CB_ARRIVALの多重モード構造（15-22ms / 25-32ms / 47-49msにピーク）
   → OSがAudioスレッドをプリエンプトし、コールバックをバッチ配送している

4. XRUNの100%が正方向Drift
   → 常に遅延が蓄積する方向にバイアス

## 11. 総合的結論と提言

### 11.1 結論

**ConvoPeq 内部のDSPCore/Publish/Logger/コンボルバは全て健全であり、XRUNの原因ではない。本セッションの定量データは、全ての観測が「OSスケジューラによるAudioスレッドのプリエンプション」と一貫性を持つ。**

初めて定量化されたCB_ARRIVAL到着間隔分布は以下の事実を明らかにした：
- **74.3%** のコールバックは期待値±25%以内に到着（良好）
- **5.9%** は軽度遅延（6.67〜8.0ms）でバッファ枯渇には至らない
- **6.0%** はXRUN閾値（>8.0ms）を超え、オーディオ出力にドロップアウトを生じる
- 遅延パターンはランダムではなく、**15〜22ms/25〜32msのバッチ配送**という明確なOSスケジューラパターンを示す
