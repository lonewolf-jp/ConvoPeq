# ConvoPeq 音声処理シグナルフロー・音響工学的妥当性検証レポート

対象: `DSPCore::processDouble()` / `processInputDouble()` / `processOutputDouble()` を中心とした実行経路。
`audioProcessorPlayer.setDoublePrecisionProcessing(true)` が無条件設定されているため、**実運用では常にdouble精度パスのみが通る**（float版 `processFloat`/`processOutput` は事実上未使用の並行実装）。フローチャートは同梱の `ConvoPeq_SignalFlow.mermaid` を参照。

---

## 1. 全体構成の妥当性（配置の検証）

段階を大きく分けると以下の10ブロックで構成されている。

1. 入力フォーマット変換・メータリング・入力DCブロック
2. アップサンプリング（任意, 2x/4x/8x）
3. 畳み込みエンジン / 20バンドEQ（**順序はユーザー選択可**）
4. OutputFilter（HC/LC/LPF、convとEQのどちらが最終段かで内容切替）
5. 出力メイクアップゲイン
6. ソフトクリップ（任意）
7. バイパスクロスフェード
8. ダウンサンプリング
9. 出力DCブロック・TruePeak/LUFSメーター・ディザ/ノイズシェイピング・ハードクランプ
10. 固定レイテンシ整列

### 1-1. 妥当と判断できる配置

- **DCブロッカーが入力・オーバーサンプル域・出力の3箇所に置かれている**のは適切。特にオーバーサンプル域に置くことで、アップサンプル補間フィルタのインパルス応答由来の微小DC/低周波ドリフトが後段（畳み込み・EQ）で蓄積するのを防いでいる。
- **ソフトクリップが「オーバーサンプル済みの場合はその領域内で適用し、オーバーサンプルなしの場合はローカルに2倍アップ→処理→ダウン」という分岐**は、非線形処理特有のエイリアシング対策として教科書的に正しい設計。非線形段を無条件にベースレートで処理する実装（帯域外高調波が折り返してノイズになる）と比べて明確に優れている。
- **畳み込み↔EQの順序をユーザーが選択できる**点は、実際のスタジオワークフロー（「マイク補正EQを先にかけてから空間畳み込み」か「畳み込みで得た音を後からEQで整える」か）双方に対応でき、プロ用途として妥当。
- **OutputFilterがconv/EQどちらが最終段かで内容を切り替える**設計（`convIsLast`判定）は、畳み込み由来の超高域上のアーティファクト整形と、EQのLPFモード整形を目的別に使い分けており、単純に固定順序でフィルタを掛けるよりも合理的。
- **固定レイテンシ整列(`applyFixedLatencyDelay`)を最終段に置く**のは、PDC（Plugin Delay Compensation）的な役割としては正しい位置（全処理が終わった後に遅延を揃える）。

### 1-2. 音響工学的に疑問が残る配置

#### (a) TruePeak/LUFSメーターの計測点が最終ゲイン適用前

`processOutputDouble()` の実行順序は次の通り（コード上の実際の順）。

```
出力DCブロック → NaN scrub → TruePeakDetector.processBlock(...)
              → LoudnessMeter.processBlock(...)
              → ディザ/ノイズシェイピング（× kOutputHeadroom ≒ -1.0dB）
              → NaN scrub → ハードクランプ(±kOutputHeadroom)
```

**TruePeak/LUFS計測が、`kOutputHeadroom`（≒ -1.0dB）のゲイン適用およびディザ加算より前の信号に対して行われている。** ITU-R BS.1770-4 のTrue Peak測定は「実際に出力・書き出しされる信号」に対して行うことが前提であり、この実装では計測対象と実際の出力サンプルの間に固定で約+1.0dBの乖離が生じる。具体的には:

- 計測値は実出力より約1.0dB高く出る（保守的な方向のズレではあるが、正確ではない）。
- LUFSも同様に、実際に書き出される信号よりシステマティックに高いラウドネス値を表示する。
- 表示メーターが「BS.1770-4/EBU R128準拠」を謳う場合、この固定オフセットは規格の趣旨（実際に配信・再生される信号を測ること）からは外れる。

**対応案**: TruePeak/LUFS計測を、ハードクランプ・ディザ適用後（＝実際に`buffer`へコピーされる直前のデータ）に移動する。ディザ後の量子化雑音を含めて測ることは実運用上も正しい（ディザ雑音自体もTrue Peak/LUFSの対象に含まれるべき）。

#### (b) True Peak "検出" はあるが True Peak "制限" が存在しない

`TruePeakDetector::processBlock()` の戻り値/検出結果は、コードを追う限り**UIメーター表示や内部テレメトリに使われるのみで、信号のゲインには一切フィードバックされていない**。その後にあるのは:

```
ディザ/ノイズシェイピング → ±kOutputHeadroom への瞬時ハードクランプ
```

つまり実運用上の「オーバー対策」は、-1.0dBFSでの**瞬時サンプルクランプ（ブリックウォールクリップ）のみ**であり、ルックアヘッド型・オーバーサンプル型の True Peak リミッターは存在しない。ハードクランプは波形を鋭く切り欠くため、発生した瞬間に高調波歪みとして耳に付きやすい。20バンドEQの強いブースト設定や畳み込みの正規化次第では、この経路が事実上の「最終リミッター」として機能してしまう可能性がある。

**対応案**: 少なくとも出力直前に軽量な look-ahead ピークリミッター（既に4xオーバーサンプルされたTruePeakDetectorの検出値をそのままゲインリダクション信号として再利用できる）を挟み、ハードクランプは「万一の際の保険」に留める設計に変更する。

#### (c) NaN/Infスクラブが同一バッファに対して3回連続で行われている

`processInputDouble()`の`sanitizeFiniteChunk`、`processOutputDouble()`前半の`NaN/Inf scrub #2`、ディザ後の`NaN/Inf scrub #3`と、**同じ信号経路上でNaN/Inf検出+ゼロ化が計3回**実施されている（加えてEQの`processBand`/`processBandStereo`内でも状態変数レベルで毎サンプルNaN/Infチェックが行われている）。

これは安全側ではあるが、裏を返せば「どの段で異常値が発生し得るかが明確に切り分けられていない」ことの表れでもある。プロ仕様DSPとしては、NaN/Infガードは(1)ユーザー操作起因の不連続点（IR切替・パラメータジャンプ）、(2)フィードバックを持つ再帰段（IIR状態変数、ノイズシェイパの積分器）の出力、の2箇所に絞り込み、単純な素通り区間（線形ゲイン適用のみの箇所）には置かないのが一般的である。3重の全バッファ走査は毎ブロック確実に発生するCPU負荷であり、「本当にどこで異常値が発生し得るか」を解析的に詰めた上で削減する余地がある。

## 2. ブロックごとのアルゴリズム検証

### 2-1. 入力段（`processInputDouble`）

- フォーマット変換 → モノラル展開 → NaN sanitize → レベル計測 → ヘッドルームゲイン → DCブロック、という順序。
- **レベル計測(`measureLevel`)がヘッドルームゲイン適用前**に行われている。これは「ユーザーが実際に入力しているソースのレベル」を見せるという意味では正しい（ヘッドルームゲインは内部処理用のゲインステージ調整であり、UIに見せたいのは生の入力レベルであるべき）。この点は(1-2)のTruePeak/LUFSの問題とは性質が異なり、妥当な設計と判断できる。
- DCブロッカーがヘッドルームゲイン適用後に置かれている点も、後段のDSP（畳み込み・EQ・オーバーサンプリング）にDCオフセットを持ち込まないという意味で正しい配置。

### 2-2. オーバーサンプリング（`CustomInputOversampler`）

- 2x/4x/8xの多段（各段2倍）ポリフェーズ・ハーフバンドFIRで構成。
- タップ数: LinearPhaseプリセットで1023/255/63（段ごと）、IIRLikeプリセットで511/127/31。
- 減衰量: LinearPhaseで140/160/120dB相当、IIRLikeで90/110/140dB相当（コード中の`attenuationForStage`）。
- `besselI0`関数の存在から、**Kaiser窓によるFIR設計**であることが読み取れる。Kaiser窓は任意の減衰量・遷移帯域を指定してタップ数を設計できる、実務で広く使われる手法であり、90〜160dBという数値はハイエンドのオーバーサンプリング実装として十分な水準。
- LinearPhase/IIRLikeの2プリセットを用意し、直線位相（レイテンシ増）と低レイテンシ（若干の位相非線形性）をユーザーが選べる設計は、マスタリング用途と低遅延モニタリング用途の両方に対応できており妥当。

**結論**: オーバーサンプリングエンジン自体の設計は音響工学的に問題なし。むしろ丁寧な実装。

### 2-3. 畳み込みエンジン（`MKLNonUniformConvolver`）

過去のセッションで作成した検証レポート（`ConvoPeq_Convolver_Validation_Report.md`）にて、TDUPCパイプライン・Radix-2 DIF/DIT FFT・Garcia(2002)のパーティション選択アルゴリズムの整合性は検証済み。今回、実際のRTホットパス(`Add`/`Get`/`processDirectBlock`/`processLayerBlock`)を直接確認した範囲でも、ヒープ確保・ロック・ログ出力は存在せず、AVX2アライメントチェックも適切に行われている。アルゴリズム面での新規の懸念は今回は見つからなかった。

### 2-4. 20バンドEQ（`EQProcessor`、TPT State-Variable Filter）

- 各バンドは Vadim Zavalishin の "The Art of VA Filter Design" に基づく **Topology-Preserving Transform (TPT) SVF** で実装されている。積分器状態を `ic1eq`/`ic2eq` として持ち、`2*v - state` という更新式は同文献の標準的な zero-delay feedback SVF の実装そのものであり、理論的に正しい。
- **直列(Serial)・並列(Parallel)の2つのフィルタ構造を切り替え可能**で、切り替え時にはEqual-Power的なクロスフェード（`equalPowerSin`によるTaylor近似sin）を用いて新旧構造の出力をブレンドしている。構造切替時の内部状態を独立して保持し、旧構造・新構造を両方計算してからブレンドするという実装は、単純に係数だけを切り替える実装（クリック音やフィルタ状態の不整合が生じやすい）よりも堅牢。
- **バンドごとにStereo/Mid/Side/Left/Rightのチャンネルモードを個別選択可能**。Mid/Side処理はM=(L+R)/2, S=(L-R)/2のエンコード→当該バンドのみ処理→L=M+S, R=M-Sのデコードという標準的な実装で数学的に正しい。
- **各バンドに歪み（`saturation`）をtanh近似でブレンドする機能が組み込まれている**点は特徴的だが、直列構造で20バンド全てにこの飽和を有効にした場合、各段が独立して非線形処理を行うため、単一の飽和段を通す場合に比べて累積的な高調波の生成パターンが予測しにくくなる。これは「バグ」ではなく明確に狙った音色付けの機能と考えられるが、音響工学的には「意図した音色設計」であることをドキュメント上明確にしておく価値がある（ユーザーが全バンドに強い飽和を掛けた場合の挙動が非直感的になり得るため）。
- AGCはブロックレート（コールバック単位）でRMSエンベロープを更新し、その後サンプル単位で線形ゲインランプを適用する設計。サンプル精度のエンベロープ追従ではなく「ブロック単位で1回だけ更新し、ブロック内は線形補間する」簡略化がされている。これはブロックサイズが大きい場合（512〜1024サンプル程度）にはアタック/リリースの実効的な時間分解能がブロック長に制限されることを意味する。一般的なAGC/コンプレッサー実装として許容範囲内ではあるが、非常に速いアタックタイムを設定した場合に、ブロックサイズ次第で意図通りの追従speedにならない可能性がある点は留意事項として記載する。

### 2-5. OutputFilter（HC/LC/LPF）

- HC（High Cut）: サンプルレートに応じてfc=19kHz（≤48kHz時）/22kHz（それ以外）のLPF。Sharp（Q=0.541と1.307の2段カスケード、実質4次特性）/Natural（Q=0.707×2段、Butterworth 4次相当）/Soft（Q=0.5の単段、1段のみで緩やかな特性）の3モード。
- LC（Low Cut）: Natural=18Hz Q0.707、Soft=15Hz Q0.5のHPF。
- 追加のLPF（`lpCoeff`）: fc=19kHz（≤48kHz）/24kHzで、Sharp/Natural/Soft相当のQ違い3種。
- **フィルタ設計自体はRBJ Cookbook系の標準的な2次biquad式（`makeLPF`/`makeHPF`）であり、係数導出に数学的な誤りはない。**
- fc=19kHz付近（48kHz動作時）はナイキスト（24kHz）に対してかなり近く、特にSharpモード（4次相当、Q1.307は比較的ピーキーな特性）の場合、可聴帯域上端（15-20kHz)にわずかなピーキングやロールオフが生じる可能性がある。マスタリング用途を意識するなら、この帯域でのマグニチュード特性（±0.1dB程度の平坦性が保たれているか）を測定・確認しておくことを推奨する。

### 2-6. ソフトクリップ（`musicalSoftClip` / `softClipBlockAVX2`）

- 前述の通り、オーバーサンプリング領域内、または未使用時はローカル2倍オーバーサンプルを介して適用されており、エイリアシング対策の観点で適切。
- 非対称パラメータ（`clipAsymmetry`）を持たせている点は、真空管的な偶数次高調波の付加を狙った設計と考えられ、意図が明確であれば問題ない。

### 2-7. 出力段（`processOutputDouble`）

- DCブロック → NaN scrub → TruePeak/LUFSメーター → Dither/NoiseShaping → NaN scrub → ハードクランプ → 固定レイテンシ整列、の順。
- TruePeak/LUFSの計測位置、およびハードクランプがリミッターの代替になっている点は前述（1-2 (a)(b)）の通り、改善の余地がある。
- ディザ/ノイズシェイピングは Fixed4Tap / Fixed15Tap / Adaptive9thOrder / plain TPDF の4方式を切り替え可能で、いずれも`kOutputHeadroom`（≒-1.0dBFS）でスケーリングしてから適用しており、量子化前に適切な安全マージンを確保する設計思想自体は正しい。Adaptive9thOrderのバンク切替時は`clampCoeff`によるNaNガードが効いているが、前セッションで指摘した通り`/fp:fast`（icxビルド）下での`std::isnan()`最適化消去リスクには注意が必要（別紙参照）。

### 2-8. 固定レイテンシ整列

- 単純なリングバッファによる遅延読み出しで、PDCとしての機能は妥当。ただし、ここで揃えられる遅延がオーバーサンプリングのFIRレイテンシ・畳み込みエンジンの内部遅延を含めた「トータルレイテンシ」の一部として正しく計算されているかは、`AudioEngineProcessor::prepareToPlay`側の`setLatencySamples()`呼び出しとの整合を別途確認する価値がある（今回のスコープでは信号処理アルゴリズム自体の検証に注力したため未確認）。

## 4. 確認された優れた設計（参考）

- 20バンドEQ: Zavalishin方式のTPT SVF、直列/並列構造切替時のクロスフェード、Mid/Side/L/Rの柔軟なチャンネルモード
- オーバーサンプリング: Kaiser窓ポリフェーズ・ハーフバンドFIR、90〜160dBの減衰、LinearPhase/IIRLikeの選択制
- DCブロッカー: 2段カスケード・周波数分散設計により位相歪みを抑制、3Hzという保守的なカットオフ
- TruePeakDetector: ITU-R BS.1770-4準拠の4xオーバーサンプル方式
- LoudnessMeter: Pre-filter(高域シェルフ) + RLBフィルタによる正しいK-weighting実装
- ソフトクリップ: 非線形段のエイリアシング対策としてオーバーサンプリング領域での適用を徹底
- RCU/EBRベースのロックフリー係数更新（EQ・畳み込み双方）
