## 1. 確定事項（反証不能）

### 1.1 DSP処理はXRUNの原因ではない

**根拠**: CALLBACK_STAGE 全2,590件を解析。単一コールバック内で予算超過（Budget>100%）したケースは**1件も存在しない**。

| 世代 | 件数 | 平均Budget | 最大Budget | 超過(Budget>100%) |
|------|------|-----------|-----------|-----------------|
| Gen=4 | 92 | 6.2% | 17.1% | **0件** |
| Gen=5 | 2,496 | 18.3% | 44.2% | **0件** |
| Gen=3 | 2 | 14.5% | 16.2% | **0件** |

**コード検証**: `Budget = callback処理時間[us] / expectedInterval[us] × 100` の定義を `BlockDouble.cpp:430-480` で確認。`CallbackTelemetryScope` の `budgetPermille` を実測。

**確度**: **確定**（ログ + コードの両立）

### 1.2 XRUNはコールバック間隔超過が原因

**根拠**: XRUN検出条件は `Interval > max(Expected × 1.5, 3.0ms)`。Expected=5.33ms → 閾値=8.0ms。
全131件のXRUNが Interval=8.00〜9.47ms で **8.0ms超過は131/131件（100%）**。

```cpp
// BlockDouble.cpp:445-453
const double intervalThreshold = std::max(expectedMs * kRatioThreshold, kFixedMarginMs);
// kRatioThreshold=1.5, kFixedMarginMs=3.0 → threshold = max(5.33*1.5, 3.0) = 8.0ms
if (intervalMs > intervalThreshold) xrunDetected = true;
```

**コード検証**: `BlockDouble.cpp:428-480` でXRUN検出ロジック確認。`kRatioThreshold=1.5`, `kFixedMarginMs=3.0` を確認。

**確度**: **確定**（ログ + コードの両立）

### 1.3 CPU Migrationが過剰

**根拠**: ログから `[CPU_MIG]` イベントを36,347件抽出。220.5秒間のコールバック約41,250回に対して約88%の確率でCPU移動。

| 世代 | Migration回数 | 使用CPU数 | 対コールバック比 |
|------|-------------|-----------|----------------|
| Gen=3 | 29 | 14/16 | — |
| Gen=4 | 1,012 | 16/16 | 高頻度 |
| Gen=5 | **35,306** | 16/16 | **~88%** |

**コード検証**: `GetCurrentProcessorNumber()` による検出は `AudioBlock.cpp:158-185` で確認。

**確度**: **確定**

### 1.4 CPU Affinityが完全未設定

**根拠**: `ThreadAffinityManager.h` の全コードパスを調査。

```
src/core/ThreadAffinityManager.h:144: std::atomic<bool> initialized_{false};
```

- `initialized_` への代入は宣言箇所以外に**1箇所も存在しない**
- `applyCurrentThreadPolicy()` は全8箇所の呼び出し全てで `if (!initialized_) return;` により即時return
- 呼び出し元: RebuildDispatch(1), LoaderThread(1), WorkerThread(1), DeferredFreeThread(1), NoiseShaperLearner(2), ProgressiveUpgradeThread(1), MainApplication(1)
- `masks_` を設定するpublicメソッドも存在しない

**コード検証**: grep全ソース(`ThreadAffinityManager|affinityManager|initialized_|masks_|SetThreadAffinityMask`) で完全確認。

**確度**: **確定**（ソースコードによる反証不能な事実）

### 1.5 Obsolete Rebuildが存在し無駄時間が発生

**根拠**: ログから3件のobsolete rebuildを確認。

### 2.1 DSPCore::prepare() の時間増加がメモリプレッシャーと相関

- 全割り当ては **条件付き**: 容量不足時にのみ再確保

**注意**: 時間増加の原因がPF支配か、memset支配か、アロケータ競合かはETW等での確認が必要。現時点では**相関**のみ確認。

**確度**: **高確度**（コード構造からはPF感受性が高いと言えるが、プロファイル未確認）

### 2.2 CPU Migrationのキャッシュ影響

**推定影響**:
- L1データキャッシュ（32KB/コア）: 毎回全ミスの可能性高い
- L2キャッシュ（256KB/コア）: DSP処理データ2048〜8192サンプル×8バイト×2ch〜2ch = 32KB〜128KB。おおむねL2に収まる可能性あり
- L3キャッシュ（共有、数MB）: 全コアで共有。CPUが変わってもL3ヒットの可能性あり

**コード検証**: DSP計算パスのデータアクセスパターン:
- Streaming access（連続メモリアクセス）: 大部分のDSP処理は線形アクセス（prefetch有効）
- DSPCore::process() 内で使用するデータは `alignedL/alignedR` などの線形バッファ
- コンボルバーは非線形アクセス（FFTベース、テーブルルックアップ含む）

**確度**: **高確度**（L1は確実に全ミス、L2は条件次第、L3はある程度緩和される）

### 2.3 リビルド廃棄が総合パフォーマンスに与える影響

**定量値**: 516.8msの無駄時間が3回の廃棄で発生。
**影響**: この間、CPUは新しいDSPCoreの構築に使用され、他スレッド（オーディオ含む）のスケジューリングに影響を与える可能性。

**確度**: **高確度**（無駄時間そのものは確定。他スレッドへの波及効果は推定）

## 3. 仮説（因果関係未証明・相関のみ確認）

### 3.1 「OS/Driver層のスケジューリング遅延 → XRUN」（残る仮説）

**深掘調査で否定された仮説**:
1. 「メモリ増加→PF→XRUN」: PFレートとXRUN発生率に相関なし（最大PF秒間232,451にXRUN=1、定常4,000/sにXRUN=0〜2）
2. 「リビルド負荷→XRUN」: BUILD_PHASE前後500msにXRUN=0件。リビルドスレッドはBELOW_NORMAL優先度
3. 「DSP処理時間→drift→XRUN」: 処理時間とdriftに無相関（DSP>1000us avg drift=+55us vs DSP<500us avg drift=-83us）
4. 「CPU Migration→キャッシュミス→XRUN」: XRUNのCallback=0.00ms（コールバック開始時点でDSP未実行）。キャッシュミスは処理時間に影響するが、コールバック到着間隔には影響しない

**残る可能性**: 現時点ではConvoPeq内部の原因は確認できず、OS/Driver層の複数要因（Windows Scheduler、MMCSS、ASIO Driver、Voicemeeter内部、DPC/ISR等）のいずれか、あるいは複合要因の可能性が残る
- コールバックスレッドが実行可能状態（Ready）になってから実際にディスパッチされるまでの時間（Ready Time）が時折5.33ms以上に伸びている
- CBSUMMARYのlossが全215件で負値（平均-5.0）であることから、システムは間引かれたコールバックを後で「追い込み」補償している
- このパターンは「OSのスケジューリング機会損失」と整合する
- 原因として考えられる要因（現時点では区別不可、ETWが必要）:
  ・Windows Scheduler
  ・MMCSS状態
  ・DPC/ISR滞留
  ・Audio Driver内部FIFO/バッファリング
  ・ASIOドライバ内部同期問題
  ・CPU Parking / Core scheduling / Interrupt Steering
  ・他プロセス・カーネルスレッドの割り込み

**検証に必要なもの**:
1. CPU affinity A/Bテスト: XRUN減少率でCPU Migration関与の有無を確認（最優先）
2. ETW (Windows Performance Recorder) による以下の測定:
   - Ready Time（スレッドが実行可能になってから実行されるまでの時間）
   - Context Switch原因と頻度
   - DPC/ISR滞留時間
   - Hard Page Faultの有無とレイテンシ

**確度**: **仮説**（但し深掘調査により他の仮説が全て否定され、残った仮説として強化された）

### 3.2 「ページフォールトがEQ_PREPARE劣化の主因」

**問題**: `GetProcessMemoryInfo()` の `PageFaultCount` は全PF種別を含む。

| PF種別 | レイテンシ | 含まれるか |
|--------|-----------|-----------|
| Soft Page Fault | 数百ns〜数μs | ✅含まれる |
| Hard Page Fault | 数ms〜数十ms | ✅含まれる |
| Transition Fault | 数百ns | ✅含まれる |
| Copy-on-Write | 数μs | ✅含まれる |

**ソフトPFの具体例**: DLLロード時、ヒープ拡張時、スタック拡張時など。
`makeAlignedArray` は `VirtualAlloc` または `_aligned_malloc` を使用する可能性が高い。どちらの場合も：
- `_aligned_malloc`: CRTヒープから割り当て → 新規ページにsoft PF発生
- `VirtualAlloc`: 直接仮想メモリ予約 → 初回アクセス時にsoft PF発生

**重要**: 34,000 PF/秒という数字は、soft PFが大半を占める可能性が高く、Hard PFの証拠にはならない。Soft PFはHDD/SSDアクセスを伴わないためレイテンシは低い。

**検証方法**: Windows Performance Recorder で `Page Fault` トレースを取得し、`HardFault` のみをフィルタリングする。

**確度**: **仮説**（ページフォールト種別の未確認）

### 3.3 「タイマーJITTERとXRUNは同一原因」

**観測事実**:
- タイマーJITTER: Timer thread が scheduler から実行機会を得られない
- XRUN: Audio callback thread が scheduler から実行機会を得られない
- 両者は独立したスレッドだが、同時期に遅延 → 共通原因（システム全体のスケジューリング問題）の可能性

**可能性A**: Timer delay → Audio delay（Timer threadがAudio callbackの開始を阻害？ → 独立スレッド）
**可能性B**: Audio負荷 → Timer delay（AudioがCPUを占有？ → DSP=18%程度）
**可能性C**: 共通原因（システム全体のスケジューリング問題）→ ✅ 最も可能性高い

**確度**: **仮説**（可能性Cが最も高いが証明されていない）

### 4.2 OSオーディオサブシステム/ドライバの影響

**問題**: 以下の要因がXRUNに関与している可能性を排除できない。
- WASAPIオーディオドライバのバッファアンダーラン
- DPC/ISRの滞留
- USBオーディオデバイスのレイテンシ変動
- 他プロセス（Windows Update, アンチウイルス等）の割り込み

**対策**: ETWトレース（`xperf` または `WPR`）で以下を計測：
- `HardFault` の発生とレイテンシ
- `Ready Thread` 時間（スレッドが実行可能になってから実際に実行されるまでの時間）
- `DPC/ISR` の滞留時間
- コンテキストスイッチ原因と頻度

**確度**: **要調査**（現ログでは不可能。ETWが必要）

### 4.3 2.5GBがプロオーディオとして異常値か

**問題**: 同等機能のプロオーディオソフトとの比較がない。

**参考**:
- ConvoPeq構成: IR(192k taps) + EQ(20 bands) + OS(2x) + Convolver + 全DSPCore世代
- 類似製品（例: Acustica Audio, LiquidSonics, FabFilter Pro-Q + Pro-R 併用）: 同様の条件下で1〜3GB程度になる場合あり

**対策**: メモリ内訳を取得した上で、合理的か判断。

**確度**: **要調査**

## 5. 改善案の優先順位再評価

### P0：ETWによるシステム全体のスケジューリング解析（最優先）

## 8. 結論

### 現時点で最も妥当な記述

> **ログ解析から以下のことは強く示唆される：**
>
> 1. DSP処理時間（平均0.97ms）はコールバック間隔（5.33ms）に比べて十分短く、**DSP負荷がXRUNの原因ではないことは確定できる**。
>
> 2. 全XRUNはコールバック間隔の超過（平均8.32ms、閾値8.0ms超過）によって発生しており、OSスケジューラがコールバックを予定時刻に開始できていないことを示している。
>
> 3. コールバックスレッドのCPU Migrationは約88%（16コア全使用）と非常に高く、メモリ使用量も初期の33倍に達している。タイマースレッドも同様に遅延しており、これらの観測はシステム全体のスケジューリング問題の存在を示唆している。
>
> **ただし、以下の因果関係は現時点では確定できない：**
>
> - 「メモリプレッシャー → スケジューラ遅延 → XRUN」の因果連鎖は、相関関係の確認に留まる。メモリ使用量の増加とXRUNの発生は時間的に一致するが、それが直接原因かは証明されていない。
> - CPU Migrationが直接の原因かどうかは、A/Bテスト（affinity on/off）が必要。
> - システム全体のスケジューリング状態を正確に把握するには、ETW（Windows Performance Recorder）によるHardFault・Ready Time・DPC/ISRの解析が必須である。
> - 2.5GBのメモリ使用量がプロオーディオアプリケーションとして異常かどうかは、メモリ内訳の詳細把握と比較対象が必要である。
> - リビルド廃棄による516msの無駄時間は測定されたが、これがXRUNに直接寄与しているかは未確認。
