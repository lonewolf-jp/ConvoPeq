## 1. 現状の計測基盤

既存の CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS=1 は17種の診断ログ（XRUN/CALLBACK_STAGE/CB_HIST/MEM/CPU_MIG等）を出力可能。詳細は第8版レポート「付録：検証方法」を参照。

**強み**: XRUN検出とinterval/Callbackの分離、DSP処理時間の予算内確認、CB_HIST直近32件追跡、CPU Migration検出、リビルド廃棄完全トレース

**弱み**:
- **ASIO bufferSwitch発行時刻が未計測**（最優先で解消）
- **JUCE→ConvoPeqディスパッチレイテンシが未計測**
- **MMCSS登録状態が未確認**（ETWで代替可能）
- メモリ使用量の内訳（サブシステム別）が不明
- timerCallback内部のどの処理が時間を消費しているか不明

## 2. DiagEvent 設計制約

本計画書の新規計測は、既存の **DiagEvent** 設計（`AudioEngine.h`）と整合させる必要がある。

```cpp
struct DiagEvent {
    DiagCategory category;    // 7種+新規
    uint64_t eventIndex;      // グローバルコールバック通し番号

static constexpr size_t kDiagEventSizeMax = 96;  // ★上限
static_assert(sizeof(DiagEvent) <= kDiagEventSizeMax);

**新規追加時の制約**:
1. **96バイト上限を超えないこと** — union最大メンバー約56バイト。新規構造体も同等以下に
2. **RT-safe** — 割り当て・ロック・I/O禁止。RingBufferへのpushのみ
3. **カテゴリ列挙値の追加のみ** — `DiagCategory::CallbackArrival = 8` 等。`Timer.cpp` 消費ループにcase追加

## 4. P0 計測項目（最優先—XRUN原因確定に必須）

### P0-A: CallbackArrival timestamp（callback 到着時刻の直接計測）

**目的**: XRUN の Interval 超過が、callback 到着間隔そのものの変動によるものか、OS/JUCE のディスパッチ遅延によるものかを切り分ける。

**原理**: 現在の `Interval = 今回callback開始 - 前回callback終了` には (a) callback 到着間隔 と (b) 前回callbackのDSP処理時間 の両方が含まれる。ConvoPeq の callback 入口でタイムスタンプを記録し、**連続する callback 間の純粋な到着間隔**を計測することで、(a)と(b)を分離する。到着間隔が期待値（5.33ms）から乖離していれば、ドライバまたはOSスケジューラ側に原因があることになる。

**計測位置**（コード確認済）:

**double パス**（Voicemeeter ASIO が使用）:
```
JUCE ASIO → AudioEngineProcessor::processBlock(double)  ← JUCE→ConvoPeq 入口
  → audioEngine.processBlockDouble(buffer)                ← ★ 計測位置
    → [XRUN 検出ブロック (t0_start)]
```

**float パス**:
```
JUCE AudioSource → AudioEngineProcessor::processBlock(float)
  → audioEngine.getNextAudioBlock(info)                   ← ★ 計測位置
    → [XRUN 検出ブロック (t0_start)]
```

**確認事項**: `processBlockDouble()` の先頭行（line 18, `BlockDouble.cpp`）が ConvoPeq が制御可能な最も早い入口である。**CallbackArrival は Audio callback 到達直後、処理分岐・RuntimeScope・ASSERT等よりも前に記録すること。**

**重要: `thisCallbackIndex` の取得タイミング**: `recordCallbackArrival()` は `eventIndex` として `thisCallbackIndex` を使用するが、これは `fetchAddAtomic(audioCallbackEpochCounter)` が先行して行われている必要がある。したがって以下の順序とする:
```cpp
void AudioEngine::processBlockDouble(...)
{
    // 1. 安全ガード (lifecycle, shutdown) — 早期return条件
    ...

    // 2. callbackIndex を先行取得
    const auto cbIdx = convo::fetchAddAtomic(
        rtLocalState_.audioCallbackEpochCounter, uint64_t{1}, std::memory_order_acq_rel);

    // 3. CallbackArrival 記録（RuntimeScopeより前）
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
    recordCallbackArrival(diagBuffer, cbIdx, numSamples, sr, cachedThreadId);
#endif

    // 4. RuntimeScope (ISR firewall, allocator)
    RuntimeScope scope(...);
    ...
}
```
これにより `eventIndex` は正しいcallback番号を持ち、かつ timestamp は最大限早期に記録される。`fetchAddAtomic` を RuntimeScope より前に移動することで、`thisCallbackIndex`（=`cbIdx`）は `recordCallbackArrival()` の時点で確定している。

JUCE 内部（`audioDeviceIOCallback` → `AudioEngineProcessor::processBlock` 間）は ConvoPeq 側からは計測不可能であり、ETW で確認する。

**新規データ構造**（20 bytes、eventIndex を活用して最小化。expectedUs はイベント発生時点の期待周期を保存）:

```cpp
struct CallbackArrivalData {
    uint64_t timestampUs;     // callback entry 時刻（getCurrentTimeUs）
    uint32_t expectedUs;      // 期待間隔（numSamples / sampleRate * 1e6、uint32_tで十分）
    uint16_t cpu;             // 実行CPU番号（ETW照合に必須）
    uint16_t reserved;        // alignment padding
    uint32_t threadId;        // スレッドID（optional、GetCurrentThreadId()の32bit完全値）
};
// 20 bytes。制限内（union最大約56byte、kDiagEventSizeMax=96）
// ★ expectedUs を uint32_t にした理由: 最大でも 185ms（44.1kHz/8192sample）であり
//   uint32_t(4,294,967,295us)で十分なため。4byte節約により構造体全体が20byteに。
// ★ threadId は optional（補助情報）。CPU/timestamp/expectedUsがETW照合の主軸。
//   16bitトランケーションはID衝突の原因になるため禁止。
// ★ callbackIndex は eventIndex（DiagEvent 共通ヘッダ）と重複するため不要。

**既存 `eventIndex` との関係（重要）**: `DiagEvent` のヘッダフィールド `eventIndex` には `thisCallbackIndex` が格納される。`CallbackArrivalData` に `callbackIndex` を含める必要はなく、既存ローカル変数 `thisCallbackIndex` を流用するべきである。`consumeAtomic` による再取得は不要である。新規 `CallbackArrivalData` に `callbackIndex` を含める必要はない。これにより構造体を 20 bytes に抑え、将来の拡張余地を確保する。

### ★ 代替設計案: deltaFromPrevious 方式（参考）

上記 primary 案（timestampUs + expectedUs 保存）の代替として、**deltaFromPrevious 方式** も検討する。この方式では Audio Thread 側で前回からの経過時間を計算して保存する。

```cpp
struct CallbackArrivalData {
    uint64_t timestampUs;     // callback entry 時刻
    uint32_t expectedUs;      // 期待間隔
    int32_t  deltaFromPrevUs; // 前回からの実際の経過時間（= nowUs - prevEntryUs）
    uint16_t cpu;             // 実行CPU番号
    uint16_t reserved;
    uint32_t threadId;        // optional
};
// 24 bytes（delta追加により拡大）
```

**利点**: 差分計算を Audio Thread 側で行うため Timer 側の static 状態が不要。`lastCallbackEntryUs`（既存 `RTLocalState`、`AudioEngine.h:1435`）を流用可能。1イベントで完結する。

**欠点**: 24byte と primary 案より大きい。`timestampUs` は ETW 照合用に別途必要。解析の単純化には寄与するが、primary 案と比較して RingBuffer 消費量が20%増える（24byte vs 20byte）。

**評価**: primary 案（timestampUs + expectedUs、20byte）を本設計の第一候補とする。deltaFromPrevious 方式は実装時に両方式を比較検討する場合の参考としてここに記す。

**recordCallbackArrival() 関数化（実装推奨）**: 書込側コードは `BlockDouble.cpp` と `AudioBlock.cpp` の2箇所に重複して記述される。これを避けるため、以下のようなインライン関数を共通ヘッダ（例: `AudioEngine.Internal.h`）に定義することを推奨する:
```cpp
inline void recordCallbackArrival(
    LockFreeRingBuffer<DiagEvent, DiagRuntimeLimits::BufferCapacity>& diagBuffer,
    uint64_t thisCallbackIndex,
    int numSamples, double sampleRate,
    uint32_t cachedThreadId) noexcept
{
    DiagEvent event{};
    event.category = DiagCategory::CallbackArrival;
    event.eventIndex = thisCallbackIndex;
    event.data.callbackArrival.timestampUs = convo::getCurrentTimeUs();
    event.data.callbackArrival.cpu = static_cast<uint16_t>(::GetCurrentProcessorNumber());
    event.data.callbackArrival.threadId = cachedThreadId;
    event.data.callbackArrival.expectedUs = (sampleRate > 0.0 && numSamples > 0)
        ? static_cast<uint32_t>(std::lround(static_cast<double>(numSamples) / sampleRate * 1e6)) : 0;
    // expectedUs: lround で四捨五入。一度だけ recordCallbackArrival() 側で算出し、Timer側では再計算しない（prepareToPlayによる変化を正しく反映するため）
    if (!diagBuffer.push(event)) {
        // drop count increment
    }
}
```
これにより2箇所の呼び出しは `recordCallbackArrival(diagBuffer, thisCallbackIndex, numSamples, sr, rtLocalState_.cachedThreadId);` の1行で済む。inline化によりRTコストは実質ゼロ。

**読出側**（`Timer.cpp` DiagEvent 消費ループ）:

```cpp
case DiagCategory::CallbackArrival: {
    const uint64_t ts = diagEv.data.callbackArrival.timestampUs;
    const uint32_t expected = diagEv.data.callbackArrival.expectedUs;
    const uint16_t cpu = diagEv.data.callbackArrival.cpu;
    const uint32_t threadId = diagEv.data.callbackArrival.threadId;

    // 前回 timestamp との差 = callback 到着間隔
    if (s_prevTimestampUs > 0) {
        const int64_t intervalUs = static_cast<int64_t>(ts - s_prevTimestampUs);
        const int64_t driftUs = intervalUs - static_cast<int64_t>(expected);
        // drift 閾値: max(1000us固定, expected*0.1)。固定値でOS遅延を評価しつつ
        // 極端な小バッファ時は比率判定で補完。
        // ★ 実効的には 64sample/96kHz でも expected=1333us→expected/10=133<1000 なので
        //   通常のバッファ設定（128smp以上）では 1000us 固定とみなしてよい。
        const int64_t driftThreshold = std::max(static_cast<int64_t>(1000),
            static_cast<int64_t>(expected) / 10);
        if (expected > 0 && std::abs(driftUs) > driftThreshold
            && (diagEv.eventIndex & CONVOPEQ_DIAG_SAMPLE_MASK) == 0) {
            diagLog("[CB_ARRIVAL] callback=" + juce::String(static_cast<int64_t>(diagEv.eventIndex))
                + " cpu=" + juce::String(static_cast<int>(cpu))
                + " interval=" + juce::String(static_cast<double>(intervalUs) / 1000.0, 2) + "ms"
                + " expected=" + juce::String(static_cast<double>(expected) / 1000.0, 2) + "ms"
                + " drift=" + juce::String(static_cast<double>(driftUs) / 1000.0, 2) + "ms");
        }  // ★ late/early フラグは drift の正負で判断可能なため出力しない
    }
    s_prevTimestampUs = ts;
    break;
}
```

**サンプリング**: 全 callback の timestamp を記録する（20 bytes のみなので RingBuffer 負荷は軽微）。ログ出力は sampling mask で間引く。

**DiagEvent 肥大化対策**: 20 bytes 構造体。既存最大メンバー（約56 bytes）より小さく、96 bytes 制限に問題なく収まる。

**影響範囲**: `AudioEngine.h`（DiagCategory+1, union+1）、`BlockDouble.cpp`（書込＋thisCallbackIndex流用）、`AudioBlock.cpp`（書込＋thisCallbackIndex流用）、`Timer.cpp`（case+1）

### P0-A補足: 設計上の補足と将来拡張候補

**イベント記録順序（重要）**: `CallbackArrival` は `processBlockDouble()` / `getNextAudioBlock()` の先頭で記録されるため、同一コールバック内で
```
CallbackArrival → XRUN検出 → CALLBACK_STAGE → DSP処理
```
の順序が保証される。すなわち、同一 `eventIndex`（=`thisCallbackIndex`）を持つ他イベント（XRUN, CALLBACK_STAGE, CPU_MIG, DspTiming, CB_HIST）よりも先に記録されており、解析時に `eventIndex` でマージすれば「callback n の到着時刻」を正確に突合できる。この順序保証を設計書に明記することで、実装者と解析者の認識を一致させる。

**Timer側状態の構造体管理**: 読出側コードの `CallbackArrivalContext` は Timer Thread のみがアクセスする static 状態として `Timer.cpp` 内部に保持する:
```cpp
// Timer.cpp 内部: Timer Thread のみアクセス
struct CallbackArrivalContext {
    uint64_t lastTimestampUs = 0;   // 前回callback到着時刻
    uint32_t lastExpectedUs = 0;    // ★将来: sampleRate変化検出用（現在は不使用）
    uint64_t intervalAccum = 0;     // 平均interval計算用（将来拡張）
    int      intervalCount = 0;     // 同上
};
static CallbackArrivalContext g_callbackArrivalCtx;
```
`lastExpectedUs` は現在の読出コードでは使用していないが、将来の sampleRate 変更検出に備えて確保している。

**最初の1イベントの扱い**: `lastTimestampUs==0` の場合は「前回値なし」とみなし interval 計算を行わない。これにより Timer 起動直後やリセット後の最初のイベントは正常にスキップされる。

この構造体のリセットは以下のいずれかのタイミングで必ず行うこと:
  - `prepareToPlay()` 末尾での Timer 起動時
  - Device Stop → Device Start の再開時
  - 上記いずれも保証できない場合は Timer 開始直後に強制リセット（`lastTimestampUs = 0`）
実装時には `prepareToPlay()` と Timer 起動の両方でリセット処理を入れることが望ましい。

**CPU番号の注意点**: `::GetCurrentProcessorNumber()` は Windows の group-aware API (`GetCurrentProcessorNumberEx()`) ではなく group 非対応版を使用する。64論理CPU未満の環境では問題ないが、将来 64CPU 超環境に対応する場合は `GetCurrentProcessorNumberEx()` への移行を検討すること。本設計書の対象範囲では後方互換性を優先して `GetCurrentProcessorNumber()` を採用する。

**ThreadID の保存方針**: Audio Thread は通常固定のため、毎イベントの4byte保存は無駄が多い。本設計では `CallbackArrivalData` の `threadId` フィールドは構造体として確保するが、値は Audio Thread 開始時に一度だけ `GetCurrentThreadId()` で取得し `RTLocalState::cachedThreadId` に保持する。以降の全callbackでは API 呼び出しをせずにキャッシュ値を流用する:
```cpp
event.data.callbackArrival.threadId = rtLocalState_.cachedThreadId;  // 毎回API呼ばない
```
`GetCurrentThreadId()` は TLS 参照程度で高速だが、192kHz/1024spb では 187回/秒の無駄な API 呼び出しとなるため避ける。ETW照合には `timestamp` + `CPU` で十分対応できる。

**ThreadID の目的**: 通常 Audio Thread は1本のみのため ThreadID は不変だが、timestamp だけでは ETW 上で複数の Audio Thread 候補が存在する場合（稀なマルチクライアント構成等）に ThreadID を使って候補を絞り込む。ほとんどの環境では不要なため `optional`（補助情報）として位置付ける。

**CallbackArrival Written/Drop カウンタ（実装推奨）**: RingBuffer が逼迫した場合、`CallbackArrival` イベントが最もドロップされやすい。現在の `diagTickDropped` は全イベント共通だが、`CallbackArrival` 専用のカウンタを追加することを推奨する。具体的には以下の2つを `rtAuxMutable_` に追加する:
  - `cbArrivalWritten` — push成功回数
  - `cbArrivalDropped` — push失敗（drop）回数
両方を Timer 側の DIAG_STAT で出力することで、`drop率 = dropped / (written+dropped)` まで把握できる。drop率が高い場合は RingBuffer サイズ増加や sampling mask 調整の判断材料となる。

**将来拡張**: 単なる drop率に加えて「最大連続Drop数」も記録すると、`4回連続drop中` と `10000callback中4回` の区別がつき解析が容易になる。実装時に `cbArrivalConsecutiveDrops` カウンタを追加することを推奨する。

**RingBuffer 逼迫時の優先度制御（将来拡張候補）**: さらに進んだ対策として、カテゴリ別のドロップ優先度を導入し、`CallbackArrival` を他の重要イベント（XRUN, CPU_MIG, DspTiming）より先にドロップする設計を将来課題として検討する。

**QPC raw counter（将来拡張候補）**: ETW との時刻照合をさらに精密化したい場合、`timestampUs` に加えて `QueryPerformanceCounter()` の raw カウント値を保存する方法がある。現在の `getCurrentTimeUs()` は `std::chrono::steady_clock` ベース（`core/TimeUtils.h:14`）であり、QPC は使用していない。QPC raw 値を保存すれば WPA とほぼ完全一致の時刻照合が可能だが、clock source の差異は通常の運用では問題にならないため、「あれば便利」レベルの拡張として位置付ける。

## 5. P1 計測項目（重要—原因特定を補強）

### P1-A: TimerCallback 内部ブロック別経過時間

**概要**: timerCallback 内の主要ブロックの前後で elapsed を計測。10回に1回（約1秒ごと）出力。

**計測区間**:
```
timer start (getMillisecondCounterHiRes)
  ├─ Jitter: 前回timerからのjitter判定
  ├─ Learning: processLearningCommands / processDeferredLearningActions
  ├─ Memory: getProcessMemoryInfo (1秒ごと)
  ├─ CBSUMMARY: callback統計の集計
  ├─ XRUNDrain: XRunBuffer消費 + CB_HISTダンプ
  ├─ DiagDrain: DiagEventリングバッファ消費 + DIAG_STAT
    ├─ LogFlush: timerCallback末尾の実行時間警告(>10ms時のみ)
    └─ Other: Total - Σ(上記各ブロック) — 未分類時間の可視化
  timer end
```

各ブロックの経過時間は `juce::Time::getMillisecondCounterHiRes()` の差分から計測する。`Other` を算出することで、新たなブロック追加や予期しない処理時間の増加を見逃さない。10回に1回（約1秒ごと）の出力で十分。
**初版から変更**: P0 → **P1（ETW推奨）**。
- `AvQueryMmThreadCharacteristics()` は現在スレッド（Timer Thread）のMMCSS状態しか確認できず、Audio Thread は確認不可
- Audio Thread 内で毎回呼ぶのは RT 違反
- **ETW の MMCSS トレースで完全に代替可能**。ETW を優先すべき

### 注意: M1 の設計上のポイント

- **eventIndex 重複排除**: DiagEvent ヘッダの `eventIndex` が `thisCallbackIndex` を保持するため、データ部に `callbackIndex` を二重保存しない
- **P0-B 統合**: Callback entry delay は P0-A の `timestampUs` で兼ねる。別計測不要
- **expectedUs は保存必須**: uint32_t で十分（最大185ms）。Timer側での再計算は prepareToPlay による sampleRate/bufferSize 変更時に不整合が生じる危険がある。また解析対象イベントは将来オフライン解析される可能性を考慮し、イベント発生時点の期待周期をイベント自身に保存することで、解析時点の設定状態に依存しない自己完結性を確保する。4byteの追加（20byte計）は96byte制限に十分収まる。
- **threadId は optional**: CPU番号・timestamp・expectedUs の3つがETW照合の主軸。ThreadID は Audio Thread 固定なことが多くXRUN解析への寄与は限定的だが、「あれば便利」程度の補助情報として保持する（+4byte）。
- **代替構成案（16byte trim）**: ThreadID を削除し、`reserved` を2byteに拡張すると `timestamp(8) + expected(4) + cpu(2) + reserved(2) = 16byte` にできる。これは2のべき乗サイズでアライメントに自然であり、RingBuffer 帯域も削減できる。本設計書では ETW 照合の利便性を重視して 20byte を推奨するが、RingBuffer 負荷が問題になった場合の代替構成として記憶しておくこと。

### P0設計への反映

上記の確定結果に基づき、CallbackArrival計測の設計は以下の方針で進める：

1. **ConvoPeq自身はASIO経路でMMCSS登録を実施していない（コード確定）**。ただしASIOドライバやJUCE内部が別経路でMMCSS登録する可能性までは否定しない。ETW Phase 0 の優先度は変わらない。
2. **P0-CallbackArrival計測**は本設計書の通り実装する。XRUN原因がOSスケジューラ遅延であっても、その証拠としてコールバック到着間隔の直接計測は必須。
3. **ThreadIDは補助情報として保持**（CPU/timestamp/expectedUsが主軸）。
4. **drift閾値はmax(1000us, expected/10)**で判定。最低1ms、小バッファ時は期待周期の10%で補完する。
