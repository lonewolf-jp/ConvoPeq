# デバウンス時間短縮による悪影響の検証報告

先ほどの修正で `REBUILD_DEBOUNCE_DEFAULT_MS` を 400ms から 20ms に短縮し、過大なキュー蓄積による音飛び（ドロップアウト）を解消しましたが、この短縮によって生じうる副作用（悪影響）を詳細に検証しました。

## ⚠️ 予想される主な悪影響

デバウンスを 20ms に極端に短くした場合、以下のような副作用がトレードオフとして発生します。

### 1. バックグラウンドスレッドの CPU 負荷増大（スラッシング）
- **現象**: 構造的リビルド（`rebuildAllIRsSynchronous` など、FIRフィルタの再計算等）は非常に重く、1回あたり **約 400ms〜540ms** の時間を要します。
- **影響**: UIのパラメータをドラッグし続けた場合、20ms ごとに新しいリビルド要求が発生します。システムは古いキューを破棄して最新の要求を 1つだけ保持する仕組み（`hasPendingTask` の単一スロットキュー）になっていますが、バックグラウンドの Rebuild スレッドは「400msの重いビルドを終えた直後に、次のビルドを即座に開始する」という処理をドラッグ中ずっと繰り返すことになります。これにより、対象のCPUコアが継続的に 100% 張り付く状態になります。

### 2. オーディオスレッドへのリソース競合（再度の音飛びリスク）
- **現象**: 上記の通りバックグラウンドスレッドがフル稼働すると、CPUキャッシュ（L2/L3）やメモリ帯域を大量に消費します。
- **影響**: コア数の少ない環境や、OSのスケジューラ分離が甘い環境では、リアルタイム処理が必須のオーディオスレッド（Message/Audio Thread）のリソースを奪ってしまいます。結果として、**キューの遅延とは別の原因（CPUリソースの枯渇）によって、再度音飛びが発生するリスク**が高まります。

### 3. 頻繁な DSP クロスフェードによるオーディオアーティファクト
- **現象**: リビルドが完了する（約 400〜500ms ごと）たびに、古い DSP 状態から新しい DSP 状態への切り替え（クロスフェード）が実行されます。
- **影響**: ユーザーがスライダーを3秒間動かし続けた場合、その間に約6〜7回のクロスフェードが連続して発生します。頻繁なクロスフェードは、コムフィルタリング（シュワシュワとした音色変化）や位相干渉（フェージング）といった不自然なオーディオアーティファクトを引き起こし、聴覚上の違和感に繋がります。

### 4. メモリアロケーションのチャーン（頻繁な確保と解放）
- **現象**: リビルド処理では、巨大な FFT バッファや IR のゼロパディング処理のために、大きなメモリブロックの動的確保・解放が発生します。
- **影響**: 500msに1回の頻度で巨大なメモリの確保・解放（メモリチャーン）が繰り返されると、OS のメモリアロケータに負担がかかり、メモリ断片化やページフォールトを誘発します。最悪の場合、これがオーディオスレッドの処理を瞬間的にブロックする要因になり得ます。

---

## 💡 今後の根本的な対策方針

デバウンスを 20ms に短縮した現在の状態は、「400ms 待機によるバッファ決壊（確実な音飛び）」を回避する緊急処置としては有効ですが、長期的には上記の副作用が問題になります。
今後は以下のようなアプローチを検討することをお勧めします。

1. **パラメータスムージングの導入**:
   頻繁に動かすパラメータ（EQゲインなど）については、構造的なリビルド（IR再計算）を行わずに DSP 内部でリアルタイムに係数を補間（スムージング）する仕組みに変更する。
2. **適切なデバウンス値の再設定**:
   UI の操作が終わったと見なせるギリギリのライン（例: `100ms` 〜 `150ms`）までデバウンス値を再調整し、ドラッグ中の無駄なリビルド回数を減らす。
3. **IR 再計算の軽量化・非同期化**:
   `rebuildAllIRsSynchronous` の処理時間を 400ms から削減するための最適化を行う（バックグラウンドでの分割計算など）。
