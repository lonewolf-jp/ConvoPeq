## 共通基盤: DiagnosticsConfig.h の拡張

`getProcessMemoryInfo()` を各 .cpp に重複実装せず、`DiagnosticsConfig.h` に集約する。これにより Timer.cpp / Commit.cpp / RebuildDispatch.cpp の全ファイルで一元利用可能になる。

### 追加する共通定義

```cpp
// src/DiagnosticsConfig.h に追加

#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS

// ---- シーケンス番号（全診断ログ共通の単調増加カウンタ） ----
inline std::atomic<uint64_t>& diagSequenceCounter() noexcept
{
    static std::atomic<uint64_t> counter { 0 };
    return counter;
}

// ---- ProcessMemoryInfo ----
#include <windows.h>
#include <psapi.h>
// ★ #pragma comment(lib, "psapi.lib") は DiagnosticsConfig.h には置かない。
//    代わりに Timer.cpp / Commit.cpp の #if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
//    ブロック内に配置する。これによりライブラリ依存が全TUへ伝播するのを防ぐ。

struct ProcessMemoryInfo {
    uint64_t privateUsageMB = 0;
    uint64_t workingSetMB = 0;
    uint64_t pageFaultCount = 0;
};

inline ProcessMemoryInfo getProcessMemoryInfo() noexcept
{
    ProcessMemoryInfo info{};
    PROCESS_MEMORY_COUNTERS_EX pmc;
    pmc.cb = sizeof(pmc);
    if (GetProcessMemoryInfo(GetCurrentProcess(),
            reinterpret_cast<PROCESS_MEMORY_COUNTERS*>(&pmc), sizeof(pmc)))
    {
        info.privateUsageMB = static_cast<uint64_t>(pmc.PrivateUsage / (1024ULL * 1024ULL));
        info.workingSetMB = static_cast<uint64_t>(pmc.WorkingSetSize / (1024ULL * 1024ULL));
        info.pageFaultCount = static_cast<uint64_t>(pmc.PageFaultCount);
    }
    return info;
}

#endif
```

### 優先度1: XRUN 検出強化

**目的**: 音飛びとAudio Thread遅延の因果関係を証明するための決定的証拠。

**現状**: XRUN検出は実装済みだが、今回のログでは一度も発火しなかった。原因として:
- Worker Thread と Audio Thread は別スレッドのため、Workerの523ms占有が直接Audioのcallback時間に影響しない
- ページフォールトによるOSレベルの停止は callbackMs/intervalMs として検出できるはずだが、
  callback時間が閾値（`max(expected*1.2, 1ms)`）を超えなかった可能性がある
- 192kHz・1024samples の場合 `expectedMs = 5.33ms`、閾値 `max(5.33*1.2, 1.0) = 6.4ms`

**改善案**: XRUN検出に加えて、**1秒単位で intervalMax / callbackMax / callbackCount の集約値**を出力する。XRUNが発生しなくとも「Audio Threadが本当に遅延したか」を直接観測可能にする。

**主要指標**: `intervalMax` を主指標、`callbackMax` を副指標とする。OS Scheduler停止はcallbackが開始されない現象であり、callbackMaxは小さなまま、intervalMaxだけが増加する。逆にcallbackMaxが大きい場合はDSP処理負荷の問題を示す。

**抑制**: intervalMax・callbackMax のいずれかに変化があった秒のみ出力（通常時は出力0行）。callbackCountは参考値として出力するが、変化判定には含めない。長時間ログでの出力増加を防ぐ。

```cpp
// Timer.cpp — MEMブロックの隣に追加（1秒1回、変化時のみ出力）
// ★ Audio callback の最大値サマリ（XRUNが発生しなくても常時計測）
//    callbackMaxUs_ / intervalMaxUs_ / callbackCount_ は AudioBlock.cpp 内で
//    Audio Thread の各callback終了時に atomic 更新される。
{
    static uint64_t lastCbSummaryTicks = 0;
    const auto nowTicks = juce::Time::getHighResolutionTicks();
    const auto ticksPerSec = juce::Time::getHighResolutionTicksPerSecond();
    if (nowTicks - lastCbSummaryTicks >= ticksPerSec)
    {
        lastCbSummaryTicks = nowTicks;
        // atomic<uint32_t> で保持 — μs単位で整数化してRT-safeに
        const uint32_t cbMaxUs = convo::exchangeAtomic(callbackMaxUs_, 0u, std::memory_order_acq_rel);
        const uint32_t ivMaxUs = convo::exchangeAtomic(intervalMaxUs_, 0u, std::memory_order_acq_rel);
        const uint32_t cbCount = convo::exchangeAtomic(callbackCount_, 0u, std::memory_order_acq_rel);

        // 期待値計算: sampleRate / blockSize
        // 固定値ではなく動的算出することで、サンプルレートやブロックサイズ変更に追従
        const uint32_t expectedCbCount = (sampleRate_ > 0 && blockSize_ > 0)
            ? static_cast<uint32_t>(sampleRate_ / blockSize_) : 0;
        const int32_t lossCount = static_cast<int32_t>(expectedCbCount) - static_cast<int32_t>(cbCount);

        // 変化があった秒のみ出力。主判定は intervalMax の変化。
        // callbackCount（loss）は参考値として出力するが変化判定には使わない。
        // ★ intervalMax を変化判定に含める — OS Scheduler停止は intervalMax の急増で検出
        static uint32_t s_prevIntervalMaxUs = 0;
        static uint32_t s_prevCbMaxUs = 0;
        const bool hasChange = (ivMaxUs != s_prevIntervalMaxUs)
            || (cbMaxUs != s_prevCbMaxUs);
        s_prevIntervalMaxUs = ivMaxUs;
        s_prevCbMaxUs = cbMaxUs;

        if (hasChange && (cbMaxUs > 0 || cbCount > 0)) {
            const auto seq = convo::fetchAddAtomic(diagSequenceCounter(), uint64_t{1}, std::memory_order_acq_rel) + 1u;
            diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
                + "] [CBSUMMARY] intervalMax=" + juce::String(static_cast<double>(ivMaxUs) / 1000.0, 3) + "ms"  // ★ 主指標
                + " callbackMax=" + juce::String(static_cast<double>(cbMaxUs) / 1000.0, 3) + "ms"  // ★ 副指標
                + " expected=" + juce::String(static_cast<juce::int64>(expectedCbCount))
                + " actual=" + juce::String(static_cast<juce::int64>(cbCount))
                + " loss=" + juce::String(static_cast<juce::int64>(lossCount)) + ")");  // ★ 参考値
        }
    }
}
```

**追加の atomic 変数**（`AudioEngine.h` に `std::atomic<uint32_t>` で追加。全変数は `memory_order_relaxed` で十分）:
- `intervalMaxUs_` — 1秒間のcallback間隔の最大値（μs、0でリセット）★主指標
- `callbackMaxUs_` — 1秒間のcallback実行時間の最大値（μs、0でリセット）★副指標
- `callbackCount_` — 1秒間のcallback実行回数（参考値）
- （`sampleRate_` / `blockSize_` はAudioEngineの既存メンバから読み取り）

**AudioBlock.cpp の更新**（各callback末尾）— `updateAtomicMaximum()` ヘルパーを使用:
```cpp
// ★ updateAtomicMaximum() ヘルパー（DiagnosticsConfig.h に追加）:
//    atomic_fetch_max 相当の操作。compare_exchange_weak ループを隠蔽。
//    C++26 の std::atomic<…>::fetch_max がない環境向け。
//    memory_order は relaxed で十分（uint32_t 単一の更新、他変数との順序依存なし）。
inline void updateAtomicMaximum(std::atomic<uint32_t>& target, uint32_t value) noexcept
{
    uint32_t expected = target.load(std::memory_order_relaxed);
    while (value > expected && !target.compare_exchange_weak(expected, value,
        std::memory_order_relaxed, std::memory_order_relaxed)) {}
}

// AudioBlock.cpp の callback 末尾:
const auto cbUs = static_cast<uint32_t>(callbackEndUs - callbackStartUs);
updateAtomicMaximum(callbackMaxUs_, cbUs);

const auto ivUs = static_cast<uint32_t>(callbackIntervalUs);
updateAtomicMaximum(intervalMaxUs_, ivUs);

callbackCount_.fetch_add(1u, std::memory_order_relaxed);
```

### 優先度2: PageFault Delta + 警告（最優先で実装）

**目的**: 333,592回のような異常なPageFault急増をリアルタイムで検出し、原因特定を迅速化する。

#### 設計

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存、`ProcessMemoryInfo` は共通基盤から）

**出力先**: `Timer.cpp` MEMブロック（1秒1回の定期ログに追記）

**抑制**: MEMログ自体が1秒1回。Delta計算は内部変数のみ。異常時のみ警告行を追加。

**閾値**: 以下の2条件の**いずれか**を満たした場合に警告。

```
absolute:   delta > 50000          ← 絶対値（必ず検出）
relative:   delta > EWMA * 5       ← 倍率（高負荷時も適応）
```

```
EWMA: avg = 0.05 * delta + 0.95 * avg_prev   ← alpha=0.05（巨大スパイクの引きずりを抑制）
```

- 通常時のPageFault/秒が平均500の場合、`EWMA_avg * 5 = 2500` → 2500超過で警告
- 333,592のスパイク後（クリッピングなし）: `EWMA_avg = 0.05*333592 + 0.95*500 ≈ 17,155` → `threshold ≈ 85,775`
  - クリッピングあり（`min(delta,50000)`）: `EWMA_avg = 0.05*50000 + 0.95*500 ≈ 2,975` → `threshold ≈ 14,875`
  - クリッピングによりベースライン維持 → スパイク後も通常の異常(20000)を検出可能

> **注**: EWMA分散（σ）による判定（`delta > EWMA + 4σ`）は設計論としては有用だが、Windows PageFaultは元々バーストしやすい特性があり、σ条件が本当に有効かは実測前には判断できない。初版では absolute + relative の2条件で運用し、必要に応じて後日σ条件を追加する。

```cpp
// Timer.cpp MEMブロック内、getProcessMemoryInfo() 呼び出し後に追加
static uint64_t s_prevPageFaults = 0;

const uint64_t pfDelta = (s_prevPageFaults > 0)
    ? (memInfo.pageFaultCount - s_prevPageFaults)
    : 0;
s_prevPageFaults = memInfo.pageFaultCount;

// ★【重要】EWMA更新は「警告判定の後」に行う
//    誤った順序: EWMA更新 → 判定（異常値自身が閾値を引き上げてしまう）
//    正しい順序: 判定 → EWMA更新（異常値は閾値計算に影響させない）
// ★ EWMA初期化: 初回の異常値でEWMAが汚染されるのを防ぐため、
//    最初の10サンプルは absolute 判定のみで運用する。
//    10サンプル集まってから relative 判定を有効にする。
static double s_pfEwmaAvg = 0.0;
static int s_pfSampleCount = 0;
static constexpr int kEwmaWarmupSamples = 10;

// 2条件の判定: absolute OR relative（EWMA更新の前に判定）
// ★ relative判定は10サンプル集まってから有効化
static constexpr uint64_t kAbsoluteThreshold = 50000;
const double thresholdRel = (s_pfSampleCount >= kEwmaWarmupSamples) ? (s_pfEwmaAvg * 5.0) : 0.0;
const uint64_t threshold = std::max<uint64_t>(kAbsoluteThreshold,
    static_cast<uint64_t>(thresholdRel));
const bool warnAbsolute = (pfDelta > kAbsoluteThreshold);
const bool warnRelative = (s_pfSampleCount >= kEwmaWarmupSamples)
    && (pfDelta > static_cast<uint64_t>(thresholdRel));

// MEMログに Private/WS/PageFaults/Delta/Seq の5点セットを必ず含める
// 変更前: "[MEM] Private=1352MB WS=1256MB PageFaults=328239"
// 変更後: "[MEM] Seq=120 Private=1352MB WS=1256MB PageFaults=328239 Delta=+0"

if (pfDelta > 0 && (warnAbsolute || warnRelative)) {
    const auto seq = convo::fetchAddAtomic(diagSequenceCounter(), uint64_t{1}, std::memory_order_acq_rel) + 1u;
    diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
        + "] [WARN] PageFault surge: +" + juce::String(static_cast<juce::int64>(pfDelta))
        + " faults (EWMA=" + juce::String(s_pfEwmaAvg, 0)
        + ", threshold=" + juce::String(static_cast<juce::int64>(threshold)) + ")");
}

// ★ 警告判定の後にEWMA更新（異常値が閾値を汚染しない）
// ★ EWMA更新値はクリッピング: min(delta, 50000) をEWMAへ入れることで
//    333kのような巨大スパイクでベースラインが壊れるのを防止する。
//    警告判定にはクリッピング前の raw delta を使用するため、検知能力は低下しない。
// ★ サンプルカウント更新はEWMA更新の前（0サンプルでも判定はabsoluteのみ）
if (s_pfSampleCount < kEwmaWarmupSamples) {
    // ウォームアップ中もクリッピング適用
    const double clippedDelta = std::min<double>(static_cast<double>(pfDelta), 50000.0);
    s_pfEwmaAvg = (s_pfEwmaAvg * static_cast<double>(s_pfSampleCount)
        + clippedDelta)
        / static_cast<double>(s_pfSampleCount + 1);
    ++s_pfSampleCount;
} else {
    // ★ EWMA更新のみクリッピング — 警告判定は raw pfDelta を使う
    const double clippedDelta = std::min<double>(static_cast<double>(pfDelta), 50000.0);
    s_pfEwmaAvg = 0.05 * clippedDelta + 0.95 * s_pfEwmaAvg;
}
```

### 優先度4: Timer jitter + XRUN連携

**目的**: Worker Threadの負荷やOSのメモリ管理がTimer Threadに影響したかを検出する。Timer jitter は XRUN イベントに集約して出力する。

#### 設計

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**出力先**: `Timer.cpp` timerCallback 先頭（計測） + XRUNログに付加

**抑制**:
- 単独の `[TIMER] jitter` は相対閾値 `max(10ms, expected×10%)` 以上乖離時のみ（正常時0行）
- XRUN発生時は現在のTimer実行時間＋直近最大値をXRUNログへ自動付加

```cpp
// timerCallback() 先頭 — 計測のみ
static double s_prevCallbackMs = 0.0;

const double nowMs = juce::Time::getMillisecondCounterHiRes();
const double timerExecStartMs = nowMs;  // ★ callback開始時刻

// ★ timerPeriodMs は AudioEngine のメンバ変数（startTimer() で設定された値を保持）
//   startTimer(100) で設定されるためデフォルトは 100ms。
//   将来 startTimerHz() などで周期が変更されても追従可能。
const double expectedMs = static_cast<double>(timerPeriodMs_);
double timerIntervalMs = expectedMs;
double timerJitterMs = 0.0;

if (s_prevCallbackMs > 0.0) {
    timerIntervalMs = nowMs - s_prevCallbackMs;
    timerJitterMs = timerIntervalMs - expectedMs;

    // 相対閾値（周期適応型）: max(10ms, expectedMs * 0.1)
    // 固定10msでは将来の周期変更(例:200ms→20ms)で不適切になるため、
    // expectedMsに比例した閾値を採用。
    // 100ms周期なら10ms、50ms周期なら10ms、500ms周期なら50ms。
    const double jitterThreshold = std::max(10.0, expectedMs * 0.1);
    if (std::abs(timerJitterMs) > jitterThreshold) {
        // ★ Timer callback自身の実行時間は callback末尾で計測
        //    intervalが大きいがexecが小さい→OS Scheduler停止
        //    intervalもexecも大きい→Timer自身が詰まっている
        //    ★ exec は timerCallback() 終了直前に計測（このifブロック内ではない）
        const uint64_t gen = (runtimeWorld != nullptr)
            ? static_cast<uint64_t>(runtimeWorld->generation) : 0;
        const auto seq = convo::fetchAddAtomic(diagSequenceCounter(), uint64_t{1}, std::memory_order_acq_rel) + 1u;
        diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
            + "] [TIMER] jitter: interval=" + juce::String(timerIntervalMs, 2) + "ms"
            + " (expected=" + juce::String(expectedMs, 0) + "ms, delta=" + juce::String(timerJitterMs, 2) + "ms");
    }
}
s_prevCallbackMs = nowMs;

// ... 全ての処理 ...

// ★ timerCallback 末尾 — 実行時間計測
//    intervalが大きいがexecが小さい→OS Scheduler停止
//    intervalもexecも大きい→Timer自身が詰まっている
{
    const double execMs = juce::Time::getMillisecondCounterHiRes() - timerExecStartMs;
    if (execMs > 10.0) {  // 10ms以上の場合のみ記録（通常は0.3ms未満）
        const uint64_t gen = (runtimeWorld != nullptr)
            ? static_cast<uint64_t>(runtimeWorld->generation) : 0;
        const auto seq = convo::fetchAddAtomic(diagSequenceCounter(), uint64_t{1}, std::memory_order_acq_rel) + 1u;
        diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
            + "] [TIMER] exec=" + juce::String(execMs, 3) + "ms");
    }
}
```

#### 出力例

```
[19:57:51.336] Gen=5 Ticks=5562739738 [TIMER] Seq=129 jitter: interval=108.43ms (expected=100ms, delta=+8.43ms)
[19:57:51.336] Gen=5 Ticks=5562739738 [TIMER] Seq=130 exec=0.324ms
```

`interval=108ms, exec=0.32ms` → Timer自身は軽量だがOS Schedulerが止まった。`interval=150ms, exec=45ms` → Timer自身が詰まっている。execは10ms以上のときのみ出力。

### 優先度5: フェーズ別メモリ

**目的**: メモリ急増が build/rebuildIR/NUC初期化 のどのフェーズで発生するかを特定する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**出力先**: `RebuildDispatch.cpp` rebuildThreadLoop 内

**抑制**: ビルド1回につき最大3行（build後/rebuildIR後/commit前）。通常は数分に1回程度。

**前提**: `RebuildDispatch.cpp` は既に `#include "DiagnosticsConfig.h"` 済み。共通基盤の `getProcessMemoryInfo()` をそのまま利用する（移植不要）。

```cpp

//    独自の移植は一切不要 — 全ファイルが同一の定義を共有する

// ... 中間処理 ...

// ... commit直前 ...
const auto memBeforeCommit = getProcessMemoryInfo();
```

```cpp
// rebuildIR 完了後
```

### 優先度6: クロスフェード開始＋終了

**目的**: DSPCore切り替え時のクロスフェードが計画通り開始・完了したかを確認する。開始と終了の両方を記録することで、途中で停止した場合も検出可能。

#### 設計

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**出力先**: `Timer.cpp` fade開始ブロック + fade完了ブロック

**抑制**: クロスフェードの開始時と完了時のみ（数分に0〜数回）。

```cpp
// ★ fade 開始時（crossfade 発行ブロック）
//    発行元の generation（old world）と発行先の generation（new world）を両方記録
//    例: "4→5" で old gen=4 から新 gen=5 への遷移が分かる
const double expectedSec = crossfadeRuntime_.getQueuedFadeTimeSec();
const auto seq = convo::fetchAddAtomic(diagSequenceCounter(), uint64_t{1}, std::memory_order_acq_rel) + 1u;
diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
    + "] [XFADE] start " + juce::String(static_cast<juce::int64>(currentGen))
    + "->" + juce::String(static_cast<juce::int64>(targetGen))
    + " expected=" + juce::String(expectedSec, 3) + "s");

// ★ fade 完了時（既存の fade 完了ブロック内）
const double elapsedSec = static_cast<double>(crossfadeRuntime_.getFadeAgeUs()) / 1'000'000.0;
diagLog(diagPrefix(gen) + " [Seq=" + juce::String(static_cast<juce::int64>(seq))
    + "] [XFADE] completed: " + juce::String(static_cast<juce::int64>(currentGen))
    + "->" + juce::String(static_cast<juce::int64>(targetGen))
    + " elapsed=" + juce::String(elapsedSec, 3) + "s"
    + " expected=" + juce::String(expectedSec, 3) + "s");

// ★ fade cancelled（中断）— 将来の使用に備えて枠を定義
//    例: 新しいpublishが入り、現在のcrossfadeが中断された場合
//    [Seq=130] [XFADE] cancelled: 4->5 elapsed=0.023s expected=0.050s
//    現時点では実装不要。枠のみ定義。
```

#### 出力例

### 優先度7: RetireQueue（変化時のみ、現状維持）

**検討結果**: 1秒ごとの定期出力には変更しない。「変化時のみ」の現状設計を維持する。理由は以下の通り。

- RetireQueueが変化しない期間（例: 0→0→0→0）は解析価値がない
- 変化した瞬間（0→1→2→5→0）だけ記録すれば滞留の発生と解消が追える
- ログ量を削減し、重要なイベント（XRUN/MEM/BUILD_PHASE）を埋もれさせない

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**変更不要**: 現状の「変化時のみ」出力を維持。

## 除外した設計

以下の設計はレビューにより「望ましくない」と判断されたため、本計画書から除外した。

| 除外した設計 | 理由 |
|------------|------|
| TimerCallback 毎tick実行時間ログ | 10行/秒の出力で重要イベントが埋もれる。閾値超過時のみ + XRUN付加情報として代替 |
| Core Affinity 重複検出 | 実装コストに対し情報価値が低い。今回の仮説検証に不要 |
| 新マクロの追加（CONVOPEQ_DIAG_VERBOSE 等） | 既存の `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` で一括制御する方針に統一 |
| RetireQueue 定期ログ（1秒毎） | 変化のない期間は解析価値がなく、ログが埋もれる。「変化時のみ」を維持 |
| getProcessMemoryInfo の重複実装 | `DiagnosticsConfig.h` の共通定義で一元化。コピー禁止 |

## 実装推奨順序

```
Step 0: DiagnosticsConfig.h 共通基盤追加     ← 全項目の前提
Step 1: Audio callback 1秒サマリ（変化時のみ出力）← 音飛び検証に必須の決定的証拠
Step 2: PageFault Delta + 2条件警告（判定→EWMA更新順）← 異常の即時検出
Step 3: Obsolete 浪費検出（taskGen/currentGen）← gen=7の無駄を確定
Step 4: Timer jitter + 実行時間(相対閾値)  ← OS停止＋Timer自身の詰まり判別
Step 5: フェーズ別メモリ（Delta付き）        ← メモリ急増フェーズの特定
Step 6: クロスフェード開始＋終了（generation）← DSP切替の確認
```

> **注**: XRUN検出・MEMログ・WORLD・BACKPRESSURE・BUILD_PHASE は既に実装済み。今回の追加項目はこれらを補完するものであり、まずは既存の診断ログが次回の音飛び事象で正しく出力されることを確認することが最優先。
