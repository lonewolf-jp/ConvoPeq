## 1. メモリ3GBの内訳（確定度別分類）

### 1.1 確実に言えること（ソースコードから直接確認可能）

| 項目 | 内訳 | 確からしさ |
|------|------|-----------|
| MKLNonUniformConvolver 1組 (stereo) | ~80 MB | バッファ数×サイズの積み上げ（Layer/L0+L1+L2の構成に依存） |
| UI + RT の2組は最低限存在 | ~160 MB | ソース確認済み（別インスタンス） |
| IR Cache (max 8 entries) | ~24 MB | `MAX_CACHE_ENTRIES=8`、各~3MB |
| ConvolverState（軽量化版） | negligible | バッファ無しメタデータのみ |
| MixedPhasePersistentCache | 0 MB (in-memory) | ディスク永続化のみ |
| FFT Plan Cache | ~数MB | `IppFFTPlan`数個（order→fftSpec） |

### 1.2 ここまでは推定が入る

| 項目 | 推定 | 確からしさ |
|------|------|-----------|
| **Retire滞留数** | **不確定** | **kMaxReclaimPerLoop=4だが、実際の滞留数は未計測** |
| **DSPCore 1組のメモリ** | ~132 MB | aligned buffers + EQ + NS + その他、実測が必要 |
| **Working Set拡大** | 不確定 | OS管理領域のため直接コードから推定不可 |
| **MKL内部領域** | 不確定 | スレッドローカル・ワークバッファ等 |
| **古いRuntimeの滞留** | **大量に滞留している可能性** | **RetireQueueDepth未計測のため不明** |

### 1.3 3GBという数字に対する評価

| シナリオ | 推定値 | 判定 |
|---------|-------|------|
| 最小構成（UI+Active+Fading） | ~400 MB | ❌ 3GB説明不可 |
| 中程度（+Retire4世代滞留） | ~960 MB | ❌ 3GB説明不可 |
| **Retire大量滞留 + Working Set拡大 + 実測較正後** | **~2-3 GB** | **可能性はあるが現時点では推測の域** |

**結論**: 3GBの内訳をソースコードのみから正確に説明することは不可能。実測（Private Bytes + Working Set + RetireQueueDepth）との組み合わせが必須。

## 2. 確定した新たな事実

### 2.1 getRuntimeBackpressureTelemetry() が未使用

```cpp
// AudioEngine.h L1201-L1214
[[nodiscard]] RuntimeBackpressureTelemetry getRuntimeBackpressureTelemetry() const noexcept
{
    return {
        consumeAtomic(retireQueueDepth_, std::memory_order_acquire),
        consumeAtomic(fallbackQueueDepth_, std::memory_order_acquire),
        // ... (quarantineResident, rebuildBacklog, saturationEnterCount,
        //      saturationExitCount, publicationRejectCount, rebuildCollapseCount,
        //      retirePressureLevel, retireEscalationCount,
        //      maxRetireDeferralEpochs, maxRetireWallClockMs まで)
    };
}
```

- `getRuntimeBackpressureTelemetry()` は完全に**未使用のdead code**
- `retireQueueDepth_` / `fallbackQueueDepth_` の値は内部的に更新されているが、**どこにもログ出力されていない**
- `maxPendingRetireObserved_` （最大Retire滞留数のHigh Watermark）も未出力

**【影響】**: ログの `[VERIFY] tx counters` に retireQueueDepth が含まれていないため、メモリ使用量とRuntime滞留の相関が全く見えない。

### 2.2 現状のVERIFYログに不足している情報

```
現在の [VERIFY] tx counters:
  lifecycle(pub/ret/reclaim)          ← ◯ ある
  rebuild(req/queued/...)             ← ◯ ある
  eqCacheMiss(create/lookup)          ← ◯ ある
  convDebounce(req/defer/sched/trigger) ← ◯ ある
  shutdownPhase                       ← ◯ ある

不足:
  retireQueueDepth / fallbackQueueDepth  ← ✗ なし
  quarantineResident                    ← ✗ なし
  maxRetireDeferralEpochs              ← ✗ なし
  maxRetireWallClockMs                 ← ✗ なし
  PrivateBytes / WorkingSet            ← ✗ なし（API未統合）
```

### 2.3 RCU Reader エポック機構

- EpochDomain が `getMinReaderEpoch()` を提供
- DeferredFreeThread はこれを呼び出し、最小エポックより古いエントリのみ解放
- Audio Thread がエポックを進めていない場合（長時間のcallbackなど）、Retireが保留される
- quarantine機構: Readerが異常終了した場合でも、quarantineフラグによりsafe-epoch計算から除外される

### 2.4 GetProcessMemoryInfo は未統合

- Windows `GetProcessMemoryInfo()` / `PROCESS_MEMORY_COUNTERS_EX` は使用されていない
- Private Bytes、Working Set、PageFault Count のいずれも出力なし
- 新規追加が必要なコード量: 約20行（ラッパー関数 + timerCallback への組み込み）

## 3. 推定メモリモデルの検証可能性

### 3.1 検証に必要な最小限の計測

以下の2つが取得できれば、原因をほぼ確定できる:

```
# 計測A: RetireQueueDepth の時系列
t=0: retireDepth=0  private=900MB  ← Runtime非滞留時
t=1: retireDepth=5  private=2.8GB ← Runtime滞留時（高相関なら原因確定）

# 計測B: 上記 + publish世代番号
publish gen=4 → retireDepth=1 private=1.2GB
publish gen=5 → retireDepth=2 private=1.5GB
publish gen=8 → retireDepth=4 private=2.8GB
...
```

### 3.2 Private BytesとWorking Setの違い

| 指標 | 意味 | 3GBの解釈 |
|------|------|-----------|
| Private Bytes = 3GB | プロセスが確保した仮想メモリ（コミット済み） | **重大**: 他プロセスに影響する |
| Working Set = 3GB | 物理メモリに常駐しているページ | **大きい**: 他プロセスのWSを圧迫 |
| Commit Size = 3GB | ページファイル込みのコミット総量 | システム全体の制限に関わる |

**現時点では Private Bytes / Working Set のどちらも不明。**

## 4. 音飛びとの関係（可能性順位・再評価）

### 4.1 現時点での順位

| 順位 | 候補 | 可能性 | 既存ログでの確認可否 |
|------|------|--------|-------------------|
| **①** | **Retire滞留によるRuntime多重保持** | **★★★★★** | **要追加計測（retireQueueDepth）** |
| ② | OSのWorking Set Trim → ページフォールト | ★★★★☆ | 要追加計測（PageFaultCount） |
| ③ | StereoConvolverが想定以上に巨大 | ★★★★☆ | ソースから計算可能 |
| **④** | **RCU Readerエポック停滞によるRetire保留** | **★★★★☆** | **要追加計測（maxRetireDeferralEpochs）** |
| ⑤ | UI側とRT側の二重保持 | ★★★★☆ | 構造的に最低2組は確定 |
| ⑥ | Memory Compression | ★★☆☆☆ | Win11では最終手段。82%ではまだ先 |
| ⑦ | 本当のメモリリーク | ★☆☆☆☆ | シャットダウン時に正常drain確認済み |

### 4.2 実運用中のメモリ推移（ログから読み取れる範囲）

ログ期間中のpublish回数:
```
rev=4: gen=4 publish (DSPCore #3完了時)
rev=5: gen=8 publish (DSPCore #6完了時)
```

- 最低2回のpublishが発生
- その間にgen=1〜8の8世代が構築された
- 各世代が完全に解放される保証はない（RCU epoch依存）

## 5. メモリリークの可能性（再評価）

| チェック項目 | 結果 | 備考 |
|-----------|------|------|
| mkl_malloc / mkl_free 対称性 | ✅ | releaseAllLayersでカバー |
| ippsMalloc / ippsFree 対称性 | ✅ | FFT Plan含め解放確認 |
| DeferredFreeThread の動作 | ⚠️ 条件付き | Reclaimはepoch依存。epochが進まないと滞留 |
| シャットダウン drain | ✅ OK | `pendingRetire=0` を確認 |
| **RCU Readerエポック進捗** | **⚠️ 未確認** | **Readerがepochを進めていない場合、Retireが永久滞留する可能性** |

**結論**: 「リーク」より「RCU epoch起因の滞留」が原因である可能性が高い。

### P0: ログ出力の追加（既存インフラ利用、工数小）

### P1: Windows API 統合（新規実装、工数中）

| 追加対象 | 実装難易度 | 必要コード量 |
|---------|-----------|------------|
| Private Bytes 定期ログ | 中 | ~20行（ラッパー関数 + timer呼び出し） |
| Working Set 定期ログ | 中 | 同上（`GetProcessMemoryInfo` で同時取得） |
| Page Fault カウント | 中 | 同上（`PROCESS_MEMORY_COUNTERS.PageFaultCount`） |

```cpp
// 実装イメージ:
#include <windows.h>
#include <psapi.h>

static void logProcessMemoryInfo()
{
    PROCESS_MEMORY_COUNTERS_EX pmc;
    pmc.cb = sizeof(pmc);
    if (GetProcessMemoryInfo(GetCurrentProcess(),
            (PROCESS_MEMORY_COUNTERS*)&pmc, sizeof(pmc)))
    {
        diagLog("[MEM] PrivateUsage=" + String(pmc.PrivateUsage / 1024 / 1024) + "MB"
            + " WorkingSet=" + String(pmc.WorkingSetSize / 1024 / 1024) + "MB"
            + " PageFaults=" + String(pmc.PageFaultCount));
    }
}
```

## 7. 追加計測による切り分けフローチャート

```
[MEM] PrivateBytes / WorkingSet 追加
  │
  ├─ PrivateBytes < 2GB → 3GBは誤認。タスクマネージャの見方を確認
  │
  ├─ PrivateBytes ≈ 3GB:
  │   ├─ retireQueueDepth 追加
  │   │   ├─ retireDepth > 0:
  │   │   │   └─ PrivateBytes と retireDepth に相関あり？
  │   │   │       ├─ YES → 原因特定：Retire滞留
  │   │   │       └─ NO  → 他の要因（Convolver/FFTバッファ等）
  │   │   └─ retireDepth = 0:
  │   │       └─ Runtime滞留ではない → Convolver/FFT構造の詳細調査
  │   └─ PageFaultCount 追加
  │       └─ PageFaults > 100/秒 かつ 音飛び時刻と相関？
  │           ├─ YES → Working Set Trim / Memory Compression 原因
  │           └─ NO  → メモリ以外の原因を優先
  │
  └─ PrivateBytes ≈ 3GB だが WorkingSet < 1GB:
      └─ 仮想メモリのコミットは多いが物理メモリ圧迫は軽度
          └→ 音飛びはメモリ以外の原因の可能性が高い
```
