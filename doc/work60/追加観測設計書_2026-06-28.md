#### callbackMax測定範囲の補足

`cbStartTicks` は `getNextAudioBlock()` / `processBlockDouble()` の末尾にある診断ブロック内で設定される。つまり測定範囲は診断ブロックの実行時間のみであり、DSPCore::process() の実処理時間は含まない。このため callbackMax=0.001ms という値が出力されている。

```
processBlock開始
  ├─ DSPCore::process()        ← 未計測
  ├─ その他処理                 ← 未計測
  └─ #if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
       ├─ t0_start (cbStartTicks) ← ★ここから計測開始
       ├─ XRUN検出
       ├─ publishAtomic
       ├─ ACTIVATE検出
       ├─ nowTicks               ← ★ここで終了
       └─ callbackMax = nowTicks - cbStartTicks
```

**設計意図**: DSP処理時間そのものは音飛びの直接原因ではない（callbackMaxが小さくても音飛びは発生している）ため、現状の診断優先度としては intervalMax のみで十分である。

ただし、将来的に以下のケースを切り分ける必要が生じた場合は、`cbStartTicks` を関数先頭（DSPCore::process() の直前）に移動することで callbackMax の測定範囲を拡張できる:
- IR切替時のFFTプラン更新がDSP時間に与える影響
- SIMD初期化やCPUキャッシュミスによるDSP時間の一時的増大
- コンボルバの内部ブロックサイズ変更に伴う処理時間変動

この改造は数行の変更で完了するため、必要に応じて実装する。

### 🔶 仮説の更新

**2026-06-29 重要**: CBSUMMARY再計測により `intervalMax=7.8〜12.0ms` が確定した。これは「Audio callbackが550ms停止した」という第1次仮説を**否定する**結果である。

従来:
```
PageFault急増 → OS Scheduler停止 → Audio callback停止(550ms) → 音飛び
```

更新後:
```
PageFault急増 → Timer停止(557ms) ＋ Worker長期占有(497ms)
                      ↓
              Publish遅延 → 切替タイミング逸脱 → 音飛び
```

または:
```
PageFault急増 → Commit Charge増大 → Publish遅延 → 音飛び
```

つまり音飛びの原因は **Audio callbackの停止ではなく、RuntimeWorldの公開・切替・適用のパイプライン遅延** にあることが現時点で最も整合する仮説である。Timer jitter 557ms はAudio ThreadではなくTimer Thread自身の停止を示しており、Worker Threadの497ms占有と合わせて「OS Schedulerが全体として不安定」であることを示すが、Audio callback自体は継続して実行されていた。

したがって「PageFault急増がAudio callbackを止めた」ではなく「PageFault急増がPublish〜切替のパイプライン全体を遅延させ、その結果として音飛びが発生した」という仮説が現時点で最も整合する。ただし、Publish遅延そのものはまだ直接測定しておらず、あくまで間接証拠に基づく推論である。

#### 調査詳細

**A. enqueueTimestampUs**:
- 定義: `RuntimePublicationOrchestrator.h` の `DeferredPublishSlot` 構造体（`uint64_t enqueueTimestampUs{0};`）
- 設定箇所: `RuntimePublicationOrchestrator.cpp:273` — 新しいdeferral発生時に `steady_clock::now()` のカウント値を格納
- 利用箇所: 同ファイル255行で滞留時間計算、322行でTTL超過チェック

**B. revision**:
- `RuntimeTransition.h:64` に `TransitionState::revision` フィールドが存在
- VERIFYログでは `publishedWorld->generation` を実質的なrevisionとして使用（`AudioEngine.h:2968`）
- 同一 `generation` 内の再publishは `worldId`（`RuntimeState::worldId`）で区別可能

**C. publishStart/publishEnd**:
- `PublicationExecutor::publish()`（`PublicationExecutor.cpp:7-38`）は `FrozenRuntimeWorld` を受け取り、`releaseState()` → `coordinator.publishWorld()` を同期的に実行
- 関数先頭と末尾（`return`直前）に `juce::Time::getHighResolutionTicks()` を追加することで実現可能
- `coordinator.publishWorld()` に渡した後の最適化(validation, swap, retire)が全区間の内訳となる

**D. 時刻系の統一（`enqueueTimestampUs`）**:
- `enqueueTimestampUs` は `steady_clock::now().time_since_epoch().count()`（ナノ秒カウント）
- 既存の診断ログは `getHighResolutionTicks()`（QueryPerformanceCounter）を使用
- **変換は行わず全て `getCurrentTimeUs()` に統一する**。publish側で `waitUs = convo::getCurrentTimeUs() - enqueueUs` と計算し、Δ値を `[PUBLISH]` ログに出力する。`enqueueTimestampUs` そのものは後まで保持しない。

### 優先度0: CBSUMMARY修正バイナリでの再計測（最優先）

**ステータス**: ✅ 実装完了・動作確認済み

#### 修正内容

CBSUMMARYが全く出力されていなかった原因は、`AudioEngineProcessor` が `supportsDoublePrecisionProcessing() = true` であるためDAWがダブル精度パス（`processBlockDouble` in `BlockDouble.cpp`）を使用していたことによる。`BlockDouble.cpp` には診断コードが一切存在せず、`AudioBlock.cpp`（float精度パス）のみに記述されていた。

修正として `BlockDouble.cpp` に以下を追加:
- `#include "DiagnosticsConfig.h"`
- CBSUMMARY/XRUN/ACTIVATE 診断ブロック全体（`AudioBlock.cpp` から移植）

また、両ファイル共通で `intervalMax` ガベージ値バグ（XRUNブロックが `lastCallbackEndTicks` を上書きした後、CBSUMMARYブロックがそれを読み取っていたため `cbStartTicks - prevEnd` が負の値になりuint64_tラップで `2576980.480ms` と表示される問題）を修正:
- XRUNブロックの前に `cbPrevEndTicks` に `lastCallbackEndTicks` を保存
- XRUN・ACTIVATE・CBSUMMARYの各ブロックで `cbPrevEndTicks` を使用

**変更ファイル**:
| ファイル | 変更内容 |
|---------|---------|
| `AudioEngine.Processing.BlockDouble.cpp` | 診断ブロック追加 + cbPrevEndTicks修正 |
| `AudioEngine.Processing.AudioBlock.cpp` | cbPrevEndTicks修正（同一バグ） |

**アクション**: 修正版バイナリで再計測済み

**確認結果**:
```
[CBSUMMARY] intervalMax=12.014ms callbackMax=0.001ms (expected=187 actual=108 loss=79)
[CBSUMMARY] intervalMax=9.221ms callbackMax=0.017ms (expected=187 actual=196 loss=-9)
[CBSUMMARY] intervalMax=8.354ms callbackMax=0.006ms (expected=187 actual=192 loss=-5)
```

`intervalMax` は 7.8〜12.0ms の正常範囲に回復。43分のガベージ値は解消。

**重要な帰結**: intervalMax が 7.8〜12ms であることは、**音飛び発生時にAudio callbackが数100ms単位で停止したわけではない**ことを示す。従来の「PageFault急増→OS Scheduler停止→Audio callback停止」という因果仮説は、CBSUMMARYの実測によって否定された。音飛びの原因はAudio callback以外の経路（Publish〜切替パイプライン）にある可能性が高い。

### 優先度1: PageFault EWMA 閾値改善

**ステータス**: ✅ 実装済み

**問題**: EWMAがゼロ付近に減衰すると `delta > EWMA×5` が常に成立し偽陽性が多発（14件/16件）

**修正内容**（`Timer.cpp` MEMブロック）:

従来:
```cpp
const double thresholdRel = (s_pfSampleCount >= kEwmaWarmupSamples)
    ? (s_pfEwmaAvg * 5.0) : 0.0;
```

修正後:
```cpp
const double thresholdRel = (s_pfSampleCount >= kEwmaWarmupSamples)
    ? std::max(1000.0, s_pfEwmaAvg * 5.0) : 0.0;
```

**変更点**: 相対閾値の下限を 1000 に設定（設計当初の 500 から実測レビューで上方修正）。
**効果**: EWMA=200 でも `threshold_rel = max(1000, 1000) = 1000`。Delta=600程度のWindows通常PageFaultでは発火しない。真の異常（Delta > 50000 の絶対条件、または Delta > 1000 かつ EWMA比5倍超過）のみ警告。

### 優先度2: WorkingSet 急減検出（新規）

**ステータス**: ✅ 実装済み

**目的**: OSがWorkingSetをトリミングしたタイミングを検出する。WSが突然減少した場合、OSがメモリ圧力を検知してページを解放した証拠になり、PageFault再発生の引き金になる。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**出力先**: MEMブロック内（1秒ログのEWMA更新後）

**設計**: 割合条件と絶対量条件のAND併用

```cpp
// 前回のWS(MB)と比較して、5%以上 かつ 50MB以上の減少を検出
static uint64_t s_prevWsMB = 0;
if (s_prevWsMB > 0 && memInfo.workingSetMB < s_prevWsMB) {
    const uint64_t wsDropMB = s_prevWsMB - memInfo.workingSetMB;
    const double wsDropPct = static_cast<double>(wsDropMB) / static_cast<double>(s_prevWsMB) * 100.0;
    if (wsDropMB >= 50 && wsDropPct >= 5.0) {  // AND条件
        diagLog(... "[WARN] WS dropped: " + wsDropMB + "MB (" + wsDropPct + "%) from XMB to YMB");
    }
}
s_prevWsMB = memInfo.workingSetMB;
```

**出力例**:
```
[WARN] WS dropped: 120MB (8.8%) from 1366MB to 1246MB
```

**抑制**: 5%以上 AND 50MB以上の両方満たす場合のみ出力（通常時0行）。

### 優先度4（追加）: Timer jitter 閾値調整

**ステータス**: ✅ 実装済み

**問題**: WindowsのJUCE Timerは通常動作でも ±10ms 程度のジッタが発生する。閾値10msでは通常動作でも頻繁に警告が出る可能性がある。

**修正**:
```cpp
// 従来:

```

**効果**: 通常のWindowsタイマーばらつき（±10ms程度）では警告が出ず、真の異常（557ms等）のみ検出。

### 優先度5（追加）: MEMログへのPagefileUsage追加

**ステータス**: ✅ 実装済み

**目的**: 現状 `PrivateUsage` のみだが、`PagefileUsage`（現在のコミットチャージ = Commit Charge）も併記することで以下の区別が可能になる:
- PrivateUsage + PagefileUsage が増加 → コミット済みメモリ（VirtualAlloc等）の増加
- WorkingSetだけが減少 → OSによるワーキングセットのトリミング（物理ページ解放）
- なお、`PagefileUsage` そのものは実際にページファイルへ書き出された量ではなく、OSがプロセスに割り当てたコミット領域の総量を示す。ページアウトの有無はこの指標だけでは断定できない。

**変更**:

`DiagnosticsConfig.h`:
```cpp
struct ProcessMemoryInfo {
    uint64_t privateUsageMB = 0;
    uint64_t workingSetMB = 0;
    uint64_t pagefileUsageMB = 0;  // ★追加
    uint64_t pageFaultCount = 0;
};
```

`Timer.cpp`:
```cpp
// MEMログに Pagefile を追加
" [MEM] Private=... WS=... Pagefile=" + pagefileUsageMB + "MB PageFaults=... Delta=..."
```

**出力例**:
```
[MEM] Private=1361MB WS=1366MB Pagefile=1400MB PageFaults=356216 Delta=+67
```

## 次期観測設計の提案（改訂版） — Publishパイプライン遅延計測（優先度: 高）

CBSUMMARYの実測により「Audio callbackは止まっていない」ことが判明した。ただし「callbackが正常」と「DSPが古いRuntimeWorldを使い続けた」は別問題である。observePublishedWorld()が更新されなければ古いDSPを延々使うため、調査対象を**RuntimeWorldの公開・切替・適用パイプライン**に移すべきである。

### 既存ログから読み取れるパイプライン

```
Worker完了 → rebuildThreadLoop
  ↓
enqueue (DeferredPublishSlot)
  ↓
Admission (Accepted / Deferred / Rejected)
  ↓
publishStart
  ├─ Validation (FrozenRuntimeWorldビルド)
  ├─ Coordinator.publishWorld
  ├─ RuntimeStore::publishAndSwap
  └─ publishEnd
  ↓
DSP readAudioRuntimeView → consumeWorldHandle → observe new world
  ↓
ACTIVATE (audio thread detects new generation)
  ↓
Audio callback が new world を使用開始
```

### 現状の課題

1. publishに500ms掛かったのか、DSP observeが500ms遅れたのかが全く区別できない
2. TimerによるobserveはUI反映用であり、DSPのobserveとは一致しない
3. 時刻系が混在（`steady_clock` / `QueryPerformanceCounter` / raw count）している

### 調査で確定した実装前提

ソースコード調査により以下の既存機構を確認:

### 最終計測案 — シンプル版（推奨）

#### 設計方針

1. **Telemetry Recorder 拡張方式**: PublicationContextを作って運搬せず、既存 `recordProgress()` のイベント列を拡張する。
2. **DSP observe をTimer observeより優先**: DSPの `readAudioRuntimeView()` がいつ新Worldを観測したかを主計測点とする。
3. **既存TelemetryにCIDを統合**: `[REBUILD_TELEMETRY]` に `cid=` を追加する。新規ログ形式は追加しない。
4. **publish は開始〜終了のみ計測**: Validation内訳等の細分化は、原因がPublish側にあると判明してから追加すれば十分。
5. **新規atomicは追加しない**: `publicationSequence` はDSPが観測した `runtimeWorld->metadata.publicationSequence` から既に取得可能。
6. **時刻系を統一**: 全時刻を `getCurrentTimeUs()` に統一。

#### 計測点

| # | 計測点 | 優先度 | 方法 |
|---|--------|--------|------|
| 1 | **waitUs**（旧enqueueUs） | ★★★★★ | publish開始時に `waitUs = publishStartUs - enqueueUs` を計算して出力。enqueue自体は後まで保持しない。`enqueueTimestampUs` の時刻系を `getCurrentTimeUs()` に統一。 |
| 2 | **publishStartUs** | ★★★★★ | `PublicationExecutor::publish()` 先頭で `getCurrentTimeUs()` |
| 3 | **publishEndUs** | ★★★★★ | `publish()` 末尾で `getCurrentTimeUs()`。同時に `PublicationSequence` 取得 |
| 4 | **DSP observeUs** | ★★★★★ | Audio callback先頭で `dspSeq != lastObservedSeq` の時のみ `getCurrentTimeUs()` |
| 5 | **DSP activateUs** | ★★★★★ | ACTIVATE検出時に `getCurrentTimeUs()` |

**publishStart → publishEnd** だけで十分。Validator/Retireへの細分化はPublishに原因があると確定してから追加する。

#### 主要指標: `pubToObserveUs` + `callbacksUntilObserve`

**`observeLag=Sequence差` は採用しない。** Sequenceはrollback/republish/retry/deferredでも増加するため、callback遅延量と一致しない。

代わりに以下の2つを主要指標とする:

| 指標 | 意味 | 算出方法 |
|------|------|---------|
| **`pubToObserveUs`** | publish完了からDSP observeまでの時間(μs) | `dspObserveUs - publishEndUs`（同一sequenceの対応を維持） |
| **`callbacksUntilObserve`** | publish完了後、DSPが新Worldを観測するまでのcallback数 | DSP側でpublishEnd後に観測したworldの `publicationSequence` が変化するまでのcallback回数 |

#### PublishTiming のRT-safe共有 — Fixed Publish Timing History（16エントリ）

**これはSPSC Ringではない。** DSPはconsume（pop）せず、固定長History Tableをlookup（search）する。

DSP側（RT reader）は `publishTimingHistory` を固定長の履歴テーブルとして扱い、`publicationSequence` が一致するエントリを検索する。RT側がatomicを書き換える必要は一切ない。名前は動作を正確に反映するため **`kPublishTimingSlots`** とする。

```cpp
struct PublishTimingEntry {
    uint64_t sequence;
    uint64_t publishStartUs;
    uint64_t publishEndUs;
    uint64_t gen;
    uint64_t worldId;
    uint64_t publishCallbackIndex;  // publish発行時のcallbackIndex
};

struct RTLocalState {
    // ... existing fields ...
    // Fixed publish timing history (writer-overwrite, reader-lookup):
    // NonRT(writer) → RT(reader) — NOT a ring buffer, no pop/consume
    std::atomic<uint64_t> publishTimingWriteCount { 0 };
    static constexpr size_t kPublishTimingSlots = 16;
    PublishTimingEntry publishTimingHistory[kPublishTimingSlots];
};
```

**動作原理**:
- `writeCount` のみ存在。NonRT writerが increment する。
- RT reader は `writeCount` を read-only で参照し、slot計算 → 一致検索を行う。
- 固定長テーブルは循環上書きされる。16エントリ（数百Byte）で安全マージン十分。
- 書き込み完了の可視性は **Release/Acquireペア** によって保証される（後述）。

**publish側 (NonRT writer, `PublicationExecutor::publish()`)**
```cpp
const auto seq = engine.getLastCommittedPublicationSequence();
const auto* world = /* 発行したRuntimeState */;
const uint64_t nowUs = convo::getCurrentTimeUs();
const uint64_t wc = convo::fetchAddAtomic(engine.rtLocalState_.publishTimingWriteCount,
    uint64_t{1}, std::memory_order_acq_rel);
const uint64_t slot = wc % RTLocalState::kPublishTimingSlots;

// ① 先にエントリを完全に書き込む（writerからはまだ見えない）
engine.rtLocalState_.publishTimingHistory[slot] = {
    static_cast<uint64_t>(seq), startUs, nowUs,
    world->generation, world->worldId, publishCallbackIndex
};
// ② release store(writeCount)で公開 — これより前の全storeがreaderから可視に
// writeCountの発行をもって「エントリ完成」を宣言
```

**Release/Acquire ペアによる可視性保証（重要）**:
```
 Writer (NonRT)                     Reader (RT)
 ─────────────                     ──────────
 publishTimingHistory[slot] = ...  (データ準備)
 std::atomic_thread_fence(seq_cst) または
 writeCount.fetch_add(1, release) ─→ writeCount.load(acquire)
                                      publishTimingHistory[slot] を読む
```

- `fetch_add(..., release)` より前の全メモリ書き込みは、reader側の `load(acquire)` 完了後に可視になる（Inter-Thread Happens-Before）。
- 安全なのは「atomic不要だから」ではなく**Release/Acquireの順序保証があるから**。
- この保証なしでは、readerが書き込み途中のpartial stateを読む可能性がある（ティアリング）。

**DSP側 (RT reader, Audio callback先頭)**
```cpp
const uint64_t dspSeq = (runtimeWorld != nullptr)
    ? runtimeWorld->metadata.publicationSequence : 0;

// acquire load(writeCount) — publish側のrelease storeとの同期点
const uint64_t wc = convo::consumeAtomic(rtLocalState_.publishTimingWriteCount,
    std::memory_order_acquire);

// lookup方式: writeCount-1 から順に遡ってsequence一致検索
const uint64_t startSlot = (wc > 0) ? (wc - 1) % RTLocalState::kPublishTimingSlots : 0;
constexpr size_t kMaxLookback = RTLocalState::kPublishTimingSlots;
PublishTimingEntry matched = {};
for (size_t i = 0; i < kMaxLookback; ++i) {
    const uint64_t idx = (startSlot >= i) ? (startSlot - i)
        : (RTLocalState::kPublishTimingSlots + startSlot - i);
    const auto& entry = rtLocalState_.publishTimingHistory[idx];
    if (entry.sequence == dspSeq && entry.sequence != 0) {
        matched = entry;
        break;  // 最新の一致を採用
    }
}
// matched: {.sequence, .publishStartUs, .publishEndUs, .gen, .worldId, .publishCallbackIndex}
// pubToObserveUs = observeUs - matched.publishEndUs
// gen/worldId は [PUBLISH]/[DSP_TIMING] クロス確認用
// publishCallbackIndex → callbacksUntilObserve = observeCallbackIndex - publishCallbackIndex
```

RT側（reader）は atomic を一切書き換えない。固定長History Tableとして read-only で参照するのみ。実装増加は約25行。

#### callbacksUntilObserve + observeLatencyUs（両方出力）

`callbacksUntilObserve` はブロックサイズ非依存だが、実際の経過時間はblockSize × sampleRateに依存する。両方を併記する。

**callbackIndexはcallback開始直後に1度だけ取得し、全診断ブロックで同じ値を使う**:
```cpp
// Audio callback開始直後（1度だけ）
const auto thisCallbackIndex = convo::consumeAtomic(
    rtLocalState_.audioCallbackEpochCounter, std::memory_order_relaxed);
// 以降のXRUN/ACTIVATE/CBSUMMARY/DSP_TIMINGは thisCallbackIndex を使う
```

**callbacksUntilObserve の計算には publishCallbackIndex も保存する**:
```cpp
// DSP_TIMING 出力（Observe検出時）
const auto* matchedEntry = /* lookup結果のPublishTimingEntry* */;
if (matchedEntry != nullptr && matchedEntry->publishCallbackIndex > 0) {
    const uint64_t callbacksUntilObserve =
        thisCallbackIndex - matchedEntry->publishCallbackIndex;
    // ... 出力に利用
}
```

**出力例**:
```
[DSP_TIMING] cid=84 seq=105 callbackIndex=28144
    publishCallbackIdx=28141
    callbacksUntilObserve=3 observeLatencyUs=15832
    observeToActivateUs=100
```

`publishCallbackIdx=28141, observeCallbackIdx=28144, callbacksUntilObserve=3` により、途中のCBSUMMARYとも照合可能。



#### DSP Observe の計測位置 — publicationSequence 変化時のみ記録 + ObserveReason

**毎callback計測する必要はない。** `readAudioRuntimeView()` が新しい `publicationSequence` を返した**瞬間のみ**Observe時刻を記録する。

また、Observe理由（Forward/Rollback/Replay）も併記する:

```cpp
// Audio callback先頭
auto runtimeReadHandle = readAudioRuntimeView();
const auto* runtimeWorld = getRuntimeWorldFromReadHandle(runtimeReadHandleRef);
const uint64_t dspSeq = (runtimeWorld != nullptr)
    ? runtimeWorld->metadata.publicationSequence : 0;

// 診断ブロック内: publicationSequence が変化した時のみObserve時刻を記録
static uint64_t s_lastObservedSeq = 0;
uint64_t observeUs = 0;
const char* observeReason = "";
if (dspSeq > 0 && dspSeq != s_lastObservedSeq) {
    if (dspSeq > s_lastObservedSeq) {
        observeReason = "Forward";   // 単調増加: 正常な前進
    } else if (dspSeq < s_lastObservedSeq) {
        observeReason = "Rollback";  // 減少: 後退（rollback/republish）
    } else {
        observeReason = "Replay";    // 等しくない→等しい（通常はここに来ない）
    }
    s_lastObservedSeq = dspSeq;
    observeUs = convo::getCurrentTimeUs();  // ★変化時のみ取得
}
// observeReason は [DSP_TIMING] ログに出力
```

これにより、不要な `getCurrentTimeUs()` 呼び出し（毎callback約5ms）を回避し、「何故sequenceが変わったか」も同時に追跡できる。

#### callbackIndex の出力 — callback開始直後に固定取得

`callbackIndex` はCBSUMMARYとの相関解析用。XRUN/ACTIVATE/DSP_TIMINGの全診断ログに追加する。

**重要: 各診断ブロックで個別に `consumeAtomic` するのではなく、callback開始直後に1度だけ取得し全ブロックで同じ値を使用する**:

```cpp
// Audio callback開始直後（1度だけ）
// memory_order_relaxed: 単なるcallback番号（カウンタ）の取得であり、
// 他のメモリ操作との順序保証は一切不要。
const auto thisCallbackIndex = convo::consumeAtomic(
    rtLocalState_.audioCallbackEpochCounter, std::memory_order_relaxed);
// 以降の全診断ブロックで thisCallbackIndex を使用
```

#### 新規atomic追加の整理

| 追加するatomic | 用途 | 必須か |
|--------------|------|--------|
| `publishTimingWriteCount` | Fixed History writer count | ✅ 必須（publish側からの新着通知） |
| `publishTimingHistory[16]` | 非atomic（writeCountのRelease/Acquire保護下） | ✅ 必須（sequence + publishEndUs等の共有） |

**追加しないもの**:
- ~~`publishTimingReadIndex`~~ — lookup方式では不要。RT側はread-onlyで参照する。
- `lastPublishedSequence` — `runtimeWorld->metadata.publicationSequence` から取得可能
- スタンドアロンの `publishEndUs` — Historyで管理

#### CID統合（既存Telemetry活用）

`CorrelationId` の型は `{uint64_t engineInstanceId, uint64_t localCounter}` であり、`shortValue()` は `localCounter`（単調増加64bitカウンタ）を返す。**16-bitではないため65535での衝突は発生しない**が、`shortValue()` という名称は誤解を招くため、設計書では `correlationId.localCounter` のように直接参照する。

**実装方法**: `emitRebuildTelemetry()` に `CorrelationId` パラメータを追加（デフォルト引数で空値）:

#### PUBLISHログ — sequence + generation + worldId のクロス確認

publish完了時に `[PUBLISH]` ログを出力し、DSP observe側でも同一情報を出力する。これにより「sequence一致だがworldId不一致」等の異常を検出できる。

**publish完了時（`PublicationExecutor::publish()` 末尾）**:
```cpp
const uint64_t seq = engine.getLastCommittedPublicationSequence();
const auto* world = /* 発行済みの RuntimeState* */;
DBG(juce::String("[PUBLISH] cid=") + juce::String(static_cast<juce::int64>(correlationId.localCounter))
    + " seq=" + juce::String(static_cast<juce::int64>(seq))
    + " gen=" + juce::String(static_cast<juce::int64>(world->generation))
    + " worldId=" + juce::String(static_cast<juce::int64>(world->worldId))
    + " publishEndUs=" + juce::String(static_cast<juce::int64>(publishEndUs)));
```

**DSP observe側（診断ブロック内、新World検出時）**:
```cpp
DBG(juce::String("[DSP_TIMING] cid=") + juce::String(static_cast<juce::int64>(correlationId.localCounter))
    + " seq=" + juce::String(static_cast<juce::int64>(dspSeq))
    + " gen=" + juce::String(static_cast<juce::int64>(runtimeWorld->generation))
    + " worldId=" + juce::String(static_cast<juce::int64>(runtimeWorld->worldId))
    + " callbackIndex=" + ...);
```

**解析例**:
```
[PUBLISH]  cid=84 seq=105 gen=8  worldId=0xABCD publishEndUs=1234567000
[DSP_TIMING] cid=84 seq=105 gen=8  worldId=0xABCD callbackIndex=28144 ...
```

- `seq` と `worldId` が両方一致 → 同一worldが正しく伝搬された ✅
- `seq` 一致だが `worldId` 不一致 → **world置き換え（rollback/republish）の証拠** ⚠️
- `gen` 不一致 → 世代管理の不整合 ❌

`worldId` は `RuntimeState` の既存フィールド（`std::uint64_t worldId`、Builderで設定済み）。新規フィールド追加不要。

#### 時刻系の統一 — 影響範囲

`getHighResolutionTicks()`（QPC）→ `convo::getCurrentTimeUs()`（`steady_clock` μs）

| ファイル | 置換箇所 | 用途 |
|---------|---------|------|
| `AudioBlock.cpp` | 9箇所（6診断 + 3 CallbackTelemetryScope） | XRUN(3) + CBSUMMARY(3) + ACTIVATE(1) + CallbackTelemetry(2) + CallbackTelemetryScope(3) |
| `BlockDouble.cpp` | 9箇所（6診断 + 3 CallbackTelemetryScope） | 同上（ダブル精度パス） |
| `Timer.cpp` | 8箇所（5診断 + 3 deferred timing） | MEM診断(2) + CBSUMMARY出力(2) + diagPrefix(1) + deferredStructural(1) + deferredFinalize(2) |

**全26箇所の置換が必要。** うち17箇所は `#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` ガード内。残り9箇所（CallbackTelemetryScope 6 + deferred timing 3）は機能コードだが、`getCurrentTimeUs()` への統一に支障はない（いずれも数百ms〜数秒のタイマーであり、マイクロ秒精度で十分）。

**加えて `enqueueTimestampUs` も `getCurrentTimeUs()` に統一する。** 現在は `steady_clock::now().time_since_epoch().count()`（QPC/100ns単位の生カウント）を使用しており、`getCurrentTimeUs()`（マイクロ秒）との直接比較ができない。`enqueue`→`publishStart` 間のΔ値計算は `publish` 側で `waitUs = getCurrentTimeUs() - enqueueUs` と計算する。

**既存バグの発見**: TTLチェック（`RuntimePublicationOrchestrator.cpp:324`）で `getCurrentTimeUs()` と `enqueueTimestampUs`（QPC生カウント）を直接減算しており、時刻系の不一致により正しいTTL超過判定ができていない。`enqueueTimestampUs` の `getCurrentTimeUs()` 統一によりこのバグも同時に修正される。

#### 確認: 新規atomic追加の要不要

**結論: publishTimingWriteCount 1個のみ。**

- `publicationSequence` はDSP側で観測した `runtimeWorld->metadata.publicationSequence` から取得可能（`RuntimeBuilder.cpp:196` で設定）
- `getLastCommittedPublicationSequence()` は既存の atomic から読み取り可能
- History Table の `publishTimingWriteCount`（1 atomic）のみが新規追加。`publishTimingHistory[16]` は非atomic（writeCountのRelease/Acquire保護下）。
- 旧案の `publishTimingReadIndex` は lookup 方式により不要。

#### 実装推奨順序

```
Step 0: CID を REBUILD_TELEMETRY に追加（1行, AudioEngine.h）
Step 1: Fixed Publish Timing History を RTLocalState に追加（writeCount + history[16], ~15行）
Step 2: publish() に timing + [PUBLISH] ログ追加（~10行, PublicationExecutor.cpp）
Step 3: DSP callbackに observeUs + callbacksUntilObserve + ObserveReason 追加（~20行）
Step 4: callbackIndexをcallback開始直後に固定取得して全ブロックで使用
Step 5: [DSP_TIMING] + [PUBLISH] のフォーマット確定
Step 6: getCurrentTimeUs() への時刻統一（26箇所、全ファイル）
Step 7: enqueueTimestampUs を getCurrentTimeUs() に変更 + waitUs計算（1行）

#### 既存バグの発見

`RuntimePublicationOrchestrator.cpp` において:
- `enqueueTimestampUs` は `steady_clock::now().time_since_epoch().count()`（QPC生カウント）で設定
- TTLチェック（324行目）は `getCurrentTimeUs()`（マイクロ秒）と `enqueueTimestampUs` を直接減算
- 時刻系の不一致により、TTL超過判定が正しく動作していない
- `enqueueTimestampUs` の `getCurrentTimeUs()` 統一により同時に修正される
```

#### 備考

- この計測は第3次追加観測として独立させる（第2次と混在させると解析が複雑になるため）。
- `callbacksUntilObserve` は callback 回数なので、サンプルレートやブロックサイズに依存しない。後から `callbacks × blockSize / sampleRate` で時間に換算可能。
