## 1. 確定事項：RTスレッドからの同期FileLogger I/O（★★★★★）

### 1.1 JUCE FileLogger::logMessage() の実体

`JUCE/modules/juce_core/logging/juce_FileLogger.cpp:61-66`:

```cpp
void FileLogger::logMessage (const String& message)
{
    const ScopedLock sl (logLock);        // ① SpinLock取得
    FileOutputStream out (logFile, 256);  // ② ファイルストリームオープン（毎回）
    out < message < newLine;            // ③ 同期ディスク書き込み
}
```

**確定判定**: RTスレッドから呼ぶと、①スピンロック待ち、②ファイルオープン（システムコール）、③同期書き込み（I/Oブロッキング）が発生。RT予算 192kHz/1024sample = **5.33ms** を容易に超過し、XRUN/glitchを引き起こす。**RT違反確定。**

## 6. ユーザー仮説の検証結果

### 仮説(a): 「Audio ThreadがFileLoggerに同期的に到達する」

**✅ 確定。**

`diagLog()` → `juce::Logger::writeToLog()` → `FileLogger::logMessage()` (SpinLock + FileOutputStream + 同期書き込み)。

AudioBlock.cppに4箇所、BlockDouble.cppに2箇所、DSPCoreFloat/Doubleに2箇所。計最低8箇所。

### 仮説(b): 「Input=CPU_MIGは未証明」

**✅ 確定。**

CPU_MIGはオーディオコールバックの冒頭（Input phase）で検出される。サンプリングなし（全件検出）、イベント駆動型。CPUマイグレーションは高頻度で発生する可能性があり、RT違反の影響は大きい。

### ユーザー優先度テーブルの検証

| 優先度 | 項目 | 検証結果 |
|--------|------|---------|
| ★★★★★ | Audio Threadから同期FileLogger完全排除 | **コード証拠で確認**: 最低8箇所（AudioBlock:4, BlockDouble:2, DSPCore:2）の直接writeToLogがRTスレッド上にある |
| ★★★★☆ | ring-buffer CPU_MIG + CALLBACK_STAGE | **既存パターン確認**: xRunBuffer（LockFreeRingBuffer）が同じ構造を実装済み。これを汎用化すれば対応可能 |

## 8. 改善計画

### フェーズ1: DiagEventRingBufferの新設（★★★★★）

**既存の `xRunBuffer` パターンを汎用化する新しいリングバッファを追加:**

```cpp
// 新設: DiagEventRingBuffer
// RT Thread: push() でイベント蓄積（ロックフリー）
// Timer Thread: pop() → diagLog() → FileLogger（非RT）
struct DiagEvent {
    uint64_t timestampUs;
    uint64_t callbackSeq;
    int cpuId;
    DiagCategory category;  // CPU_MIG, CB_SEQ, DSP_TIMING, etc.
    char message[256];      // or juce::String preformatted on RT thread
};
LockFreeRingBuffer<DiagEvent, 256> diagEventBuffer;
```

**修正対象（RT→リングバッファpushに変更）:**

| 3 | `AudioBlock.cpp` | 557 | `[DSP_TIMING]` | `writeToLog` → `diagEventBuffer.push(...)` |

### フェーズ2: `diagLog()` 関数のRT安全化（★★★★☆）

各TUの無名名前空間にある `diagLog()` を、RTスレッド用と非RTスレッド用で分離:

- **RT用**: メッセージを構築してリングバッファにpush（文字列構築もRTスレッドで行うか、数値データのみpushしてTimerスレッドでフォーマット）
- **非RT用**: 現状通り `juce::Logger::writeToLog()` を直接呼ぶ

### フェーズ3: CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICSのデフォルト変更（★★★☆☆）

現在 `DiagnosticsConfig.h` で `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS=1`（デフォルト有効）。Releaseビルドでは `0` にすることでRT違反を完全排除。ただし、診断機能が必要な場合はフェーズ1-2の実装でRT安全化すべき。

## 9. 結論

**ユーザーの仮説は完全に確定した。** 診断ログシステムはRTボトルネックになっている:

1. **Audio Threadから同期FileLogger I/Oが確実に存在** — 最低8箇所の直接 `writeToLog` 呼び出しがRTスレッド上にある
2. **FileLogger::logMessage()はSpinLock + ファイルオープン + 同期書き込み** — RTスレッドから呼ぶと確実にXRUNを引き起こす
3. **既存のリングバッファ非同期パターン（xRunBuffer）が動作中** — これを汎用化することでRT安全な診断ログシステムを実現可能
4. **改善コストは低い** — 既存のLockFreeRingBufferインフラとTimer drainパイプラインが完成済み。新しいDiagEvent構造体とリングバッファを追加し、8箇所のwriteToLogをpushに変更するだけで完了
