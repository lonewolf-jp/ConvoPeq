## 7. 根本原因分析

### 原因分類

| 原因カテゴリ | 件数 | 代表例 |
| --- | --- | --- |
| OSスケジューラ遅延（Windows） | 231件（98.3%） | 8.0〜9.7msのXRUN全般 |
| 適応フィルタ係数切替 | 1件（0.4%） | XRUN#56（32.14ms） |
| システム初期化中 | 1件（0.4%） | XRUN#1（11.04ms） |
| PageFaultサージ相関 | 2件（0.9%） | XRUN#231-235 |

### ルートコーズ

1. **Windowsオーディオコールバックのスケジューリングジッター（主原因）**
   - 5.33ms周期のコールバックがOSにより8-9ms遅延
   - これはConvoPeqの制御外 — OSのタイムクリティカルスレッド管理に依存
   - 高負荷時（メモリ2.5GB、ページフォールト多発）で悪化

2. **適応フィルタ係数切替のブロッキング（異常値原因）**
   - COEFF_AUTH + ADAPTIVE_SWITCHにより32.14msのコールバック遅延
   - 係数認証プロセスがオーディオスレッドをブロックした可能性

3. **パイプライン設計の健全性確認**
   - Publish→Observeのレイテンシは283〜4,624μsで正常
   - RCUベースのRuntimeは正しく機能
   - プレッシャー指標は全て正常

### 対応方針

| 優先度 | 対策 | 期待効果 | 難易度 |
| --- | --- | --- | --- |
| ★★★ | 適応フィルタ係数切替の非ブロッキング化 | 32ms異常値の防止 | 中 |
| ★★☆ | オーディオスレッド優先度の確認・最適化 | 平均ジッター低減 | 低 |
| ★☆☆ | バッファサイズ拡張（512→1024サンプル） | 実効許容遅延が2倍に | 低（ただしレイテンシ増） |
| ★☆☆ | ハードウェアバッファの設定最適化 | スケジューラ余裕増 | 低 |

## 8. 推奨アクション

### 短期的（即日対応可能）

1. **COEFF_AUTH + ADAPTIVE_SWITCHの非同期実行**を検討
   - 係数認証処理をオーディオコールバック外に移動
   - または認証をバックグラウンドスレッドに委譲

2. **オーディオコールバックスレッドの優先度確認**
   - `SetThreadPriority` / `AvSetMmThreadPriority` が適切に設定されているか確認
   - Windows Multimedia Class Scheduler（MMCSS）の利用状況確認

### 中期的（次回観測まで）

3. **バッファサイズ512サンプル（@44100Hz = 11.6ms）の許容性評価**
   - 現状の8ms平均ジッターに対して余裕は3.6ms
   - バッファサイズ1024に拡張すると許容範囲が23.2msに拡大

4. **PageFault分散の最適化**
   - メモリ2.5GBのワーキングセットを事前フォールト（prefault）で確保
   - 起動時のページフォールトサージ（302,256件）を平準化

### 長期的（設計見直し）

5. **リアルタイムアンセーフな処理の分離**
   - COEFF_AUTH / ADAPTIVE_SWITCH処理の完全非同期化
   - ダブルバッファリングによるブロッキング排除
