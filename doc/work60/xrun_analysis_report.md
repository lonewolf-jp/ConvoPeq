### Callback 処理時間（CB_HIST 全101,056サンプル）

**パーセンタイル分布（昇順ソート、正確な値）:**

| 指標 | 処理時間 | 予算(5,333μs)比 |
|------|---------|----------------|
| 最小 | 215 μs | 4% |
| P50（中央値） | **2,938 μs** | **55%** |
| P90 | **6,920 μs** | **130% ❌** |
| P95 | **8,768 μs** | **164% ❌** |
| P99 | **13,550 μs** | **254% ❌** |
| P99.9 | **21,414 μs** | **401% ❌** |
| 最大 | 833,164 μs | 15,625% |

**⚠️ 重要**: 中央値2,938μsは予算内だが、**P90で既に6,920μsと予算超過**。10%のコールバックが予算超過しており、これがXRUNの直接原因。P95/P99/P99.9は深刻な超過。前回の「P90=1,452μs」は降順ソートの誤りで、正確には6,920μs。

**CPU Migration × 処理時間の関係（パーセンタイル比較）:**

| パーセンタイル | MIG有 | MIG無 | MIG有 vs 予算 |
|---------------|-------|-------|-------------|
| P50 | 3,219 μs (60%) | 1,964 μs (37%) | 予算内 |
| P90 | **7,123 μs (134%)** | **5,376 μs (101%)** | ❌ 超過 |
| P95 | **8,977 μs (168%)** | **6,944 μs (130%)** | ❌ 超過 |
| P99 | **13,828 μs (259%)** | **11,386 μs (214%)** | ❌ 超過 |
| 最大 | 833,164 μs | 26,162 μs | ❌ 異常 |

**核心**: MIG無しでもP90=5,376μsで既に予算超過。CPU Migrationは「根本原因」ではなく「増幅器」。MIGが無くても10%のコールバックは予算超過している。

**確定: CONV_TIME/STCONV_TIME/EQ_TIME 未発火の原因**
- 診断コード自体は `DiagnosticsConfig.h` の `#ifndef` フォールバックにより **実質的に常時ON**
- 各閾値が高すぎて発火しなかった:
  | タグ | 閾値（192kHz/1024spb時） | 結果 |
  |------|--------------------------|------|
  | EQ_TIME | max(100, 5333×0.1) = **533μs** | ✅ EQは高速 |
  | STCONV_TIME | max(200, chunk×0.15) ≈ **800μs** | ✅ Convチャンク高速 |
  | CONV_TIME | max(300, 5333×0.2) = **1067μs** | ✅ Conv合計高速 |
- **Pure DSPは閾値未満だが、callback全体のP90=6,920μs** → DSP個別処理と全体の乖離は、**コールバック開始以前（OS/Driver区間）**または**DSP外処理（コピー/ミキシング/メータリング/Observer）**の可能性がある。閾値≠実測値のため差分定量は不可。

### Drift（コールバック開始時刻偏差）分布

**CB_HIST 全101,056サンプルからのdrift値:**

| 指標 | Drift値 | 解釈 |
|------|---------|------|
| P50（中央値） | **-67 μs** | 半数のコールバックはほぼ時刻通りに開始 |
| P25 | -978 μs | 25%は1ms早く開始 |
| P75 | +822 μs | 25%は0.8ms遅れて開始 |
| P95 | **+3,402 μs** | 5%は3.4ms以上遅れて開始 |
| P99 | **+8,146 μs** | 1%は8.1ms以上遅れて開始 |
| 最大 | +827,852 μs | 異常値 |
| 平均 | +120 μs | わずかに正方向 |

**重要**: 中央値は-67μsとほぼ0。つまりOS/Driverは**通常時は5.33ms周期を正確に守れている**。しかしテール部分で大幅な遅延が発生（P95=+3,402μs, P99=+8,146μs）。このテール遅延がXRUNの直接原因。

**Drift × CPU Migration の関係（新たな重要発見）:**

| グループ | P50 | P95 | P99 | 最大 |
|---------|-----|-----|-----|------|
| MIG有 | -52 μs | 3,251 μs | 7,986 μs | 107,694 μs |
| MIG無 | **-201 μs** | **4,392 μs** | **9,124 μs** | **827,852 μs** |

**→ CPU Migrationがないコールバックの方がDriftが大きい。** MIG有はP95=3,251μs、MIG無はP95=4,392μsと、MIG無の方が1.1msも遅延が大きい。この観測事実は「CPU Migration→cache miss→callback遅延」の単純な因果とは整合しない。**因果方向は未確定**であり、以下の両方が説明可能: (a) Migrationによりcache missが発生し処理時間が増加、(b) 処理時間が長いcallbackをOSが別コアに移動。現時点では**相関のみ**であり、両方の可能性がある。

### CPU Migration

| 指標 | 値 |
|------|------|
| 総CPU Migration数 | **50,232 回** |
| Migrationレート | **157回/秒** |
| ユニークcallback数 | 50,232（各callback最大1回のMigration） |
| 全callback数 | ~288,000（推測） |
| Migration率 | ~17%のcallbackでMigration発生 |

**確定: CPU Migration × Proc Time の相関あり**

CPU Migration 発生時のコールバック処理時間と非発生時の比較:

| グループ | 件数 | 平均proc時間 | 最大proc時間 |
|---------|------|------------|------------|
| Migration有り | 50,159 | **4,027 μs** | **833,164 μs** |
| Migration無し | 7,978 | **2,912 μs** | 26,162 μs |
| 差分 | — | **+1,115 μs (+38.3%)** | +807,002 μs |

- 最大833msのスパイクは**Migration有りの時のみ発生**
- 平均でも**38.3%の処理時間増加**
- ただし **相関と因果は異なる**:
  - Migration → cache miss → 処理時間増加 の可能性
  - 処理時間が長い → OSがスレッドを別コアに移そうとする の可能性
  - 両者に共通原因（システム負荷上昇）の可能性

### メモリ推移

| フェーズ | Private | WS | PageFaults | Delta |
|---------|---------|-----|-----------|-------|
| 初期化直後 | 1,352 MB | 1,255 MB | 327,916 | — |
| IR+Peq設定後 | **2,497 MB** | **2,343 MB** | **671,443** | **+320,903** |
| 定常状態 | ~2,491 MB | ~2,336 MB | ~710K→780K | 3K-5K/30s |

**確定: メモリプレッシャーは定常状態では問題にならない**
- IR読み込み時に **+320,903 PageFaults** のスパイク → これは初期化時の1回限り
- 定常状態では **100-167 PageFaults/秒** → 無視可能
- Working Set 2.3GBは大きいが、**ページングは発生していない**
- メモリ帯域の contention は未測定だが、少なくともページフォールトが原因ではない

## 🔍 既存コード調査結果

### 1. ThreadAffinityManager — 完全な No-Op

`ThreadAffinityManager` はアフィニティ設定のインフラのみ実装されたスタブ:

- `initialized_` が**永遠に `false`**（trueにする代入が存在しない）
- 全 `ThreadAffinityMasks` フィールドが**デフォルト `0`**

**CMakeLists.txt:55**: `option(CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS OFF)` — デフォルトOFF

## 🎯 根本原因分析

### 観測事実（確定）

**事実1: XRUN Intervalの最小値=8ms、5.33msを下回るケースが1件もない**

全7,691件のXRUNにおいて、Intervalが8ms未満のものは**1件も存在しない**。期待値5.33msを下回るケースが皆無であることは、以下の3つの可能性に絞られる:
- ① OS/Driverのスケジューリング周期そのものが5.33msに対応できていない
- ② DSP処理が毎回必ず一定量の時間を消費し、その結果コールバック間隔が延びている
- ③ なんらかの共通原因により常に同じ量の遅延が発生している

**事実2: XRUNレートは全320秒間を通じて一定（約25 events/秒）**

10秒バケットごとのXRUN数は170〜298で、時間経過による増減傾向なし。初期化直後も320秒後も変わらない。これは「初期化の影響」「メモリ増加の影響」「経年劣化」などを否定する。

**事実3: 最初の100msで97 callbacksが損失**

CBSUMMARY初回（prepareToPlay直後最初のタイマー起動）: expected=187, actual=90, loss=97。オーディオ開始直後から既にコールバックの52%が失われている。
それ以降のCBSUMMARYは loss=-4〜+11 で安定。つまり初期の大規模損失は一過性であり、その後はコールバック損失率2.03%で安定。

**事実4: DSP個別処理は閾値未満、しかしCallback全体P90=6,920μsで予算超過**

CONV_TIME/EQ_TIME/STCONV_TIMEが全く発火していないことから、Convolver/EQの個別処理時間はそれぞれ**閾値未満**（1067μs/533μs/800μs未満）。しかしCallback全体のP90は6,920μsと予算を30%超過している。**閾値は実測値ではないため差分の定量は不可。** DSP個別処理と全体との乖離は、コールバック開始以前（OS/Driver区間）とDSP外処理（コピー/ミキシング/メータリング/Observer）の2つの可能性がある。

**事実5: CPU Migrationは「増幅器」であり、かつ呼び出し遅延の結果でもある**

MIG有: P90=7,123μs, MIG無: P90=5,376μs。MIGが無くても10%のコールバックは予算超過。MIGは+32%の追加ペナルティを与えるが、根本原因ではない。

**さらに、Drift×Migration分析により以下が判明:**

| 指標 | MIG有 | MIG無 | 差 |
|------|-------|-------|-----|
| Drift P50 | -52 μs | -201 μs | MIG無の方が良い |
| Drift P95 | 3,251 μs | **4,392 μs** | MIG無が**1.1ms悪い** |
| Drift P99 | 7,986 μs | **9,124 μs** | MIG無が**1.1ms悪い** |
| Drift 最大 | 107,694 μs | **827,852 μs** | MIG無が致命的 |

**→ Migration有とMigration無でproc時間に差はあるが、因果方向は未確定。** 「Migration→cache miss→callback遅延」と「callback遅延→OSがMigration→別コアで継続」の両方が観測事実と整合する。**相関は確認済みだが、因果の検証にはAffinity固定実験またはcallbackEntryUs計測が必要。**

### 総合評価

```
                           5,333μs（予算）
                               │
     P50(2,938μs)     P90(6,920μs)
       │                 │
       ▼                 ▼
   余裕あり ───→ 予算超過 ──→ XRUN
                   │         │
                   │    MIG有: +32% → 7,123μs
                   │    MIG無: 5,376μs（それでも超過）
                   │
                   ▼
          CPU Migrationは「増幅器」
          原因は別にある
```

### 残る仮説（優先順位順）

**真因は以下の二段構造のどちらか（または両方）に絞られる。v7のcallbackEntryUs計測により境界が初めて定量可能になる。**

| # | 仮説 | 層 | 根拠 | 検証方法 |
|---|------|----|------|---------|
| 1 | **Callback開始以前の遅延（OS/Driver区間）** | ① | min Interval=8msが示す固定的遅延 | v7 callbackEntryUs計測 |
| 2 | **OSスケジューリングマージン不足** | ① | 最小Interval=8ms（5.33msにならない） | spb=2048実験 |
| 3 | **WASAPIバッファリング/Driver遅延** | ① | コールバック開始時刻自体が遅延 | DirectSound切替え |
| 4 | **Callback内部のDSP外処理が支配的** | ② | 全体P90=6,920μs > DSP閾値 | v7計測で全内訳取得 |
| 5 | **CPU Migration（増幅器）** | ② | +38.3%のペナルティ確認済み | Affinity固定実験（診断目的） |

## 💡 次に行うべき計測（Practical Stable ISR Bridge Runtime 推奨順序）

### Phase 1: 既存v7診断の閾値調整（コード修正不要の設定変更）

| タグ | 現状の閾値 | 推奨値 | 理由 |
|------|-----------|--------|------|
| CONV_TIME | max(300μs, expected×0.2) | **max(100μs, expected×0.05)** | 発火率向上 |
| STCONV_TIME | max(200μs, expected×0.15) | **max(50μs, expected×0.05)** | 発火率向上 |
| EQ_TIME | max(100μs, expected×0.1) | **max(30μs, expected×0.03)** | 発火率向上 |

または **閾値を完全に撤廃**（常時出力）して1度だけ観測する。

### Phase 2: CPU Migration × Callback時間の相関分析

既存ログで以下の相関を確認可能:

1. `[CPU_MIG]` の有無で `[CB_HIST] proc=` の値をグループ化
2. Migration発生時のproc時間 vs 非発生時のproc時間
3. Migration先CPUとproc時間の関係

### Phase 3: メモリプレッシャー測定

以下の追加計測:
- `GetProcessMemoryInfo` のPageFaultCount（既に取得可能）
- Hard Fault率（Performance Counter）
- TLB Miss率（Performance Counter）

### Phase 4: スケジューリング実験（順序厳守）

```
① CONV_TIME/STCONV_TIME/EQ_TIME 閾値引き下げ
    ↓ 実測
② CPU Migration × proc時間 相関分析
    ↓ 実測
③ 必要なら音飛び区間のDSP内訳特定
    ↓ 特定後
④ CPU固定実験（診断目的、改善目的ではない）
    ↓
⑤ MMCSS "Pro Audio" → AVRT_PRIORITY_HIGH 昇格
    ↓
⑥ WASAPI → DirectSound 切替え試験
    ↓
⑦ spb=2048 緩和試験
```

**重要な原則（Practical Stable ISR Bridge Runtime）:**
- CPU固定は**診断の前に symptom を隠す**ため、最後に行う
- 計測 → 分析 → 仮説検証 → 対策 の順序を厳守
- 一つの変更で一つの仮説を検証する
