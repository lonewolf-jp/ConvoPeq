## F: diagLog Release対応（基盤整備）

### 目的
diagLog() の実装がファイル間で不統一な問題を修正。CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS=1 でReleaseビルド時に全診断ログが確実に動作する。

### 現状の問題
AudioEngine.Timer.cpp: 常時出力（Releaseでも動作可能）
他9ファイル: JUCE_DEBUG/CONVO_CI_BUILD でガード（Releaseでは無効）

### 改修内容
全ファイルの diagLog 定義を Timer.cpp 準拠に統一（9ファイル: Commit, Parameters, Snapshot, UIEvents, RebuildDispatch, Init, PrepareToPlay, ReleaseResources, DSPCoreLifecycle, CtorDtor）

```cpp
void diagLog(const juce::String& message)
{
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
    DBG(message);
    juce::Logger::writeToLog(message);
#else
    juce::ignoreUnused(message);
#endif
}
```

## A: getNextAudioBlock() コールバック受信タイムスタンプ（最重要）

### 目的
OS/Driver → MMCSS → JUCE callback dispatch 区間のジッターを可視化。期待受信時刻と実際の受信時刻の差（driftUs）を測定。

### 設計上の注意
- atomic<double> 禁止 → atomic<int64_t> で整数μs保持
- exchangeAtomic 不要 → load + store で十分（診断専用値のため厳密な同期は不要）

### 改修内容（AudioBlock.cpp / BlockDouble.cpp）
getNextAudioBlock() 冒頭、AudioCallbackRuntimeScope 構築直後に追加:
```cpp
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
{
    const uint64_t nowUs = convo::getCurrentTimeUs();
    const uint64_t prevEntryUs = convo::consumeAtomic(
        rtLocalState_.lastCallbackEntryUs, std::memory_order_relaxed);
    convo::publishAtomic(rtLocalState_.lastCallbackEntryUs,
        nowUs, std::memory_order_relaxed);
    if (prevEntryUs > 0 && numSamples > 0) {
        // ...期待値計算、driftUs を int64_t で保持...
        convo::publishAtomic(rtLocalState_.lastCallbackDriftUs,
            driftUs, std::memory_order_relaxed);
    }
}
#endif
```

### 追加メンバ変数（AudioEngine.h / RTLocalState）
```cpp
std::atomic<uint64_t> lastCallbackEntryUs { 0 };
std::atomic<int64_t> lastCallbackDriftUs { 0 };
```

### ログ出力（既存 [XRUN] ブロック内に追記）
```cpp
if (xrunDetected) {
    const int64_t driftUs = convo::consumeAtomic(
        rtLocalState_.lastCallbackDriftUs, std::memory_order_relaxed);
    log += " DriftUs=" + juce::String(static_cast<juce::int64>(driftUs));
}
```

### 期待効果
DriftUs大（正）→ OS/Driver区間遅延。DriftUs小 → ConvoPeq内部原因。この1つで原因切分け可能。

## G: CPU migration 記録

### 目的
GetCurrentProcessorNumber() でAudio ThreadのCPUコア遷移を検出。MMCSS固定失敗・DPC/ISR競合の兆候。

### 改修内容（AudioBlock.cpp / BlockDouble.cpp）
```cpp
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
{
    const uint32_t currentProc = static_cast<uint32_t>(::GetCurrentProcessorNumber());
    const uint32_t prevProc = convo::exchangeAtomic(
        rtLocalState_.lastCallbackProcessor, currentProc, std::memory_order_relaxed);
    if (prevProc != currentProc && prevProc != UINT32_MAX) {
        convo::fetchAddAtomic(rtLocalState_.cpuMigrationCount, uint64_t{1}, relaxed);
        juce::Logger::writeToLog("[CPU_MIG] CPU=" + juce::String((int)currentProc)
            + " prev=" + juce::String((int)prevProc));
    }
}
#endif
```

### 追加メンバ変数
```cpp
std::atomic<uint32_t> lastCallbackProcessor { UINT32_MAX };
std::atomic<uint64_t> cpuMigrationCount { 0 };
```

### 期待効果
XRUN直前にCPU migration → OS問題。コストは数ns〜数十ns。

## 仮説判定マトリクス

| XRUN時観測 | 示唆する原因 |
| --- | --- |
| DriftUs大 + CPU_MIG有 | OSスケジューラ問題（CPU migration + 遅延） |
| DriftUs大 + CPU_MIG無 | OS/Driver DPC/ISR問題（同一CPU割込み遅延） |
| DriftUs小 + CONV_TIME大 + STCONV_TIME大 | FFT畳み込み遅延 |
| DriftUs小 + CONV_TIME大 + STCONV_TIME小 | Convolver管理処理遅延（RCU等） |
| DriftUs小 + EQ_TIME大 | EQ IIRフィルタ遅延 |
| DriftUs小 + CB_PROC_TIME大 + 全内部計測小 | 未計測内部要因 |
| 全計測値小 | 偶発的OSジッター |
