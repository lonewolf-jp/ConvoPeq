## 1. 総評

### 1.1 推定メモリ内訳

| カテゴリ | 推定サイズ | 確定度 | 備考 |
|---------|-----------|--------|------|
| MKLNonUniformConvolver (1ch) | ~40 MB | ★★★★☆ | 3 Layer NUP, 2ch時 ~80MB/組 |
| StereoConvolver 合計 | ~80 MB/組 | ★★★★☆ | direct + ring + FFT + FDL バッファ |
| DSPCore 1インスタンス | ~132 MB | ★★★☆☆ | convolver + aligned + EQ + NS + その他 |
| UI ConvolverProcessor | ~80 MB | ★★★★☆ | UI側1組（別インスタンス） |
| IR Cache (max 8 entry) | ~24 MB | ★★★★★ | AudioBuffer<double> × 8 |
| DSPCore 合計 (UI+Active+Fading) | ~396 MB | ★★★☆☆ | 3インスタンス同時 |
| Retire滞留 (旧DSPCore群) | **~数百MB〜1GB+** | ★★★☆☆ | **最大の不確定要素** |
| FFT Plan Cache | ~数MB | ★★★★★ | IppFFTPlan 数個のみ |
| MixedPhasePersistentCache | 0 MB (in-memory) | ★★★★★ | ディスク永続化のみ |
| EQ Cache | ~数MB | ★★★☆☆ | スナップショットハッシュ管理 |

### 1.2 3GBの説明可能性

| シナリオ | 推定使用量 | 3GB説明可能か |
|---------|-----------|-------------|
| 最小（UI+Active+Fading+通常Retire） | ~800 MB | ❌ 不可能 |
| 中程度（加えて過去5世代滞留） | **~1.8 GB** | ⚠️ 可能性あり |
| 最大（IR Cache満杯＋多数世代滞留＋Working Set拡大） | **~2.5-3.5 GB** | ✅ 説明可能 |

**結論**: 3GBのメモリ使用量は、複数世代のDSPCore/RuntimeがDeferredFreeによって解放されず滞留している場合に説明可能。

## 2. 最有力：Retire滞留による複数世代の同時保持

### 2.1 ログ上のGeneration数

ログから以下の8世代が確認できる:

| Generation | DSPCORE_PREPARE | publish | Retire時期 |
|-----------|----------------|---------|-----------|
| 1 | #1 (sr=48k) | ✅ gen=1 | prepareToPlay直後 |
| 2 | #2 (sr=192k) | ❌ | gen=4の前 |
| 3 | #3 (sr=192k) | ❌ | gen=4の前 |
| 4 | #4 (sr=192k) | ✅ gen=4 | gen=8の前 |
| 5 | - | ❌ | gen=6の前 |
| 6 | - | ❌ | gen=7の前 |
| 7 | #5 (sr=192k) | ❌ | gen=8の前 |
| 8 | #6 (sr=192k) | ✅ gen=8 (rev=5) | **アクティブ** |

### 2.2 DeferredFreeThreadの動作

```cpp
// DeferredFreeThread.h
static constexpr int kMaxReclaimPerLoop = 4;
```

- DeferredFreeThreadは1ループあたり最大**4個**しか解放しない
- ポーリング間隔は明示的なsleepなし（`std::this_thread::yield()`）
- つまりrebuildThreadが次々と新しいRuntimeをpublishすると、DeferredFreeが追いつかずに**RetireキューにRuntimeが滞留する**

### 2.3 滞留時のメモリインパクト

```
publish gen=4 → gen=1,2,3 が retire される
↓
DeferredFree: 4個まで解放 → gen=1,2,3 は解放される（3個なので問題なし）
↓
publish gen=5 → gen=4 が retire
publish gen=6 → gen=5 が retire
publish gen=7 → gen=6 が retire
publish gen=8 → gen=7 が retire
↓
この間、DeferredFreeが1ループで4個解放可能 → gen=4,5,6,7 は解放される
```

もしrebuildが高速に連続すると:

```
gen=4 (publish直後に gen=5 が要求される)
gen=5 (publish直後に gen=6 が要求される)
gen=6 (publish直後に gen=7 が要求される)
gen=7 (publish直後に gen=8 が要求される)
gen=8 (publish成功)
```

各publish時に旧RuntimeがDeferredFreeキューに入る。最大 **4世代 + current + fading = 6インスタンス** 同時保持の可能性がある。

### 2.4 メモリ試算（滞留時）

| インスタンス | 数 | 単価 | 小計 |
|------------|----|------|-----|
| UI ConvolverProcessor | 1 | ~80 MB | 80 MB |
| Active Runtime DSPCore | 1 | ~132 MB | 132 MB |
| Fading Runtime DSPCore | 1 | ~132 MB | 132 MB |
| Retire滞留 (gen=4,5,6,7) | 4 | ~80 MB (StereoConvolverのみ) | 320 MB |
| IR Cache | 8 | ~3 MB | 24 MB |
| アラインドバッファ (DSPCore×3) | 3 | ~4 MB | 12 MB |
| オーバーサンプリングIR | - | ~50 MB | 50 MB |
| ノイズシェイパー状態 | - | ~10 MB | 10 MB |
| MKL内部/ワーキングセット拡大 | - | ~200 MB | 200 MB |
| **合計** | | | **~960 MB** |

しかし3GBにはまだ遠い。さらに調査が必要。

### 2.5 追加のメモリ消費要因（調査中）

以下の要因が3GBへの不足分を説明できる可能性がある:

| 要因 | 説明 | 推定インパクト |
|-----|------|-------------|
| StereoConvolver の Layer 2 が有効時 | IR長によってはL2が構築される | +~50MB/組 |
| Multi-Instance Retire 蓄積 | 上記試算は4世代だが、実際は更多い可能性 | +~200MB/世代 |
| Aligned Buffer の過剰確保 | `internalMaxBlock = inputMaxBlock × 8` = 524288 | +~8MB/組 |
| MKL スレッドローカルストレージ | MKL per-thread scratch buffer | +~50MB |
| Working Set の自然拡大 | OSのメモリ管理によるWS拡大 | +~200-500MB |
| DSPCore の `dryBypassBuffer` | 追加のaligned buffer × 2ch | +~8MB/組 |
| DSPCore の `ramp`/`history` バッファ | 複数の内部状態バッファ | +~16MB/組 |

## 3. 次点：Convolverメモリの重複

### 3.1 UI ConvolverProcessor と Runtime DSPCore の関係

```cpp
// AudioEngine.h
ConvolverProcessor uiConvolverProcessor;  // UI側 (メッセージスレッド)
struct DSPCore {
    // ...
    StereoConvolver convolver;  // RT側 (StereoConvolver を内包)
    // ...
};
```

- `uiConvolverProcessor` はUI操作（ファイル選択・パラメータ変更）用の**独立したConvolverProcessorインスタンス**
- RT側のDSPCoreは音声処理用の**別インスタンス**
- 両者は**独立したStereoConvolver（=MKLNonUniformConvolver × 2ch）** を持つ

つまり常に最低2組のNUCエンジン（UI + RT）が存在する。

### 3.2 IR読み込み時のbuild連鎖

```
UI側: convolverParamsChanged → IR読み込み → MixedPhase → NUC build
↓
AudioEngine: convolverParamsChanged → rebuild intent
↓
Worker: RuntimeBuilder.build() → newDSP->convolverRt().rebuildAllIRsSynchronous()
  → RT側でも別途NUC build
```

IR変更1回につき、**UI側とRT側で2回**NUC buildが走る。

### 3.3 publish → retire のライフサイクル

```
build() → publishWorld() → 旧Runtime retire → DeferredFreeThread で解放
```

publish後、旧RuntimeのDSPCoreは即座にdeferred freeキューに入るが、DeferredFreeThreadが**4個/ループ**でしか解放しないため、連続publishが高速だと滞留する。

## 4. 補足：その他のメモリ消費

### 4.1 オーバーサンプリング用IR

CustomInputOversamplerが内部で保持するフィルタIR:

```cpp
// DSPCore::prepare()
oversampling.prepare(inputMaxBlock, factor, preset);
// → OSフィルタ用FIR係数テーブルを確保
```

OS 4x時: ~数十MBのFIR係数テーブル。

### 4.2 DC Blocker 状態

```cpp
struct DCBlockerRuntimeState {
    UltraHighRateDCBlocker outputL, outputR;
    UltraHighRateDCBlocker inputL, inputR;
    UltraHighRateDCBlocker oversampledL, oversampledR;
};
```

6個のDCBlockerインスタンスがDSPCoreごとに存在。各DCBlockerがIIR状態を保持。

### 4.3 IQ不公平性・TruePeakDetector

```cpp
truePeakDetector.prepare(newSampleRate, maxInternalBlockSize);
loudnessMeter.prepare(newSampleRate, maxInternalBlockSize);
```

TruePeakDetectorとLoudnessMeterはオーバーサンプリング用の内部バッファを持つ。

## 5. メモリリークの可能性

### 5.1 調査結果

| チェック項目 | 結果 | 根拠 |
|-----------|------|------|
| mkl_malloc / mkl_free 対称性 | ✅ 対応あり | 全Layer解放は `releaseAllLayers()` でカバー |
| ippsMalloc_8u / ippsFree 対称性 | ✅ 対応あり | FFT Plan含め解放パス確認済み |
| aligned_make_unique / aligned_free | ✅ 対応あり | unique_ptr + custom deleter |
| DeferredFreeThread の確実な実行 | ⚠️ 条件付き | reclaim は minEpoch依存、Readerがepochを進めないと滞留 |
| DSPCore の reset() | ✅ 呼ばれている | rebuild毎に呼ばれる |
| シャットダウン時の解放 | ⚠️ タイムアウト | `kGracefulDrainMaxMs=5000ms`、タイムアウト時は強制drain |

### 5.2 シャットダウンログの確認

ログのシャットダウンセクション:
```
releaseResources: EmergencyDrain -- pendingPub=0 pendingRetire=0 stuckReaders=0
releaseResources: VerifyDrained -- collecting drain audit
```

- `pendingRetire=0` で正常にdrain完了している
- つまりシャットダウン時は全て解放されている

→ **メモリリークよりは「稼働中の滞留」が原因である可能性が高い。**

## 6. 音飛びとの関係評価

### 6.1 影響シナリオ

| # | シナリオ | 発生確率 | 音飛び影響 |
|---|---------|---------|-----------|
| ① | DeferredFree 追いつかず Retire 滞留 | 中〜高 | 中: Workerのpage fault増加 |
| ② | Working Set Trim → Soft Fault | 中 | 高: Audio Threadのcallback遅延 |
| ③ | Memory Compression → 展開CPU負荷 | 低〜中 | 高: Audio ThreadのCPU奪取 |
| ④ | 物理メモリ不足 → スワップ | 低（16GB以上想定） | 致命的: 数十ms〜数百msの停止 |
| ⑤ | MKLメモリ確保時のページフォールト | 中 | 中: build中のWorker遅延→publish遅延 |

### 6.2 最も疑わしいシナリオ

```
ConvoPeq 3GB + 他プロセス → システム82%使用
↓
OSが Working Set Trim を開始
↓
ConvoPeqの一部ページが Standby List に移動
↓
Worker Thread or Audio Thread がトリミングされたページにアクセス
↓
Soft Fault → ページ復元 (数ms〜数十ms)
↓
★ これが publish 後の callback と重なった場合、音飛び発生
```

### 6.3 確認すべきこと

| 確認項目 | 確認方法 |
|---------|---------|
| 実際のPrivate Bytes / Working Set | プロセスモニタ or タスクマネージャ |
| Retire待ちRuntime数 | ログ: `retireQueueDepth` の値 |
| Page Fault / Hard Fault 発生率 | Performance Monitor (perfmon) |
| Memory Compression の有無 | `GetMemoryCompressionInformation` / RAMMAP |

## 7. 追加すべき計測

### 7.1 メモリ観点の追加ログ

```cpp
// 理想的なログ出力例:
[MEM] PrivateBytes=3200MB WorkingSet=2800MB PageFaults/s=120
[MEM] RetireDepth=4 QuarantineResident=2
[MEM] ConvolverInstances: UI=1 Active=1 Fading=1 Retired=4
[MEM] IRCache: entries=3/8 size=9MB
[MEM] MKLAlloc: total=420MB peak=480MB
```

### 7.2 既存のメモリ関連診断

以下の診断が既にコードに存在する:

| 診断 | 場所 | 状態 |
|-----|------|------|
| `retireQueueDepth_` | `AudioEngine.Retire.cpp` | atomic保持、Timerでログ出力可能 |
| `quarantineResident_` | `AudioEngine.Retire.cpp` | 同上 |
| `pendingRetireCount` | `ISRRetireRouter` | 同上 |
| `DeferredFreeThread::snapshotBoundaryTelemetry()` | `DeferredFreeThread.h` | 呼び出し側で未使用の可能性 |

### 7.3 推奨する追加計測（優先度順）

1. **Private Bytes 定期ログ出力**（例: 1秒ごと）
   - Windows: `GetProcessMemoryInfo()` → `PROCESS_MEMORY_COUNTERS`
   - 既存のtimerCallback (100ms) に追加するのが最も簡単

2. **Retire待ちRuntime数の定期ログ出力**
   - `retireQueueDepth_` / `pendingRetireCount` を `[VERIFY]` ログに追加
   - 音飛び発生時刻とRetire滞留の相関確認

3. **Page Fault カウント**
   - `GetProcessMemoryInfo()` の `PageFaultCount` を定期的に取得
   - Hard Fault（ページングI/O発生）の有無を確認

4. **publish 直後のメモリ状況スナップショット**
   - `[DIAG] trySubmit: executor_.publish SUCCEEDED` 発行時に
     同時に Private Bytes と Retire カウントを出力
