## 2. 確定した事実

以下の項目はソースコードとログにより完全に確認・確定した。

### 2.1 setUseIncrementalRebuild のバグ

```cpp
// src/convolver/ConvolverProcessor.Rebuild.cpp L274-L278
void ConvolverProcessor::setUseIncrementalRebuild(bool enable) noexcept
{
    juce::ignoreUnused(enable);                          // enable を完全に無視
    convo::publishAtomic(useIncrementalRebuild, false);  // 常に false
}
```

- `setUseIncrementalRebuild(true)` と呼び出しても `false` しか書き込まれない
- この関数を呼び出すコードが存在しない
- `IncrementalRebuildJob`, `runIncrementalBuildStep()`, `runIncrementalFinalizeStep()` の実装は存在するが使用されていない
- コンパイルガード `CONVOPEQ_ENABLE_CONVOLVER_SPLIT_REBUILD=1` は定義済み

**【評価】**: これは確かにバグだが、**音飛びの直接原因ではない。** 性能改善経路が殺されているだけ。

### 2.2 rebuildThreadLoop は同期リビルドを使用

```cpp
// src/audioengine/AudioEngine.RebuildDispatch.cpp L809
newDSP->convolverRt().rebuildAllIRsSynchronous(isObsolete);
```

- `rebuildThreadLoop` は Worker Thread 上で動作
- 現在は `IncrementalRebuild` ではなく `rebuildAllIRsSynchronous()` を使用
- ログの処理時間: generation=8 で build=95.9ms + rebuildIR=405.2ms

### 2.3 Worker Thread は Audio Thread から隔離されている

```
ISR Bridge Architecture:

Worker Thread (Non-RT):                     Audio Thread (RT):
     │                                            │
     ├─ RuntimeBuilder.build() ← 95ms             │ (古い世界で処理継続)
     ├─ rebuildAllIRsSynchronous() ← 405ms        │
     ├─ publishWorld() ← atomic swap (O(1)) ─────→│ 新しい世界を原子ロード
     │                                            ├─ DSPCore::process()
     │                                            ├─ Crossfade (fadeTimeSec)
     │                                            └─ 古い世界は順次 retire
```

- **405ms は Worker Thread が停止するだけ。Audio Thread はブロックされない。**
- `publishWorld()` は `WriteAccess::publishAndSwap()` による **O(1) の atomic ポインタスワップ**
- そのため、ISR Bridge Runtime の観点では **RT 安全性は維持されている**

### 2.4 DSPCore::prepare() は非RT安全

- `DSPCore::prepare()` 内の `mkl_malloc`, `ippsMalloc_8u`, aligned buffer確保はすべて PrepareToPlay / rebuildThread の non-RT 経路で実行
- RT Allocator Firewall により Audio Thread からの動的確保は禁止されている

### 2.5 CrossfadeRuntime による安全な遷移

- 新しい Runtime が publish されると、Audio Thread は `CrossfadeRuntime::start()` でクロスフェードを開始
- `runLatencyAlignedCrossfadeMixLoop()` により**遅延差を考慮した**クロスフェードを実行
- クロスフェードは `LinearRamp` で gain を 0→1 にスムージング
- 完了は `notifyFadeComplete()` → `complete()` で pending フラグをクリア
- キュー溢れ検出: `crossfadeEventDropCount_` でモニタリング

### 2.6 DSPCORE_PREPARE の各ステップ

ログに各ステップの開始・終了が出力されている:

```
ramp.prepare → oversampling.prepare → softClipOS.prepareSingleStage
→ convolverState->prepare → eqState->prepare → dcBlockers().init
→ noiseShaper prepare → outputFilter.prepare → truePeakDetector.prepare
→ loudnessMeter.prepare → setFixedLatencySamples
```

各ステップは Worker Thread で実行されるため **Audio Thread を直接ブロックしない。**

### 2.7 IppFFTPlanCache

- FFT Plan は `IppFFTPlanCache` (global singleton, order → {fftSpec, sizeWork}) でキャッシュ
- **初回のみ** `ippsFFTInit_R_64f` で計画生成 (~15ms/order)
- 2回目以降の rebuildIR では FFT plan 作成時間はかからない
- FFT/IFFT そのものは毎回実行

### 2.8 NoiseShaperLearner バッファ

- `AudioSegmentBuffer::kCapacity = 5秒 × 768000Hz = 3,840,000 samples`
- バッファ飽和は設計上の最大値到達であり、**正常動作の範囲内**
- リングバッファ (`LockFreeRingBuffer<AudioBlock, 4096>`) で Audio Thread と Worker Thread は分離済み
- **音飛びの原因ではない**

### 2.9 convDebounce / queueFull

- ログカウンタ: `req/defer/sched/trigger=0/0/0/0`, `blockP/blockR/queueFull/drain/match/fallback=0/0/0/0/0/0`
- 一度も発火していない。リビルドキューに溢れなし。

## 3. 未確定の仮説

以下の項目は現時点では**証拠不足**であり、追加計測が必要。

### 3.1 音飛びの原因候補（7つ）

ログからは特定できず、以下の候補が考えられる:

| # | 仮説 | 推定影響度 | 検証方法 |
|---|------|-----------|---------|
| ① | Crossfade 中の過渡的乱れ | 中 | `runLatencyAlignedCrossfadeMixLoop` 前後で歪み計測 |
| ② | prepareToPlay → placeholder DSP 構築 | 高 | 初回 prepareToPlay の各ステップ時間をログ化 |
| ③ | Latency 更新 (setLatencySamples) | 中 | publish 前後のレイテンシ変更値をログ化 |
| ④ | ASIO Buffer 切替時のドライバ動作 | 高 | 外部計測器またはASIOドライバログが必要 |
| ⑤ | Oversampling 変更 (OS係数切替) | 中 | OS変更時のフィルタ状態初期化を確認 |
| ⑥ | Convolver Warmup (processLayerBlock) | 低 | warmupCompleted フラグの状態をログ化 |
| ⑦ | FFT Plan 初回生成の15ms | 低 | 2回目以降との差分確認 |

### 3.2 DSPCORE_PREPARE 各処理時間

現在のログでは各ステップの**開始・終了のみ**が出力され、**各ステップの所要時間は記録されていない。** 以下の行間の時間を計測する必要がある:

```
[DSPCORE_PREPARE] calling oversampling.prepare
〜 時間不明 〜
[DSPCORE_PREPARE] oversampling.prepare done
```

### 3.3 publish 直後の Audio Thread 動作時間

- `getNextAudioBlock()` に `CallbackTelemetryScope` で処理時間計測機構は存在する
- CLI テレメトリが有効 (`isCliProcessingTelemetryEnabled()`) の場合のみ記録
- 現在のログではテレメトリ出力が確認できない → **有効化すればpublish直後のコールバック処理時間を測定可能**

### 3.4 405ms 内訳の数値精度

MixedPhase 176ms、NUC構築 228ms という数字は**ログから直接取得した値ではなく**、以下の根拠に基づく推定:

```
generation=7: build=105.2ms, rebuildIR=106.3ms
generation=8: build=95.9ms,  rebuildIR=405.2ms
```

- generation=7: rebuildIR=106.3ms（MixedPhaseのみ、NUC未構築）
- generation=8: rebuildIR=405.2ms（MixedPhase + NUC構築）

→ MixedPhase ≈ 176ms（gen=7→8の差分と MixedPhase ログ 176.3ms からの推定）
→ NUC構築 ≈ 228ms（405.2 - 176.3 ≈ 228.9ms）

**ただし**: generation=7 と generation=8 ではIR長・処理条件が異なる可能性があるため、この推定には不確定要素が含まれる。

## 4. 具体的な調査結果

### 4.1 prepareToPlay の呼び出し経路

```
ホストASIOドライバ
  └→ AudioEngineProcessor::prepareToPlay(sampleRate, samplesPerBlock)  [L39]
      └→ AudioEngine::prepareToPlay(samplesPerBlock, sampleRate)
          ├─ 既存 Runtime からプレースホルダ DSP 構築
          ├─ publishWorld(HardReset) で即時切り替え
          ├─ uiConvolverProcessor.prepareToPlay(safeSampleRate, bufferSize)
          └─ submitRebuildIntent(Structural) → rebuildThreadLoop
```

- `AudioEngine.Init.cpp` L62 の `prepareToPlay(48000.0, 512)` は暫定値
- 実際のデバイス開始後に `AudioEngine.Processing.PrepareToPlay.cpp` で正しい値で再呼び出し
- 再呼び出し時に placeholder DSP を HardReset で publish → rebuild intent 発行

### 4.2 publishWorld の動作

```cpp
// RuntimePublicationCoordinator::publishWorld()
auto* newWorld = worldOwner.release();
std::atomic_thread_fence(std::memory_order_release);
auto* oldWorld = writeAccess_.publishAndSwap(newWorld);
// → publishAndSwap は atomic なポインタスワップ (O(1))
// → Audio Thread からは RCU 読み取りで安全に参照
bridge.retireRuntimePublishWorldNonRt(oldWorld, false);
// → 旧 world は非同期に retire される
```

### 4.3 クロスフェードの動作

```
publishWorld
  └→ DSPTransition::onPublishCompleted()
      └→ CrossfadeRuntime::start(fadeTimeSec, sampleRate)
          └─ gain_.reset(sampleRate, fadeTimeSec)
          └─ gain_.setCurrentAndTargetValue(0.0)  ← 現在値0、目標値1
          └─ pending_ = true

Audio Thread の各コールバック:
  └→ armCrossfadeIfPending()
      └─ gain_.setTargetValue(1.0)
  └→ processCrossfadeDelayGateIfPending()  ← startDelayBlocks 分待機
  └→ runLatencyAlignedCrossfadeMixLoop()   ← 遅延補正 + gainスムージング
  └→ finalizeCrossfadeMixPath()             ← gain完了後、pendingクリア
```

### 4.4 startDelayBlocks の動作

```cpp
// AudioEngine.h L3165-L3179
// startDelayBlocks > 0 の場合、そのブロック数だけ
// 古い DSP のみで処理し、新しい DSP の出力を使わない
// → レイテンシ補正用の遅延バッファが十分に蓄積されるのを待つ
```

### 4.5 FFT Plan キャッシュの状態

| 呼び出し | FFT Plan 作成 | 時間 |
|---------|-------------|------|
| 初回 IR Load | `IppFFTPlanCache::getOrCreate()` で新規作成 | ~15ms |
| 2回目以降の rebuildIR | キャッシュヒット | ~0ms |
| DSPが異なる場合（別インスタンス） | 共有キャッシュから再利用 | ~0ms |

## 5. Practical Stable ISR Bridge Runtime 評価

### 5.1 合致している点

| 原則 | 状態 | 根拠 |
|------|------|------|
| Build/Publish 分離 | ✅ 完全分離 | Worker Thread で build、atomic publish |
| Audio Thread の動的確保禁止 | ✅ 担保 | RTAllocatorFirewall |
| RCU による Runtime 安全参照 | ✅ 実装済み | audioThreadRcuReader |
| クロスフェードによる遷移 | ✅ 実装済み | CrossfadeRuntime + 遅延補正 |
| Retire の非同期化 | ✅ 実装済み | ISRRetireRouter / DeferredFreeThread |

### 5.2 改善余地がある点

| 原則 | 状態 | 問題 |
|------|------|------|
| 初回buildの延期 | ⚠️ 未実装 | `prepareToPlay(48000)` で placeholder DSP を即時build |
| Incremental build | ⚠️ Dead Code | 実装は存在するが未使用 |
| FFT Plan 事前ウォームアップ | ⚠️ 未実装 | rebuildIR 経路内で初回作成 |
| publish 回数の最小化 | ⚠️ 改善余地 | REBUILD_MERGED が6回 → 過剰な可能性 |

## 6. 優先順位とアクションプラン

### 6.1 優先順位（Practical Stable ISR Bridge Runtime 観点）

| 優先度 | 項目 | 理由 |
|--------|------|------|
| **P0** | `setUseIncrementalRebuild` バグ修正 | バグであり、ISRの改善経路を塞いでいる |
| **P0** | Publish直後の DSPCore 切替時間を計測 | 音飛び原因特定の第一歩 |
| **P0** | DSPCORE_PREPARE 各処理時間をログ化 | 各ステップの時間を把握しボトルネック特定 |
| P1 | IncrementalRebuild 有効化 | Worker Thread のブロッキング時間短縮 |
| P2 | 初回 build の延期（prepareToPlay改善） | SampleRate変更時の無駄な初回build削減 |
| P3 | FFT Plan 事前ウォームアップ | rebuildIR の初回時間を削減 |

### 6.3 音飛び原因切り分け手順

```
Step 1: (A)+(D) 追加ログ出力 → Debug ビルドで実行
  ├─ DSPCORE_PREPARE のどのステップが最も時間がかかるか特定
  ├─ rebuildIR の Transform/Build 内訳を確定
  └─ Worker Thread のボトルネックを定量化

Step 2: (B) 有効化 → Release ビルドで音飛び再現
  ├─ publish 直後の Audio Thread callback 時間が異常か確認
  ├─ publish 時以外にも callback 時間が延びている箇所がないか確認
  └─ 「Worker Thread の処理時間」と「Audio Thread の乱れ」の相関を確認

Step 3: 原因特定後、適切な対策を実装
  ├─ Worker Thread 起因 → IncrementalRebuild 有効化 (+ FFT Plan 事前生成)
  ├─ Audio Thread の Crossfade 起因 → 遷移パラメータ調整
  ├─ prepareToPlay 起因 → 初回build延期
  └─ ASIO ドライバ起因 → バッファサイズ調整 / ドライバ変更
```
