---

## エグゼクティブサマリー

**音飛びの直接原因**: `gen=8` の `rebuildAllIRsSynchronous` 実行中に発生した **大規模ページフォールトストーム**（333,592回）。メモリ使用量が **1,377MB→2,484MB（+1.1GB）** に急増し、OSのメモリ管理がプロセス全体を一時停止させた。

---

## 全音飛び発生箇所の特定

ログを精査した結果、**5つの音飛び発生候補**が特定された。下記#1〜#4は MEM ログ開始前（or 直後）に密集して発生し、#5が最大の原因。

### 音飛び候補 #1: gen=1 初回ビルド（startup）

| 項目 | 値 |
|------|-----|
| 時刻 | 19:57:43 付近（ログ開始直後） |
| DSPCore prepare | 79.23ms（48kHz, 65536block） |
| BUILD_PHASE e2e | **86.9ms** |
| 再現性 | 初回起動時のみ |

startup直後の初回DSPCore構築。サンプルレート48kHzで65536ブロックと大きいが、IRなし（rebuildIR=0ms）のため軽量。ただし、初回のmkl_malloc連続（aligned buffers, EQ, 全DSPチェーン）によりページフォールトが集中する可能性あり。

### 音飛び候補 #2: gen=2 サンプルレート変更時ビルド

| 項目 | 値 |
|------|-----|
| 時刻 | 19:57:43〜19:57:44（prepareToPlay中） |
| DSPCore prepare | **99.05ms**（192kHz, OS=4x→768kHz） |
| BUILD_PHASE | なし（prepareToPlayに伴う内部再構築） |
| 再現性 | サンプルレート変更時毎回 |

`prepareToPlay` で48kHz→192kHzにレート変更。OS倍率が変わり（8x→4x→2x）、`prossingRate=768kHz` と高い。EQ_PREPARE での mkl_malloc（scratch: 4,194,304 → 大容量）がページフォールトを誘発。

### 音飛び候補 #3: gen=4 2回目のビルド（IRなし）

| 項目 | 値 |
|------|-----|
| 時刻 | 19:57:44 付近 |
| DSPCore prepare | 81.49ms |
| BUILD_PHASE e2e | **89.0ms** |
| IR | なし（rebuildIR=0ms） |
| 再現性 | startup時の再調整 |

IRなしでDSPCoreのみの再構築。gen=1と同等の軽量ビルドだが、この時点で新旧2つのDSPCoreが共存する可能性あり。

### 音飛び候補 #4: gen=7 初回IRビルド（破棄された中間ビルド）<-- NEW!

| 項目 | 値 |
|------|-----|
| 時刻 | 19:57:48〜19:57:49（MEMログ開始後） |
| DSPCore prepare | 91.02ms + 90.69ms（2回連続） |
| build | 97.7ms |
| rebuildIR | **111.3ms**（IR転送 + NUC初期化） |
| MixedPhase | **なし** |
| BUILD_PHASE | **なし（破棄されてpublishに至らず）** |
| 再現性 | IRロード＋構造変更時毎回 |

**重要**: `[BUILD_PHASE]` が出力されていない = publishに至らなかった。gen=8の到来により gen=7 は `isObsolete()` で破棄された（`continue`）。にもかかわらず、gen=7は DSPCore構築(97.7ms) + IR再ビルド(111.3ms) = **合計209ms** ものWorker Thread時間を消費。この間、NUCエンジンのメモリ確保＋ページフォールトが発生し、**無駄なメモリプレッシャー**を生んだ。

### 音飛び候補 #5: gen=8 最終IRビルド + MixedPhase（最大原因）

| 項目 | 値 |
|------|-----|
| 時刻 | 19:57:49〜19:57:51 |
| DSPCore prepare | 91.13ms |
| build | 97.3ms |
| rebuildIR | **422.7ms**（MixedPhase 185.9ms含む） |
| BUILD_PHASE e2e | **523.5ms** ← 最大 |
| MixedPhase | **185.9ms**（GreedyAdaGrad, 12freq, maxIter=4） |
| メモリ増加 | **1,377MB→2,484MB（+1,107MB）** |
| PageFault増加 | **349,100→718,363（+369,263回）** |
| 再現性 | IRファイル変更＋MixedPhase最適化時毎回 |

**最大の原因かつ唯一 MEM ログで全貌を追跡できた事象**。MixedPhase最適化(185.9ms)を含む422.7msのrebuildIRと、新旧DSPCore共存による1.1GBのメモリ急増が同時発生し、OSレベルのメモリ圧力→Audio Threadコールバック遅延→音飛びに至った。

## 根本原因：4つの増幅因子

### 因子①：新旧DSPCore同時存在（最重要）

`RuntimeBuilder::build()` が新DSPCoreの全メモリを確保するが、旧DSPCoreはAudio Threadが参照中のため解放できない。

```
[状態]               [旧DSPCore]  [新DSPCore]  [合計メモリ]
gen=4 安定稼働中       ~1.3GB       —            ~1.3GB
gen=8 build開始       ~1.3GB       ~0.3GB(構築中) ~1.6GB
gen=8 rebuildIR中     ~1.3GB       ~1.0GB(増加中) ~2.3GB
gen=8 publish直後     ~1.3GB       ~1.2GB(完成)   ~2.5GB  ← ピーク
gen=8 retire開始      ~0.5GB(退役中) ~1.2GB       ~1.7GB
gen=8 retire完了      —            ~1.2GB        ~1.2GB
```

新DSPCoreのNUCエンジン初期化だけで約1.2GBのメモリが必要。旧DSPCoreが同時に存在することで最大2.5GBに達する。

**関連コード**:
- `src/audioengine/RuntimeBuilder.cpp` L459-502: `RuntimeBuilder::build()` → DSPCore新規生成
- `src/ConvolverProcessor.h` L632-742: `StereoConvolver::init()` → mkl_malloc

### 因子②：MixedPhase 全量最適化（高コスト）

`src/convolver/ConvolverProcessor.MixedPhase.cpp` L394-479:

```cpp
// ライブ再構成かどうかの判定
const bool liveReconfigure = (owner != nullptr) &&
    convo::consumeAtomic(owner->isPrepared, ...) && hasPreviousIR;

// ★ 初回IRロード時は liveReconfigure==false となり、全量最適化が走る
const int optimFreqPoints = liveReconfigure ? (highRateLive ? 12 : 64) : 256;
                                          //  通常ライブ=64    初回=256
designer_config.numSections = liveReconfigure ? (highRateLive ? 2 : 8) : 20;
designer_config.cmaesMaxGenerations = liveReconfigure ? (highRateLive ? 3 : 12) : 160;
designer_config.cmaesPopulationSize = liveReconfigure ? (highRateLive ? 6 : 12) : 64;
```

初回IRロード時:
- `cmaesMaxGenerations=160` × `populationSize=64` = **10,240回のMKL FFT評価**
- `numSections=20` の allpass フィルタ設計
- 各評価: `fftSize = nextPowerOfTwo(numSamples * 4)` のFFT

これにより **185.9ms** ものCPU専有時間が発生する。

### 因子③：NUCエンジン mkl_malloc 爆発

`src/MKLNonUniformConvolver.cpp` の `SetImpulse()` 内部:

- 3レイヤー（L0/L1/L2）の非一様パーティショニング
- 各レイヤーごとに以下のバッファを mkl_malloc:
  - `irFreqDomain`（numParts × partStride doubles）
  - `irFreqReal/irFreqImag`（numParts × complexSize）
  - `fdlBuf/fdlReal/fdlImag`
  - `fftTimeBuf/fftOutBuf`
  - `prevInputBuf/accumBuf/accumReal/accumImag`
- これらを2チャンネル分

IR長25,906サンプル × 192kHz の場合、チャンネルあたり約**600MB**確保。

`mkl_malloc` は Win32 `VirtualAlloc` を使用するため、初回アクセス時にハードページフォールトが発生する。

### 因子④：DeferredFreeThread 解放速度不足

`src/DeferredFreeThread.h` L75:

```cpp
static constexpr int kMaxReclaimPerLoop = 4;
```

退役ループ（L112-L131）:
```cpp
while (auto* ptr = swapperRef.tryReclaim(minEpoch)) {
    delete ptr;
    if (++reclaimCount >= kMaxReclaimPerLoop) break;  // ← 4回で中断
}
if (reclaimCount == 0)
    std::this_thread::sleep_for(1ms);  // ← アイドル時は1msスリープ
```

- 1ループあたり最大4個しか解放しない
- アイドル時は1msスリープ（再開が遅れる）
- `SafeStateSwapper` のリングバッファ `kMaxRetired=256` を超えると fallback queue へ

結果: 289エントリの退役待ちが発生し、メモリ解放が著しく遅延。

## XRUN未検出の理由

| 診断ログ | 有無 | 理由 |
|---------|------|------|
| `[XRUN#N]` | ❌ なし | Audio ThreadはWorker Threadと別スレッドで動作。Workerの523ms占有はAudio Threadのcallback時間に直接影響しない。 |
| `[BACKPRESSURE]` | ✅ あり | retireDepth=1→0 と正常範囲。退役の滞留は軽微。 |
| `[MEM]` | ✅ あり | 1秒ごとのPageFault急増を正確に記録。 |
| `[ACTIVATE]` | ❌ なし | gen=4→5 の切替はログ出力タイミングと一致していた可能性。 |
| `[BUILD_PHASE]` | ✅ あり | gen=8 e2e=523.5ms を正確に記録。 |

**結論**: 音飛びはAudio Threadの処理遅延ではなく、**Worker Threadのmkl_malloc/VirtualAllocによるページフォールトがOSレベルでプロセス全体を停止させた**ために発生。XRUN閾値（callback時間・interval超過）では捕捉できない種類のドロップアウト。

### 各ログの詳細設計

#### A. PageFault Delta + 急増警告（最優先）

現在の MEM ログ（1秒1回）に PageFault の前回差分を追記。合わせて閾値超過時に警告行を追加する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: MEMログ自体が1秒1回。Delta計算のための前回値保持のみで追加出力はなし。閾値超過時の警告行も1秒1回が上限。

```cpp
// Timer.cpp MEMブロック内、getProcessMemoryInfo() 呼び出し後に追加
static uint64_t prevPageFaults = 0;
const uint64_t pfDelta = (prevPageFaults > 0)
    ? (memInfo.pageFaultCount - prevPageFaults)
    : 0;
prevPageFaults = memInfo.pageFaultCount;

// 既存MEM行に Delta を追記
// 変更前: "PageFaults=349100"
// 変更後: "PageFaults=349100 Delta=+856"

// 閾値超過時のみ別枠で警告（50,000/秒 = 約200MB/秒の新規コミットに相当）
if (pfDelta > 50000) {
    diagLog(diagPrefix(gen) + " [WARN] PageFault surge: +"
        + juce::String(static_cast<juce::int64>(pfDelta)) + " faults in 1s");
}
```

---

#### B. TimerCallback 実行時間

timerCallback の開始〜終了までの経過時間を計測。Timer Thread 自体の負荷が可視化される。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: 毎tick（100ms毎）出力。ただし1行のみで十分軽量。timerCallbackの処理時間増大はXRUNにも影響するため、基本統計として常時取得に値する。

```cpp
// timerCallback() 先頭（emitEvidenceTickNonRt より前）
const auto cbStartTicks = juce::Time::getHighResolutionTicks();

// ... 既存の全処理 ...

// timerCallback() 末尾、} の直前（#endif の直前）
const auto cbElapsedTicks = juce::Time::getHighResolutionTicks() - cbStartTicks;
const double cbElapsedMs = (static_cast<double>(cbElapsedTicks) * 1000.0)
    / static_cast<double>(juce::Time::getHighResolutionTicksPerSecond());
diagLog(diagPrefix(gen) + " [DIAG] timerCb=" + juce::String(cbElapsedMs, 3) + "ms");
```

#### C. Obsolete ビルド浪費検出

rebuildThreadLoop 内で `isObsolete()` によりビルドが破棄された時点で、そこまでに費やした時間を記録する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: 発生時のみ。ビルド破棄は頻繁には起こらない（数分に1回程度）。

```cpp
// 各 isObsolete() の continue; の直前に追加
const double wastedMs = juce::Time::getMillisecondCounterHiRes() - buildStartMs;
diagLog(diagPrefix(0) + " [DIAG] rebuild obsolete at <phase> gen="
    + juce::String(task.generation)
    + " wasted=" + juce::String(wastedMs, 1) + "ms");
```

追加箇所（4箇所）:
1. Prepare直後（`if (isObsolete()) continue;` 前）- L795付近
2. rebuildIR前（`if (isObsolete()) continue;` 前）- L807付近
3. warmup後（`if (isObsolete()) continue;` 前）- L827付近
4. commit前（`if (isObsolete()) continue;` 前）- L867付近

#### D. Timer jitter 監視

timerCallback の実際の呼び出し間隔を計測し、期待値100msからの乖離が大きい場合のみ警告。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: 5ms以上の乖離があったときのみ出力。正常動作中は一切出力されない。

```cpp
// timerCallback() 先頭
static double prevCallbackMs = 0.0;
const double nowMs = juce::Time::getMillisecondCounterHiRes();
if (prevCallbackMs > 0.0) {
    const double actualIntervalMs = nowMs - prevCallbackMs;
    const double jitterMs = actualIntervalMs - 100.0;
    if (std::abs(jitterMs) > 5.0) {
        diagLog("[DIAG] timer jitter: interval="
            + juce::String(actualIntervalMs, 2) + "ms (expected=100ms)");
    }
}
prevCallbackMs = nowMs;
```

#### E. Worker フェーズ別メモリ

rebuildThreadLoop 内の各重要フェーズで `getProcessMemoryInfo()` を呼び、メモリ増加がどの段階で発生するかを特定する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS` + `getProcessMemoryInfo()` の RebulsDispatch.cpp への移植が必要

**抑制**: ビルド1回につき最大5回。IRロードが頻繁に発生しない限り、数分に1回程度。

```cpp
// 移植: RebuildDispatch.cpp に getProcessMemoryInfo() を追加（Timer.cpp からコピー）
// 各フェーズで以下を追加:
// Prepare完了直後:
//   const auto memAfterBuild = getProcessMemoryInfo();
// rebuildIR完了直後:
//   const auto memAfterIR = getProcessMemoryInfo();
// commit直前:
//   const auto memBeforeCommit = getProcessMemoryInfo();

// 出力:
diagLog("[MEM] buildPhase gen=" + juce::String(task.generation)
    + " afterBuild=" + juce::String(static_cast<juce::int64>(memAfterBuild.privateUsageMB)) + "MB"
    + " afterIR=" + juce::String(static_cast<juce::int64>(memAfterIR.privateUsageMB)) + "MB");
```

#### F. クロスフェード完了時間

既存の `CrossfadeRuntime::getFadeAgeUs()` と `getQueuedFadeTimeSec()` を利用して、クロスフェードの実所要時間を記録する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: クロスフェード完了時のみ。IRロード・publishのたびに最大1回。頻度は低い。

```cpp
// Timer.cpp の fade 完了ブロック内（L474-522付近）
const double elapsedSec = static_cast<double>(crossfadeRuntime_.getFadeAgeUs()) / 1'000'000.0;
const double expectedSec = crossfadeRuntime_.getQueuedFadeTimeSec();
diagLog(diagPrefix(gen) + " [DIAG] crossfade done: elapsed="
    + juce::String(elapsedSec, 3) + "s"
    + " expected=" + juce::String(expectedSec, 3) + "s");
```

#### G. RetireQueue 定期ログ（1秒毎）

現在の `BACKPRESSURE` ログは「変化時のみ」出力。これを1秒毎の定期出力に変更する。

**制御**: `CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS`（既存）

**抑制**: 1秒1回。変化時のみ→毎秒に変更することでログ量は増えるが、1行/秒は許容範囲。

```cpp
// 現在:
// if (backpressure.retireQueueDepth != rtAuxMutable_.debugLastReportedRetireQueueDepth || ...) { ... }

// 変更後: 周期カウンタで1秒毎に制御
static int bpTick = 0;
if (++bpTick >= 10) {  // 10tick = 1秒
    bpTick = 0;
    rtAuxMutable_.debugLastReportedRetireQueueDepth = backpressure.retireQueueDepth;
    // ... 既存のログ出力 ...
}
```
