## 3. 原因① IncrementalRebuild 未使用

### 3.1 発見事項

**`setUseIncrementalRebuild` にバグがあり、かつ呼び出し元が存在しない。**

```cpp
// src/ConvolverProcessor.h L792
std::atomic<bool> useIncrementalRebuild { false };  // デフォルト false

// src/convolver/ConvolverProcessor.Rebuild.cpp L274-L278
void ConvolverProcessor::setUseIncrementalRebuild(bool enable) noexcept
{
    // ★ BUG: enable 引数を無視して常に false を書き込んでいる
    convo::publishAtomic(useIncrementalRebuild, false, std::memory_order_release);
}
```

- `setUseIncrementalRebuild(true)` を呼び出すコードが **どこにも存在しない**
- コンパイルガード `CONVOPEQ_ENABLE_CONVOLVER_SPLIT_REBUILD=1` は CMakeLists.txt で定義済み
- `IncrementalRebuildJob` 構造体と `beginIncrementalRebuild`/`advanceIncrementalRebuild`/`runIncrementalBuildStep`/`runIncrementalFinalizeStep` の完全な実装は存在するが、**呼ばれていない**

### 3.2 現在の動作パス

```
rebuildThreadLoop
  └→ convolverRt().rebuildAllIRsSynchronous()    ← 同期的・全ステップ逐次実行
      └→ LoaderThread.runSynchronously()
          ├── doLoadIRStep()      ▶ ~1ms
          ├── doTrimStep()        ▶ ~1ms
          ├── doTransformStep()   ▶ MixedPhase ~176ms    ★
          └── doBuildStep()       ▶ NUC構築 ~228ms       ★
          Total: ~405ms
```

### 3.3 期待されるインクリメンタルパス（未使用）

```
rebuildThreadLoop 内でループ呼び出し:
  while (advanceIncrementalRebuild() == IncrementalRebuildSliceResult::InProgress)
  {
      // 各スライスは短時間で完了 → Audio Thread のブロッキング最小化
  }
```

### 3.4 修正方法

**ファイル: `src/convolver/ConvolverProcessor.Rebuild.cpp`**

```cpp
// 修正前 (L277):
convo::publishAtomic(useIncrementalRebuild, false, std::memory_order_release);

// 修正後:
convo::publishAtomic(useIncrementalRebuild, enable, std::memory_order_release);
```

**ファイル: `src/audioengine/AudioEngine.RebuildDispatch.cpp`**

`rebuildThreadLoop` 内で、`rebuildAllIRsSynchronous` を呼び出す代わりにインクリメンタルパスを使用するよう変更。

## 4. 原因② rebuildIR 405ms の内訳

### 4.1 LoaderThread 処理時間内訳

| ステップ | 関数 | 時間 | 割合 |
|---------|------|------|------|
| LoadIR | `doLoadIRStep()` | ~1ms | 0.2% |
| Trim | `doTrimStep()` | ~1ms | 0.2% |
| **Transform** | **`doTransformStep()`** | **~176ms** | **43.5%** |
| **Build** | **`doBuildStep()`** | **~228ms** | **56.3%** |
| **Total** | | **~405ms** | **100%** |

### 4.2 MixedPhase 176ms の詳細

`doTransformStep()` → `convertToMixedPhase()` → `convertToMixedPhaseAllpass()`

| サブ処理 | 時間 | 備考 |
|---------|------|------|
| AllpassDesigner (GreedyAdaGrad) | ~140ms | maxIter=4, freqPoints=12, numSections=2 |
| AllpassResponse 計算 (FFT) | ~15ms | fftSize=262144 クラス |
| 混合位相IR合成 (IFFT) | ~15ms | |
| スケール計算 | ~6ms | |
| **Total** | **~176ms** | |

**GreedyAdaGrad パラメータ:**

```cpp
// src/convolver/ConvolverProcessor.MixedPhase.cpp L501-L509
designer_config.numSections = liveReconfigure ? (highRateLive ? 2 : 8) : 20;
designer_config.method = convo::OptimizationMethod::CMAES;
// ...
if (preferGreedyForLive)
{
    designer_config.method = convo::OptimizationMethod::GreedyAdaGrad;
    designer_config.maxIterations = 4;
    designer_config.learningRate = 0.006;
}
```

- ライブ時: GreedyAdaGrad, maxIter=4, numSections=2
- 非ライブ時（初回IR読み込み）: CMAES, maxGenerations=160, population=64, numSections=20

### 4.3 NUCエンジン構築 228ms の詳細

`doBuildStep()` → `buildConvolverFromTrimmed()` → `initializeConvolverSynchronously()` → `StereoConvolver::init()` → `MKLNonUniformConvolver::SetImpulse()` × 2ch

```cpp
// src/MKLNonUniformConvolver.cpp L492
bool MKLNonUniformConvolver::SetImpulse(const double* impulse, int irLen,
    int blockSize, double scale, bool enableDirectHead, const FilterSpec* filterSpec)
```

| サブ処理 | 時間 | 備考 |
|---------|------|------|
| DirectForm 設定 (先頭32taps) | ~5ms | mkl_malloc + memcpy |
| Layer構成決定 | ~1ms | L0/L1/L2 パーティション分割 |
| IppFFTPlanCache 取得 | ~15ms | **初回のみ** `ippsFFTInit_R_64f` |
| IR FFT (L0/L1/L2) | ~80ms | `ippsFFTFwd_R_64f` × 2ch × 3Layer |
| SpectrumFilter 適用 | ~40ms | TailProfile, Crossfade |
| ワークバッファ確保 | ~20ms | `ippsMalloc_8u` |
| Warmup | ~60ms | `processLayerBlock` 状態初期化 |
| **Total per ch** | **~114ms** | |
| **Total × 2ch** | **~228ms** | |

**IppFFTPlanCache** はグローバルシングルトンで FFT 計画をキャッシュ:
```cpp
// src/MKLNonUniformConvolver.cpp L84-L132
class IppFFTPlanCache {
    static std::unordered_map<int, std::unique_ptr<IppFFTPlan>> cacheStorage_;
    // order → {fftSpec, sizeWork} のマップ
};
```

- 初回のみ `ippsFFTInit_R_64f` で計画生成 (~15ms)
- 2回目以降はキャッシュヒット（時間ほぼゼロ）
- ただし MKL メモリ確保 (`mkl_malloc`) と FFT/IFFT、Warmup は毎回実行

### 4.4 音飛び発生メカニズム

```
rebuildThread (Worker Thread):           Audio Thread (Real-time):
     │                                        │
     │-- rebuildAllIRsSynchronous() --│        │ (古いRuntimeで処理継続)
     │   ├─ ~176ms MixedPhase         │        │
     │   ├─ ~228ms NUC構築           │        │
     │   └─ ~405ms 経過              │        │
     │                                        │
     │-- trySubmit: publish SUCCEEDED ────────│→ Runtime切り替え
     │                                        │   DSPCore_PREPARE
     │                                        │   ランプ/クロスフェード
     │                                        │   ★ 過渡的な音の乱れ
```

**重要**: rebuildThread は Audio Thread と別スレッドのため、405ms の間も Audio Thread は古い Runtime で処理を継続できる。音飛びは **publish 後の Runtime 切り替え + DSPCore_PREPARE + クロスフェード遷移時** に発生する。

## 5. 原因③ 起動時処理レート変動

### 5.1 問題箇所

```cpp
// src/audioengine/AudioEngine.Init.cpp L69
uiConvolverProcessor.prepareToPlay(48000.0, 512);  // 暫定値 48kHz
```

この後、実際のオーディオデバイス開始時に192kHzに変更される。

### 5.2 リビルド連鎖の詳細

| 順序 | トリガー | Generation | DSPCORE_PREPARE |
|------|---------|-----------|----------------|
| 1 | `prepareToPlay(48k, 512)` → rebuild gen=1 | gen=1 | #1 (sr=48k, spb=65536) |
| 2 | デバイス開始 `prepareToPlay(192k, 1024)` | gen=1→4 | #2 (sr=192k, **768kHz処理**) |
| 3 | オーバーサンプリング有効化 (osFactor 0→2) | gen=2→4 | #3 (sr=192k, 384kHz処理) |
| 4 | convolverParamsChanged (#1) | gen=5→6 | #4 (sr=192k, 384kHz) |
| 5 | convolverParamsChanged (#2, impulse_hpf) | gen=7 | #5 (sr=192k, 384kHz) |
| 6 | DEFERRED 解放 → 最終確定 | gen=8 | #6 (sr=192k, 384kHz) |
| | **publish成功** | **rev=5** | |

各 DSPCORE_PREPARE で以下をフル再初期化:
```
ramp → oversampling → softClipOS → convolver → EQ → DCBlocker
→ noiseShaper → outputFilter → truePeakDetector → loudnessMeter
```

### 5.3 修正方法

```cpp
// 修正案: 暫定値を実運用レートに近づける
// AudioEngine.Init.cpp L69:
// 修正前:
uiConvolverProcessor.prepareToPlay(48000.0, 512);
// 修正後:
uiConvolverProcessor.prepareToPlay(192000.0, 1024);
```

この変更により、起動時の gen=1→4 までの 4 回の DSPCORE_PREPARE 連鎖を削減できる可能性がある。

## 6. 正常動作確認項目

### 6.1 DEFERRED 200ms window

```cpp
// AudioEngine.UIEvents.cpp L134
const int64_t minDeltaTicks = juce::Time::getHighResolutionTicksPerSecond() / 5; // 200ms
```

- `convolverParamsChanged` が `PreparedIRApplyWindow` (200ms以内) に再度呼ばれた場合に Structural rebuild を延期
- 200ms後、`timerCallback` (100ms polling) で `dueTicks >= nowTicks` をチェックして解放
- ログ上も正常動作: `REBUILD_DEFERRED → REBUILD_DEFERRED (released) → REBUILD_REQUESTED`
- 最大遅延: 200ms + 100ms (timer polling) = 最大300ms
- **連鎖リビルドを抑制する意図通りの動作で問題なし**

### 6.2 NoiseShaperLearner バッファ飽和

```
AudioSegmentBuffer::kCapacity = 5 [秒] × 768000 [Hz] = 3,840,000 [samples]
```

- ログの `bufferedSamples=3840000` は **設計上の最大値到達＝正常**
- リングバッファがフルの場合、新しいブロックは古いブロックを上書き（古いデータは破棄）
- ワーカースレッドで非同期動作 → Audio Thread に影響なし
- **音飛びの原因ではない**

### 6.3 convDebounce

- ログ: `convDebounce(req/defer/sched/trigger)=0/0/0/0`
- 1 度も発火していないため影響なし

### 6.4 キュー状態

- `blockP/blockR/queueFull/drain/match/fallback` = 全て 0
- リビルドキューに溢れや輻輳は発生していない

## 7. 総合評価と改善優先順位

### 7.1 影響度マッピング

```
音飛びの程度
  │
  │  ★ 原因② rebuildIR 405ms (80%)
  │  │
  │  │  ★ 原因③ 起動時連鎖 (15%)
  │  │  │
  │  │  │  ★ NoiseShaperLearner (5%)
  │  │  │  │
  └──┴──┴──┴──→ 発生頻度（起動時＝高、運用中＝低）
```

### 7.2 改善優先順位

| 優先度 | 項目 | 効果 | 難易度 | ファイル |
|--------|------|------|--------|---------|
| **P0** | `setUseIncrementalRebuild` バグ修正 | rebuildIR分割 | 低 | `ConvolverProcessor.Rebuild.cpp` |
| **P1** | IncrementalRebuild を rebuildThreadLoop で使用 | 405msブロッキング解消 | 中 | `AudioEngine.RebuildDispatch.cpp` |
| **P2** | `prepareToPlay` 暫定値を192kHzに変更 | 起動時連鎖削減 | 低 | `AudioEngine.Init.cpp` |
| **P3** | IppFFTPlanCache の事前ウォームアップ | rebuildIR 15ms削減 | 低 | `AudioEngine.Init.cpp`, `MKLNonUniformConvolver.cpp` |
| P4 | MixedPhasePersistentCache 活用 | 同一IR再リビルド高速化 | 中 | `ConvolverProcessor.MixedPhase.cpp` |
| P5 | rebuildDebounceMs 調整 | パラメータ変更の集約 | 低 | `ConvolverProcessor.h` |

|------|-------------|-------------------|------------|
| P0+P1 | 405ms → スライス分割（各~50ms） | - | **~60%** |
| P2 | - | 6回→2回 | **~20%** |
