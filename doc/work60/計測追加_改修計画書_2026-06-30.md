## 設計原則

### 共通キー
全ログに callbackIndex, gen, seq を付与。

### budget + expectedUs
全時間計測に budget% と expectedIntervalUs（uint32_t）の両方を付与。

### 共有データの整数化
リングバッファの budget は uint16_t permille、expectedIntervalUs は uint32_t。

### memory_order
- 診断専用値: 全て memory_order_relaxed
- リングバッファ commit sequence: store(release) / load(acquire)
- callbackTimingWriteCount: load(relaxed) — commit sequence が整合性を保証

### sequence 設計
リングバッファの commit sequence 値には callbackIndex を使用（v7）。
writeCount, callbackIndex, sequence の3種類を callbackIndex に統一し、ログ解析時の相関を容易にする。

**将来拡張性**: 現在は callbackIndex で問題ないが、将来 callbackIndex の採番方法が変わった場合に備え、
commit sequence は独立した writeCount ベースの世代番号として分離する設計も選択肢として考慮すること
（sequence本来の役割は「slotの更新世代」であり、callbackIndexは副次的情報）。

## B: Callback処理時間 リングバッファ（32件 + callbackIndex sequence）

### v6からの改善点
- callbackTimingWriteCount の読取: consumeAtomic → **load(relaxed)**
  （commit sequence の release/acquire が整合性を保証するため writeCount 自体に acquire は不要）
- commit sequence 値: writeCount (wc) → **callbackIndex**
  （sequence, callbackIndex, writeCount を callbackIndex に統一し解析性向上）

### 構造体定義
```cpp
static constexpr size_t kCallbackTimingSlots = 32;
struct CallbackTimingEntry {
    uint64_t callbackIndex = 0;
    uint64_t processTimeUs = 0;
    int64_t driftUs = 0;
    uint32_t cpu = UINT32_MAX;
    uint16_t budgetPermille = 0;
    uint32_t expectedIntervalUs = 0;
    std::atomic<uint64_t> sequence{0};  // ★ callbackIndex を格納
};
CallbackTimingEntry callbackTimingHistory[kCallbackTimingSlots];
std::atomic<uint64_t> callbackTimingWriteCount{0};
```

### 書き込み側
```cpp
const uint64_t wc = convo::fetchAddAtomic(
    rtLocalState_.callbackTimingWriteCount, uint64_t{1}, std::memory_order_relaxed);
const size_t idx = static_cast<size_t>((wc - 1) % kCallbackTimingSlots);
auto& entry = rtLocalState_.callbackTimingHistory[idx];
// 1) 非atomicフィールドを先に書く
entry.callbackIndex = thisCallbackIndex;
entry.processTimeUs = processTime;
entry.driftUs = rtLocalState_.lastCallbackDriftUs.load(std::memory_order_relaxed);
entry.cpu = rtLocalState_.lastCallbackProcessor.load(std::memory_order_relaxed);
entry.budgetPermille = static_cast<uint16_t>(
    std::min(999.0, std::round(budgetPct * 10.0)));
entry.expectedIntervalUs = static_cast<uint32_t>(expectedIntervalUs);
// 2) 最後に sequence を更新（commit）— ★ callbackIndex を使用
entry.sequence.store(thisCallbackIndex, std::memory_order_release);
```

### 読み取り側（XRUN時）
```cpp
if (xrunDetected) {
    // ★ load(relaxed) — commit sequence が整合性を保証
    const uint64_t wc = rtLocalState_.callbackTimingWriteCount.load(
        std::memory_order_relaxed);
    const uint64_t start = (wc > kCallbackTimingSlots)
        ? wc - kCallbackTimingSlots : 0;
    for (uint64_t i = start; i < wc; ++i) {
        const size_t idx = i % kCallbackTimingSlots;
        const auto& e = rtLocalState_.callbackTimingHistory[idx];
        const uint64_t seq = e.sequence.load(std::memory_order_acquire);
        if (seq == 0) continue;
        // ★ seq は callbackIndex なので entry.callbackIndex と直接比較可能
        if (e.sequence.load(std::memory_order_acquire) == seq) {
            juce::Logger::writeToLog(
                "[CB_HIST] idx=" + juce::String((int64_t)seq)
                + " proc=" + juce::String((int64_t)e.processTimeUs)
                + " drift=" + juce::String((int64_t)e.driftUs)
                + " cpu=" + juce::String((int)e.cpu)
                + " budget="
                + juce::String(static_cast<double>(e.budgetPermille) / 10.0, 1)
                + "% expected=" + juce::String((int64_t)e.expectedIntervalUs));
        }
    }
}
```

## 仮説判定マトリクス

| 観測パターン | 示唆する原因 |
| --- | --- |
| DriftUs大 + CPU_MIG有 | OSスケジューラ問題 |
| DriftUs大 + CPU_MIG無 | OS/Driver DPC/ISR問題 |
| DriftUs小 + CONV_TIME大 + STCONV_TIME大 | FFT畳み込み遅延 |
| DriftUs小 + CONV_TIME大 + STCONV_TIME小 | Convolver管理処理遅延（RCU等） |
| DriftUs小 + EQ_TIME大 | EQ IIRフィルタ遅延 |
| DriftUs小 + CB_HIST漸増傾向 + budget増加 | メモリプレッシャー |
| DriftUs小 + CB_HIST急増 + CPU_MIG有 | OS要因（コンテキストスイッチ） |
| 全計測値小 + budget余裕あり | 偶発的OSジッター |
