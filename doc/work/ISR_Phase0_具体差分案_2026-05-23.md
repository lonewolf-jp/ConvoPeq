## 0. 目的

Phase 0（Snapshot整合性の一本化）を、実装に直結する粒度で定義する。

- 変更対象: 構造体1件 + 関数群
- 出力: ファイル別の具体編集点、差分イメージ、適用順

対象不具合:

- N1: 1ブロック内 snapshot 二重取得
- N2: Audio Thread で control reader index 使用
- K: snapshot処理で寿命検証レイヤ不一致

---

## 1. 変更サマリ（最小差分）

### 1.1 構造体変更

- `src/audioengine/AudioEngine.h`
  - `AudioEngine::DSPCore::ProcessingState` を拡張

- `const convo::EQParameters* eqParams = nullptr;`

設計意図:

- DSPCore 内で snapshot を再取得せず、呼び出し元で確定した EQ 参照情報を注入する。

## 2. ファイル別 具体差分案

## 2.1 `AudioEngine.h`（構造体とビルダー）

### 編集ポイント（AudioBlock）

1. `DSPCore::ProcessingState` へ EQ注入フィールド追加
2. `buildAudioThreadProcessingState(...)` の生成ロジックに EQ注入パラメータを追加

### 差分イメージ（概念）

- `ProcessingState` 末尾に以下を追加:
  - `eqParams / eqCache / eqCoeffHash / eqFromSnapshot`
- `buildAudioThreadProcessingState` に以下入力を追加:
  - `const convo::EQParameters* eqParamsToUse`
  - `const EQCoeffCache* eqCacheToUse`
  - `uint64_t eqCoeffHash`
  - `bool eqFromSnapshot`

### 注意

- 既存呼び出し箇所が全て新引数を渡すまでコンパイルが通らないため、
  ヘッダ修正と呼び出し元修正は同一コミットで行う。

---

## 2.2 `AudioEngine.Processing.AudioBlock.cpp`（float入口）

### 編集ポイント

`getNextAudioBlock(...)` 内で snapshot を1回取得した時点で、EQ参照も確定して `procState` に注入する。

### 具体手順（AudioBlock）

1. 既存の
   - `observedSnapshot = m_coordinator.observeCurrentRuntime(kAudioEpochReaderIndex)`
   を維持。
2. `snap` から `eqCoeffHash` と `eqParams` を取り、`eqCacheManager.get(hash)` を解決。
3. `buildAudioThreadProcessingState(...)` 呼び出し時に EQ情報を渡す。
4. 以降の DSPCore 側で再取得しない前提にする。

### DoD直結チェック（AudioBlock）

- 本関数内の snapshot 取得は1回のみ。
- `procState` の EQ情報が null でない（通常系）。

---

## 2.3 `AudioEngine.Processing.BlockDouble.cpp`（double入口）

### 編集ポイント（BlockDouble）

`processBlockDouble(...)` でも float と同じ責務を持たせる。

### 具体手順（BlockDouble）

1. `observedSnapshot = m_coordinator.observeCurrentRuntime(kAudioEpochReaderIndex)` を起点に EQ情報を確定。
2. `buildAudioThreadProcessingState(...)` へ注入。
3. 後段 `DSPCore::processDouble(...)` に渡す state は「完成済み状態」のみ。

### DoD直結チェック（BlockDouble）

- double経路で `kControlEpochReaderIndex` 使用なし。
- EQ情報は入口でのみ確定。

---

## 2.4 `AudioEngine.Processing.DSPCoreFloat.cpp`（再取得廃止）

### 編集ポイント（DSPCoreFloat）

`DSPCore::process(...)` 内の以下ブロックを削除/置換:

- `ownerEngine->observeCurrentRuntime()`
- `snap->eqCoeffHash` 経由の `eqParamsToUse/eqCacheToUse` 解決

### 置換方針（DSPCoreFloat）

- `state.eqParams` / `state.eqCache` を直接使用。
- fallback は原則入口で解決済みのため不要。
  （防御として `state.eqParams==nullptr` 時に既存の `eqRt().process(processBlock)` フォールバックは維持可）

### DoD直結チェック（DSPCoreFloat）

- 本関数内に `observeCurrentRuntime` 呼び出しが残らない。

---

## 2.5 `AudioEngine.Processing.DSPCoreDouble.cpp`（再取得廃止）

### 編集ポイント（DSPCoreDouble）

`DSPCore::processDouble(...)` も float と同様に再取得を削除。

### 置換方針（DSPCoreDouble）

- `state.eqParams` / `state.eqCache` / `state.eqCoeffHash` を使用。
- `eqRt().process(...)` 呼び出し分岐を state駆動へ一本化。

### DoD直結チェック（DSPCoreDouble）

- 本関数内 `ownerEngine->observeCurrentRuntime()` ゼロ。

---

## 2.6 `AudioEngine.Processing.Snapshot.cpp`（寿命検証レイヤ統一）

### 編集ポイント（Snapshot）

`processWithSnapshot(...)` の active/fading 解決方針を通常経路と整合させる。

### 具体手順（Snapshot）

1. `activeNode` 直参照優先を廃止し、resolverベースを優先。
2. old/new同一実体検出を追加:
   - `if (isFadingTarget && dsp == active)` 相当を検知したら
   - `jassertfalse` + safe fallback（dry-old or clear）
3. isFadingTarget かつ fading未解決時の fallback を明示。

### DoD直結チェック（Snapshot）

- old/new 同一実体を許容しない。
- snapshot経路の寿命検証方針が通常経路と一致。

---

## 3. 変更適用順（推奨コミット順）

1. `AudioEngine.h` の `ProcessingState` 拡張（呼び出し箇所未修正で赤になる前提）
2. `getNextAudioBlock` / `processBlockDouble` の state 注入対応
3. `DSPCore::process` / `processDouble` の再取得削除
4. `processWithSnapshot` の resolver統一と old/new同一禁止
5. ビルド・スキャン・hash一致テスト

## 4. 検証項目（Phase 0専用）

## 4.1 静的確認

- `src/audioengine/**` で Audio Thread 経路の `kControlEpochReaderIndex` 使用ゼロ
- `DSPCoreFloat.cpp` / `DSPCoreDouble.cpp` に `observeCurrentRuntime(` 残存ゼロ

## 4.2 実行確認

- Debug/Release ビルド成功
- Strict Atomic Dot-Call Scan pass
- 1ブロック内 hash一致:
  - `procState.eqCoeffHash`
  - 実際に適用した EQ hash
  が一致

## 4.3 回帰確認

- snapshot fade が従来どおり動作（品質劣化なし）
- bypass/通常経路の音量・位相に異常なし

---

## 5. 実装メモ（レビューで見られる点）

- `ProcessingState` の追加は ABI 影響範囲が大きいため、初期値付きで安全導入。
- EQ cache pointer の寿命は「同一block処理中有効」を契約し、所有権移譲しない。
- fallbackは「入口で解決」が原則だが、防御的 null-safe 分岐は残してよい。
