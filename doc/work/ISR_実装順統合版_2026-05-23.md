## 0. 目的

Phase 0〜4 を **実装順で横断管理** するための統合版。

- 各Phaseの開始条件（ゲート）
- 実装順（PR順）
- ファイル別の主要変更点
- 検証と完了判定

を1ファイルで追跡できるようにする。

参照先テンプレとの表記整合として、**Phase 0〜3 も軽量G1〜G5ゲート番号スタイルを採用済み**（Phase 4 と同一語彙体系）とする。

---

## 1. 全体方針（ISR不変条件）

全Phaseで以下を維持する。

1. 同一Audio blockで snapshot は1回のみ取得。
2. Audio Thread は `kAudioEpochReaderIndex` のみ利用。
3. `DSPCore::process*` から runtime snapshot を再取得しない。
4. snapshot fade の old/new 同一実体を禁止。
5. 異常入力（0ch/null/非正samples）で U.B. に入らない。
6. Audio Thread で lock/libm/dynamic alloc/blocking を導入しない。

## 3. 実装順詳細（横断版）

## 3.1 PR-0 / Phase 0（Snapshot整合性一本化）

### 主要変更点（Phase 0）

- `AudioEngine::DSPCore::ProcessingState` に EQ参照情報を注入
- callback入口で snapshot 1回取得 + state構築
- `DSPCore::process*` の runtime再取得廃止
- `processWithSnapshot` の寿命検証レイヤ統一

### 主対象ファイル（Phase 0）

- `src/audioengine/AudioEngine.h`
- `src/audioengine/AudioEngine.Processing.AudioBlock.cpp`
- `src/audioengine/AudioEngine.Processing.BlockDouble.cpp`
- `src/audioengine/AudioEngine.Processing.DSPCoreFloat.cpp`
- `src/audioengine/AudioEngine.Processing.DSPCoreDouble.cpp`
- `src/audioengine/AudioEngine.Processing.Snapshot.cpp`

### DoD（Phase 0）

- Audio Thread 経路の `kControlEpochReaderIndex` 使用ゼロ
- `DSPCore::process*` 内 `observeCurrentRuntime()` ゼロ
- 1block内の EQ hash 一致

---

## 3.2 PR-1 / Phase 1（double経路の安全対称化）

### 主要変更点（Phase 1）

- `processBlockDouble` 入口に shutdown/lifecycle/firewall/RT mark を対称導入
- double経路で snapshot fade を実音声へ反映
- snapshot専用経路に sample-rate mismatch guard 追加
- OS>1 分岐でも Dry/Wet 合成を適用し OS=1 と特性統一

### 主対象ファイル（Phase 1）

- `src/audioengine/AudioEngine.Processing.BlockDouble.cpp`
- `src/audioengine/AudioEngine.Processing.Snapshot.cpp`
- `src/audioengine/AudioEngine.Processing.DSPCoreDouble.cpp`
- （必要時）`src/audioengine/AudioEngine.h`

### DoD（Phase 1）

- double callback の入口安全ガードが float相当
- snapshot fade が double実音声に反映
- OS=1 / OS>1 の bypass遷移特性が一致

---

## 3.3 PR-2 / Phase 2（異常系防御）

### 主要変更点（Phase 2）

- 0ch/null 入力の早期防御を I/O 経路へ統一
- `numSamples < 1` の clear呼び出しを禁止（即return）
- EQバイパス遷移を dryCopy 有無から分離し前進保証

### 主対象ファイル（Phase 2）

- `src/audioengine/AudioEngine.Processing.DSPCoreIO.cpp`
- `src/audioengine/AudioEngine.Processing.DSPCoreDouble.cpp`
- `src/eqprocessor/EQProcessor.Processing.cpp`

### DoD（Phase 2）

- 0ch/null/非正samples でクラッシュ/U.B. なし
- dryCopy 未確保時もバイパス遷移が停止しない

---

## 3.4 PR-3 / Phase 3（RT規約適合: libm排除）

### 主要変更点（Phase 3）

- `std::floor` ベース正規化を no-libm 実装へ置換
- wrap/負値/frac近傍で置換前同等の連続性を維持

### 主対象ファイル（Phase 3）

- `src/convolver/ConvolverProcessor.Runtime.cpp`

### DoD（Phase 3）

- Audio Thread 実行経路で `std::floor` 残存ゼロ
- 境界ケースでクリック/段差悪化なし

---

## 3.5 PR-4 / Phase 4（仕様FIX後最適化: mix=0）

### 着手前ゲート（必須 / 未承認なら未着手）

> 実装PR作成・コード変更は、以下 G1〜G5 を全て満たすまで禁止（文書化のみ可）。

- [ ] **G1 方針固定（単一選択）**
      `完全skip` / `軽量prewarm` のいずれか1つを確定し、仕様IDを記載する

- [ ] **G2 定量要件固定（事前確定）**
      CPU平均改善 ≥ 15%、P95悪化 ≤ 3%、クリック ≤ -70 dBFS、20ms窓RMS差 ≤ 1.5 dB、追加遅延 0 samples

- [ ] **G3 RT安全制約遵守**
      lock / blocking / dynamic alloc / libm再導入なし

- [ ] **G4 3者承認完了**
      Audio Tech Lead / Audio QA / Product(Spec) Owner
      （承認者・日時・コメント・仕様IDを記録）

- [ ] **G5 証跡添付完了**
      Build（Debug/Release）・Static Scan・CPU比較・連続性評価・承認情報をPRに添付

**Go/No-Go:** G1〜G5 を全充足で Go。1つでも未達なら No-Go（= 未承認なら未着手）。

### 主要変更点（Phase 4）

- `needsConvolution` 判定と `conv->process(...)` 実行条件の整合
- dry-only 時の不要畳み込み抑制
- 再mix連続性の仕様適合（クリック/立上り要件）

- G1〜G5 の形式ゲートは依然として Phase 4 の完了要件として残る。
- 正式な Phase 4 完了判定は、G4/G5 の証跡が揃ってから更新する。

### 主対象ファイル（Phase 4）

- `src/convolver/ConvolverProcessor.Runtime.cpp`

### DoD（Phase 4）

- dry-only 時の不要畳み込みが抑制される
- 再mix連続性が仕様許容範囲内
