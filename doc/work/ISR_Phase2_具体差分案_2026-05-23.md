## 0. 目的

Phase 2（異常系防御）を、実装に直結する粒度で定義する。

- 変更対象: 関数群（異常入力防御と状態機械前進保証）
- 出力: ファイル別の具体編集点、差分イメージ、適用順

対象不具合:

- B1: 0ch/null 入力で I/O 経路が未定義動作に進む余地
- B2: EQ バイパス遷移が dryCopy 未確保時に停止し得る
- B3: `numSamples <= 0` で clear 呼び出しが不正条件に到達し得る

## 2. ファイル別 具体差分案

## 2.1 `AudioEngine.Processing.DSPCoreIO.cpp`（0ch/null 防御の統一）

### 編集ポイント（DSPCoreIOGuard）

I/O 出力経路で `numChannels <= 0` / `dataL == nullptr` / `dstL == nullptr` を早期防御する。

### 具体手順（DSPCoreIOGuard）

1. 関数入口で `numChannels <= 0` を判定し、即 return。
2. `dataL` / `dstL` / `dc.output*` 利用前に null-safe 条件を導入。
3. null 前提で到達する copy/accumulate ループを封鎖（条件分岐で迂回）。
4. 既存の通常系（2ch前提など）には影響しないよう、ガードは入口寄せで最小化。

### 差分イメージ（DSPCoreIOGuard）

- Before:
  - チャンネル/ポインタ前提が暗黙で、異常入力時に危険経路へ到達し得る。
- After:
  - 入口ガードで危険経路を閉じ、正常入力時のみ既存ループへ進む。

### DoD直結チェック（DSPCoreIOGuard）

- `numChannels <= 0` で出力処理に進まない。
- null ポインタ条件で `dc.output*` が呼ばれない。

## 2.2 `EQProcessor.Processing.cpp`（バイパス遷移の前進保証）

### 編集ポイント（EQBypassProgress）

dryCopy の確保可否に関わらず、ramp消費と `rtBypassedShadow` 更新を前進させる。

### 具体手順（EQBypassProgress）

1. 既存ロジックから「dryCopyがあるときだけ進行」依存を分離。
2. ramp進行更新をデータ処理と独立した制御ブロックへ移動。
3. `rtBypassedShadow` 更新を dryCopy 有無と非連動化。
4. dryCopy 不足時フォールバックを明示（wet-only継続または即時遷移確定）。
5. フォールバック時も遷移が停止しないことを保証。

### 差分イメージ（EQBypassProgress）

- Before:
  - バッファ条件で遷移進行が止まり得る。
- After:
  - 状態機械は常に前進し、データ経路は可用性に応じてフォールバック。

### DoD直結チェック（EQBypassProgress）

- dryCopy 未確保時でも ramp が進行する。
- `rtBypassedShadow` が遷移完了方向へ更新される。

## 2.3 `AudioEngine.Processing.DSPCoreDouble.cpp`（非正 samples 防御）

### 編集ポイント（DSPCoreDoubleSamplesGuard）

`processDouble(...)` の `numSamples < 1` 分岐を、clear を伴わない安全 return へ統一する。

### 具体手順（DSPCoreDoubleSamplesGuard）

1. 入口で `numSamples < 1` を判定し、即 return。
2. clear 呼び出しは `numSamples > 0` が保証される分岐に限定。
3. 既存 clear 呼び出し箇所に前提条件コメント/assert を追加（必要最小限）。
4. 0 samples のとき状態機械が不整合にならないよう、進行更新の要否を明確化。

### 差分イメージ（DSPCoreDoubleSamplesGuard）

- Before:
  - 非正 samples で clear 経路に入り得る。
- After:
  - 非正 samples は副作用最小で return、clear は正サンプル時のみ。

### DoD直結チェック（DSPCoreDoubleSamplesGuard）

- `numSamples < 1` で不正 clear が実行されない。
- 非正 samples 入力でクラッシュ/U.B. が発生しない。

## 4. 検証項目（Phase 2専用）

## 4.1 静的確認

- `numChannels <= 0` 防御が I/O 出力経路に存在
- `dataL` / `dstL` / `dc.output*` 利用前の null-safe 条件が存在
- `numSamples < 1` で clear を呼ばない分岐が存在
- EQ遷移進行更新が dryCopy 有無に依存しない

## 4.2 実行確認

- Debug/Release ビルド成功
- Strict Atomic Dot-Call Scan pass
- 0ch 入力でクラッシュしない
- `numSamples <= 0` で不正 clear が起きない
- dryCopy 未確保でもバイパス遷移が完了する

## 4.3 回帰確認

- 通常入力（2ch/正samples）で処理結果が従来と一致
- 異常系防御追加により通常経路レイテンシが悪化しない
- EQ バイパス遷移の聴感上の discontinuity が悪化しない

## 5. 実装メモ（レビューで見られる点）

- Phase 2 は「異常系防御」に限定し、Phase 3（libm排除）を混在させない。
- Audio Thread 制約（no lock / no libm / no dynamic alloc / no blocking）を維持する。
- 防御分岐は入口集中で追加し、既存の通常系分岐の可読性を崩さない。
