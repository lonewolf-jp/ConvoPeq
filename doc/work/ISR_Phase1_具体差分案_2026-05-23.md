## 0. 目的

Phase 1（double経路の安全対称化）を、実装に直結する粒度で定義する。

- 変更対象: 関数群（必要時のみ最小のヘッダ追補）
- 出力: ファイル別の具体編集点、差分イメージ、適用順

対象不具合:

- A1: `processBlockDouble` 入口の安全ガード非対称
- A2: double経路で snapshot fade 状態のみ進行し、音声反映が不十分
- A3: snapshot専用経路の sample-rate 不整合時ガード不足
- A4: OS>1 の bypass 合成が OS=1 と非対称

## 2. ファイル別 具体差分案

## 2.1 `AudioEngine.Processing.BlockDouble.cpp`（callback入口の安全対称化）

### 編集ポイント（BlockDouble）

`processBlockDouble(...)` を float側 callback と同等の保護シーケンスに合わせる。

### 具体手順（BlockDouble）

1. 先頭で `isShutdownInProgress()` を評価し、必要時は `clear + return`。
2. lifecycle enter/leave を導入し、途中return経路でも対称に release される構造へ。
3. firewall enter/leave（または同等の診断ガード）を導入。
4. RT context mark（active callback count / thread mark）を float側と同一条件で実施。
5. 例外/異常早期return経路が guard unbalanced を起こさないことを整理。

### 差分イメージ（BlockDouble）

- Before:
  - double callback 入口で guard 適用が部分的、または経路依存。
- After:
  - shutdown/lifecycle/firewall/RT mark を統一順で適用し、
    どの return 経路でも対称に unwind。

### DoD直結チェック（BlockDouble）

- shutdown進行時、double callback は無音化して安全に return。
- guard の enter/leave 対が崩れない。

## 2.2 `AudioEngine.Processing.Snapshot.cpp`（snapshot専用経路の rate guard）

### 編集ポイント（SnapshotRateGuard）

`processWithSnapshot(...)` の snapshot専用処理に sample-rate 一致チェックを追加する。

### 具体手順（SnapshotRateGuard）

1. active/fading snapshot の sample-rate（または等価な rate識別子）を参照。
2. 処理対象 buffer の rate 条件と不一致なら、
   - `clear`
   - `return`
   を即時実行（中途半端な処理を行わない）。
3. mismatch時のログ/診断は既存ポリシー範囲で最小限（Audio Threadで非RT安全処理を増やさない）。

### 差分イメージ（SnapshotRateGuard）

- Before:
  - rate不整合でも snapshot経路を進行し得る。
- After:
  - mismatch を入口で遮断して無音return。

### DoD直結チェック（SnapshotRateGuard）

- rate mismatch ケースで clear + return が必ず発火。
- 不整合時に old/new snapshot 処理へ入らない。

---

## 2.3 `AudioEngine.Processing.DSPCoreDouble.cpp`（snapshot fade の実音声反映 + OS>1合成対称化）

### 編集ポイント（DSPCoreDoubleFade）

`DSPCore::processDouble(...)` で snapshot fade 進行値を実際の old/new 合成へ接続する。

### 具体手順（DSPCoreDoubleFade）

1. `updateAudioThreadSnapshotFade(...)` の結果（from/to/alpha）を取得。
2. double buffer で old/new の処理結果を保持できる経路を用意。
3. `snapshotAlpha` を使って old/new を実際にクロスフェード合成。
4. state進行のみで音声未反映になる枝を除去。

### 差分イメージ（DSPCoreDoubleFade）

- Before:
  - fade state は更新されるが、合成経路が限定的/未接続。
- After:
  - fade state と音声合成が一体化し、遷移が実音へ反映。

### DoD直結チェック（DSPCoreDoubleFade）

- double経路で snapshot遷移が可聴上連続になる。
- `snapshotFrom/snapshotTo/snapshotAlpha` が「計算のみ」で終わらない。

---

### 編集ポイント（DSPCoreDoubleBypassOS）

OS>1 分岐でも OS=1 と同等の Dry/Wet 合成特性に揃える。

### 具体手順（DSPCoreDoubleBypassOS）

1. processDown 後の分岐に Dry/Wet 合成を適用。
2. OS=1 と OS>1 で bypass 遷移時の gain カーブと適用タイミングを揃える。
3. 既存の oversampling 処理順を壊さないよう、合成位置を最小変更で固定。

### 差分イメージ（DSPCoreDoubleBypassOS）

- Before:
  - OS=1 のみ遷移品質が安定、OS>1 は合成差で体感差が出る余地。
- After:
  - OS on/off で同等の遷移特性。

### DoD直結チェック（DSPCoreDoubleBypassOS）

- OS=1 / OS>1 で bypass遷移のレベル挙動が一致。
- クリック/段差の悪化がない。

### 編集ポイント（HeaderAdjust）

### DoD直結チェック（HeaderAdjust）

- ヘッダ差分が最小であり、Phase 2以降の責務を混在させない。

## 4. 検証項目（Phase 1専用）

## 4.1 静的確認

- `processBlockDouble` に shutdown/lifecycle/firewall/RT mark が揃っている
- `processWithSnapshot` に rate mismatch guard がある
- `processDouble` で snapshot fade 値が実際の合成へ接続されている
- OS>1 分岐に Dry/Wet 合成が存在する

## 4.2 実行確認

- Debug/Release ビルド成功
- Strict Atomic Dot-Call Scan pass
- shutdown遷移中 callback で安全無音化を確認
- rate mismatch 入力で clear + return を確認

## 4.3 回帰確認

- snapshot遷移時の可聴連続性が維持/改善
- OS=1 / OS>1 で bypass遷移の体感差が縮小
- double経路追加ガードにより通常経路性能が劣化しない

---

## 5. 実装メモ（レビューで見られる点）

- Phase 1 は「double経路の対称化」に限定し、Phase 2（異常系横展開）を混在させない。
- Audio Thread 制約（no lock / no libm / no dynamic alloc / no blocking）を守る。
- snapshot fade 導入時、old/new 同時実行の一時バッファ寿命を callback スコープ内で閉じる。
