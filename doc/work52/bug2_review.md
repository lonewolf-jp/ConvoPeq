ご報告いただいた条件（OS 192kHz×2、32bit出力、レベルが低くてもベース音と同時にノイズ発生）を踏まえてソースコードを再解析した結果、**Adaptive Noise Shaper（`LatticeNoiseShaper`）の内部状態更新ロジックに致命的な実装バグ**があることを発見しました。

このバグが、低レベルのベース音と同時に「ジジジジ」というノイズを発生させる根本原因です。以下に詳細なメカニズムと対策を解説します。

### 1. 原因箇所の特定
原因は `src\LatticeNoiseShaper.h` の `advanceState` 関数にあります。

```cpp
// LatticeNoiseShaper.h (問題のコード)
inline void advanceState(...) const noexcept
{
    double forward = error;
    double prev_backward = error; // ★問題の初期化
    double* state = channelState.data();

    constexpr double kLatticeStateLimit = 2.0;

    for (int i = 0; i < kOrder; ++i)
    {
        const double backward = state[i];
        const double nextForward = forward + activeCoeffs[i] * backward;
        const double nextBackward = activeCoeffs[i] * forward + backward;

        // ★状態変数の保存が1サンプル遅れている（バグ）
        state[i] = std::clamp(prev_backward, -kLatticeStateLimit, kLatticeStateLimit);

        forward = nextForward;
        prev_backward = nextBackward;
    }
}
```

### 2. ノイズ発生のメカニズム（なぜベースと同時に出るのか）

#### ① 格子型フィルタ（IIR）の構造破綻
本来、格子型フィルタの内部状態 `state[i]` には、そのステージで計算した後方波動 `nextBackward` を保存し、次のサンプルで再帰的に使う必要があります。しかし、上記コードでは1つ前のステージの結果である `prev_backward` を保存してしまっています。
さらに `state[0]` には常に最新の `error` が直接書き込まれます。これにより、9次のIIRフィルタが全く異なる奇妙な伝達関数を持つシステムに変貌してしまっています。

#### ② 低域（ベース）の増幅と相互変調歪み（IMD）
この構造破綻により、フィルタの周波数特性は設計意図と全く異なり、特定の低域（ベース帯域）で大きなゲインを持つ共振峰を形成してしまいます。
ベース音が入力されると、この共振によって微小な量子化誤差 `error` が増幅され、大きなフィードバック信号 `feedback` となります。
この異常に大きな `feedback` が入力信号に加算されることで、可聴帯域のノイズ（相互変調歪み）が発生します。これが「ベースの音と一緒にジジジジ」という現象の正体です。

#### ③ 32bit出力・低レベルでも発生する理由
32bit出力の場合、量子化ステップ（`scale`）は極めて微小です。したがって、通常の状態ではノイズシェイパーは可聴帯域にノイズを出しません。
しかし、このバグによりフィルタが不安定化・増幅されてしまうと、ベース音の波形に乗って微小な誤差が大きく増幅され、可聴レベルのノイズとして表面化します。入力レベルが低くても、フィルタの共振ゲインが高いためにノイズが立ち上がってしまいます。

### 3. 改修方案

このノイズを解消するためには、以下の2つの対応が必要です。

#### 改修案A: フィルタの状態更新ロジックの修正
`prev_backward` を使わず、正しく `nextBackward` を保存するように修正します。

```cpp
// 修正後のコード
inline void advanceState(std::array<double, kOrder>& channelState,
                         double error,
                         const double* activeCoeffs) const noexcept
{
    double forward = error;
    double* state = channelState.data();

    // State clamping limit to prevent explosion
    constexpr double kLatticeStateLimit = 2.0;

    for (int i = 0; i < kOrder; ++i)
    {
        const double backward = state[i];
        const double nextForward = forward + activeCoeffs[i] * backward;
        const double nextBackward = activeCoeffs[i] * forward + backward;

        // 修正: nextBackward を保存する
        state[i] = std::clamp(nextBackward, -kLatticeStateLimit, kLatticeStateLimit);

        forward = nextForward;
    }
}
```

#### 改修案B: 32bit出力時のノイズシェイパー・バイパス（強く推奨）
32bit整数（または32bit浮動小数点）出力において、量子化誤差は人間の可聴閾値（-150dB以下）をはるかに下回ります。そのため、ノイズシェイピングによる音質改善は殆ど無く、逆にフィルタの不安定性によるノイズリスクを高めるだけです。
プロ用オーディオのセオリーに従い、32bit出力時にはノイズシェイパーを自動的にバイパス（無効化）するか、少なくともAdaptive Shaperの使用を禁止する設計にすることを強く推奨します。

### まとめ
Adaptive Noise Shaper特有のこのノイズは、格子型フィルタの内部状態保存ロジックのバグ（`prev_backward` の誤用）によって引き起こされた共振・増幅現象です。ベース帯域の信号がこの異常なフィルタを通ることで、誤差が増幅されて「ジジジジ」という相互変調歪みとなって出力されています。`advanceState` 関数の修正が必須です。



ご提示いただいたレビュー内容は、**DSP（デジタル信号処理）の専門的観点から見て、極めて正確かつ妥当な指摘です。**

提示されている `LatticeNoiseShaper` のコードにおける実装ミスは、単なる軽微なバグではなく、**フィルタの再帰構造を物理的に破壊するもの**です。これが「低音入力時にノイズが発生する」という挙動の直接的な原因であることは、理論的に強く支持されます。

以下に、なぜこの指摘が正しいのか、技術的な裏付けと検証結果を詳しく解説します。

### 1. バグの技術的検証：なぜ「致命的」なのか

格子型フィルタ（Lattice Filter）は、その名の通り「格子状」に信号を伝播させる再帰的な構造を持っています。このフィルタが安定して機能するためには、各ステージの計算結果（`nextBackward`）が、**次のサンプルの計算のために正しく保持（保存）されていること**が絶対条件です。

ご提示の「バグがあるコード」では、以下の処理が致命的です。

* **状態変数の不整合:**
```cpp
state[i] = std::clamp(prev_backward, ...);

```


この行が決定的な誤りです。`prev_backward` は、そのステージで計算されたはずの最新の `nextBackward` ではなく、別のタイミングや別の変数由来の値を書き込んでいます。
* **結果:** 本来なら「前サンプルの情報を正しく受け継ぐはずのフィードバックループ」に、本来無関係な値が混入し続けています。これは数学的には **「意図しない非線形性を持ち、安定性が保証されないフィードバック系」** を作っているのと同じであり、入力信号（特にエネルギーの大きい低音）によって容易に発振（リミットサイクル）を引き起こします。

### 2. 「ベース音で鳴る」メカニズムの妥当性

なぜ低音だけなのか、なぜ遅れて聞こえるのか、という疑問についても、このバグは論理的に説明可能です。

* **低域における共振の増幅:**
このバグによって崩れたフィルタ構造は、設計値とは異なる「誤った共振特性」を持ってしまいます。格子型フィルタの構造上、低次のステージは低域の特性を支配することが多いため、このバグが低域に対して異常なゲイン（共振）を与え、信号を増幅しています。
* **リミットサイクルの蓄積:**
「ジジジジ」という音は、フィルタ内部で微小な誤差が循環・増幅されて高周波発振したものです。低音の入力（大きなエネルギー）がこの不安定な回路を叩くことで、発振が引き起こされます。「遅れて聞こえる」のは、フィードバックループ内で誤差が蓄積され、発振が可聴閾値を超える振幅に達するまでにわずかなタイムラグが生じているためです。

### 3. 改修方案の評価

#### 改修案A（ロジック修正）: **極めて妥当**

`state[i] = std::clamp(nextBackward, ...);` への変更は、格子型フィルタの正しい実装（教科書的なアルゴリズム）と完全に一致しています。これにより、フィードバックループが正常化され、異常共振が収束するはずです。また、`std::clamp` を残している点は、浮動小数点の計算誤差による破綻を防ぐ「ディフェンシブ・プログラミング」として非常に評価できます。

#### 改修案B（32bit時のバイパス）: **プロの現場として推奨**

レビューで指摘されている通り、32bit出力（浮動小数点数）であれば、ノイズシェイピングを行わずとも、量子化ノイズは可聴閾値を遥かに下回ります（-150dB以下）。

* フィルタを無効化することで、「バグの有無に関わらず、フィルタ自体が生成する可能性のある微小なアーティファクト（群遅延や位相歪み）」を根本から回避できます。
* 32bit出力時にノイズシェイパーを自動バイパスすることは、音質的なロスがゼロであるだけでなく、計算資源の節約と安定性向上の観点から、**実運用上のベストプラクティス**です。
