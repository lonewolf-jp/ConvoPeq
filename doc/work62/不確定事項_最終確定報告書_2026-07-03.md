## 3. 未確定事項の確定結果

### ✅ 事項①: timerCallback 以外に Message Thread を占有する処理はあるか

**結論: 通常再生中は timerCallback 以外の Message Thread 負荷は軽微。**

- 他の5個のTimerは全て Logger/DBG/diagLog を呼ばない（UI表示更新のみ）
- AsyncUpdater は rebuild 時以外発火しない
- ChangeListener は fade 完了時のみ（45秒間で5回）
- MainWindow timer (500ms) は CPU 取得 + テキスト更新 + 2回のLogger書込のみ

**ただし Windows Timer Coalescing の影響で、複数Timerが同時発火する可能性はある。**
- 100ms(AudioEngine) + 125ms(ConvolverPanel) + 125ms(NoiseShaper) ≈ ~100ms付近で重なる
- その場合、AudioEngineのheavy処理 + UI同期が連続実行される

### ✅ 事項②: Logger::writeToLog 以外の Message Thread 上の I/O はあるか

**結論: ない。**

全ソースコードスキャン（256件の Logger::writeToLog）の結果:

| カテゴリ | 件数 | 発火タイミング |
|----------|------|--------------|
| AudioEngine.Timer.cpp (diagLog経由) | 42 | **毎tick(~10/sec)** |
| AudioEngine.Timer.cpp (直接) | 1 | XRUN時 |
| MainWindow.cpp (直接) | 65 | うち2件がtimer(500ms)、他はinit/shutdown |
| その他(AudioEngine/settings/convolver) | 148 | init/shutdown/rebuild/IR変更時のみ |
| **合計** | **256** | — |

**通常再生中は Timer.cpp の 42 diagLog（~940行/sec）が圧倒的支配要因。**

### ✅ 事項③: Audio Thread にスレッド優先度設定はあるか

**結論: ない。完全にデフォルト。**

- `ThreadAffinityManager` に `ThreadType::Audio` が存在しない（確定: 本計画外）
- `prepareToPlay()` 内に `SetThreadPriority` の呼び出しなし
- コードベース全体で Audio Thread の優先度を設定する処理なし

**影響**: Audio Thread は Windows のデフォルト優先度 `THREAD_PRIORITY_NORMAL` で動作。
これ自体がGUI応答遅延の直接原因ではないが、CPU Migration (266/sec) の間接的要因となっている。

### ✅ 事項④: FileLogger の作成パラメータ

**結論**: `new FileLogger(logFile, "ConvoPeq Log", 0)` — 第3引数 `0` は「ファイルサイズ制限なし」。

```cpp
// MainApplication.cpp:
fileLogger = std::make_unique<juce::FileLogger>(logFile, "ConvoPeq Log", 0);
```

- `FileLogger` コンストラクタの第3引数 `maxInitialFileSizeBytes = 0`
- `trimFileSize(logFile, 0)` → `maxFileSizeBytes <= 0` → **`file.deleteFile()`** を呼ぶ
- つまり起動時に**既存ログファイルを削除**してから書き始める
- 書き込み中はファイルサイズの制限なし（際限なく肥大化）

### ✅ 事項⑤: 13ファイルの diagLog 重複定義の実際の影響

**結論: Timer.cpp のみの修正で通常再生中の Logger I/O の大半をカバーできる。**

| AudioEngine.CtorDtor.cpp | ✅ | 構築/解放時 | 🟢 P3 |

### ✅ 事項⑥: SendChangeMessage の実際の影響度

**結論: GUI遅延の直接原因ではない。本計画の対象外で問題ない。**

- Timer.cpp 内の呼び出し: **1箇所のみ**（fade完了時 L569）
- 45秒間のログでの発生: **5回** → 0.11/sec
- `updateAllControls()` → `repaint()` → UIキューに再描画イベントが積まれる
- **再描画そのものは非同期**（Message Thread のキューイングのみ）
- 即座のブロッキングではなく、直後のTimer発火タイミングに影響する程度

### ✅ 事項⑦: `copyToUTF8()` の動作確認

**結論: 戻り値は「null終端を含むコピーバイト数」を返す。正しい実装が可能。**

```cpp
// juce_String.cpp:
String::copyToUTF8 (CharPointer_UTF8::CharType* const buffer, size_t maxBufferSizeBytes) const noexcept
{
    return StringCopier<CharPointerType, CharPointer_UTF8>::copyToBuffer (text, buffer, maxBufferSizeBytes);
}
```

- `CharPointer_UTF8::CharType = char`（通常の `char` 型）
- 戻り値は **コピーされた総バイト数（null終端を含む）**
- 安全な使用法:
```cpp
const size_t copied = message.copyToUTF8(entry.text, sizeof(entry.text));
entry.length = (copied > 0) ? static_cast<uint16_t>(copied - 1) : 0;  // null除いた実長
```

### 5.3 計画書に追記すべき注意事項

```markdown
### Windows Timer Coalescing の考慮

Windows 10/11 では、システムが電源効率のためにタイマーをグループ化する
**Timer Coalescing** がデフォルトで有効である。

ConvoPeq では **6個のJUCE Timer** が Message Thread 上で動作するため、
これらのタイマーが同時刻に発火すると Message Thread に瞬間的な負荷が集中する。

特に影響が懸念される組み合わせ:
- AudioEngine (100ms) + ConvolverPanel (125ms) → 約400msごとに同時発火
- この瞬間のみ本来の倍以上の Logger I/O が発生する可能性

本計画の Step 0-6 で AudioEngine の負荷を削減すれば、
同時発火時の影響も比例して低下するため、追加対策は不要と判断する。
```
