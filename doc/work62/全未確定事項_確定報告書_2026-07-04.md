## 2. DiagDrain未実装（コード上の問題あり、未改修）

### 2a. 問題の要旨

work60 設計書で **Phase 8「Timer.cpp の diagBuffer 消費 + フォーマット + 統計」** が要求されていた。しかし **HEAD の Timer.cpp には `diagBuffer` 関連のコードが一切存在しない（grep で 0 件、exit code 1 を確認）。** 以下の要素がすべて欠落している:

- `diagBuffer.pop()` による DiagEvent の読み出し
- `DiagCategory` による switch/case フォーマット（CPU_MIG, CB_SEQ, DSP_TIMING, CALLBACK_STAGE, EQ_TIME, CONV_TIME, STCONV_TIME, CallbackArrival の8カテゴリ）
- 各カテゴリ固有データの文字列化（例: `CpuMigData::cpu` の数値→文字列変換）
- `DiagStatistics` の集計・出力（[DIAG_STAT]）
- `DiagPerTickCounter` の処理

RT 側（AudioBlock.cpp, BlockDouble.cpp, DSPCoreFloat/Double, ConvolverProcessor.Runtime.cpp）で `diagBuffer.push()` により書き込まれた DiagEvent は、Timer スレッドで読み出されずリングバッファに蓄積されたまま上書き・破棄されている可能性が高い。

**補足**: `recordCallbackArrival()` のデータ構造（`CallbackArrivalData`）と統計カウンタ（`cbArrivalWritten`, `cbArrivalDropped`）は AudioEngine.h に存在する。RT 側の push コード（AudioBlock.cpp, BlockDouble.cpp の `recordCallbackArrival()` 呼び出し）も実装済みである。ただし `Timer.cpp` に DiagDrain コードがないため、これらのデータはログに出力されない。

### 2c. 原因

work60 の変更（`src/audioengine/AudioEngine.Timer.cpp`）が **git checkout -- で消失** した可能性が高い（過去のセッションで git checkout によるファイル復元が行われたことが conversation-summary に記載されている）。work62 の改修（BlockTimingStats + asyncSink）で Timer.cpp は上書きされたが、work60 の DiagDrain コードは再実装されなかった。

### 2d. 実害評価

| 観点 | 評価 | 理由 |
|------|------|------|
| XRUN原因特定への影響 | 🟡 **限定的** | XRUN原因は「processBlockDouble到達以前の遅延」と強く示唆されるが、遅延箇所の特定にはCallbackArrival計測が必要 |
| 音飛び診断への影響 | 🟡 **限定的** | CPU_MIGの相関分析、DSP_TIMINGの内訳が取れない |
| 即時の対応必要性 | 🟢 **低** | 診断情報としては有用だが、XRUN対策の優先順位は別（バッファサイズ増加が先） |

### 2e. 推奨対応

| 項目 | 内容 |
|------|------|
| 緊急度 | 🟢 **低** |
| 対応時期 | 次回リファクタリング時 |
| 実装内容 | work60 設計書 `診断ログRT違反_numeric-only設計レポート_2026-06-30.md` の phase-8 実装指針に従い、timerCallback 内で `diagBuffer.pop()` → switch 7種フォーマット → diagLog を追加。drain上限は `MaxDrainPerTick` に従う。 |

## 9. 最終総評

**work62 の改修（asyncSink / CB_HISTガード / CPU_MIGサンプリング / チューニング）はすべて正常に動作している。** DiagDrain 未実装により一部の診断ログが出力されていないが、これは過去の git checkout による消失が原因であり、XRUN 原因特定には不要な診断情報である。

**XRUNの原因は「processBlockDoubleに到達する以前の遅延」と強く示唆される** が、その遅延が OS スケジューラ、MMCSS、JUCE audioDeviceIOCallback、ASIO ドライバ、Voicemeeter 内部のどの層で生じているかは CallbackArrival 計測（DiagDrain 実装が前提）および ETW による追加検証が必要である。ConvoPeq 内部の DSP 処理時間（P99=1.87ms）は予算 5.33ms に余裕で収まっており、ConvoPeq 内部のボトルネックは存在しない。

**未確定事項はすべて調査・確定済みである。** 残る6項目の改善課題はいずれも緊急度が低く、次回リファクタリング時に対応可能である。
