## 3. Step 8: DiagDrain 再実装

### 3.1 概要

work60 設計書 `診断ログRT違反_numeric-only設計レポート_2026-06-30.md` の **Phase 8** を再実装する。Timer.cpp に `diagBuffer` からの DiagEvent 読み出し・フォーマット・統計出力のコードを追加する。

**ファイル**: `src/audioengine/AudioEngine.Timer.cpp`（既存の timerCallback 末尾、flushLogBuffer の前に追加）

### 3.2 DiagDrain コード構造

```cpp
// AudioEngine::timerCallback() 内、flushLogBuffer() の前
#if CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS
    // ★ DiagEvent drain: RTスレッドが書き込んだバッファを消費
    {
        const uint64_t gen = (runtimeWorld != nullptr)
            ? static_cast<uint64_t>(runtimeWorld->generation) : 0;

        // --- Phase A: DiagEvent pop + format (up to MaxDrainPerTick) ---
        DiagEvent event;
        int drained = 0;
        while (drained < static_cast<int>(DiagRuntimeLimits::MaxDrainPerTick)
               && diagBuffer.pop(event))
        {
            ++drained;
            // diagTickPopped.value.fetch_add(1, std::memory_order_relaxed);

            switch (event.category)
            {
                case DiagCategory::CpuMig:
                {
                    const auto& d = event.data.cpuMig;
                    diagLog(diagPrefix(gen) + " [CPU_MIG] cbIdx="
                        + juce::String(static_cast<juce::int64>(event.eventIndex))
                        + " pubSeq=" + juce::String(static_cast<juce::int64>(d.pubSeq))
                        + " gen=" + juce::String(static_cast<juce::int64>(d.generation))
                        + " cpu=" + juce::String(static_cast<int>(d.cpu))
                        + " prev=" + juce::String(static_cast<int>(d.prevCpu)));
                    break;
                }

                case DiagCategory::CallbackArrival:
                {
                    const auto& d = event.data.callbackArrival;
                    diagLog(diagPrefix(gen) + " [CB_ARRIVAL] cbIdx="
                        + juce::String(static_cast<juce::int64>(event.eventIndex))
                        + " ts=" + juce::String(static_cast<juce::int64>(d.timestampUs))
                        + " expected=" + juce::String(static_cast<juce::int64>(d.expectedUs))
                        + " cpu=" + juce::String(static_cast<int>(d.cpu))
                        + " tid=" + juce::String(static_cast<int>(d.threadId)));
                    break;
                }

                // case DiagCategory::CallbackSequence:
                // case DiagCategory::DspTiming:
                // case DiagCategory::CallbackStage:
                // case DiagCategory::EqTime:
                // case DiagCategory::ConvTime:
                // case DiagCategory::StereoConvTime:
                // ★ 上記6カテゴリも同様に実装（work60 設計書の format 文字列に従う）
                //   ただし最初の実装では CpuMig + CallbackArrival のみでも可
                //   （残りは段階的に追加）

                default:
                    // 未知のカテゴリ
                    diagLog(diagPrefix(gen) + " [UNKNOWN] category="
                        + juce::String(static_cast<int>(event.category)));
                    break;
            }
        }
    }
#endif
```

### 3.3 実装ステップ

| # | 作業 | 内容 | 難易度 |
|---|------|------|--------|
| 8a | **ベース構造追加** | XRUN drain ループの直後、flushLogBuffer の前に DiagDrain ブロックを追加（diagBuffer.pop() + switch）。まず `DiagCategory::CpuMig` と `DiagCategory::CallbackArrival` の2カテゴリのみ実装。 | ★★☆ |
| 8b | **全カテゴリ実装** | 残り6カテゴリ（CallbackSequence, DspTiming, CallbackStage, EqTime, ConvTime, StereoConvTime）のフォーマット文字列を work60 設計書に従い実装。 | ★★☆ |
| 8c | **DiagStatistics 追加** | diagTickPushed/Popped/Dropped の exchangeAtomic + DIAG_STAT ログ出力（work60 設計書 L640-700 参照）。 | ★☆☆ |
| 8d | **ビルド確認** | Release ビルド成功を確認。sizeof(DiagEvent) の実測値を確認し、static_assert を `<=` から `==` に更新。 | ★☆☆ |

### 3.4 注意点

- 既存の `s_logBuffer`（asyncSink 用）と `diagBuffer`（DiagEvent 用）は**別物**。混同しないこと
- `diagBuffer.pop()` は SPSC なので **mutex 不要**（Timer Thread が唯一の Consumer）
- `diagTickPushed/Popped/Dropped` は `rtAuxMutable_` のメンバとして定義済み（AudioEngine.h）
- `diagBuffer` は `AudioEngine` クラスのメンバ（`LockFreeRingBuffer<DiagEvent, 512>`）として定義済み

## 4. Step 9: XRUN 原因特定のための追加検証

### 4.1 Step 9a: CallbackArrival ログ取得と分析

**前提条件**: Step 8 (DiagDrain) 完了

**目的**: JUCE callback 到着時刻（`recordCallbackArrival()` の timestampUs）と processBlockDouble 先頭の XRUN Interval 計測との差分から、遅延が ConvoPeq 外部か内部かを判定する。

**判定ロジック**:
```
Case A: CallbackArrival ts が 5.33ms ± 誤差 → 遅延は ConvoPeq 内部（RuntimeScope/preamble）
Case B: CallbackArrival ts が 8〜10ms に乖離   → 遅延は ConvoPeq 外部（OS/JUCE/Driver）
```

**期待される成果**: 現在「強く示唆」レベルの「processBlockDouble 以前の遅延」が、ConvoPeq 外部か内部かで確定する。

### 4.2 Step 9b: ETW トレース

**前提条件**: Step 9a で「外部」と判定された場合

**取得対象**:
| トレース | 対象 |
|---------|------|
| `Thread::ReadyThread` | スレッドが実行可能になったがディスパッチされなかった時間 |
| `Thread::CSwitch` | コンテキストスイッチのタイミングと待ち時間 |
| `DPC::Routine` | 遅延プロシージャコールの実行時間 |
| `ISR::Routine` | 割り込みサービスルーチンの実行時間 |

**ツール**: Windows Performance Recorder (WPR) + Windows Performance Analyzer (WPA)

### 4.3 Step 9c: ASIO 比較試験

**目的**: Voicemeeter BANANA Virtual ASIO 固有の問題かどうかを切り分ける。

**比較対象**:
| ドライバ | 種別 | 入手性 |
|---------|------|--------|
| ASIO4ALL | WASAPI wrapper | 無料、即座に利用可能 |
| FL Studio ASIO | バーチャルASIO | FL Studio ユーザー |
| ハードウェア ASIO (RME/Focusrite) | 物理ASIO | 要実機 |

## 5. Step 10: XRUN 緩和策

### 5.1 Step 10a: MMCSS AVRT_PRIORITY_HIGH 設定追加

**ファイル**: `src/audioengine/AudioEngine.Init.cpp` または `src/audioengine/AudioEngine.Threading.cpp`

```cpp
#include <avrt.h>
#pragma comment(lib, "avrt.lib")

// ★ [work62] MMCSS "Pro Audio" 優先度設定
//   prepareToPlay() の先頭、または Audio Thread 起動時に呼び出す。
//   デフォルトの AVRT_PRIORITY_NORMAL から AVRT_PRIORITY_HIGH に昇格し、
//   OS のオーディオコールバックディスパッチ優先度を向上させる。
//   効果は未検証のため、実験的に追加して改善幅を計測する。
void AudioEngine::applyMmcssPriority()
{
    DWORD taskIndex = 0;
    HANDLE hTask = AvSetMmThreadCharacteristicsA("Pro Audio", &taskIndex);
    if (hTask != nullptr)
    {
        AvSetMmThreadPriority(hTask, AVRT_PRIORITY_HIGH);
    }
}
```

**検証方法**: MMCSS 設定の有無で XRUN レートを比較。

### 5.2 Step 10b: バッファサイズ 2048 試験

**目的**: 現象緩和。現在 XRUN の原因となっている 8〜10ms の dispatch 遅延が、10.67ms の予算内に収まるかを確認する。

**方法**: ASIO ドライバ設定のバッファサイズを 1024→2048 に変更（ConvoPeq のコード変更不要）。

**注意**: これは原因特定ではなく現象緩和が目的。CallbackArrival/ETW による原因特定を優先すること。
