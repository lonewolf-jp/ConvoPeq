## 1. エグゼクティブサマリ

ConvoPeq の音飛び（XRUN）問題について、延べ **29文書** にわたる調査・解析・実機計測を統合した結果、以下の結論に至った。

### 総合結論

**ConvoPeq 内部の主要な候補（DSP負荷、Publish、Crossfade、Logger I/O、CPU Migration、メモリプレッシャー）が XRUN の主因である可能性は大きく低下した。現在の計測結果は「オーディオコールバックが processBlockDouble に到達する以前の遅延」が支配的であることを強く示唆している。ただし、その遅延がどの層（Windows スケジューラ、MMCSS、JUCE audioDeviceIOCallback、ASIO ドライバ、Voicemeeter 内部）で生じているかは、CallbackArrival 計測（DiagDrain 実装が必要）および ETW による追加検証が必要である。**

現時点で確実に言えることは以下の通り:

- XRUN として観測された callback では DSPCore 以降の処理時間に十分余裕があった（CB_HIST P99=1.87ms << 5.33ms 予算、超過率0.0%）
- Publish/Crossfade、Logger I/O、CPU Migration、メモリプレッシャーが主因である証拠は見つかっていない
- 遅延は少なくとも `processBlockDouble()` より前の区間で発生していることが強く示唆される
- その遅延が Windows スケジューラ、MMCSS、JUCE、ASIO ドライバ、Voicemeeter のどこで発生しているかは、現在の計測だけでは切り分けられない
- 切り分けには CallbackArrival 計測（DiagDrain 実装が前提）と ETW が最も有効である

### 核心エビデンス

```
Callback予算（192kHz/1024spb）: 5.33ms
DSP実測 P99:                    1.87ms  ← 予算の35%、余裕あり
DSP予算超過率:                   0.0%    ← 全コールバックが予算内
XRUN閾値:                       8.0ms   ← intervalがこれを超えると発報
XRUN実測 Interval 最小値:       8.0ms   ← XRUN閾値が8.0msのため、8.0ms未満のintervalはXRUNとして記録されない（正常時intervalの分布は未計測）
```

DSP 処理が 5.33ms の予算内に収まっているにもかかわらず、コールバック間隔が常に 8ms 以上であるということは、**コールバックが processBlockDouble に到達する以前に何らかの遅延が発生している** ことを示す。この遅延が OS スケジューラ、MMCSS、JUCE の audioDeviceIOCallback、ASIO ドライバ、あるいは Voicemeeter 内部のどこで生じているかは、CallbackArrival 計測（work61 で設計済み、RT 側の実装は完了しているが DiagDrain 未実装のため出力されていない）によって初めて定量可能になる。

### XRUN件数の推移（改善の経緯）

| フェーズ | XRUN/秒 | 備考 |
|---------|---------|------|
| 改修前（work60 初回） | 24/秒 | 旧ログシステム+CB_HIST毎tick+CPU_MIG毎回 |
| 閾値改善後（work60 第3回） | ~1.7/秒 | XRUN検出閾値適正化により98.8%「見かけ上」削減 |
| **改修後（work62 Step5）** | **1.0/秒** | asyncSink+CB_HISTガード+CPU_MIGサンプリング。不変 |

XRUN レートは Message Thread の改修（Logger I/O 99.8%削減）後も **全く減少していない**。これは XRUN が Message Thread と無関係であることの直接証拠である。

### 2.3 不変の観測事実（3セッション共通）

**事実①: XRUN として記録される interval の最小値は 8.0ms**

```
XRUN検出閾値: interval > max(expectedMs * 1.5, 3.0) = max(8.0, 3.0) = 8.0ms
→ XRUNは interval > 8.0ms で発報
→ 8.0ms 未満の interval は記録されない
```

## 3. XRUN検出機構の正確な理解

### 3.1 検出位置と方法

XRUN 検出は **Audio Callback 内**（`processBlockDouble` / `getNextAudioBlock`）で行われる。監視タイマースレッドではない。

```cpp
// AudioBlock.cpp (processBlockDouble内のXRUN検出)
{
    const auto t0_start = convo::getCurrentTimeUs();
    const uint64_t prevEnd = convo::consumeAtomic(lastCallbackEndTicks, ...);
    const double intervalMs = static_cast<double>(t0_start - prevEnd) / 1000.0;

    const double expectedMs = (numSamples / sampleRate) * 1000.0;  // = 5.33ms

    // XRUN検出: interval > max(expected * 1.5, 3.0ms)
    const double threshold = std::max(expectedMs * 1.5, 3.0);
    if (intervalMs > threshold) → XRUNイベントとしてリングバッファにpush

    const auto t1_end = convo::getCurrentTimeUs();
    publishAtomic(lastCallbackEndTicks, t1_end, ...);
}
```

### 3.2 Interval = start-to-start 計測

```
intervalMs(N) = t0_start(N) - t1_end(N-1)
              ≈ t0_start(N) - t0_start(N-1)  （preamble ≪ 5μs）
```

つまり **前回コールバックの開始から今回コールバックの開始までの経過時間**。DSP 処理時間そのものではなく、OS/Driver がコールバックをディスパッチする間隔である。

### 3.3 閾値の変遷

| 版 | 閾値式 | 実効値(192k/1024) | XRUN件数/秒 |
|----|--------|-------------------|------------|
| 旧 | `expectedMs * kRatio + kFixedMargin` （詳細不明） | 不明（過小） | 24/秒（過剰検出） |
| 現行 | `max(expectedMs * 1.5, 3.0)` | **8.0ms** | 1.0/秒 |
| 参考 | `max(expectedMs * 2.0, 5.0)` | **10.66ms** | さらに減少見込 |

### 3.4 `[TIMER] exec` との関係

`[TIMER] exec` は **Message Thread の timerCallback 実行時間**（avg=10.989ms）であり、Audio Thread のコールバック処理時間ではない。両者は **独立したスレッド** である。

## 4. 否定された仮説（5件）

### 仮説①: Logger I/O が Message Thread を占有 → XRUN を誘発 ❌

| 根拠 | 詳細 |
|------|------|
| asyncSink 導入後、XRUN レート不変 | 1.0/秒 → 1.0/秒。**全く変わらず** |
| `[TIMER] exec` 安定（avg=10.989ms） | 100ms 予算に対し 11%、余裕あり |
| CB_HIST の XRUN ガード後も XRUN 不変 | ログ出力削減は XRUN に影響なし |

**結論**: Logger I/O は GUI 応答性の原因だったが、**XRUN とは無関係**。

### 仮説②: CPU Migration によるキャッシュミス → DSP遅延 → XRUN ❌

| 根拠 | 詳細 |
|------|------|
| Same-CPU vs Diff-CPU の差 20μs | 予算 5.33ms の **0.4%**、無視可能 |
| Gen=5 では逆転（Diff の方が速い） | 一貫した傾向なし |
| MIG 無の方が Drift 大（P95: MIG有3,251μs vs MIG無4,392μs） | **因果が逆**（遅延→OSがMigration）の可能性 |
| `ThreadAffinityManager` は完全な No-Op | アフィニティ設定自体が初期化されていない |

**結論**: CPU Migration は「原因」ではなく「結果」。根本原因ではない。

### 仮説③: DSP 処理負荷が予算超過 → XRUN ❌

| 指標 | work60 | work62 |
|------|--------|--------|
| DSP P90 | 6,920μs（旧CB_HIST全数） | **1,311μs**（XRUN時のみ） |
| DSP P99 | 13,550μs | **1,874μs** |
| 予算超過率 | ~10% | **0.0%** |

work60 では CB_HIST が全 tick 出力だったため、Work60 の CB_HIST P90=6,920μs は **IR 未ロード時の Gen=4（proc=289μs）と IR ロード後の Gen=5（proc=10,270μs）の混合** であった。work62 の CB_HIST は XRUN ガードにより XRUN 発生時のみに限定されるが、それでも **P99=1,874μs** と余裕がある。

**結論**: XRUN として記録された callback における DSPCore 処理時間は予算内であり、DSPCore 以降の処理が XRUN の直接原因である可能性は低い。

### 仮説④: メモリプレッシャー（PageFaults/WS増加）→ XRUN ❌

| 根拠 | 詳細 |
|------|------|
| 定常状態の PageFaults | 100〜167/秒（無視可能） |
| XRUN レート時間不変 | 初期化直後（Private=1,354MB）も 6分後（Private=2,497MB）も同レート |
| IR 読み込み時の大規模 PageFault | **初期化時1回限り**（+320,903 PF）。定常状態では影響なし |
| Working Set 2.5GB | ページングは発生していない（メモリ80%使用だが安定） |

**結論**: メモリプレッシャーは定常状態では XRUN の原因ではない。

### 仮説⑤: Publish の瞬間的な影響（Crossfade/Latency更新）→ XRUN ❌

| 根拠 | 詳細 |
|------|------|
| Publish は O(1) atomic ポインタスワップ | RCU Reader ノンブロッキング確認済み |
| Crossfade は LinearRamp で 0→1 にスムージング | 過渡的乱れは設計上最小限 |
| Latency 更新は startDelayBlocks で補償 | レイテンシ差を吸収してからクロスフェード開始 |
| Publish 前後で XRUN レートの変化なし | Session3 で gen=4→gen=5 の前後で変化なし |

**結論**: Publish/Crossfade の瞬間的影響は XRUN の原因ではない。

## 5. 原因分析 — 確定したこと・確定していないこと

### 5.1 DSPCore 以降の要因については否定的な結果

以下の要因は、コード調査・実機計測・ログ解析の3方向から否定的な結果が得られた（注: いずれも DSPCore::process() 以降の区間についての評価であり、processBlockDouble 入口から DSPCore までの区間および JUCE callback ディスパッチ以前の区間は含まない）:

| 仮説 | 判定 | 根拠 |
|------|------|------|
| DSPCore 処理負荷が予算超過 | ❌ XRUN時では否定的 | CB_HIST P99=1.87ms << 5.33ms予算、超過率0.0%（ただし正常時DSP時間の分布は未計測） |
| Logger I/O (Message Thread) | ❌ 否定 | asyncSink導入後もXRUNレート不変 |
| CPU Migration によるキャッシュミス | ❌ 否定 | MIG無の方がDrift大、差は予算の0.4% |
| メモリプレッシャー | ❌ 否定 | XRUNレート時間不変、PageFaults安定 |
| Publish/Crossfade の瞬間的影響 | ❌ 否定 | publish前後で変化なし |

### 5.2 強く示唆されるが未確定: コールバック到達以前の遅延

```
ASIO Driver ISR (Voicemeeter)
  → DPC (Deferred Procedure Call)
  → MMCSS → スレッドディスパッチ
  → JUCE audioDeviceIOCallback
  → ★ この区間が未計測 ★
  → AudioEngine::processBlockDouble()
    [XRUN計測はここから] → DSPCore::process()
```

現在の XRUN 計測は `processBlockDouble` の先頭で行われるため、それ以前の全区間（ドライバ ISR から JUCE ディスパッチまで）がまとめて Interval に含まれる。以下の観測事実は「processBlockDouble 到達以前に何らかの固定的な遅延が存在する」ことを強く示唆する:

| 観測事実 | 示唆 |
|---------|------|
| XRUN検出閾値が8.0msのため、XRUNとして記録されるintervalは常に8.0ms以上。8.0ms未満のintervalが「存在しない」のか「存在するがXRUNとして記録されていないだけ」かは、現状のログからは判断できない（正常時intervalの分布は未計測）。 | 5.33〜8.0msの区間に正常なJUCE callback到着が存在する可能性があるが確認不可。CallbackArrival計測によりJUCE callback到着時刻の実測分布が初めて定量可能になる。 |
| XRUN レートが時間不変（初期化直後も6分後も同一） | OSリソース変化ではなく固定的なオーバーヘッド |
| DSP P99=1.87ms << 5.33ms にもかかわらず XRUN 発生 | DSP時間以外の区間が支配的 |
| Voicemeeter + SAVIHost + MConvolutionEZ では音飛びなし | ConvoPeq 固有の問題ではない可能性 |

### 5.3 遅延の発生箇所を特定するには

遅延の原因が以下のどの層にあるかは、現状のログだけでは特定できない。CallbackArrival 計測（work61 で設計済み、RT側の実装は完了）と ETW による外部計測が必要である:

| 層 | 検証方法 |
|----|---------|
| Windows スケジューラ | ETW: Ready Thread / Context Switch / DPC / ISR |
| MMCSS "Pro Audio" | AVRT_PRIORITY 変更試験 + 比較計測 |
| JUCE audioDeviceIOCallback | CallbackArrival で processBlockDouble 直前にタイムスタンプ取得 |
| ASIO ドライバ (Voicemeeter) | 他ASIO ドライバとの比較試験 |

## 6. 未確定要素（原因特定に必要な追加検証）

以下の項目は、現状のログ・コード調査だけでは確定できない。**原因の完全な特定にはこれらの追加検証が必須である。**

### 6.1 🔴 CallbackArrival 計測（DiagDrain 実装が必要）

work61 で設計された `recordCallbackArrival()` は **RT 側（AudioBlock.cpp, BlockDouble.cpp）の実装は完了** しており、`diagBuffer` に DiagEvent を push している。しかし **Timer.cpp の DiagDrain が未実装** のため、このデータは読み出されず破棄されている。（work60 Phase 8 のコードが git checkout で消失した可能性が高い。）

CallbackArrival が取得できれば、以下の差分が定量化可能になる:

```
JUCE audioDeviceIOCallback入口
  → [JUCE内部処理]
  → recordCallbackArrival() の timestampUs
  → [preamble: ここが未計測]
  → processBlockDouble の先頭
  → XRUN計測（Interval）
```

→ timestampUs の実測値から、「JUCE ディスパッチから processBlockDouble までの時間」と「callback 間隔の変動」が分離できる。

### 6.2 🔴 ETW による OS 層の計測

CallbackArrival では「ConvoPeq 内」までの計測しかできない。OS スケジューラやドライバレベルの遅延を直接見るには ETW（Event Tracing for Windows）が必要:

| トレース | 対象 |
|---------|------|
| `Thread::ReadyThread` | スレッドが実行可能になったがディスパッチされなかった時間 |
| `Thread::CSwitch` | コンテキストスイッチのタイミングと待ち時間 |
| `DPC::Routine` | 遅延プロシージャコールの実行時間 |
| `ISR::Routine` | 割り込みサービスルーチンの実行時間 |

### 6.3 MMCSS AVRT_PRIORITY の現在値

- `AVRT_PRIORITY_NORMAL`（=7）がデフォルトで設定されている可能性が高い
- コード内に `AvSetMmThreadCharacteristics()` の呼び出しは存在しない（JUCE 内部で設定されている可能性はある）
- AVRT_PRIORITY_HIGH（=8）以上に昇格した場合の効果は未検証

### 6.4 他 ASIO ドライバとの比較試験

- Voicemeeter BANANA + SAVIHost + MConvolutionEZ では音飛びが発生しないことが確認されている
- ただし ConvoPeq と Voicemeeter 以外の ASIO ドライバ（ASIO4ALL, FL Studio ASIO, ハードウェア ASIO）での比較は未実施
- **Voicemeeter が原因であるとは断定できない**（比較データがないため）

## 7. 対策推奨

### Priority 3: バッファサイズ増加試験（原因特定より現象緩和）

2048 samples（10.67ms）に増加することで、現在 XRUN の原因となっている 8〜10ms の dispatch 遅延が予算内に収まる可能性がある。ただしこれは現象の緩和であって原因の特定には寄与しない。また、ドライバや OS の挙動がバッファサイズ変更により変化する可能性もあるため、実機での確認が必要。

## 8. 追加検証計画（原因確定のため）

### Step A: DiagDrain 実装（最優先 — CallbackArrival 出力のための前提作業）

Timer.cpp に `diagBuffer.pop()` + フォーマットコード（8カテゴリ + CallbackArrival）を追加する。これにより:
- CallbackArrival の timestampUs が出力され、JUCE callback 到着時刻の実測分布が初めて定量可能に
- CPU_MIG / DSP_TIMING / EQ_TIME / CONV_TIME / STCONV_TIME / CALLBACK_STAGE / CB_SEQ の診断情報が復活
- 実装指針は work60 設計書 `診断ログRT違反_numeric-only設計レポート_2026-06-30.md` Phase 8 に従う

### Step B: CallbackArrival ログ取得

DiagDrain 完成後、JUCE callback 到着時刻（timestampUs）を実測する。これにより:
```
JUCE callback到着時刻 (recordCallbackArrival)
  → processBlockDouble 先頭 (XRUN Interval計測)
  → この差分が初めて定量可能に
```
- 到着時刻自体が 5.33ms から乖離 → 遅延は ConvoPeq 外部（OS/JUCE/Driver）
- 到着時刻は 5.33ms だが processBlockDouble 到着が遅延 → 遅延は ConvoPeq 内部（RuntimeScope/ASSERT/preamble）

### Step C: ETW トレース（CallbackArrival 判定後）

CallbackArrival で「外部」と判定された場合のみ、ETW で OS/Driver 層の遅延を特定する。

### Step D: MMCSS AVRT_PRIORITY_HIGH 試験

コードに `AvSetMmThreadCharacteristics("Pro Audio")` + `AvSetMmThreadPriority(AVRT_PRIORITY_HIGH)` を追加し、XRUN レートの変化を確認する。

### Step E: バッファサイズ 1024→2048 試験

現象緩和策として最も即効性が期待できる。ただし原因特定には寄与しない。
