# ConvoPeq GUI応答遅延 — 再評価 確定報告書

**作成日**: 2026-07-03 (v2.0 - 優先順位を全面的に改訂)
**根拠ログ**: 45秒間, 45,635行, ~1,014行/sec
**更新理由**: 初版(CB_HIST主犯説)からLogger::writeToLog主犯説へ修正

---

## 重要訂正: 初版分析の誤り

初版報告書では「CB_HISTダンプ(48%)とCPU_MIG(38%)が主原因」としたが、**これは誤りだった。**

実際の因果関係:

```
CB_HISTの32件全ダンプ(XRUNなしでも毎tick)
  + CPU_MIGの全件記録(266/sec)
  + その他イベント(CALLBACK_STAGE等)
→ 1tickあたり ~94行の Logger::writeToLog 呼び出し
→ FileLogger::logMessage() で 94回の CreateFile + WriteFile + CloseHandle
→ Message Thread 上の同期ファイルI/O負荷 → Timer遅延 + GUIブロック
```

**すなわちCB_HIST/CPU_MIGは「発端」ではなく「増幅器」であり、真の原因は `Logger::writeToLog` → `FileLogger::logMessage` の1行ごとのファイルI/Oである。**

### 🔴 重大発見: Timer実行時間計測がバグで動作していない

```cpp
// AudioEngine.Timer.cpp (timerCallback末尾)
static double s_timerStartMs = 0.0;  // ← ここがバグ！
if (s_timerStartMs > 0.0) {  // ← s_timerStartMs は 0.0 のまま NEVER SET
    const double execMs = juce::Time::getMillisecondCounterHiRes() - s_timerExecStartMs;
    if (execMs > 10.0) {
        // ★ このブロックは絶対に実行されない！
        diagLog(… "[TIMER] exec=…");
    }
}
s_timerExecStartMs = juce::Time::getMillisecondCounterHiRes();  // s_timerExecStartMs はセットされている
```

**原因**: 開発者は `s_timerExecStartMs` をチェックするつもりだったが、`s_timerStartMs` と書いてしまった（0.0のまま初期値で固定）。
**結果**: 条件が常に偽 → `[TIMER] exec` 行は絶対に出力されない → **timerCallbackの実実行時間が全くわからない。**

このバグにより、初版分析では「timerCallback < 10ms」と推定したが、**これは誤りであった可能性が高い。**

## 原因ランキング（確定版）

### 🥇 P0: `Logger::writeToLog()` → `FileLogger::logMessage()` の1行ごとのファイルI/O

**メカニズム**: `diagLog()` → `Logger::writeToLog()` → `FileLogger::logMessage()` のフルパス:

```cpp
// JUCE/modules/juce_core/logging/juce_Logger.cpp
void Logger::writeToLog (const String& message) {
    if (currentLogger != nullptr)
        currentLogger->logMessage (message);  // virtual dispatch
}

// JUCE/modules/juce_core/logging/juce_FileLogger.cpp
void FileLogger::logMessage (const String& message) {
    const ScopedLock sl (logLock);       // ① Mutex Lock (CriticalSection::enter)
    DBG (message);                        // ② OutputDebugStringW (Windows API)
    FileOutputStream out (logFile, 256);  // ③ openHandle() → CreateFileW (Windows API)
    out << message << newLine;            // ④ writeInternal() → WriteFile (Windows API)
}                                          // ⑤ デストラクタ → flushBuffer() → closeHandle() → CloseHandle (Windows API)
```

**1回のlogMessageで5回のシステムコール＋1回のMutexロック。**
**~940回/sec → ~4,700回のシステムコール/sec がMessage Threadから発行される。**

#### Windowsでの推定コスト（実測値の推定）

| 操作 | 推定レイテンシ | 要因 |
|------|---------------|------|
| CriticalSection::enter | ~0.001ms | ユーザーモード |
| OutputDebugStringW | ~0.01ms | カーネルトランジション |
| **CreateFileW** | **~0.05-0.5ms** | **カーネル + リアルタイムスキャン(Defender)** |
| WriteFile | ~0.001ms | キャッシュ書き込み |
| **CloseHandle** | **~0.05-0.5ms** | **カーネル + リアルタイムスキャン** |
| 合計(1回) | **~0.11-1.01ms** | AVスキャンの有無で変動 |
| **合計(94回/tick)** | **~10-95ms** | ← Timer周期100msを超える可能性 |

**推定**: リアルタイムスキャン(Windows Defender等)が有効な場合、1回のファイルI/Oに 0.3-0.5ms かかる。94回/tick → **28-47ms/tick がLoggerで消費**。これがTimerジッター55msの主要因。

### 🥈 P1: `sendChangeMessage()` → `updateAllControls()`

**発火回数**: 45秒間で **5回**（初回起動時の初期状態反映 + 後続のUI変更）

**5回/45sec = 0.11/sec では直接的な原因とは言えないが、問題を増幅させる:**

```
sendChangeMessage()
→ MainWindow::changeListenerCallback()
    → eqPanel->updateAllControls()     // UI全要素の再描画をトリガー
    → convolverPanel->updateIRInfo()   // IR情報更新
```

`updateAllControls()` は全てのEQ UI要素のラベル・値を再設定し、`repaint()` をキューイングする。これによりメッセージキューに大量の再描画イベントが積まれ、直後のTimer発火が遅延する。

### 🥉 P2: CB_HISTの無条件全tickダンプ

**問題**: XRUN発生時のみの意図が、`writeCount != lastDumpedCount` 条件により**全tickで発火**。
**34倍過剰な出力**（予定10/sec → 実際339/sec）が、Logger I/O呼び出しを水増ししている。

**真の影響**: CB_HISTそのものが重いのではなく、**Loggerへの書き込み回数を水増ししている**点が問題。

### P3: CPU_MIG 全件記録（CPU affinity欠如）

**問題**: Audio ThreadにCPU affinity設定なし → 1.1 callbackに1回のCPU Migration → **266件/secのログ**

真の影響はCB_HIST同様、Loggerへの書き込み回数増加。

### P4: `diagLog()` が DBG + Logger::writeToLog の2重出力

**各diagLog呼び出しで2つのアウトプット**:
- `DBG()` → OutputDebugString (軽量, 無視可能)
- `Logger::writeToLog()` → File I/O (重量, 上記P0)

### P5: `diagPrefix()` が重い

`juce::Time::getCurrentTime()` + `formatted()` + `getCurrentTimeUs()` + 5回のString連結を **16回/tick**実行。

### P6: `processLearningCommands()` 毎tick実行

173行の関数を毎tick実行。DBG 5回 + Logger 7回を含む。OutPutDebugString（DBG）は軽量だが7回のLogger呼び出しは無視できない。

## 改訂 修正優先順位

### 🔴 P0: Logger::writeToLog を timerCallback から排除 or 無効化

**3つの選択肢:**

#### A. Releaseビルドで CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS を OFF にする
```cmake
if(CMAKE_BUILD_TYPE STREQUAL "Release")
    target_compile_definitions(... PUBLIC CONVOPEQ_ENABLE_RUNTIME_DIAGNOSTICS=0)
endif()
```
- **効果**: 全診断ログがコンパイル時に消失。**ファイルI/Oゼロ。**
- **リスク**: デバッグ情報が得られない
- **実装時間**: CMakeLists.txt の1行変更

#### B. diagLog をリングバッファ蓄積＋別スレッド書き出しに変更
```cpp
// 案: LockFreeQueue<String> にプッシュし、別スレッドが consume してまとめて writeToLog
static LockFreeQueue<juce::String, 1024> s_diagQueue;
void diagLog(const juce::String& message) {
    s_diagQueue.push(message);  // これだけ → ~0.001ms
}
// 別スレッドで100msごとに:
//   while (s_diagQueue.pop(msg)) batch.append(msg);
//   Logger::writeToLog(batch);  // 1回のI/Oにまとめる
```
- **効果**: Logger I/OをMessage Threadから完全排除。~940回/秒 → 10回/秒
- **リスク**: アプリケーションクラッシュ時に最新ログが消失
- **実装時間**: ~半日

#### C. FileLoggerをやめてOutputDebugStringのみにする（DBGのみ）
```cpp
void diagLog(const juce::String& message) {
    DBG(message);  // OutputDebugStringのみ（ファイル書き込みなし）
}
```
- **効果**: ファイルI/Oが完全になくなる。DBGは軽量
- **リスク**: ファイルにログが残らない（DebugView等で代替可能）
- **実装時間**: 5分

### 🟡 P1: timerCallback の実行時間計測バグを修正

```cpp
// BEFORE (broken):
static double s_timerStartMs = 0.0;
if (s_timerStartMs > 0.0) {  // s_timerStartMs は常に0.0

// AFTER (fixed):
static double s_timerExecStartMs = 0.0;  // 変数名も統一推奨
if (s_timerExecStartMs > 0.0) {
```
- **効果**: 実際のtimerCallback実行時間が可視化され、以後の最適化が定量的に可能に

### 🟡 P2: CB_HISTダンプ条件の修正

```cpp
// BEFORE: 毎tick 32件ダンプ
const uint64_t wc = rtLocalState_.callbackTimingWriteCount.load(...);
if (wc != rtAuxMutable_.lastCbHistDumpedWriteCount) {  // ← 条件が毎回成立

// AFTER (案A): XRUN検出時のみ
if (xRunDetected) {
    // dump CB_HIST
}

// AFTER (案B): sampling mask で間引き（例: 10tickに1回）
static int cbHistTickCounter = 0;
if (++cbHistTickCounter % 10 == 0) {
    // dump CB_HIST
}
```

### 🟢 P3: Audio Thread に CPU affinity 設定

```cpp
// ThreadAffinityManager.h
enum class ThreadType {
    Audio,  // ★ 追加
    Worker, LearnerMain, LearnerEval, HeavyBackground, LightBackground, UI
};

// AudioEngine.Processing.PrepareToPlay.cpp
// prepareToPlay() 内で:
affinityManager.applyCurrentThreadPolicy(ThreadType::Audio);
```

### 🟢 P4: `sendChangeMessage()` 間引き

```cpp
// timerCallback 内:
static uint32_t lastSendChangeMs = 0;
const uint32_t now = juce::Time::getMillisecondCounter();
if (now - lastSendChangeMs >= 500) {  // 最大2Hzに制限
    sendChangeMessage();
    lastSendChangeMs = now;
}
```

### 🟢 P5: CONVOPEQ_DIAG_SAMPLE_MASK 調整

0xF(1/16) → 0x3F(1/64) で CPU_MIG/CB_ARRIVAL/CALLBACK_STAGE の出力を1/4に削減
