### 3.2 実装詳細

**ファイル**: `src/audioengine/AudioEngine.Timer.cpp`

**挿入位置**: timerCallback 末尾、`flushLogBuffer()` の直前

**設計方針**: 単一の `formatDiagEvent()` 関数にswitchをカプセル化し、Timer.cpp 側は呼び出しのみとする。これにより Timer.cpp のコード増加を最小限に抑えつつ、保守性を確保する（個別フォーマッタ関数への分割は Step 8.1）。

**注意**: DiagStatistics の出力は **Step 8a には含めない**（Step 8.1 として後日、dropRate確認後に判断）。Step 8a は pop + format + diagLog の最小復旧に徹する。

```cpp
    // ★ DiagDrain: RTスレッドが書き込んだ DiagEvent リングバッファを消費
    {
        const uint64_t gen = (runtimeWorld != nullptr)
            ? static_cast<uint64_t>(runtimeWorld->generation) : 0;

        DiagEvent event;
        int drained = 0;
        const uint64_t drainStartUs = convo::getCurrentTimeUs();  // DiagDrain処理時間計測開始
        while (drained < static_cast<int>(DiagRuntimeLimits::MaxDrainPerTick)
               && diagBuffer.pop(event))
        {
            ++drained;
            diagLog(formatDiagEvent(event, gen));
        }
        const uint64_t drainUs = convo::getCurrentTimeUs() - drainStartUs;

        // ★ Step 8a: drained件数+drain時間+占有数を記録（500tick周期）
        //   drained==0 → バッファ空（過剰なDrain枠）
        //   drained==MaxDrainPerTick → 上限到達。ただしDrain不足とは限らない（全件処理済みの可能性もある）
        if ((s_blockTickCount % 500) == 0) {
            const size_t approxOcc = diagBuffer.size();  // ベストエフォート近似占有数（lock-freeのため正確値ではない）
            diagLog(diagPrefix(gen) + " [DIAG_DRAIN] tick=" + juce::String(static_cast<juce::int64>(s_blockTickCount))
                + " drained=" + juce::String(drained)
                + " drainUs=" + juce::String(static_cast<juce::int64>(drainUs))
                + " max=" + juce::String(static_cast<int>(DiagRuntimeLimits::MaxDrainPerTick))
                + " hitLimit=" + juce::String(drained == static_cast<int>(DiagRuntimeLimits::MaxDrainPerTick) ? "true" : "false")
                + " approxOcc=" + juce::String(static_cast<juce::int64>(approxOcc)));
        }
    }
```

`drained` の per-tick 記録により、以下の分析が事後可能:
- `drained==0` → このtickでDrainすべきイベントなし（Drain枠に対してイベント量が少ない）
- `drained==MaxDrainPerTick`（`hitLimit=true`）→ 上限に達してループ終了。`approxOcc` が大きければDrain不足の可能性（ただし全件処理済みの可能性もあり）
- `approxOcc`（drain直後のバッファ近似占有数、lock-freeのため正確値ではない）により、Drain能力の過不足を推定できる（例: drained=64, approxOcc=130 → Drain不足の可能性が高い）
- `tick` 番号によりETWの時系列データと突き合わせ可能
- `drainUs`（DiagDrainループ全体の処理時間）により、Logger負荷・format負荷・DiagDrain自身のコストが定量可能。Timer全体の実行時間（BLOCK_TIMING）とは別に、DiagDrain単体の時間を追跡できる

### 3.3 フォーマッタ関数 `formatDiagEvent()`

switch を内包する単一のフォーマッタ関数。匿名 namespace 内に定義し、Timer.cpp の DiagDrain ループから呼ばれる。

```cpp
namespace {

juce::String formatDiagEvent(const DiagEvent& event, uint64_t gen) {
    switch (event.category) {
        case DiagCategory::CpuMig:
            return diagPrefix(gen) + " [CPU_MIG] cbIdx=" + juce::String(static_cast<juce::int64>(event.eventIndex))
                + " pubSeq=" + juce::String(static_cast<juce::int64>(event.data.cpuMig.pubSeq))
                + " gen=" + juce::String(static_cast<juce::int64>(event.data.cpuMig.generation))
                + " cpu=" + juce::String(static_cast<int>(event.data.cpuMig.cpu))
                + " prev=" + juce::String(static_cast<int>(event.data.cpuMig.prevCpu));
        case DiagCategory::CallbackArrival:
            return diagPrefix(gen) + " [CB_ARRIVAL] cbIdx=" + juce::String(static_cast<juce::int64>(event.eventIndex))
                + " ts=" + juce::String(static_cast<juce::int64>(event.data.callbackArrival.timestampUs))
                + " expected=" + juce::String(static_cast<juce::int64>(event.data.callbackArrival.expectedUs))
                + " cpu=" + juce::String(static_cast<int>(event.data.callbackArrival.cpu))
                + " tid=" + juce::String(static_cast<int>(event.data.callbackArrival.threadId));
        // ↑ 残り6カテゴリも同様（CallbackSequence, DspTiming, CallbackStage, EqTime, ConvTime, StereoConvTime）
        default:
            return diagPrefix(gen) + " [UNKNOWN] category=" + juce::String(static_cast<int>(event.category));
    }
}

} // namespace
```

**設計意図**: 単一関数にswitchをカプセル化することで、Timer.cpp のDiagDrainループは `diagLog(formatDiagEvent(event, gen))` のみとなる。カテゴリ追加時もこの関数のみ修正すればよい。個別フォーマッタ関数への分割（formatCpuMig, formatCallbackArrival等）は Step 8.1 で実施。

**将来性**: `DiagCategory::Count` センチネル値（=9）に対する `static_assert` により、カテゴリ追加時の `formatDiagEvent` 更新忘れをコンパイル時に検出。

## 4. Step 9: ETW 取得（コード変更不要、すぐに分岐可能）

### 4.1 なぜ ETW が先か（並行実行可能）

ETW は **コード変更が一切不要** であり、ConvoPeq 起動中に Windows Performance Recorder でトレースを取得できる。Step 8 (DiagDrain) との依存関係はなく、**並行して実行可能**。

**分岐ロジック**:

```
ETW で ReadyThread > 3ms のディスパッチ待ちが確認できる
├── Yes → Schedulerによるディスパッチ遅延を強く示唆（ただしMMCSS不足/CPU飽和/高IRQL/他RTスレッドの
  │         影響など複数原因があり、ETW単体では断定できない。追加分析または段階的改善で検証）
  └── No  → ConvoPeq/JUCE 側の精査へ。CallbackArrival で到着間隔の実測。
```

つまり ETW は「ショートカット」になる可能性がある。DiagDrain と並行して取得することで、原因特定までの最短経路を確保する。

### 4.2 取得対象

| トレース | 対象 |
|---------|------|
| `Thread::ReadyThread` | スレッドが実行可能になったがディスパッチされなかった時間 |
| `Thread::CSwitch` | コンテキストスイッチのタイミングと待ち時間 |
| `DPC::Routine` | 遅延プロシージャコールの実行時間 |
| `ISR::Routine` | 割り込みサービスルーチンの実行時間 |

### 4.3 取得方法

```powershell
# Windows Performance Recorder でトレース開始
wpr -start Audio -start ConvoPeqProfile
# ConvoPeq を起動・動作
# トレース停止
wpr -stop ConvoPeqTrace.etl
# WPA で解析
wpa ConvoPeqTrace.etl
```

### 4.4 評価指標

ReadyThread は **P99 を最優先指標** として評価する。単一の固定閾値（例: 3ms）ではなく以下を参照:
- **P99 ReadyTime** — 最優先。例: P99=4ms, 平均=30μs → 異常（テール遅延）
- 平均 ReadyTime
- P95 ReadyTime
- 最大 ReadyTime
- ReadyTime の時系列プロット

### 4.5 期待される成果（分布で評価）

| ETW 結果 | 結論 |
|---------|------|
| ReadyThread に数msの待ちが頻発（P95 >> 1ms） | **Schedulerによるディスパッチ遅延を強く示唆**（MMCSS/CPU飽和/高IRQL/他RTスレッド等の追加分析が必要） |
| 加えて: XRUN発生時刻とReady待ちが一致 | **因果関係の強い証拠**。Schedulerでのディスパッチ遅延がXRUN原因と特定可能 |
| ReadyThread はほぼゼロ（P95 < 100μs） | **ConvoPeq/JUCE 入口に原因**（CallbackArrival で精査） |
| DPC/ISR が長時間占有 | **ドライバ（Voicemeeter）が原因** |

ReadyThread の評価は **分布（平均・P95・P99・最大）** に加え、**XRUN発生時刻との一致** を確認する。XRUNが起きた callback と Ready待ちが一致していれば Scheduler によるディスパッチ遅延を強く示唆する。

## 5. Step 10: CallbackArrival 解析

### 5.1 Step 10a: 到着間隔 deltaArrival 解析（中核）

**前提条件**: Step 8a (DiagDrain) 完了

現在の XRUN Interval は `processBlockDouble` 先頭で計測している。`CallbackArrival.timestampUs` は `processBlockDouble` より前に `recordCallbackArrival()` で記録される。

これにより以下が初めて定量可能になる:

| 計測値 | 意味 |
|--------|------|
| `deltaArrival = timestampUs(N) - timestampUs(N-1)` | **JUCE callback の実到着間隔** |
| `interval = t0_start(N) - t1_end(N-1)` | processBlockDouble の start-to-start（現行のXRUN計測） |
| `deltaArrival - interval` | **ConvoPeq入口〜DSP開始までの差分指標（proxy metric）**（測定区間が一致しないため近似値として扱う） |

**判定ロジック**:
```
Case A: deltaArrival ≈ 5.33ms（期待値通り）
   → recordCallbackArrival() までは正常。遅延は「recordCallbackArrival〜processBlockDouble」の間
     （ConvoPeq内部オーバーヘッドまたは JUCE ディスパッチ直後）
   → deltaArrival - interval でConvoPeq入口〜DSP開始までの差分指標（proxy metric）を得る

注意: `interval = t0_start(N) - t1_end(N-1)` であり、「前回終了→今回 processBlockDouble 開始」の差。厳密な preamble 時間（Arrival→t0_start）は、別途イベントを追加しない限り完全には求められない。したがって deltaArrival - interval は「差分指標（proxy metric）」として扱う。

Case B: deltaArrival ≈ 8〜10ms（XRUN interval と同程度）
   → recordCallbackArrival() より前の区間（OS/JUCE/processBlockDouble入口）に遅延が存在
   → ETW 結果と突き合わせて特定

補足:
  deltaArrival は recordCallbackArrival() の timestampUs から算出されるため、
  「JUCE callback 到着直後」を計測している。Driver ISR から JUCE callback までの区間
  （OS Scheduler/MMCSS/DPC/ISR）はこの計測範囲外であり、別途 ETW が必要。
```

### 5.2 Step 10b: 正常時サンプリング取得（段階的、dropRate確認後）

現在 `CB_HIST` は「XRUN 時のみ」のため、正常時の DSP 時間分布が不明。

**方法**: まず全 CB_ARRIVAL（187 events/sec）を push し、DiagStatistics の dropRate を確認する。dropRate が許容範囲（< 1%）であれば sampling 不要。dropRate が大きい場合は必要に応じて検討する。

**方針**:
```
Step 8a 完了
  ↓
実機10分: DIAG_STAT で dropRate 確認
  ├── dropRate ≈ 0% → sampling不要、全CB_ARRIVAL取得可能
  └── dropRate > 1% → 必要なら sampling導入を検討
```

**現時点で固定的な段階設計は行わない**。理由:
- 実際のイベント量は運用条件（CPU_MIG/DSP_TIMING等の発生頻度）で変わる
- dropRate 実測後に初めて必要なsampling率が判断できる
- 最初から full-rate で取得できれば XRUN 解析の情報量が最大になる

```
正常時 deltaArrival の実測分布例（仮）:
  5.3ms, 5.4ms, 5.2ms, 9.1ms, 5.3ms ...
   → P50=5.3ms, P99=9.1ms → 通常は正常、テールのみ異常

  7.8ms, 7.9ms, 8.1ms, 7.7ms ...
   → 全て予算超過気味 → 固定的なオフセット遅延
```

### 5.3 Step 10f: Arrival〜Exit タイムライン拡張（将来拡張、優先度低）

現状の `recordCallbackArrival()` で記録しているのは `timestampUs`（到着時刻）のみ。以下の拡張で完全なタイムラインが取得可能:

| 計測点 | タイミング | 差分の意味 |
|--------|-----------|-----------|
| `timestampUs` (recordCallbackArrival) | **JUCE callback 到着直後** | — |
| `t0_start` (XRUN計測部) | processBlockDouble 先頭 | **ConvoPeq入口〜DSP開始までの差分指標** |
| DSPCore::process() 終了後 | — | DSP 処理時間（既存: CB_HIST proc） |
| `t1_end` (XRUN計測部) | block 終了 | **block 後処理時間** |
| `lastCallbackEndTicks` | callback 完全終了 | **callback 最終出口** |

この拡張は「影響度の高い」または「原因が特定できなかった場合のみ」実施すればよい。まずは deltaArrival の基本分析から開始する。

### 5.5 Step 10d: deltaArrival ヒストグラム（100μs刻み）

**目的**: deltaArrival の生ログだけでは分布の全体像を把握しにくい。100μs 刻みのヒストグラムを生成することで、以下の識別が容易になる:

```
パターンA: 5.1ms〜5.4ms に集中 + たまに9ms
  → 通常は正常、テールのみ異常（OS スケジューラの一時的停滞）

パターンB: 7.8ms〜8.2ms に集中
  → 固定的なオフセット遅延（MMCSS/Driver/JUCEの定常的オーバーヘッド）

パターンC: 全域に分散（5〜12ms）
  → システム負荷に依存した変動的遅延
```

**方法**: deltaArrival の実測値をログ解析時にスクリプトでヒストグラム化する（ConvoPeq のコード変更不要、事後解析）。50/100/250μs を切り替え可能にしておくと後で便利。

**優先度**: Step 10e（callbackIndex相関）より低い。ヒストグラムは後からログ再解析で何度でも作成可能だが、callbackIndex 相関は実機データがなければ分析できないため。

### 5.4 Step 10e: callbackIndex 相関分析（Step 10dより優先）

DiagEvent は全カテゴリに `eventIndex`（callbackIndex）が含まれている。これをキーに以下の相関分析が可能:

| 相関 | 分析内容 |
|------|---------|
| CPU_MIG × deltaArrival | CPU migration 直後だけ到着間隔が悪化するか |
| XRUN × deltaArrival | XRUN 発生時の到着間隔が特に悪いか |
| DSP_TIMING × deltaArrival | Observe 前後で到着間隔が変化するか |

```
例: 同一 callbackIndex で突き合わせ
  callbackIndex=12345: deltaArrival=9.1ms, CPU_MIG(cpu=4→15), XRUN#42
  callbackIndex=12346: deltaArrival=5.3ms, (CPU_MIGなし)
  → migration直後だけ到着間隔が悪化 → 相関あり
```

**優先理由**: ヒストグラム（Step 10d）は後からログ再解析で何度でも作成可能。一方 callbackIndex 相関は実機データがなければ分析できず、原因特定への貢献も大きい。

## 6. Step 11: XRUN 緩和策（原因特定後）

### 6.1 Step 11a: MMCSS AVRT_PRIORITY_HIGH

**位置づけ**: 原因特定ではなく「改善確認」。CallbackArrival/ETW で Scheduler によるディスパッチ遅延が強く示唆された場合に実施。

**実装場所**: `AudioEngine.Processing.AudioBlock.cpp` の `getNextAudioBlock()` 内（初回コールのみ `std::atomic<bool>` でガード）。`prepareToPlay`（Message Thread）ではなく Audio Thread で適用する。同期には `std::atomic<bool>` を使用（`std::call_once` は内部mutexのためRTスレッド非推奨）。

```cpp
#include <avrt.h>
#pragma comment(lib, "avrt.lib")

void AudioEngine::applyMmcssPriority() noexcept
{
    DWORD taskIndex = 0;
    HANDLE hTask = AvSetMmThreadCharacteristicsA("Pro Audio", &taskIndex);
    if (hTask != nullptr) {
        AvSetMmThreadPriority(hTask, AVRT_PRIORITY_HIGH);
        const int win32Priority = ::GetThreadPriority(::GetCurrentThread());
        const DWORD pc = ::GetPriorityClass(::GetCurrentProcess());
        diagLog("[MMCSS] registered: taskIndex=" + juce::String(static_cast<int>(taskIndex))
            + " priority=AVRT_PRIORITY_HIGH"
            + " win32Prio=" + juce::String(win32Priority)
            + " procClass=" + juce::String(static_cast<int>(pc)));
        // 参考: AvQuerySystemResponsiveness() で Windows全体のMMCSS状態も取得可能
    } else {
        const DWORD err = ::GetLastError();
        diagLog("[MMCSS] FAILED: GetLastError=" + juce::String(static_cast<int>(err))
            + " taskIndex=" + juce::String(static_cast<int>(taskIndex)));
    }
}
```

**ログ出力項目**:
- `AvSetMmThreadCharacteristicsA` の戻り値（hTask != nullptr の成否）
- `GetLastError()` — 失敗理由の特定（権限不足/登録済み/未対応）
- `TaskIndex` — 正常登録時は 0-2 の値。既存のMMCSSカテゴリ数で変化
- `GetThreadPriority()` — 成功時のWin32スレッド優先度。`AVRT_PRIORITY_HIGH` 適用後に実際に変化したか確認
- `GetPriorityClass()` — プロセスの優先度クラス（NORMAL/HIGH/REALTIME）。MMCSS登録後のプロセス全体の状態確認
- `AvQuerySystemResponsiveness()` [参考] — Windows全体のMMCSS応答性（0%=完全応答、100%=完全不応答）。システム負荷状態の確認になるが、XRUN原因特定への寄与は限定的。必要に応じて追加。

### 6.2 Step 11b: バッファサイズ 2048 試験

**位置づけ**: 現象緩和。原因特定には寄与しない。

ASIO ドライバ設定のバッファサイズを 1024→2048 に変更（コード変更不要）。現在の 8〜10ms の dispatch 遅延が 10.67ms 予算内に収まるかを確認。
