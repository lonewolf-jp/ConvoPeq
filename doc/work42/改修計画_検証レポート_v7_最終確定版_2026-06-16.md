### 1.2 Sprint3-B 全面再定義: World Reclaim Visibility → Grace Period Visibility

**v6の問題**: `retiredCount - activeWorldCount` を reclaim 数の推定式としたが、これは **理論的に不成立**。

**不成立の証明**:

```cpp
// WorldLifecycleAudit
void onWorldRetired(...) {
    convo::fetchSubAtomic(activeWorldCount_, 1u, ...);  // ★ retire 時点で -1
    convo::fetchAddAtomic(retiredCount_, 1u, ...);
}
```

`activeWorldCount` は **retire 時点** でデクリメントされる。reclaim (メモリ解放) はその後、Reader の grace period 完了後に発生する。

```
時系列:
  published:  activeWorldCount++
  retired:   activeWorldCount--, retiredCount++    ← ここで active は減る
                    ↓
             EBR Grace Period (Reader が epoch を保持)
                    ↓
  reclaimed: memory actually freed                  ← activeWorldCount は既に減っている
```

したがって:
```text
retired - active = retired - (published - retired)  (定常状態)
                = 2*retired - published
                ≠ reclaimed count
```

**v7の修正**: Sprint3-B を **Grace Period Visibility** に再定義。

| 項目 | v6 (誤り) | v7 (正しい) |
| --- | --- | --- |
| 名称 | World Reclaim Visibility | **Grace Period Visibility** |
| 計測対象 | World 解放数 (推定) | EBR Grace Period の健全性 |
| 指標 | `retired - active` (不成立) | `reclaimAttemptCount`, `reclaimSuccessCount`, `reclaimEfficiency` |
| World 追跡 | 試みた | **行わない** (現状の EBR 設計では World 単位の reclaim 追跡不能) |

## 2. 確定設計: Grace Period Visibility

### 2.1 追加指標

| 指標 | 型 | 意味 | 取得元 | 正確性 |
| --- | --- | --- | --- | --- |
| `reclaimAttemptCount` | `uint64_t` | `tryReclaim()` / `reclaim()` が呼ばれた延べ回数 | `EpochDomain` (新設) | ✅ 正確 |
| `reclaimSuccessCount` | `uint64_t` | 実際に `deleter(ptr)` が実行された延べ DeletionEntry 数 | `DeferredDeletionQueue::reclaim()` 戻り値 (累積) | ✅ 正確 |
| `reclaimEfficiency` | `double` | `successCount / attemptCount` | 事後計算 | ✅ |
| `stuckReaderCount` | `uint64_t` | 閾値超過で滞留中の Reader 数 | 既存 (`collectDrainAudit()`) | ✅ 既存 |
| `pendingRetireCount` | `uint32_t` | キュー内の未処理エントリ数 | 既存 (`ISRRetireRouter`) | ✅ 既存 |

### 2.2 EpochDomain 拡張

```cpp
// EpochDomain.h — 追加フィールド
private:
    std::atomic<uint64_t> globalEpoch;
    std::array<ReaderSlot, kMaxReaders> readers;
    DeferredDeletionQueue deferredDeletionQueue;
    std::atomic<uint64_t> reclaimAttemptCount_{0};
    std::atomic<uint64_t> reclaimSuccessCount_{0};
```

```cpp
// EpochDomain.h — tryReclaim() 変更
void tryReclaim() noexcept override
{
    reclaimAttemptCount_.fetch_add(1, std::memory_order_relaxed);  // ★ 試行回数
    const auto n = deferredDeletionQueue.reclaim(getMinReaderEpoch());
    reclaimSuccessCount_.fetch_add(n, std::memory_order_release);  // ★ 成功数
}
```

```cpp

};
```

## 4. 168h 試験判定式（確定版）

### 条件①: World 整合性 ✅

```text
publishedCount ≈ retiredCount + activeWorldCount
```
- 偏差5%以内
- `WorldLifecycleAudit` から測定

### 条件②: Grace Period 健全性 ✅（全面再設計）

```text
条件②a: reclaimSuccessCount が単調増加していること
        (試験期間中に減少する瞬間がない)

条件②b: reclaimEfficiency = reclaimSuccessCount / reclaimAttemptCount
        が 0.5 以上を維持していること
        (reclaim 呼び出しの過半数が実際にエントリを解放できている)

条件②c: stuckReaderCount が常に 0 であること
        (Reader 滞留による Grace Period 延伸がない)
```

### 条件③: Backlog 非増加 ✅

```text
pendingRetireCount の最大値が試験開始時の2倍を超えない
```

### 条件④: Shutdown 収束性 ✅

```text
routerPendingRetire がシャットダウン開始後60秒以内に0に収束
```

### 条件⑤: Recovery 健全性 ✅

```text
強制タイムアウト後、publishedCount / retiredCount / reclaimSuccessCount が
60秒以内に正常増加を再開する
```

## 5. 将来課題

以下の項目は Sprint3 のスコープ外とし、将来の課題として管理する:

| 課題 | 理由 | トリガー |
| --- | --- | --- |
| **World 単位の reclaim 追跡** | デリータが stateless lambda のため通知不可。`DeferredDeletionQueue` に型情報を持たせる大規模変更が必要 | World ↔ DeletionEntry の対応関係が必要になった場合 |
| **WorldLifecycleAudit への reclaim 情報統合** | 現状の EBR 設計では World 数と DeletionEntry 数が一致しない | より精密な運用監視が必要になった場合 |
| **`reclaimed ≈ retired - pendingRetire` 再評価** | EBR 理論上不成立だが、Reader が常に0なら成立する特殊ケース | Reader 不在が証明された場合 |
