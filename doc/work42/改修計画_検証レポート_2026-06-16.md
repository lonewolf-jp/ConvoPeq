### 2.5 RuntimeDrainAudit責務分離 (B-1) — 検証

**計画の主張**: `ShutdownCompletionState` 新設、`routerPendingRetire == 0` を完了条件に含める

**検証結果**: ⚠️ **修正の余地あり、計画に一部不正確な点あり**

**現状の責務分離**:

```cpp
// RuntimeDrainAudit::isAllZero() — 現在の状態
bool isAllZero() const noexcept {
    return pendingPublication == 0
        && pendingRetire == 0
        && activeCrossfadeCount == 0
        && deferredPublish == 0;
    // ★ routerPendingRetire が含まれていない！
    // ★ quarantineResident も含まれていない
}
```

**計画に対する検証**:

| 主張 | 検証結果 | 詳細 |
| --- | --- | --- |
| 完了判定と診断情報の分離が必要 | ✅ **妥当** | 現状もヘッダコメントで「isAllZeroは監査ログ出力専用」と明記。ただし完了判定関数自体が存在しない |
| `ShutdownCompletionState` 新設 | ✅ **妥当だが代替案あり** | 新規構造体ではなく、`RuntimeDrainAudit` に `canShutdown()` メソッド追加でも良い |
| `routerPendingRetire == 0` を含める | ✅ **妥当** | `isAllZero()` から確かに漏れている |
| 保守性向上 | ✅ **妥当** | 分離により shutdown 完了ロジックと診断ロジックが明確に分かれる |

**注意点**: 計画では「新規構造体 `ShutdownCompletionState`」と言っているが、実際には `RuntimeDrainAudit` に `canShutdown()` メソッドを追加するか、`ShutdownCompletionState` を別途定義して `collectDrainAudit()` から両方を返す方が、影響範囲が限定される。どちらでもよいが、**`ShutdownCompletionState` と `RuntimeDrainAudit` でデータの二重管理にならないように注意**。

### 2.6 World Reclaim Tracking (A-1) — 検証

**計画の主張**: `reclaimedCount` を監視対象に追加、`std::atomic<uint64_t> reclaimedCount_` 追加

**検証結果**: ⚠️ **計画は部分的に不正確。reclaimedCount は既に別コンポーネントに存在する**

**既存の `reclaimedCount`**:

| 場所 | 用途 |
| --- | --- |
| `RuntimePublicationState::PublicationLedger::reclaimedCount` | Orchestrator の出版ライフサイクル管理 |
| `RuntimePublicationState::OrchestratorForwardProgress::reclaimedCount` | 7段階進捗の最終段階（Reclaim停滞検出に使用） |
| `RuntimePublicationStateOwner::onReclaimed()` | カウント更新 |
| `TelemetryRecorder::OrchestratorHealthSnapshot::reclaimedCount` | テレメトリスナップショット |
| `ISREvidenceExporter.cpp:209` | evidence JSON 出力 |

**不足している場所**: `WorldLifecycleAudit` には `reclaimedCount_` がない

| `WorldLifecycleAudit` の現状 | 計画の要求 |
| --- | --- |
| `publishedCount_` ✅ | `publishedCount` ✅ |
| `retiredCount_` ✅ | `retiredCount` ✅ |
| `activeWorldCount_` ✅ | `activeWorldCount` ✅ |
| ❌ `reclaimedCount_` **未存在** | `reclaimedCount` 追加 ⬅️ **要実装** |

**重要な発見**: `WorldLifecycleAudit` の `onWorldRetired()` は `retiredCount_` をインクリメントするが、実際のメモリ解放（reclaim）とは独立している。`reclaimedCount` を追跡するには、**`DeferredDeletionQueue` からエントリが実際に削除されたタイミング** でカウントする必要がある。これは `retireDSP` の後段で発生するため、`WorldLifecycleAudit` に単純に `reclaimedCount_` を追加するだけでは不十分 — 実際の reclaim イベントをフックする仕組み（例: `AudioEngine::deleteRetiredDSP` や `RuntimePublicationStateOwner::onReclaimed()` からの通知）が必要。

**結論**: 「統計版（選択肢A）」のアプローチは正しいが、**実装には reclaim イベントのフックポイントの追加が必要**。計画が言う「単に `reclaimedCount_` 追加」だけでは実現しない。

## 5. 最終評価

### 総評: **計画は概ね妥当。ただし部分的な修正を推奨**

#### ✅ 強く支持する点

1. **B-2 ISRBarrierOptimizer削除**: コードベース調査で完全にデッドコードであることを確認。**真のゼロリスク**。Sprint 1 最初に実施すべき。
2. **A-2 優先度 ★★★★★**: Shutdown時の診断能力不足は最も深刻なギャップであり、スコープ・実装ガードレール共に妥当。
3. **A-3 優先度 ★★★★☆**: 時系列データにより運用診断が飛躍的に向上。サイズ64推奨は適切。
4. **Sprint 順序**: B-2→A-2, A-3→B-1, A-1 → 依存関係と合致。

#### ⚠️ 修正を推奨する点

1. **A-1 の記述を補正**: `reclaimedCount` が既存コンポーネントに存在することを明記し、`WorldLifecycleAudit` への追加が本当に必要か再評価すべき。
2. **168h試験の具体的手順を追記**: 環境・負荷・監視・閾値を明文化しないと「完成宣言」の客観性が担保できない。
3. **Evidence出力形式の統合方針**: A-2/A-3 の出力先と既存 `evidence/` 構造との整合を事前に定義すべき。
