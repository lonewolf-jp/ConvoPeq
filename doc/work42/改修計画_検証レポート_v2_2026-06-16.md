## 2. B-1 RuntimeDrainAudit責務分離 — コードベース検証

### 2.1 現状分析

#### `RuntimeDrainAudit` — 定義と現在の使われ方

```cpp
// RuntimeDrainAudit.h (コメント抜粋)
// RuntimeDrainAudit: Shutdown 完了条件の監査構造体。
// isAllZero() は監査ログ出力専用。shutdown 完了判定の authority にはしない。
```

**しかし、実際の使用箇所を調べた結果:**

| 呼び出し元 | 行 | 用途 | 実態 |
| --- | --- | --- | --- |
| `AudioEngine.Processing.ReleaseResources.cpp:259` | `!audit.isAllZero() \|\| audit.stuckReaderCount > 0` | DiagnosticMode の診断分岐 | 診断（許容範囲） |
| `AudioEngine.Processing.ReleaseResources.cpp:350` | `!drainedWithinBudget \|\| !audit.isAllZero()` | **シャットダウン完了判定** | 🚨 **コメントと矛盾** |

**コードのコメントと実装が矛盾している**:
- コメント: 「isAllZero() は監査ログ出力専用。shutdown 完了判定の authority にはしない」
- 実装: `isAllZero()` をシャットダウン完了判定に使用している

#### `isAllZero()` の現在の実装

```cpp
bool isAllZero() const noexcept {
    return pendingPublication == 0
        && pendingRetire == 0
        && activeCrossfadeCount == 0
        && deferredPublish == 0;
    // ★ 欠落: routerPendingRetire
    // ★ 欠落: quarantineResident
    // ★ 欠落: stuckReaderCount (line 259で別途チェック)
}
```

**欠落している完了条件**:
| フィールド | isAllZero() に含まれるか | 判定 |
| --- | --- | --- |
| `pendingPublication` | ✅ | 含まれている |
| `pendingRetire` | ✅ | 含まれている |
| `activeCrossfadeCount` | ✅ | 含まれている |
| `deferredPublish` | ✅ | 含まれている |
| `routerPendingRetire` | ❌ | **不足** |
| `quarantineResident` | ❌ | **不足** |
| `stuckReaderCount` | ❌（259行で別途チェック） | **統合すべき** |

### 2.2 設計判断

**結論: 元計画の `ShutdownCompletionState` 完全分離が正しい。**

Copilotの「`canShutdown()` を `RuntimeDrainAudit` に追加」案は以下の理由で不適切:

1. **責務混在の永続化**: 現在の「コメントと実装の矛盾」が `canShutdown()` 追加により解消されず、むしろ恒久化する
2. **将来の誤用リスク**: `RuntimeDrainAudit` が診断＋完了判定の二重責務を持ち続けると、新規開発者が誤った authority と見なす可能性がある
3. **拡張性**: `ShutdownCompletionState` を独立させることで、将来の完了条件追加が `RuntimeDrainAudit` の診断フィールド追加と独立して行える

### 2.3 実装設計（確定）

```cpp
// 新設: ShutdownCompletionState — 完了判定専用
struct ShutdownCompletionState {
    bool canShutdown() const noexcept {
        return pendingPublication == 0
            && pendingRetire == 0
            && activeCrossfadeCount == 0
            && deferredPublish == 0
            && routerPendingRetire == 0
            && quarantineResident == 0;
    }

    // 個別公開: どの阻害要因が残っているか
    [[nodiscard]] convo::isr::ShutdownBlockingReason getPrimaryBlocker() const noexcept;

    uint64_t pendingPublication;
    uint64_t pendingRetire;
    uint64_t activeCrossfadeCount;
    uint64_t deferredPublish;
    uint64_t routerPendingRetire;
    uint64_t quarantineResident;
    uint64_t stuckReaderCount;   // ReaderActive は isAllZero とは別枠
};

// collectDrainAudit() が両方を返す
struct DrainAuditResult {
    RuntimeDrainAudit diagnostics;       // 診断専用（従来の全フィールド）
    ShutdownCompletionState completion;  // 完了判定専用
};
```

**既存コードの変更**:
- `AudioEngine::collectDrainAudit()` → `DrainAuditResult` を返すように変更
- `ReleaseResources.cpp:259` → `result.completion.canShutdown()` を使用
- `ReleaseResources.cpp:350` → `result.completion.canShutdown()` を使用
- `RuntimeDrainAudit::isAllZero()` → 維持（診断ログの互換性のため、deprecated 化推奨）

## 4. A-1 の3実装選択肢の比較評価

### 選択肢A（ユーザー提案・推奨）: RuntimePublicationState の可視化 + 差分監視方式

**内容**:
1. `publishHealthSnapshot()` を `emitEvidenceTickNonRt()` から定期呼び出し
2. `stateOwner_.onRetired()` の呼び出し元を追加（retire完了時）
3. `reclaimedCount` は既存の `retiredCount` と `pendingRetire` からの **差分推定** で代用:
   ```cpp
   reclaimedCount ≈ retiredCount - pendingRetire
   ```
4. 168h試験でデータ収集 → 不足が判明した場合のみ WorldLifecycleAudit 拡張

**メリット**:
- 最小侵襲: reclaim パイプライン内部に触れない
- 実装量最小: `publishHealthSnapshot()` の呼び出し追加 + `onRetired()` 呼び出し追加のみ
- リスク最小: 既存の reclaim 経路に変更を加えない

**デメリット**:
- `reclaimedCount` が推定値になる（`pendingRetire` の瞬間値を引くため厳密ではない）
- 168h試験で「不足が判明」した場合に追加実装が必要

**実装手順**:
```cpp
// Step 1: emitEvidenceTickNonRt() で publishHealthSnapshot を呼び出す
void AudioEngine::emitEvidenceTickNonRt(bool force) noexcept {
    // ... 既存コード ...
    if (runtimeOrchestrator_) {
        runtimeOrchestrator_->publishHealthSnapshot();  // 追加
    }
}

// Step 2: onRetired() の呼び出し元を追加（retire完了時に RuntimePublicationOrchestrator から）
// RuntimePublicationOrchestrator.cpp の retire 処理内
stateOwner_.onRetired(correlationId.shortValue());  // 追加

// Step 3: Evidence 出力に retired - pendingRetire 差分を追加（既存の retired と reclaimed で代用可）
```

**呼び出し関連図**:
```
emitEvidenceTickNonRt()
  → runtimeOrchestrator_->publishHealthSnapshot()  // 追加
    → state.progress.retiredCount   // onRetired() が呼ばれれば正しい値
    → state.progress.reclaimedCount // 現状は常に0
    → state.progress.publishedCount
```

### 選択肢B（元計画）: WorldLifecycleAudit に reclaimedCount 追加

**内容**:
1. `WorldLifecycleAudit` に `std::atomic<uint64_t> reclaimedCount_` を追加
2. reclaim完了時に呼ばれる `onWorldReclaimed()` メソッドを追加
3. reclaimフックを `ISRRetireRouter::tryReclaim()` または `DeferredDeletionQueue::dequeue()` に追加

**問題点**:
- reclaimフックの追加には **内部実装の変更** が必要（`ISRRetireRouter` または `DeferredDeletionQueue`）
- `DeferredDeletionQueue` は MPMC lock-free キューであり、callback追加は **lock-free性を損なう可能性がある**
- `ISRRetireRouter` は `IEpochProvider` インターフェース経由で動作し、callback追加はインターフェース変更につながる

### 選択肢C（Copilot初版案）: `WorldLifecycleAudit` + reclaim完了フック + RuntimePublicationState 同期

**内容**:
1. `WorldLifecycleAudit::onWorldReclaimed()` 追加
2. reclaim完了を `WorldLifecycleAudit` と `RuntimePublicationStateOwner::onReclaimed()` の両方に通知
3. フックポイントを `AudioEngine::drainDeferredRetireQueues()` または `ISRRetireRouter::tryReclaim()` に追加

**問題点**:
- 最も侵襲的: 2系統の通知経路が必要
- フックポイントの選定が困難（前述の lock-free 性の問題）

### 選択肢比較表

| 評価軸 | 選択肢A（推奨） | 選択肢B（元計画） | 選択肢C（Copilot案） |
| --- | --- | --- | --- |
| 実装量 | **最小**（2行追加 + 1関数呼び出し追加） | 中（新規フィールド + メソッド + フック） | **最大**（2系統の通知 + フック） |
| 侵襲性 | **低**（既存コード変更最小） | 中（DeferredDeletionQueue/ISRRetireRouter変更） | **高**（内部構造変更） |
| 正確性 | 中（差分推定） | **高**（実測値） | **高**（実測値） |
| lock-free性への影響 | **なし** | **リスクあり**（callback追加） | **リスクあり** |
| 将来の拡張性 | 低（必要なら後からB/Cに移行） | 中 | 中 |
| 168h試験への適合性 | 中（推定値で暫定対応） | **高** | **高** |

### 結論: 選択肢A（ユーザー提案）を推奨

> **A-1 の実装は以下の順で進める:**
> 1. `publishHealthSnapshot()` を `emitEvidenceTickNonRt()` から呼び出す
> 2. `stateOwner_.onRetired()` の呼び出し元を追加する
> 3. Evidence出力の `reclaimedCount` が `retiredCount - pendingRetire` 相当になることを確認
> 4. 168h試験を実施
> 5. 不足が判明した場合のみ、選択肢BまたはCを再評価

### 8.3 実際の reclaim 経路（完全トレース）

**RuntimePublishWorld の解放経路**:

```
retireRuntimePublishWorldNonRt(world)
  → engine_->enqueueDeferredDeleteNonRt(world, deleter)
    → m_retireRouter->enqueueRetire(ptr, deleter, epoch)
      → EpochDomain::enqueueRetire(ptr, deleter, epoch)
        → DeferredDeletionQueue::enqueue(ptr, deleter, epoch)
          → ringBuffer[slot] = {ptr, deleter, epoch, ...}

drainDeferredRetireQueues(allowDuringShutdown)
  → m_retireRouter->tryReclaim()
    → EpochDomain::tryReclaim()
      → deferredDeletionQueue.reclaim(getMinReaderEpoch())
        → isOlder(entry.epoch, minReaderEpoch)?
          → YES: entry.deleter(entry.ptr)  // ← RuntimePublishWorld::~RuntimePublishWorld() + aligned_free()
          → NO:  スキップ（Readerがエポックを保持中）
  → m_coordinator.reclaim(m_retireRouter->getMinReaderEpoch())
    → SnapshotRetireManager::reclaim(minReaderEpoch)
      → DeletionQueue::reclaim(minReaderEpoch)
        → e.deleter(e.ptr)  // ← GlobalSnapshot 解放
```

**解放経路は2系統存在する**:
| 系統 | 管理対象 | キュー | キュー特性 |
| --- | --- | --- | --- |
| `ISRRetireRouter → EpochDomain` | `RuntimePublishWorld`, DSPCore等 | `DeferredDeletionQueue` | **Lock-free MPMC** |
| `RuntimePublicationCoordinator → SnapshotRetireManager` | `GlobalSnapshot` | `DeletionQueue` | **Mutex** |

**いずれの経路にも「reclaim完了通知」のフックポイントは存在しない** 🚨
