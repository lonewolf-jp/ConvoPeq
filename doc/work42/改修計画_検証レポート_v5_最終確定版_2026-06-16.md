## 1. v4 からの主要な修正点

### 1.1 reclaim 計数方式の修正

**v4の問題**: `sizeApprox()` の before/after 差分は concurrent enqueue により不正確

```cpp
// v4の誤った方式
before = sizeApprox();  // = enqueuePos - dequeuePos (例: 100)
reclaim();              // 5件解放。同時に別スレッドが3件enqueue
after = sizeApprox();   // = (100+3-5) = 98
reclaimed = 100 - 98;   // = 2 ❌ 実際は5
```

**v5の確定方式**: `DeferredDeletionQueue::reclaim()` の戻り値を `void` → `uint32_t` に変更し、CAS成功時に `++reclaimed` する。

```cpp
// DeferredDeletionQueue.h
uint32_t reclaim(uint64_t minReaderEpoch) {
    uint32_t reclaimed = 0;
    // ... 既存のループ ...
    if (/* CAS成功 */) {
        entry.deleter(entry.ptr);  // 実際の解放
        ++reclaimed;               // ★ 実際に解放した件数をカウント
        // ...
    }
    // ...
    return reclaimed;  // ★ 正確な解放数を返す
}
```

これにより concurrent enqueue の影響を受けず、**実際に `deleter(ptr)` を呼び出した正確な件数** が得られる。

伝搬経路:
```
DeferredDeletionQueue::reclaim() → uint32_t
  → EpochDomain::tryReclaim()
    → reclaimedCount_ += deferredDeletionQueue.reclaim(getMinReaderEpoch());
  → ISRRetireRouter::reclaimedCount() [委譲追加]
  → AudioEngine::collectDrainAudit() [フィールド追加]
  → RuntimeDrainAudit::reclaimedCount [構造体フィールド追加]
  → Evidence出力
```

### 1.2 B-1 評価の確定

**v3までの誤り**: 「`isAllZero()` をシャットダウン完了判定に使用している → コメントと実装が矛盾 → 分離必要」

**v5の確定**: 設計は既に正しい。

- `waitForDrain()` 内の `isFullyDrained()` が真の Shutdown Authority
- `isAllZero()` は `"(observation only)"` 注釈付きの診断ログ条件
- コメント「shutdown 完了判定の authority にはしない」は正しい
- 設計文書 `doc/work33/remediation_plan.md` も `collectDrainAudit()` を Authority と規定

**Sprint2での軽微修正**: `isAllZero()` に `routerPendingRetire` のチェックを追加するのみ。

### 1.3 WorldLifecycleAudit 拡張の評価修正

**v4の誤り**: 「WorldLifecycleAudit拡張は不要と確定」

**v5の修正**: 「Sprint3では不要。将来拡張候補」

理由: World publish/retire/reclaim の3点整合性を見たい場合、最も自然な場所は `WorldLifecycleAudit` である。ただし現時点では `EpochDomain` レベルの reclaim 計測で十分であり、Sprint3 のスコープ外とする。

### 1.4 onRetired/onReclaimed の Sprint3 での復旧

**v4の問題**: 「優先度低」としていた

**v5の修正**: Sprint3 で `publishHealthSnapshot()` と合わせて復旧する

`onRetired()` の接続箇所:
```
AudioEngine::onRuntimeRetiredNonRt()
  → runtimeOrchestrator_->notifyWorldRetired(worldId) [新設]
    → RuntimePublicationOrchestrator
      → stateOwner_.onRetired(correlationId.shortValue())
```

`publishHealthSnapshot()` の接続箇所:
```

  → runtimeOrchestrator_->publishHealthSnapshot() [追加]
```

## 3. 確定設計: 168h 試験判定式

### 条件①: World 整合性

```text
publishedCount ≈ retiredCount + activeWorldCount
```

- 偏差5%以内を合格
- 出典: `WorldLifecycleAudit` (既存, 正確)
- Evidence: `evidence/world_lifecycle_audit.json`

### 条件②: EBR 健全性（暫定式 + 実測補正）

```text
publishedCount ~ reclaimedCount + pendingRetire + activeWorldCount
```

- **`reclaimedCount` は `EpochDomain::reclaim()` の実解放数** (v5方式で正確)
- **`pendingRetire` は `ISRRetireRouter::pendingRetireCount()`** (正確)
- 右辺と左辺の偏差が10%以内を暫定合格
- **Reader滞留がある瞬間は乖離が大きくなる** ため、168h全体の累積値で評価する
- Evidence: `evidence/world_lifecycle_audit.json` に `reclaimedCount` 追加

### 条件③: Backlog 単調非増加

```text
pendingRetire の最大値が試験開始時の2倍を超えない
```

- 出典: `ISRRetireRouter::pendingRetireCount()`
- Evidence: `evidence/retire_timeline.json`

### 条件④: Shutdown 収束性

```text
routerPendingRetire がシャットダウン開始後60秒以内に0に収束
```

- 出典: `collectDrainAudit().routerPendingRetire`
- Evidence: `evidence/shutdown_trace.json`

### 条件⑤: Recovery 健全性

```text
強制タイムアウト後、publishedCount / retiredCount / reclaimedCount が
60秒以内に正常増加を再開する
```

- `reclaimedCount` は v5方式の正確な値で評価可能

## 付録: コード変更詳細（実装設計）

### DeferredDeletionQueue::reclaim()

```cpp
// 変更前: void reclaim(uint64_t minReaderEpoch)
// 変更後: uint32_t reclaim(uint64_t minReaderEpoch)
uint32_t reclaim(uint64_t minReaderEpoch) {
    uint32_t reclaimed = 0;           // ★ 追加
    // ... 既存の変数 ...
    while (scanned < kMaxScan) {
        // ... 既存のループ ...
        if (canDelete && scanPos == deqPos) {
            if (CAS成功) {
                if (entry.deleter && entry.ptr) {
                    entry.deleter(entry.ptr);
                }
                ++reclaimed;          // ★ 追加: 実際に解放した件数
                // ... 既存の後処理 ...
            }
        }
    }
    return reclaimed;                 // ★ 戻り値追加
}
```

### EpochDomain::tryReclaim()

```cpp
// 変更前
void tryReclaim() noexcept override {
    deferredDeletionQueue.reclaim(getMinReaderEpoch());
}

// 変更後
void tryReclaim() noexcept override {
    const uint32_t n = deferredDeletionQueue.reclaim(getMinReaderEpoch());
    convo::fetchAddAtomic(reclaimedCount_, static_cast<uint64_t>(n), std::memory_order_release);
}

// 新規
[[nodiscard]] uint64_t reclaimedCount() const noexcept {
    return convo::consumeAtomic(reclaimedCount_, std::memory_order_acquire);
}

// メンバ追加
std::atomic<uint64_t> reclaimedCount_{0};
```
