## 1. B-1: ShutdownCompletionState 分離は不要と確定

### 1.1 決定的事実

**調査により、以下の事実が判明した:**

1. **設計文書 `doc/work33/remediation_plan.md` の明記**:
   - `collectDrainAudit()` / `ShutdownBlockingReason` → **Shutdown 状態判定** (Shutdown Authority)
   - `onHealthEvent()` / `HealthEvent` → **診断イベント** (Evidence 出力)

2. **ガバナンスポリシー `.github/isr-ai-governance-policy.json` の規定**:
   - "single drain authority API" — 唯一のドレイン権威として `isFullyDrained()` を指定
   - "releaseResources shutdown completion must reference single drain authority"

3. **`isAllZero()` の実際の使用箇所は診断ログのみ**:
   ```cpp
   // AudioEngine.Processing.ReleaseResources.cpp:349-359
   const auto audit = collectDrainAudit();
   if (!drainedWithinBudget || !audit.isAllZero()) {
       diagLog("[ISR][Shutdown] Drain incomplete: ..."
               " (observation only)");  // ← 明示的に "(observation only)"
   }
   ```
   - `"(observation only)"` 注釈が診断目的であることを明示
   - 実際のシャットダウン完了判定は `waitForDrain()` 内の `isFullyDrained()` ループで行われている

4. **`isAllZero()` のコメントは正しい**:
   > `isAllZero()` は監査ログ出力専用。shutdown 完了判定の authority にはしない。
   - ✅ コメントと実装に矛盾なし

### 1.2 改善点（軽微）

`isAllZero()` のチェック項目に `routerPendingRetire` が欠けている。ただしこれは診断ログの条件なので実害はない。

### 1.3 結論

**`ShutdownCompletionState` の新設は不要。** 現状の設計で正しい。

| コンポーネント | 責務 | 状態 |
| --- | --- | --- |
| `isFullyDrained()` | Shutdown Authority | ✅ 正常 |
| `isAllZero()` | Diagnostic snapshot | ✅ (routerPendingRetire欠落は軽微) |
| `collectDrainAudit()` | 両方のデータ収集 | ✅ 正常 |
| `ShutdownRuntime FSM` | 状態遷移管理 | ✅ 正常 |

## 2. Actual Reclaim 通知方式の設計確定

### 2.1 決定事項

**`EpochDomain::tryReclaim()` に reclaim 成功数カウンタを追加する。**

### 2.2 設計

```cpp
// EpochDomain.h — 追加フィールド
private:
    std::atomic<uint64_t> globalEpoch;
    std::array<ReaderSlot, kMaxReaders> readers;
    DeferredDeletionQueue deferredDeletionQueue;
    std::atomic<uint64_t> reclaimedCount_{0};  // ★ A-1: 累積 reclaim 成功数
```

```cpp
// EpochDomain.h — tryReclaim() の変更
void tryReclaim() noexcept override
{
    // ★ A-1: reclaim 前後のキューサイズ差分から成功数を算出
    const uint64_t before = static_cast<uint64_t>(deferredDeletionQueue.sizeApprox());
    deferredDeletionQueue.reclaim(getMinReaderEpoch());
    const uint64_t after = static_cast<uint64_t>(deferredDeletionQueue.sizeApprox());
    const uint64_t reclaimed = (before > after) ? (before - after) : 0;
    convo::fetchAddAtomic(reclaimedCount_, reclaimed, std::memory_order_release);
}

[[nodiscard]] uint64_t reclaimedCount() const noexcept {
    return convo::consumeAtomic(reclaimedCount_, std::memory_order_acquire);
}
```

### 2.3 精度の根拠

`DeferredDeletionQueue::sizeApprox()` は `enqueuePos - dequeuePos` を返す。`reclaim()` は実際に解放したエントリの分だけ `dequeuePos` を進める。したがって `before - after` は **reclaim() 1回の呼び出しで実際に解放したエントリ数の正確な値** となる。

### 2.4 露出経路

```
EpochDomain::reclaimedCount()
  → ISRRetireRouter::reclaimedCount()        [委譲メソッド追加]
    → AudioEngine::collectDrainAudit()       [.reclaimedCount フィールド追加]
      → RuntimeDrainAudit::reclaimedCount    [構造体フィールド追加]
        → evidence/world_lifecycle_audit.json  [出力追加]
```

**ISRRetireRouter への委譲追加**:
```cpp
// ISRRetireRouter.h
[[nodiscard]] uint64_t reclaimedCount() const noexcept {
    assert(provider_ != nullptr);
    return provider_->reclaimedCount();
}
```

**IEpochProvider / IRetireProvider への追加は不要** — `ISRRetireRouter` が EpochDomain の完全型を知っているため、そこからの委譲で十分。

### 2.5 非採用となった方式

| 方式 | 理由 |
| --- | --- |
| `DeferredDeletionQueue` への callback 追加 | Lock-free MPMC への変更リスク大 |
| `WorldLifecycleAudit` への `onWorldReclaimed()` 追加 | 別責務の混在。World追跡とreclaim追跡は独立 |
| `reclaimed ≈ retired - pendingRetire` | EBR理論上不成立 |

## 3. WorldLifecycleAudit 拡張は不要と確定

### 3.1 理由

`WorldLifecycleAudit` は **World ライフサイクル** の診断用構造体であり、設計上 `Shutdown 完了判定の Authority にはしない` と明記されている。

一方、reclaim 追跡は **EpochDomain レベル** で行うべきであり、WorldLifecycleAudit に追加する理由はない。

| 責務 | 担当コンポーネント |
| --- | --- |
| World publish/retire 追跡 | `WorldLifecycleAudit` |
| EBR reclaim 追跡 | `EpochDomain` (※新設) |
| Orchestrator pipeline 追跡 | `RuntimePublicationState` (onRetired/onReclaimed呼び出し追加で復旧) |

### 3.2 ただし `onRetired()` / `onReclaimed()` は Sprint3 で復旧する

これらは未到達コードだが、`RuntimePublicationState::PublicationLedger` の完全性のためには復旧が望ましい。ただし Sprint3 の優先度は reclaim 追跡 > pipeline ledger 復旧 である。

## 4. 168h 試験判定式の確定

### 4.1 測定可能な値 (v4)

| カウンタ | 取得元 | 測定可能性 |
| --- | --- | --- |
| `publishedCount` | `WorldLifecycleAudit` | ✅ 正確 |
| `retiredCount` | `WorldLifecycleAudit` | ✅ 正確 |
| `activeWorldCount` | `WorldLifecycleAudit` | ✅ 正確 |
| `reclaimedCount` | `EpochDomain` (新設) | ✅ 正確 (§2の方式) |
| `pendingRetire` | `ISRRetireRouter` | ✅ 正確 |
| `routerPendingRetire` | `collectDrainAudit()` | ✅ 正確 |
| `backlog (retireDepth)` | `AudioEngine` | ✅ 正確 |

### 4.2 合格条件（確定版）

#### 条件①: World 整合性

```text
publishedCount ≈ retiredCount + activeWorldCount
```

- 偏差が5%以内を合格
- `WorldLifecycleAudit` からのみ測定（既存）
- Evidence: `evidence/world_lifecycle_audit.json`

#### 条件②: EBR 健全性

```text
publishedCount ≈ reclaimedCount + pendingRetire + activeWorldCount
```

- 右辺と左辺の偏差が10%以内を合格
- `reclaimedCount` は `EpochDomain` (新設)、`pendingRetire` は `ISRRetireRouter` から取得
- EBRの性質上、Reader滞留により `reclaimed < retired` となる瞬間があるため許容範囲を広めに設定
- Evidence: 新規 `evidence/epoch_reclaim_audit.json` または `world_lifecycle_audit.json` 拡張

#### 条件③: Backlog 単調非増加

```text
pendingRetire の最大値が試験開始時の2倍を超えない
```

- `ISRRetireRouter::pendingRetireCount()` を定期的にサンプリング
- Evidence: `evidence/retire_timeline.json`

#### 条件④: Shutdown 収束性

```text
routerPendingRetire がシャットダウン開始後60秒以内に 0 に収束
```

- `collectDrainAudit().routerPendingRetire` で測定
- Evidence: `evidence/shutdown_trace.json`

#### 条件⑤: Recovery 健全性

```text
強制タイムアウト後、publishedCount / retiredCount / reclaimedCount が
60秒以内に正常増加を再開する
```

- 全3カウンタの増加再開を確認
- `reclaimedCount` は §2の新設カウンタで測定可能

### 4.3 判定フロー

```
168h 連続運転試験開始
  ↓
条件①〜③ を常時監視（1分間隔で evidence 出力）
  ↓
168h 経過後、全 evidence を分析
  ↓
全5条件を満たす → Production-Proven Practical Stable ISR Bridge Runtime 到達
いずれか不达标 → 不合格箇所の原因分析 + 修正後再試験
```
