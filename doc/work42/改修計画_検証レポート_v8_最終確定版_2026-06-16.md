## 1. v7→v8 の修正点

### 1.1 `reclaimEfficiency > 0.5` 削除（根拠不足）

**v7の問題**: `reclaimEfficiency = reclaimSuccessCount / reclaimAttemptCount > 0.5` を合格条件としたが、根拠がない。

**調査結果**: `tryReclaim()` は以下の全箇所から呼ばれる:

| 呼び出し元 | 状況 |
| --- | --- |
| `AudioEngine.CtorDtor.cpp` | コンストラクタ |
| `AudioEngine.Processing.ReleaseResources.cpp` (×3) | シャットダウン |
| `EQProcessor.Core.cpp` (×3) | Queue pressure 時 |
| `drainDeferredRetireQueues()` 内部 | 定期 drain |
| `ISRRetireRouter::enqueueRetire()` (×1~2) | QueueFull 時の強制 reclaim |

EBR の性質上、grace period 中は `reclaimSuccessCount` が増えない。試行回数に対して解放数が少ないのは **正常動作** であり、`efficiency=0.01` でも問題ない。

**v8の修正**: `reclaimEfficiency` 指標を **削除**。代わりに「単調増加」のみを監視。

### 1.2 `stuckReaderCount == 0` → `isChronic` ベースに緩和

**v7の問題**: 168h 試験中に `stuckReaderCount == 0` は厳しすぎる。

**調査結果**: `detectStuckReaders()` に `isChronic` フラグが既に存在 (residency > 30秒)。

```cpp
// IEpochProvider.h:36
bool isChronic{false};  // ★ work37: 30秒超の慢性滞留
```

**v8の修正**: 条件②c を「`stuckReaderCount == 0`」から「`isChronic == false`」に変更。

### 1.3 `onReclaimed()` 復旧設計の確定

**v7の問題**: `onReclaimed()` の復旧方法が未設計だった。

**制約の再確認**: `retireRuntimePublishWorldNonRt()` のデリータは stateless lambda (`[](void* p)`) であり、this をキャプチャできない。したがってデリータ内から `stateOwner_.onReclaimed()` を呼ぶことは **設計上不可能**。

**v8の解決策**: `publishHealthSnapshot()` のシグネチャを `void publishHealthSnapshot(uint64_t externalReclaimedCount)` に変更し、EpochDomain から直接取得した正確な `reclaimSuccessCount` を受け取る。

```cpp
// RuntimePublicationOrchestrator.h
void publishHealthSnapshot(uint64_t externalReclaimedCount) noexcept;

// RuntimePublicationOrchestrator.cpp
void RuntimePublicationOrchestrator::publishHealthSnapshot(
    uint64_t externalReclaimedCount) noexcept
{
    OrchestratorHealthSnapshot snapshot;
    snapshot.submittedCount = state.progress.submittedCount;
    snapshot.publishedCount = state.progress.publishedCount;
    snapshot.retiredCount = state.progress.retiredCount;
    snapshot.reclaimedCount = externalReclaimedCount;  // ★ EpochDomain から
    // ...
    telemetryRecorder_.recordHealth(snapshot);
}

// AudioEngine::emitEvidenceTickNonRt()
void AudioEngine::emitEvidenceTickNonRt(bool force) noexcept {
    // ...
    if (runtimeOrchestrator_) {
        const uint64_t reclaimed = m_retireRouter
            ? m_retireRouter->reclaimSuccessCount() : 0;
        runtimeOrchestrator_->publishHealthSnapshot(reclaimed);
    }
}
```

これにより:
- `onReclaimed()` を呼び出す必要がない（stateless lambda 制約を回避）
- `publishHealthSnapshot()` が常に正確な reclaim 数を報告
- `detectStuckStage()` の Reclaim 停滞検出も正しく動作

## 2. 確定設計: Grace Period Visibility

### 2.1 指標定義

| 指標 | 型 | 意味 | 正確性 | 168h閾値 |
| --- | --- | --- | --- | --- |
| `reclaimAttemptCount` | `uint64_t` | `reclaim()` が呼ばれた延べ回数 | ✅ 正確 | 単調増加のみ監視 |
| `reclaimSuccessCount` | `uint64_t` | 実際に `deleter(ptr)` が実行された延べ数 | ✅ 正確 | 単調増加のみ監視 |
| `stuckReaderCount` | `uint64_t` | 滞留中の Reader 数 (depth>0) | ✅ 既存 | 情報提供のみ |
| `isChronic` | `bool` | 30秒超の慢性滞留 Reader 有無 | ✅ 既存 | **== false を合格条件** |
| `pendingRetireCount` | `uint32_t` | キュー内未処理エントリ数 | ✅ 既存 | max < 2×初期値 |
| `overflowCount` | `uint64_t` | キュー溢れ回数 | ✅ 既存 (`ISRRetireRouter`) | == 0 を合格条件 |

### 2.2 IEpochProvider 拡張（最終版）

```cpp
class IEpochProvider : public IReaderEpochProvider,
                       public IPublicationProvider,
                       public IRetireProvider
{
public:
    // ... 既存 ...
    [[nodiscard]] virtual uint64_t reclaimAttemptCount() const noexcept { return 0; }
    [[nodiscard]] virtual uint64_t reclaimSuccessCount() const noexcept { return 0; }
};
```

### 2.3 EpochDomain 拡張（最終版）

```cpp
class EpochDomain : public IEpochProvider {
    // ...
    void tryReclaim() noexcept override {
        reclaimAttemptCount_.fetch_add(1, std::memory_order_relaxed);
        const auto n = deferredDeletionQueue.reclaim(getMinReaderEpoch());
        reclaimSuccessCount_.fetch_add(n, std::memory_order_release);
    }
    [[nodiscard]] uint64_t reclaimAttemptCount() const noexcept override {
        return convo::consumeAtomic(reclaimAttemptCount_, std::memory_order_acquire);
    }
    [[nodiscard]] uint64_t reclaimSuccessCount() const noexcept override {
        return convo::consumeAtomic(reclaimSuccessCount_, std::memory_order_acquire);
    }
private:
    std::atomic<uint64_t> reclaimAttemptCount_{0};
    std::atomic<uint64_t> reclaimSuccessCount_{0};
};
```

## 4. 168h 試験判定式（最終確定版）

### 条件①: World 整合性 ✅

```text
publishedCount ≈ retiredCount + activeWorldCount
```
- 偏差5%以内
- `WorldLifecycleAudit` から測定

### 条件②: Grace Period 健全性 ✅（v8 修正版）

```text
条件②a: reclaimSuccessCount が単調増加していること
        (試験期間中に減少する瞬間がない)
        ※ reclaimEfficiency は監視しない (EBR の性質上、効率が低くても正常)

条件②b: isChronic == false であること
        (30秒超の慢性 Reader 滞留がない)
        ※ 一時的な Reader 滞留 (depth>0) は許容する
        ※ isChronic は detectStuckReaders() の既存フィールド

条件②c: overflowCount == 0 であること
        (キュー溢れがない)
```

### 条件③: Backlog 非増加 ✅（変更なし）

```text
pendingRetireCount の最大値が試験開始時の2倍を超えない
```

### 条件④: Shutdown 収束性 ✅（変更なし）

```text
routerPendingRetire がシャットダウン開始後60秒以内に0に収束
```

### 条件⑤: Recovery 健全性 ✅（変更なし）

```text
強制タイムアウト後、publishedCount / retiredCount / reclaimSuccessCount が
60秒以内に正常増加を再開する
```
