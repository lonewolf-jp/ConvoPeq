## 2. 重要な設計発見

### 2.1 DeferredDeletionQueue に混在するオブジェクト種別

調査結果: **最低7種類** のオブジェクトが同一キューに混在する。

| # | オブジェクト種別 | 登録箇所 | エントリ数/回 |
| --- | --- | --- | --- |
| 1 | `RuntimePublishWorld` | `AudioEngine.h:2764` | 1 world = 1 entry |
| 2 | `DSPCore*` | `AudioEngine.h:3316` | 1 DSP = 1 entry |
| 3 | `EQCoeffCache*` | `AudioEngine.Cache.cpp:10` | 1 cache = 1 entry |
| 4 | `CacheMap*` | `AudioEngine.Cache.cpp:25,50` | 1 map = 1 entry |
| 5 | ConvolverProcessor state | `ConvolverProcessor.Lifecycle.cpp:50` | 不定 |
| 6 | StereoConvolver | `ConvolverProcessor.Lifecycle.cpp:63` | 1 = 1 entry |
| 7 | EQProcessor state | `EQProcessor.Core.cpp:86` | 1 state = 1 entry |

**結論**: `EpochDomain::reclaimedCount_` は「全 DeletionEntry の解放数」であり、「World 解放数」ではない。両者を同一視する式 (`published ≈ reclaimed + pending + active`) は**理論的に成立しない**。

### 2.2 RuntimePublishWorld デリータの制約

```cpp
// AudioEngine.h:2764 — RuntimePublishWorld のデリータ
engine_->enqueueDeferredDeleteNonRt(world, [](void* p) {
    auto* ptr = static_cast<RuntimePublishWorld*>(p);
    ptr->~RuntimePublishWorld();
    convo::aligned_free(ptr);
    // ★ ここに worldLifecycleAudit_.onWorldReclaimed(worldId) は書けない
    //   理由: デリータは this (AudioEngine) をキャプチャできない。
    //   シャットダウン時に AudioEngine が先に破棄される可能性があるため。
});
```

デリータ内で AudioEngine や WorldLifecycleAudit のメソッドを呼ぶことは**設計上不可能**。したがって World レベルの reclaim 通知は、デリータ外部（AudioEngine の drainDeferredRetireQueues 後など）で集計する必要がある。

### 2.3 `publishHealthSnapshot()` の安全性確認

```cpp
// TelemetryRecorder.cpp:85
void TelemetryRecorder::recordHealth(const OrchestratorHealthSnapshot& snapshot) noexcept {
    convo::publishAtomic(lastHealthSnapshot_, snapshot, std::memory_order_release);
}
```

- **ロック**: なし（`publishAtomic` は lock-free）
- **I/O**: なし（ファイル書き込みなし）
- **アロケーション**: なし（スタック上の構造体を atomic store）
- **レイテンシ**: 数十 ns

→ **1秒タイマからの定期呼び出しは完全に安全**。

## 3. A-1 の再定義: 2層構成

### 層1: EBR Queue Visibility（必須、低コスト）

| 項目 | 内容 |
| --- | --- |
| 目的 | EBR キュー (DeferredDeletionQueue) の健康状態を可視化する |
| 計数対象 | **全 DeletionEntry** (7種以上のオブジェクト) |
| カウンタ | `EpochDomain::reclaimedCount_` (新設) |
| 取得方法 | `DeferredDeletionQueue::reclaim()` が戻り値 `uint32_t` で実解放数を返す |
| 公開IF | `IEpochProvider::reclaimedCount()` (新設 virtual) |
| 経路 | `EpochDomain → ISRRetireRouter → collectDrainAudit() → Evidence` |

### 層2: World Reclaim Visibility（推奨、やや高コスト）

| 項目 | 内容 |
| --- | --- |
| 目的 | RuntimePublishWorld の解放完了を可視化する |
| 計数対象 | **RuntimePublishWorld のみ** |
| カウンタ | `AudioEngine` または `WorldLifecycleAudit` に `worldReclaimedCount_` を新設 |
| 取得方法 | `drainDeferredRetireQueues()` または `tryReclaimResources()` の**後**に 差分推定 |
| 推定式 | `worldReclaimed ≈ retiredCount - activeWorldCount(drain後)` |
| 正確性 | 推定値（厳密な per-World 追跡ではない） |

**推定式の根拠**:
- `retiredCount` (WorldLifecycleAudit) = 退役した延べ World 数
- `activeWorldCount` = 現在アクティブな World 数
- `drainDeferredRetireQueues()` は全ての safe なエントリを解放する
- drain 後に `retiredCount - activeWorldCount` = 退役済みでメモリ解放まで完了した World 数（近似）

**注意**: この推定は、RuntimePublishWorld 以外のオブジェクトが World 数に影響しないことを前提とする。実際には DSPCore 等も retire されるが、これらは World カウントに影響しないため、推定式は成立する。

```text
publishedCount ≈ retiredCount + activeWorldCount
```
- 偏差5%以内
- 出典: `WorldLifecycleAudit`

### 条件②: EBR Queue 健全性 ✅（v5から全面変更）

```text
EpochDomain::reclaimedCount が単調増加していること
かつ
pendingRetire が試験期間中に発散しないこと
```
- **「単調増加」**: reclaim() が呼ばれるたびに増加。減少しないことを確認
- **「発散しない」**: `pendingRetireCount()` の最大値が初期値の2倍以内
- World 数との直接比較は実施しない（単位不一致のため）

### 条件③: Backlog 非増加 ✅（変更なし）

```text
pendingRetire の最大値が試験開始時の2倍を超えない
```

### 条件④: Shutdown 収束性 ✅（変更なし）

```text
routerPendingRetire がシャットダウン開始後60秒以内に0に収束
```

### 条件⑤: Recovery 健全性 ✅（変更なし）

```text
強制タイムアウト後、publishedCount / retiredCount / reclaimedCount が
60秒以内に正常増加を再開する
```

## 付録: デリータ非キャプチャ制約の詳細

`retireRuntimePublishWorldNonRt()` のデリータは **this をキャプチャできない**。これは意図的な設計判断であり、理由は以下の通り:

```cpp
// AudioEngine.h:2764
void retireRuntimePublishWorldNonRt(RuntimePublishWorld* world, bool resetRevision) noexcept
{
    engine_->enqueueDeferredDeleteNonRt(world, [](void* p)    // ← stateless lambda
    {
        auto* ptr = static_cast<RuntimePublishWorld*>(p);
        ptr->~RuntimePublishWorld();
        convo::aligned_free(ptr);
    });
}
```

- デリータが実行されるタイミングは `DeferredDeletionQueue::reclaim()` 内
- `reclaim()` は `AudioEngine::drainDeferredRetireQueues()` や `EpochDomain::tryReclaim()` から呼ばれる
- シャットダウンシーケンスでは `AudioEngine` が部分的に破棄された後でも `reclaim()` が実行され得る
- `this` (AudioEngine*) をキャプチャすると、解放時に AudioEngine が生きている保証がない

**代替案**: World reclaim 通知は `drainDeferredRetireQueues()` の呼び出し側（AudioEngine）で行う。

```cpp
// AudioEngine — drainDeferredRetireQueues() 呼び出し後
void AudioEngine::onDrainCompleted() noexcept {
    const uint64_t retired = worldLifecycleAudit_.retiredCount();
    const uint64_t active = worldLifecycleAudit_.activeWorldCount();
    // 推定: drain 後に retired - active が reclaim された World 数に近い
    const uint64_t estimatedReclaimed = (retired > active) ? (retired - active) : 0;
    convo::publishAtomic(worldReclaimedCount_, estimatedReclaimed, std::memory_order_release);
}
```
