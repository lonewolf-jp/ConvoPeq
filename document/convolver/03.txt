「計算リクエストを投げるタイミングと、結果を受け取るタイミングの同期設計」
768kHz駆動のNon-uniform（不均一分割）コンボルバーにおいて、最も難易度が高いのがこの「時間軸のパズル」です。
FFTによる計算は「ブロック単位」で行われるため、計算が終わって結果が返ってくるまでには必ず物理的な遅延（レイテンシ）が発生します。これをDirect Form（時間領域）の結果と完璧に繋ぎ合わせるための決定論的同期設計を解説します。
1. 同期の基本原理：相対デッドライン
Non-uniform方式では、各パーティション（分割セグメント）ごとに「いつまでに計算が終わっていなければならないか」というデッドラインが厳密に決まっています。
Partition 0 (Direct Form): 即時（遅延 0）。
Partition 1 (FFT 小): サイズ 
。入力が 
 溜まった瞬間に計算開始。次の 
 サンプルが鳴り終わるまでに結果が必要。

。入力が 
 溜まった瞬間に計算開始。次の 
 サンプルが鳴り終わるまでに結果が必要。
2. 計算リクエストのタイミング（入出力のフェーズ同期）
オーディオスレッドは、入力サンプルを各階層の「入力バッファ」に蓄積します。バッファが満杯になった瞬間に、ワーカースレッドへタスクを投げます。
cpp
// オーディオスレッド内 (768kHz processBlock)
void processBlock(const double* input, double* output, int n) {
    // 1. 各階層のバッファにサンプルを投入
    for (auto& part : partitions) {
        part.buffer.write(input, n);

        // 2. バッファが満杯（パーティションサイズ L）になったか？
        if (part.buffer.isFull()) {
            // [リクエスト] ワーカースレッドへ計算指示を出す
            // 現在の入力バッファのポインタと、書き戻し先のバッファを渡す
            taskQueue.push({ part.id, part.buffer.getReadPtr(), part.targetOutputBuffer });
            
            // バッファをスワップ（ダブルバッファリング）して次へ
            part.buffer.swap();
        }
    }
}
コードは注意してご使用ください。

3. 結果を受け取るタイミング（Overlap-Add の同期）
ワーカースレッドから戻ってきた結果は、「その計算の元になった入力から数えて L サンプル後」の出力に足し合わせる必要があります。
タイムラインの例 (FFTサイズ = 1024 の場合)
t = 0〜1023: サンプルを蓄積。
t = 1024: ワーカースレッドへ計算リクエストを投げる。
t = 1024〜2047: ワーカースレッドが MKL FFT を実行中。
t = 2048: 計算結果を現在の出力バッファに Overlap-Add（足し合わせ） 開始。
重要: 768kHzでは、ワーカースレッドが 
 サンプル分（1024なら約1.3ms）の間に計算を終えられないと「計算落ち（ドロップアウト）」が発生します。
4. 静的メモリプールによる同期の安定化
768kHzで動的に new や malloc を行うと同期が崩れます。すべてのバッファを prepareToPlay で一括確保します。
cpp
struct PartitionBuffer {
    // ダブルバッファリング用
    alignas(64) double* inputBuffers[2]; 
    alignas(64) double* outputBuffers[2];
    
    // 現在オーディオスレッドが書き込んでいるインデックス
    int activeIndex = 0; 
    
    void swap() { activeIndex = 1 - activeIndex; }
    double* getWritePtr() { return inputBuffers[activeIndex]; }
    double* getPrevCalculatedResult() { return outputBuffers[1 - activeIndex]; }
};
コードは注意してご使用ください。

5. 768kHz 特有の「ジッター」対策
ワーカースレッドの計算時間がバラつく（OSの割り込み等）ことを考慮し、「安全マージン（Look-ahead）」を1ブロック分設ける設計がプロ用では一般的です。
マージンなし: 計算が終わった瞬間に即座に足す（非常にタイト）。
マージンあり: 計算が終わっても、あえて次のブロックまで待ってから足す。
代償: プラグイン全体のレイテンシがその分増える。
恩恵: CPU負荷がスパイクしても音が切れない。
