<Coding Guidelines>
* Build using Windows x64, Visual Studio 2022 MSVC (MSVC Toolset version v143) environment.
* The operating environment assumes an Intel X64 or AMD64 CPU with AVX2 functionality.
* Use C++20.
* Obtain information from <Required Information> before coding.
* Obtain necessary information regarding signal processing algorithms from <References>.
* When using JUCE Framework functions, be sure to verify that the function exists in the latest JUCE version at the URL mentioned above.
* It is crucial to **fully understand** the API specifications of JUCE framework V8.0.12. In particular, thoroughly investigate the meaning of return values, side effects, thread safety, and prerequisites before coding.
* Note that the sample code on the official JUCE framework GitHub is basically VST3, etc., and not a standalone application like this one. * Do not implement excessive stabilization mechanisms not found in the sample code on the official JUCE framework GitHub repository.
* Never edit files within the JUCE folder in the project root or the r8brain-free-src folder.
* Never use Structured Exception Handling (SEH).
* Within the Audio thread (the thread where getNextAudioBlock() is called), absolutely do not perform any operations that have even a 1% chance of waiting (memory allocation (new / malloc / vector::resize / mkl_malloc / mkl_free / _aligned_malloc / vslNewStream), try-catch, DftiCommitDescriptor, mkl_set_interface_layer, IR reloading, mutex lock, critical section, condition_variable, file I/O, console output, std::exp(), functions that call libm, MMCSS settings, access to MessageManager, use of std::shared_ptr). * **Never use AudioBlock::allocate, AudioBlock::copyFrom, or FFT::performFrequencyOnlyForwardTransform (without pre-allocation)** within the Audio thread (the thread calling getNextAudioBlock()).
* **Outside of the Audio thread (the thread calling getNextAudioBlock()), and where INTEL oneAPI MKL (one MKL) is used, do not use new, std::vector, or std::make_unique. Instead, use mkl_malloc / mkl_free, _aligned_malloc(64), or std::pmr + custom allocator.**
* **Data exchange and sharing between the Audio thread (the thread calling getNextAudioBlock()) and other threads must be conducted in a way that reliably guarantees the real-time performance and thread safety of the Audio thread.**
* **All data processing other than external input and output must be performed using 64-bit doubles.** However, floats are acceptable for spectrum analyzer calculations.
* Where INTEL oneAPI MKL (one MKL) is used, ensure memory is aligned to 64 bytes, and if memory allocation is required, call it only from a message thread such as prepareToPlay(), not from the Audio Thread.
* Ensure proper denormalization countermeasures are in place.
* Be careful not to forget to set destructors.
* Pay attention to memory release to prevent memory leaks.
