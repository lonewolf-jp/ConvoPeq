<Coding Guidelines>
* Build using Windows x64, Visual Studio 2022 MSVC (MSVC Toolset version v143) environment.
* The operating environment assumes an Intel X64 or AMD64 CPU with AVX2 functionality.
* Use C++20.
* Obtain information from <Required Information> before coding.
* Obtain necessary information regarding signal processing algorithms from <References>.
* When using JUCE Framework functions, be sure to verify that the function exists in the latest JUCE version at the URL mentioned above.
* It is crucial to **fully understand** the API specifications of JUCE framework V8.0.12. In particular, thoroughly investigate the meaning of return values, side effects, thread safety, and prerequisites before coding.
* Note that the sample code on the official JUCE framework GitHub is basically VST3, etc., and not a standalone application like this one. * Do not implement excessive stabilization mechanisms not found in the sample code on the official JUCE framework GitHub repository.

* Never edit files within the JUCE folder in the project root or the r8brain-free-src folder.

* Never use Structured Exception Handling (SEH).

* Within the Audio thread (the thread where getNextAudioBlock() is called), absolutely do not perform any operations that have even a 1% chance of waiting (memory allocation (new / malloc / vector::resize / mkl_malloc / mkl_free / _aligned_malloc / vslNewStream), try-catch, DftiCommitDescriptor, mkl_set_interface_layer, IR reloading, mutex lock, critical section, condition_variable, file I/O, console output, std::exp(), functions that call libm, MMCSS settings, access to MessageManager, use of std::shared_ptr). However, debug builds are exempt from these restrictions.

* **Never use AudioBlock::allocate, AudioBlock::copyFrom, or FFT::performFrequencyOnlyForwardTransform (without pre-allocation)** within the Audio thread (the thread where getNextAudioBlock() is called).

* **Outside of the Audio thread (the thread where getNextAudioBlock() is called), and where INTEL oneAPI MKL (one MKL) is used, do not use new, std::vector, or std::make_unique. Instead, use mkl_malloc / mkl_free, _aligned_malloc(64), or std::pmr + custom allocator.**

* **Data exchange and sharing between the Audio thread (the thread where getNextAudioBlock() is called) and other threads must be conducted in a way that reliably guarantees the real-time performance and thread safety of the Audio thread.** However, this does not apply to debug builds.


* * All data processing, except for external input and output, should be performed using 64-bit doubles. However, floats are acceptable for spectrum analyzer calculations.

* Implement data access according to the data access rules.

* Implement the state machine and its surrounding parts according to the <State Monitoring Technology Design Guidelines>.

* Where INTEL oneAPI MKL (one MKL) is used, the memory should be 64-byte aligned, and if memory allocation is required, call it only from message threads such as prepareToPlay(), not from the Audio Thread.

* Ensure proper denormalization countermeasures are in place.

* Be careful not to forget to install destructors.

* Pay attention to memory release to prevent memory leaks.
<Data Access Rules>
1. State Machine ⇄ Real-time (RT) Thread
Principle: Read-only, or atomic notification
Method of interaction: Flag reference using std::atomic variables.
Specific method:
The state machine itself (complex transition logic) is placed on the non-RT side.
The RT thread will only check `std::atomic<State>` at the beginning of each cycle.
If the RT thread needs to change the state (e.g., error detection), it will only set an atomic "error flag," leaving the actual state transition (e.g., resource release) to the non-RT thread.
2. State Machine ⇄ Non-Real-Time (Non-RT) Threads
Principle: Execution of logic and management of resource lifecycle
Method of interaction: Mutex and standard conditional branching.
Specific method:
Heavy processing (IPP initialization and MKL memory allocation) associated with user operations and file I/O is performed here.
After completely completing the creation and destruction of resources associated with state transitions, the atomic "state flag" seen by the RT thread is finally rewritten.
Since there are no real-time constraints here, it is acceptable to use locks to prioritize safety. 3. Real-time ⇄ Non-Real-time (RT ⇄ Non-RT) Threads
This is the most important part; two "types" are used depending on the nature of the data.
A. Configuration values ​​and parameters (small data)
Method: std::atomic<float> or std::atomic<double>.
Specific method: Non-RT writes, RT reads. Locking is not required.
B. Signals and large buffers (large data / IPP coefficients, etc.)
Method: SPSC (Single-Producer Single-Consumer) FIFO.
Specific method:
Data transmission: The non-RT side allocates and processes memory, and sends its pointer to the FIFO. The RT side retrieves it and replaces it with the current processing.
Memory return: It is strictly forbidden to free used memory in the RT thread. Use a "return FIFO" to return the old pointer to the non-RT thread, where it is safely discarded.
<State Monitoring Technology Design Guidelines>
1. Unidirectional Data Flow (Read-Only Access)
• Viewing threads (UI/Log, etc.) do not have direct write privileges to internal variables of the state machine.
• State changes should be limited to asynchronous requests via command queues or message buses.
2. Non-Blocking Reads (Lock-Free Snapshots)
• In principle, exclusive locks using synchronization primitives such as std::mutex should be prohibited when accessing from the viewing side.
• Scalar values ​​(State, Level, etc.) should be obtained lock-free using std::atomic<T> to ensure consistency.
• For complex structures, the state machine should provide double buffering or a copy (Snapshot), and the viewing side should atomically obtain only a copy of the pointer.
3. Pull-based Observation Model

- Avoids push notifications (callbacks) from the state machine and employs a "polling method" where the viewer uses its own timer (low priority) to retrieve the latest value.

- This eliminates the risk that high display load will interfere with the deterministic operation of the core logic (state transitions and signal processing).

4. Responsibility Separation

- All data processing for display (string formatting, scaling, visualization calculations) is performed using the computing resources of the viewer thread.

- The state machine focuses solely on writing raw data and has no overhead from formatting processing.

5. Best-effort Consistency

- Prioritizes "processing independence" over "lateness" of information.

- The viewing side is allowed to temporarily access old data (stale data) to prevent the effects of jitter and packet loss from spreading to the state machine.
